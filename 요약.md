# 쿡북 집필 채팅방 — 핵심 인사이트 정리

> 대화 기간: 2024년 11월 9일 ~ 2025년 1월 21일
> 참여자: 최주호(강사/팀장), 김주혁, 류재성 (k디지털 4회차 수강생)

---

## 개요

이 채팅방은 요리 레시피가 아닌 **IT 기술서(개발 실습서) 집필 프로젝트** 관련 대화다.
"쿡북(Cookbook)"은 요리책이 아니라 **실습형 개발 기술서 시리즈**의 명칭으로 사용된다.
강사 최주호가 수강생 두 명과 함께 총 18권 분량의 IT 실습서를 공동 집필하는 프로젝트다.

### 1.1 집필 분업

## 2. 책 컨셉 (공지 원문)

**따라하기 책이다.**

### 책 쓰는 법 — 필수 구조 (반복)

1. **한 두 줄 정의** (필수)
2. **정의에 대한 예시** — 일상 비유로 쉽게
3. **간단한 그림 설명** — AI(GPT, Excalidraw 다이어그램 AI) 적극 활용
4. **코드** (필수)

### 피드백 사이클

1. 팀장이 부연 설명이 필요한 부분을 지적
2. 저자가 수정

### 핵심 원칙

- **대충 써라**: 과도하게 잘 쓰면 책이 길어진다. 따라하기 책이므로 간결함이 최우선
- **내용보다 흐름**: 요즘 트렌드는 간결함 + 전체 흐름
- **따라하기 자아**: "이 내용도 들어가야 하지 않나?"라는 생각을 지금은 무시한다
- **독자가 보고 따라할 수 있게만 쓰면 된다**

## 3. 집필 워크플로우

### 3.1 단계

```
1. 코드 완성 (돌아가는지 확인)
2. 코드 최대한 줄이기 (try-catch 등 제거, 순수 로직만)
3. 전체 프로세스를 주어+목적어 명확히 해서 그림+글로 팀장에게 전달
4. 피드백 반영
5. 책으로 옮기기 (튜토리얼 형식)
```

### 3.2 그림 작성 기준

- 각 장 서론: 일상 비유 그림 (예: 호텔, 배달 시나리오)
- 각 섹션: 실제 프로그래밍 흐름 그림 (시퀀스 다이어그램 등)
- 도구: GPT, Excalidraw Diagram AI, Whimsical
- 최초 집필 시 그림 자리만 표시하고 넘어가도 됨 → 피드백 시 추가

### 3.3 각 장 시작 필수 안내

```
1. 이 장은 docker compose up으로 실행합니다.
2. 이 장은 git clone 후 직접 서버를 실행합니다.
```

### 3.4 코드 작성 기준

- `@Repository` 등 불필요한 어노테이션 제거
- `try-catch` 제거하고 순수 로직만
- 파일 수 최소화 (ValueObject 분리 같은 과도한 설계 지양)
- 코드 소개인지 코드 작성 지시인지 명확히 구분

## 5. 기술 주제별 집필 인사이트

### 5.1 S3 이미지 업로드 + Lambda 리사이징 (김주혁 담당)

**흐름 (최종 확정)**
```
1. React → Spring: Presigned URL 요청
2. Spring → React: Presigned URL 응답
3. React → S3: 해당 URL로 업로드 (비동기)
4. S3 → Lambda: 자동 발동 → 이미지 리사이징 (약 5초 소요)
5. 브라우저 HTTP 캐시에 맡김 (Cache-Control 헤더 활용)
   - 상세/목록 보기 시 같은 URL로 재요청 → 캐시에서 재사용
```

**책 목차 구성 (확정)**
```
1. 전체 아키텍처 그림
2. IAM 세팅
3. AWS 세팅
4. 코드
5. React 코드 (Tip 섹션으로 마지막에)
```

**핵심 포인트**
- Postman으로 테스트 가능: Spring에 Presigned URL 요청 → S3 업로드 → DB 저장 → 목록/상세 조회
- Postman 테스트 시나리오를 장 맨 앞에 배치

### 5.2 도커 / 쿠버네티스 (류재성 담당)

**집필 구조 원칙**
- 추상적 개념 설명(소설처럼) → 실습 코드 순서
- 도커란? 장에서 일상 비유 예시 필수
- ConfigMap: data → metadata 순서 준수
- 터널링: 외부 IP를 풀어서 내부 캡슐을 해석하는 핵심 설명 포함

**쿠버네티스 구분**
- ClusterIP: 내부에서만 접근 가능 (실서비스에서는 외부 직접 접근 불가)
- LoadBalancer: 게이트웨이에만 적용
- minikube service 명령은 로컬 편의용 — 실서비스에서는 막힘

**Docker 우분투 이미지**
- `jrei/systemd-ubuntu:24.04` 사용
- Root 사용자이므로 sudo 불필요 (설명만 추가)

### 5.3 MSA (류재성 담당)

**챕터 구성 (확정)**
```
챕터 1: 게이트웨이 없는 동기 마이크로서비스
챕터 2: 게이트웨이 있는 동기 마이크로서비스
챕터 3: 카프카 기반 크레오그래피(Choreography) 패턴
챕터 4: 카프카 + 중앙 오케스트레이션 + 웹소켓
```

**보상 트랜잭션 패턴**
- 외부 API 호출 실패 시 성공한 호출을 역순으로 롤백
- 플래그 변수(`quantityDecreased`, `deliveryId`)로 완료 여부 추적
- 보상 실패는 로그만 남기고 계속 진행

**주문 시나리오 (웹소켓 챕터 4)**
```
고객 화면 → 주문하기
고객 화면 → 배달 중 (웹소켓 연결 유지)
배달기사 → 배달완료 버튼 클릭
고객 화면 → 배달완료 알림 수신
```

**Status 컬럼 사용 기준**
- 다른 API와 함께 트랜잭션이 필요한 엔티티에만 적용 (orders 등)
- 회원가입(user) 같이 단순 CRUD에는 불필요

### 5.4 OAuth2 / OIDC 로그인 (김주혁 담당)

**책 내용 구성 원칙**
- 코드 방식(Authorization Code) 사용
- 크리덴셜 방식과의 차이는 책 마지막 정리 섹션에 수록
- 카카오 공식 문서 자료 활용 가능
- React 코드는 참고용(마지막 Tip), Postman 위주로 설명

**흐름 설명 예시**
```
1. 정의 한 두 줄
2. 예시 (리다이렉트 주소 설정 등 간단한 캡처 + 한 줄 설명)
3. 코드
```

### 5.5 웹소켓 / SSE / 폴링 (김주혁 담당)

**구분 및 그림 요구사항**
- 폴링: "1초마다 계속 서버에 물어보는" 그림
- SSE: 서버 → 클라이언트 단방향 스트림
- 웹소켓: 양방향 실시간

**STOMP 핵심**
- 심플 vs 컨트롤러 타는 방식 구분이 핵심

## 6. 수익 구조 계획

| 항목 | 내용 |
|------|------|
| 출판 인세 | 책 1권당 정가의 10~20% |
| 교육비 수익 | 수입의 30% |
| 인프런 강의 | 추후 영상 업로드 |
| 법인 수익 | 월급 + 잔여 배당 구조 |

**현실적 계산 예시**
- 18권 × 월 100부 = 1,800부
- 권당 수익 2,000원 × 1,800 = 월 360만원 (이상적 시나리오)
- 책은 돈보다 **유명세 확보 수단** — 교육 사업의 발판

## 9. 주요 실수 유형 & 교훈

| 상황 | 교훈 |
|------|------|
| 책 내용을 너무 잘 쓰려 함 | 따라하기 책은 간결함이 우선. 부연설명은 팀장이 짚어준다 |
| 흐름에 주어 없음 | "누가 누구에게" 주어+목적어를 명확히 |
| 완성 후 피드백 요청 | 진행 중에 피드백 요청해야 수정 범위 줄어듦 |
| 코드가 복잡해짐 | try-catch, ValueObject 분리 등 생략, 순수 로직만 |
| Notion 링크 관리 어려움 | Slack으로 이전, 채널별 역할 명확화 |
