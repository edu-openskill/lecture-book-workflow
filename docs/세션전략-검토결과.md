# 워크플로우 순서 검증 및 세션 전략 설계

집필 워크플로우 순서(1~7)와 세션 전환 전략을 검토한 결과.

---

## 1. 되는 부분

### 워크플로우 순서 (1~5번)

현재 CLAUDE.md의 STEP 1~4와 완전 일치한다.

| 순서 | 명령어 | STEP | 정합 |
|------|--------|------|------|
| 1 | 새 책 만들기 | 프로젝트 생성 | 일치 |
| 2 | 씨앗 심기 | STEP 1 | 일치 |
| 3 | 코드 해부 | STEP 2 | 일치 |
| 4 | 시나리오 설계 | STEP 3 | 일치 |
| 5 | 뼈대 세우기 | STEP 4 | 일치 |

### 인쇄소를 새 세션으로 분리

PDF 빌드는 컨텐츠 작성 컨텍스트가 필요 없다. publisher 스킬(pub-build, pub-layout-check 등)만 있으면 된다. 새 세션에서 프롬프트 파일만 읽고 바로 시작 가능.

### 이미지 작업 분리

현재 시스템에 이미 "이미지 2단계 파이프라인"이 존재한다.

- **1단계**(집필 시): `[GEMINI PROMPT]`, `[CAPTURE NEEDED]` 플레이스홀더 삽입
- **2단계**(집필 후): 실제 이미지로 교체

글 작성과 이미지 작업을 분리하려는 제안은 이 구조와 정합한다.

### 챕터 작성 + 검토 + 소규모 수정은 한 세션에서 가능

한 챕터의 글 작성 → 편집잡이 검토 → 수정 2~3회까지는 한 세션에서 충분하다. 단, 수정 요청과 이유를 `review/revision-log.md`에 기록하면 컨텍스트 압축 시에도 맥락이 보존된다.

## 2. 안 되는 부분

### 2-1. 서브에이전트 MCP 제약

웹 검색으로 확인한 팩트:

| 제약 | 상태 | 출처 |
|------|------|------|
| 커스텀 서브에이전트(.claude/agents/)는 MCP 도구 사용 불가 | 확인 | [Issue #13898](https://github.com/anthropics/claude-code/issues/13898) |
| MCP 호출 시 실패가 아니라 가짜 결과를 환각(hallucinate) | 확인 | 동일 이슈 |
| 백그라운드 서브에이전트도 MCP 불가 | 확인 | [Issue #13254](https://github.com/anthropics/claude-code/issues/13254) |
| 빌트인 general-purpose 타입은 MCP 가능 (foreground 한정) | 확인 | [Issue #13605](https://github.com/anthropics/claude-code/issues/13605) |

집필에이전트에서 MCP가 필요한 작업:
- Playwright MCP → 브라우저 스크린샷
- Context7 MCP → 라이브러리 문서 조회

→ 브라우저 캡처가 필요한 이미지 작업은 서브에이전트로 불가. 새 세션 필수.

### 2-2. 서브에이전트로 챕터 집필 불가

챕터 집필은 8개 변수 수집 질문 + 인사이트 질문이 필수인데, 서브에이전트는 중간 대화 없이 결과만 반환한다. 유저와의 대화가 핵심인 작업에 서브에이전트를 쓸 수 없다.

### 2-3. progress.json에 이미지 상태 추적 부재

chapter 항목에 review 상태만 있고, 이미지 작업 완료 여부 필드가 없다. 이미지 작업을 별도로 관리하려면 추적이 필수.

### 2-4. STEP 6~7 누락

사용자 제안의 6번(챕터 집필) 다음이 바로 7번(인쇄소)인데, 프롤로그/에필로그/부록 작업이 빠져 있다.

수정된 순서:
```
6. 챕터 집필        → STEP 5
7. 프롤로그 + 마무리 → STEP 6~7 (프롤로그, 에필로그, 부록)
8. 인쇄소           → 별도 세션 (PDF 빌드)
```

### 2-5. answers.md가 제대로 동작하지 않음 (버그)

answers.md는 모든 STEP의 질문 답변을 누적해야 하지만, 실제로는 STEP 1(씨앗)만 기록되는 버그가 있다.

- `사내AI비서/answers.md` → STEP 1만
- `사내AI비서_v2/answers.md` → STEP 1만 (복사된 것)
- `사내AI비서_test/answers.md` → STEP 1 + STEP 3만

모든 STEP의 답변이 기록되도록 수정 필요.

별도로, 유저의 수정 요청이나 검토 후 결정 사항을 기록하는 메커니즘이 없다. 컨텍스트 압축 시 대화의 뉘앙스가 소실된다.

해결 방안: `review/revision-log.md` 추가

### 2-6. CLAUDE.md vs 에이전트 시스템 이중 설계

CLAUDE.md: "에이전트는 없다. 하나의 AI가 스킬을 쓴다"
.claude/agents/: 6개 에이전트 정의 (meta, writer, editor, illustrator, publisher, analyst-architect)

같은 시스템에서 두 가지 동작 방식이 공존한다.

## 3. 작업별 실행 방식 판단표

| 작업 | MCP 필요? | 유저 대화? | 권장 방식 | 이유 |
|------|----------|----------|----------|------|
| 챕터 글 작성 | X | 필수 | 새 세션 | 변수 수집 8개 질문 + 인사이트 질문 |
| 이미지 — Mermaid | X | X | 서브에이전트 OK | Bash만 사용 |
| 이미지 — 터미널 캡처 | X | X | 서브에이전트 OK | scripts/capture.py 실행 |
| 이미지 — 브라우저 캡처 | O (Playwright) | X | 새 세션 필수 | 서브에이전트 MCP 불가 |
| 이미지 — 개념도 | X | X | 서브에이전트 OK | 외부 API 호출 |
| 검토 (3개 모드) | X | 결과만 | 서브에이전트 OK | 판정+피드백 반환 |
| 인쇄소 PDF | X | X | 새 세션 | 컨텍스트 분리 + 독립 작업 |

## 4. book/ 폴더 재정의

### 기존 문제

chapters/ → book/body/로 파일을 이동/복사하면 assets/ 경로가 깨진다.

### 새 구조

book/은 front/body/back으로 나누지 않는다. 평탄 구조로, 인쇄소 작업 시에만 사용한다.

```
book/
├── 프롤로그.md              ← STEP 6에서 생성
├── 01-챕터제목.md           ← 인쇄소 작업 시 chapters/에서 복사
├── 02-챕터제목.md
├── ...
├── 에필로그.md              ← STEP 7에서 생성 (유저 선택)
├── 부록.md                  ← STEP 7에서 생성 (유저 선택)
└── output/                  ← 인쇄소에서 생성
    └── book.pdf
```

핵심 원칙:
- chapters/ = 원본 (Mermaid 포함, 에셋 경로 유지)
- book/ = 출판용 복사본 (인쇄소 작업 시 Mermaid→D2 변환, 이미지 경로 재매핑, Typst 전처리)
- 집필 중(STEP 5)에는 book/을 건드리지 않음

## 5. 프롬프트 파일 저장 방식

### 위치 및 규칙

- 저장 위치: `projects/[책이름]/prompts/`
- 파일명: `next-session-[작업명].md`
- 새 세션 시작: `이어하기` 명령 → prompts/ 최신 파일 읽어 맥락 복구

### 자동 생성 시점

| 완료 시점 | 생성 파일 |
|----------|----------|
| STEP 4 완료 | `next-session-CH01.md` |
| CH01 완료 | `next-session-CH02.md` |
| 마지막 챕터 완료 | `next-session-마무리.md` |
| STEP 7 완료 | `next-session-인쇄소.md` |

## 6. 컨텍스트 유실과 대응

### 유실되는 것 vs 안 되는 것

| | 유실 안 됨 | 유실됨 |
|---|---|---|
| 파일 | 디스크의 모든 산출물 | 절대 안 날아감 |
| 대화 | 최근 몇 턴 | 이전 대화의 뉘앙스 |
| 판단 근거 | 검토 결과 파일 | 추론 과정 |

### 대응: revision-log.md

수정 요청과 이유를 파일로 기록하면 컨텍스트 압축과 무관하게 맥락 보존. 스킬/룰 시스템과 결합하면 반복 수정도 안전.

```
review/
├── feedback-log.md          ← 검토 모드 피드백 (기존)
└── revision-log.md          ← 유저 수정 요청 + 반영 결과 (추가)
```

## 7. 최종 권장 워크플로우

```
세션 A ─ 설계 (STEP 1~4)
│  1. 새 책 만들기
│  2. 씨앗 심기
│  3. 코드 해부
│  4. 시나리오 설계
│  5. 뼈대 세우기
│  └─ prompts/next-session-CH01.md 자동 생성
│
세션 B ─ CH01 집필
│  Phase 5a: 글 작성 (유저 대화)
│  Phase 5b: 이미지 디스크립션 검토/수정
│  Phase 5c: 검토 (3개 모드)
│  수정 요청 → revision-log.md 기록
│  └─ prompts/next-session-CH02.md 자동 생성
│
세션 C~N ─ CH02~N 집필 (반복)
│
세션 O ─ 마무리 (STEP 6~7)
│  프롤로그 작성 + 에필로그/부록 필요 여부 확인 후 생성
│  └─ prompts/next-session-인쇄소.md 자동 생성
│
세션 P ─ 인쇄소 (PDF 빌드)
   chapters/ → book/ 복사
   Mermaid → D2 변환 + 이미지 경로 재매핑
   pub 스킬로 PDF 생성 + 레이아웃 검증
```

## 8. 폴더 구조 (최종)

```
projects/[책이름]/
├── progress.json               ← 상태 관리
├── answers.md                  ← 모든 STEP 질문 답변 누적
├── planning/                   ← STEP 1~4 산출물
├── chapters/                   ← STEP 5 산출물 (원본)
├── book/                       ← 인쇄소 조립 폴더 (출판용 복사본)
│   ├── 프롤로그.md
│   ├── 01~N-챕터.md
│   └── output/book.pdf
├── code/                       ← 유저가 제공한 원천 소스코드
├── versions/v0.N/              ← code/를 기반으로 만든 버전별 예제 코드
├── assets/CH01~/               ← 챕터별 이미지
├── questions/pending,done/     ← 인사이트 질문
├── prompts/                    ← 세션 전환용 프롬프트 파일
└── review/
    ├── feedback-log.md         ← 검토 모드 피드백
    └── revision-log.md         ← 유저 수정 요청 + 반영 결과
```
