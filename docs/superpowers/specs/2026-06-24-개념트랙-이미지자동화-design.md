# 설계: 개념 트랙(code-free) + 이미지 생성 자동화

> 작성: 2026-06-24
> 대상 하네스: `book-workflow` (집필에이전트 v3)
> 한 줄 요약: **코드 없이 목차/주제만으로 수학·개념서를 쓰는 "개념 트랙"을 추가하고(기존 코드 트랙은 그대로 유지), 플레이스홀더 기반 이미지 생성을 Codex CLI로 자동화한다.**

---

## 0. 배경과 문제

현재 `book-workflow`는 **완성 코드(source code)를 재료로 전제**한다. 7-STEP 중 4곳이 코드에 구조적으로 묶여 있다.

| STEP | 코드 의존 |
|------|----------|
| STEP 2 코드 해부 | 완성 코드를 의도 필터로 분석 |
| STEP 3 시나리오+버전 | 버전별 예제 코드 생성 |
| STEP 4 뼈대 | 코드 [실습/설명/참고] 분류, 기술 매핑 |
| STEP 5 챕터 집필 | 챕터 = 이야기 파트 → **기술 파트(코드)**, "챕터=한 버전" 성장 |

목표: **코드 없이 목차(예: `수학·LLM_별책_목차.md`)나 주제만으로** 수학·개념서를 쓰고 싶다. 단, **기존 코드 트랙도 그대로 작동**해야 한다.

추가로, 현재 이미지(개념 일러스트)는 `[GEMINI PROMPT]` 플레이스홀더만 꽂고 **사람이 수동으로** 생성한다. 이를 자동화하고자 한다.

| # | 항목 | 결정 |
|---|------|------|

## 2. 아키텍처 — `트랙` 변수 도입

하네스에 최상위 변수 `트랙`을 도입한다. 값: `코드`(기본) | `개념`.

- **코드 트랙 = 기존 동작 그대로.** 기본값이므로 손대지 않으면 회귀 없음.
- **개념 트랙 = 신규 재료 처리 경로.**
- `트랙`은 `planning/seed.md`와 `progress.json`에 기록한다 → 세션 복구 시 어느 트랙인지 식별.

### 설계 원칙

이 하네스의 기존 철학("STEP·스킬·검토모드 3개념 + 상수/변수")을 따른다. `트랙`은 새로운 4번째 개념이 아니라 **변수 하나**다. 일부 상수가 트랙에 종속될 뿐이다.

## 3. 상수 분기 (트랙 종속 상수)

CLAUDE.md의 상수 표에서 아래 항목만 트랙 종속으로 바뀐다. **문체 상수(스토리텔링·대화체·쉽게·비유필수)는 양쪽 동일** — 이것이 통합의 핵심이다.

| 상수 | 코드 트랙 | 개념 트랙 |
|------|----------|----------|
| 재료 | 완성 코드 | 목차/주제 |
| 성장 방식 | 버전별 성장 (챕터=한 버전) | **의존성 누적** (강=선수 개념 위에 쌓기) |
| 기술 파트 채움 | 실습/설명/참고 코드 | **수식(LaTeX)·계산예제·다이어그램·연습문제** |
| 자료 분류 | [실습/설명/참고] | **[핵심수식/계산예제/그림/연습]** |
| 코드 비중 | 낮음 | 없음 (수식·그림으로 대체) |

> 근거: 목차의 "비유로 직관 → 정식 표기로 확인" 원칙이 이미 기존 문체("비유 → 기술 파트 정식 정의")와 일치한다. 그래서 문체는 공유 가능하다.

## 4. STEP별 처리

| STEP | 처리 방식 | 개념 트랙 동작 | 산출물 |
|------|----------|---------------|--------|
| **1 씨앗** | 공유 + 소폭 분기 | 시작 시 **트랙 질문** 추가. 코드 사전 스캔 대신 **목차/주제 스캔**: 목차가 있으면 강·개념·비유후크·의존관계를 파싱해 질문 선택지에 반영, 없으면 주제만으로 진행. seed에 `트랙: 개념` 기록 | `planning/seed.md` |
| **2 재료 파악** | 분기 | **개념 해부**: 목차를 의도 필터로 분석 → ① 강별 핵심 개념 ② 선수 개념(의존성) 맵 ③ 비유↔정식정의 쌍 ④ 의도 밖 가지치기(예: ε-δ 엄밀증명 제외) | `planning/concept-analysis.md` |
| **3 시나리오** | 분기 | 버전코드 자리 → **개념 의존성 그래프 + 강별 계산 예제 설계** | `planning/scenario.md` + `examples/` |
| **4 뼈대** | 분기 | 코드분류 자리 → **자료 분류 [핵심수식/계산예제/그림/연습]** + 난이도 곡선(목차의 "13·14강만 어렵다"를 곡선화) + 의존성 매핑(완성코드 대신 선수 개념) | `planning/outline.md` |
| **5 챕터 집필** | 이야기=공유 / 기술=분기 | 이야기 파트 동일("이번 버전" → "이번 강에서 뭘 새로 쌓나"). 기술 파트 → **용어정리(비유→정식정의) + 수식 전개(LaTeX) + 단계별 계산예제 + 다이어그램 + 연습문제 + "이게 AI 어디 쓰이나" 한 줄** | `chapters/NN-제목.md` |
| **6 프롤로그** | 공유 | 거의 그대로 (이미 코드 없이 개념을 엮음) | `book/front/prologue.md`, `roadmap.md` |
| **7 마무리** | 공유 + 보강 | 부록에 **기호표 + 연습문제 해답** 추가 | `book/back/appendix.md` 등 |

### 챕터 내부 구조 (개념 트랙)

```
┌─────────────────────────────────────┐
│  이야기 파트 (수식 없음)             │
│  - 도입 (경험/상황)                  │
│  - 개념 (비유 후크)                  │
│  - 이번 강에서 뭘 새로 쌓나          │
│  - "이것만은 기억하자" + 다음 예고    │
├─────────────────────────────────────┤
│  기술 파트                           │
│  - 용어 정리 (비유 → 정식 정의)       │
│  - 수식 전개 (LaTeX)                 │
│  - 계산 예제 (단계별 worked example)  │
│  - 다이어그램/그래프                  │
│  - 연습문제                          │
│  - "이게 AI 어디에 쓰이나" 한 줄      │
└─────────────────────────────────────┘
```

## 7. 수식 처리 (LaTeX → Typst)

빌드 파이프라인은 `마크다운 → Pandoc(-t typst) → Typst 컴파일`이며, **Pandoc은 마크다운 안의 LaTeX 수식을 Typst math로 자동 변환**한다.

- 저자/AI는 본문에 **LaTeX**로 쓴다: `$f'(x)=0$`, `$$L=\frac{1}{n}\sum (y-\hat y)^2$$`
- 빌드 시 Pandoc이 Typst 수식으로 번역 → 최종 PDF 렌더
- 장점: 표준 표기, 마크다운 미리보기(VS Code/GitHub)에서 즉시 렌더, 목차 표기와 일치, 타 파이프라인 호환

### 구현 시 필요한 것 2가지

1. Pandoc math 확장 확인 (`markdown` 입력이면 기본 ON)
2. **후처리 보정 규칙 추가** — Pandoc이 잘못 변환할 수 있는 매크로(`\begin{align}`, `\xrightarrow`, 일부 패키지 명령)용. 기존 build-guide의 "3단계: 후처리"에 규칙을 추가한다.

## 8. 이미지 생성 자동화

### 8.1 현재 상태

집필 시 플레이스홀더만 삽입하고 사람이 수동 생성한다. (`visual` 스킬 `references/image.md` §1 방식 A)

```markdown
<!-- [GEMINI PROMPT: 03_rag-flow]
path: assets/CH03/03_rag-flow.png
{프롬프트}
Style: ...
-->
![캡션](../assets/CH03/03_rag-flow.png)
*그림 3-2: ...*
```

### 8.2 백엔드 — Codex CLI (확정·검증됨)

- **Codex CLI(gpt-5.5)가 이미지를 생성**하고 PNG로 저장한다.
- 저장 위치: `~/.codex/generated_images/{세션-uuid}/ig_{hash}.png`
- **구독 로그인으로 동작 — API 키 불필요.**
- 주의: `codex.exe`가 **버전 해시 디렉토리로 자동 갱신**됨(`...\Codex\bin\{hash}\codex.exe`). 자동화는 PATH의 `codex`를 사용하거나 최신 bin 디렉토리를 동적으로 찾는다.

### 8.3 오케스트레이션 (코드·개념 트랙 공용)

신규 스킬/스크립트가 아래를 수행한다.

1. **스캔**: 완성 챕터에서 이미지 프롬프트 플레이스홀더 블록을 추출 → `(id, target_path, prompt)`
2. **생성**: 각 항목마다 `codex exec`(headless) spawn → 프롬프트 전달 → PNG 생성
3. **경로 파싱**: Codex 출력에서 저장된 `~/.codex/generated_images/.../ig_*.png` 경로를 추출
4. **이동**: 그 PNG를 `assets/CH{N}/{id}.png`(= 플레이스홀더의 `path:`)로 이동/리네임
5. **교체**: 플레이스홀더 주석 + `![]()`를 `<img src="../assets/CH{N}/{id}.png" width="720" alt="...">` + 캡션으로 치환 (`image.md` §5 규칙)

### 8.4 어댑터 구조 (확장성 유지)

`이미지 생성 어댑터` 인터페이스: `(prompt, target_path) → PNG`

| 어댑터 | 상태 |
|--------|------|
| **codex** (기본) | 확정 — gpt-5.5 이미지 생성 |
| manual (폴백) | 현행 유지 — 백엔드 실패 시 플레이스홀더 보존, 스캔/교체만 수동 |
| gemini (옵션) | 추후 |

### 8.5 플레이스홀더 태그 명칭

기존 태그명은 `[GEMINI PROMPT]`이나 백엔드는 Codex가 된다. **하위 호환을 위해 기존 태그를 계속 인식**하고, 문서상 "백엔드 무관"임을 명시한다. 신규 별칭 `[IMAGE PROMPT]`를 추가로 허용한다(선택). 기존 프로젝트 파일은 변경하지 않는다.

## 11. 검증 스파이크 (구현 플랜에서 먼저 처리)

| 스파이크 | 확인할 것 | 통과 기준 |
|----------|----------|----------|
| S1. Codex 헤드리스 이미지 | `codex exec`(비대화)에서도 이미지가 생성되고, 저장 경로를 **파싱 가능한 형태로** 출력하는가 (인터랙티브는 "Saved to file://..." 확인됨; `--output-format json` 등 동작 확인) | exec로 PNG 1장 생성 + 경로 자동 추출 성공 |
| S2. LaTeX→Typst | 목차의 대표 수식(예: `e^{iπ}+1=0`, `\frac{1}{n}\sum`, 행렬)이 Pandoc→Typst→PDF로 깨지지 않고 렌더되는가 | 샘플 수식 PDF 정상 렌더, 깨지는 매크로 목록화 |

> 두 스파이크 모두 설계를 막지 않는다. S1 실패 시 manual 어댑터로 폴백(스캔/교체 자동화는 유지). S2의 깨지는 매크로는 후처리 규칙으로 보정.

## 12. 범위 밖 (YAGNI)

- 시뮬레이터·슬라이드·판서대본 제작 (→ `hams:lec`)
- 재료 추상화 전면 일반화(C안) — 트랙 2종에는 과설계
- gemini 어댑터 구현 (codex로 충분)
- 기존 코드 트랙 프로젝트 파일 마이그레이션

## 13. 성공 기준
