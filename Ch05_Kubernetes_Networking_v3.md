# Ch.5 Kubernetes 네트워킹

다음 날 아침, 사무실 자리에 앉은 오픈이는 어제 성공적으로 테스트했던 네 개의 파드(Pod)를 떠올렸습니다. 자동 복구는 물론이고 새 버전으로의 교체까지 직접 확인하며 쿠버네티스의 위력을 실감했죠. 

하지만 기쁨도 잠시, 오픈이의 머릿속에는 실질적인 의문이 하나 스쳤습니다.

> **'그런데... 이걸 실제로 어떻게 사용해야 하지?'**

파드 네 대를 띄워두긴 했지만, 정작 어느 주소로 요청을 보내야 할지 막막했던 것입니다. 도커 실습 때 엔진엑스(Nginx)로 만들었던 단일 진입점이 생각났습니다. 파드 IP는 재시작될 때마다 수시로 바뀔 텐데, 그 앞단에서 변치 않고 자리를 지켜줄 '고정 진입점'이 절실했습니다.

**'쿠버네티스에서는 이 문제를 어떻게 해결할까?'**

궁금해진 오픈이는 다시 공식 문서를 펼쳤습니다.

## 5.1 Service — Pod의 고정 주소

### 5.1.1 Service가 필요한 이유
문서를 훑던 오픈이의 눈에 **Service**라는 항목이 들어왔습니다. 파드의 상태나 위치와 상관없이 변하지 않는 고정 진입점 역할을 하는 것, 그것이 바로 Service였습니다.

마치 가맹점의 점장이나 직원이 누구인지와 관계없이, 고객은 **브랜드 대표 번호** 하나로 언제든 주문을 할 수 있는 것과 같습니다. 이처럼 Service는 쿠버네티스 환경에서 파드들의 변하지 않는 '대표 창구' 역할을 수행합니다.

> **[그림 5-1. Service는 고정 주소를 제공. Pod IP가 바뀌어도 Service 주소는 그대로]**
> (구성: 외부 요청 → Service(고정 진입점) → Pod 1, 2, 3로 트래픽 분배)

### 5.1.2 Service 생성
본격적인 실습을 위해 우선 대상이 되는 파드들을 준비하겠습니다. 오픈이는 `ex11` 폴더를 열어, 어제 사용했던 `deploy-ex02.yml`을 다시 실행해 파드 네 개를 복구했습니다.
* **실습 코드 (GitHub)** : [https://github.com/metacoding-10-linux-docker/docker/tree/master/ex11](https://github.com/metacoding-10-linux-docker/docker/tree/master/ex11)

```bash
kubectl apply -f ex11/deploy-ex02.yml # Pod 4개 생성
```

이제 그 앞에 세울 Service 설정을 살펴보겠습니다.

**ex11/service-ex01.yml**
```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: NodePort       # 노드 IP와 포트로 외부 접근이 가능한 타입
  selector:
    app: nginx         # 이 라벨을 가진 Pod들을 뒤에 연결함
  ports:
    - port: 80         # 클러스터 내부에서 Service로 진입하는 포트
      targetPort: 80   # Service가 Pod로 요청을 전달하는 포트
      nodePort: 30080  # 외부에서 노드로 진입할 때 사용하는 포트 (30000~32767)
```

Service가 파드를 찾는 메커니즘은 디플로이먼트와 동일한 **라벨(Label) 매칭** 방식입니다. IP 주소가 아니라 라벨을 기준으로 연결하기 때문에, 파드가 새로 생성되어 IP가 바뀌더라도 라벨만 일치하면 Service는 요청을 정확히 전달할 수 있습니다.

> **[그림 5-2. selector가 지정한 라벨(app: nginx)을 가진 Pod만 골라 매칭. 다른 라벨(app: db)은 제외]**

### 5.1.3 Service 타입과 접근 범위
오픈이는 YAML을 작성하던 도중, 포트 설정 부분에서 잠시 멈췄습니다.

> **'type: NodePort는 무슨 뜻일까? 그리고 포트 80번 외에 targetPort와 nodePort가 또 있네. 각각 어떤 역할을 하는 거지?'**

조금 더 파헤쳐 보니, Service에는 **'누구에게 공개할 것인가'**에 따라 크게 세 가지 타입을 선택할 수 있었습니다.

가족끼리 집 안에서만 대화할 때는 **ClusterIP**, 초대받은 고객에게 현관 비밀번호를 알려줄 때는 **NodePort**, 그리고 누구나 자유롭게 드나들도록 정문을 활짝 열어줄 때는 **LoadBalancer**를 선택하면 됩니다.

#### ① ClusterIP
서비스 타입의 기본값입니다. 외부에서는 아예 접근이 불가능하며, 클러스터 내부의 파드끼리 서로를 호출할 때만 사용합니다.

*'백엔드 서버만 DB에 접속하면 되지, 굳이 외부 고객에게 DB 주소를 노출할 필요는 없으니까. 그런 보안적인 용도로 사용하는 타입이구나.'*

> **[그림 5-3. ClusterIP — 외부 요청은 닿지 못하고 내부 Pod끼리만 통신]**

#### ② NodePort
오픈이가 이번 실습에서 사용한 방식으로, 노드(서버)의 실제 IP에 특정 포트를 열어 외부 접근을 허용합니다.

*'YAML에 30080처럼 nodePort를 지정하면 이 포트로 외부에서 들어올 수 있구나. 하지만 서비스마다 이런 노드포트를 일일이 열어주면 관리가 꽤 복잡해지겠어.'*

> **[그림 5-4. NodePort — 노드의 특정 포트로 외부 접근 허용]**

#### ③ LoadBalancer
실제 운영 환경에서 가장 흔히 사용하는 방식입니다. AWS나 GCP 같은 클라우드 서비스를 이용 중이라면, 쿠버네티스가 자동으로 외부용 공인 IP를 발급받아 서비스에 연결해 줍니다. 덕분에 사용자는 복잡한 포트 번호를 외울 필요 없이 발급된 대표 IP 하나만 알면 접속할 수 있습니다.

> **[그림 5-5. LoadBalancer — 클라우드가 공인 IP를 발급하고 여러 노드에 분산]**

**[오픈이가 정리한 서비스 타입 요약]**

| 타입 | 접근 범위 | 사용 사례 |
| :--- | :--- | :--- |
| **ClusterIP** | 클러스터 내부만 | 백엔드·DB 등 외부 노출이 불필요한 서비스 |
| **NodePort** | 노드IP:포트로 외부 접근 가능 | 테스트 및 개발 환경 |
| **LoadBalancer** | 공인 IP로 외부 접근 가능 | 실제 클라우드 운영 환경 |

---

#### 포트 흐름의 이해
타입은 정리됐지만, `port`, `targetPort`, `nodePort`가 한데 적힌 이유는 여전히 궁금했습니다. 사실 이 셋은 **각자 주인이 다른 포트**입니다.

* **nodePort**: 노드(서버)의 포트
* **port**: Service 자체의 포트
* **targetPort**: 실제 Pod의 포트

외부에서 파드까지 이어지는 전체 라우팅 경로를 Service가 한 곳에서 선언하기 위해 이 셋을 함께 정의하는 것입니다.

> **[그림 5-6. 노드 안에 NodePort, Service, Pod가 자리하고 외부 사용자가 NodePort로 진입해 Service를 거쳐 Pod 안 컨테이너에 닿는 구조]**

| 포트 종류 | 소유 주체 | 역할 | 생략 시 |
| :--- | :--- | :--- | :--- |
| **nodePort** | 노드(서버) | 외부에서 노드 IP로 접근할 때 열리는 포트 | 30000~32767 중 자동 할당 |
| **port** | Service | 클러스터 내부에서 Service를 부를 때 쓰는 포트 | **생략 불가 (필수)** |
| **targetPort** | Pod(컨테이너) | 컨테이너 앱이 사용하는 포트와 일치해야 함 | `port` 값과 동일하게 설정됨 |

### 5.1.4 외부에서 Service 접속해 보기
오픈이는 이제 작성한 YAML을 적용했습니다.

```bash
kubectl apply -f ex11/service-ex01.yml # Service YAML 적용
```

> **[그림 5-7. Service 생성 결과: service/nginx-service created]**

*'생성 완료! 그럼 이제 브라우저에서 바로 접속해 봐야지.'*

하지만 여기서 예상치 못한 문제에 부딪혔습니다. 노드포트를 30080으로 분명히 열었는데도 `localhost:30080` 접속이 되지 않았습니다. 

> **'분명 포트를 열었는데 왜 안 되지? 아, 미니큐브가 내 노트북이랑 한 단계 떨어져 있어서 그런 건가...?'**

그 예감이 맞았습니다. 미니큐브(Minikube)라는 가상 세계와 우리 PC는 별도의 네트워크로 분리되어 있습니다. 이를 해결하기 위해 미니큐브는 외부 접속을 돕는 전용 명령어를 제공하고 있습니다.

| 방법 | 명령어 | 설명 |
| :--- | :--- | :--- |
| **URL 생성** | `minikube service <이름> --url` | 서비스에 접근할 수 있는 전용 URL 생성 |
| **터널 개방** | `minikube tunnel` | LoadBalancer 타입 서비스에 외부 IP 부여 |
| **포트 포워딩** | `kubectl port-forward ...` | 호스트 포트와 서비스 포트를 직접 연결 |

> **💡 미니큐브는 왜 localhost로 바로 접속되지 않을까요?**
> 미니큐브는 컴퓨터 내부의 가상 환경(VM 또는 컨테이너)에서 돌아갑니다. 도커 컨테이너를 포트포워딩으로 연결했던 것처럼, 미니큐브 환경과 우리 PC를 잇는 별도의 통로가 필요하기 때문입니다.

오픈이는 `minikube service --url` 명령어를 사용해 접속 주소를 받아오기로 했습니다.

```bash
minikube service nginx-service --url # 접속 URL 생성
```

> **[그림 5-8. minikube service URL 생성 결과: http://127.0.0.1:10345]**

명령어를 실행하자 터미널에 URL이 나타나고 커서가 멈췄습니다. 이 명령어는 연결을 유지하기 위해 터미널을 계속 점유하는 방식입니다. 생성된 URL을 브라우저에 입력하자, 드디어 기다리던 NGINX 화면이 환하게 나타났습니다.

> **[그림 5-9. 브라우저에서 nginx 접속 확인]**

확인을 마친 오픈이는 통로를 닫고 마지막 테스트를 진행했습니다. **'파드가 죽어도 서비스가 고정 진입점 역할을 유지하는가'**를 확인하기 위해 실행 중인 파드를 모두 삭제해 본 것입니다.

```bash
kubectl delete pod --all # 모든 Pod 삭제 (디플로이먼트가 자동 재생성)
minikube service nginx-service --url # 다시 접속 URL 확인
```

> **[그림 5-10. Pod 삭제 후 Service 접속 확인]**

잠시 후 새로 생성된 주소로 접속해 보니, 파드들이 완전히 교체되었음에도 서비스는 변함없이 Nginx 페이지를 보여주었습니다. Service가 고정 진입점 역할을 충실히 수행하고 있음을 확인한 순간이었습니다.

## 5.2 Ingress — 도메인 라우팅

### 5.2.1 Service의 한계
서비스 하나를 성공적으로 띄웠지만, 실제 서비스는 주문, 매장, 결제 등 수많은 기능이 하나의 도메인 아래에서 유기적으로 돌아가야 합니다. 하지만 이를 각각의 NodePort로 관리하기에는 포트 번호를 일일이 외워야 한다는 명확한 한계가 있었습니다.

> **'그냥 Service에서 경로별로 라우팅을 하고 싶은데, 마땅한 설정이 없네. 도커 때는 Nginx 한 대를 앞단에 두고 해결했었는데...'**

쿠버네티스에서도 그 역할을 담당하는 전용 리소스가 존재합니다. 바로 **Ingress**입니다.

### 5.2.2 Ingress의 역할
가맹점이 늘어나면 고객이 모든 직통 번호를 외우기란 불가능에 가깝습니다. 이때 본사가 외부 요청을 일괄적으로 받는 **콜센터**를 두면, 고객은 대표 번호 하나만 알아도 원하는 부서로 연결될 수 있습니다.

Ingress가 바로 이 콜센터 역할을 합니다. 고객이 `/order`로 접속하면 주문 서비스로, `/stores`로 접속하면 매장 서비스로 트래픽을 똑똑하게 나눠 보냅니다.

> **[그림 5-11. Ingress가 도메인과 경로를 읽어 요청을 적절한 Service로 연결하는 구조]**

* **Ingress**: 클러스터 외부의 HTTP/HTTPS 요청을 도메인과 URL 경로를 기준으로 내부 서비스에 연결해 주는 라우팅 규칙입니다.

### 5.2.3 Ingress 적용하기
실습을 위해 미니큐브에 인그레스를 활성화해 보겠습니다. 
* **실습 코드 (GitHub)** : [https://github.com/metacoding-10-linux-docker/docker/tree/master/ex12](https://github.com/metacoding-10-linux-docker/docker/tree/master/ex12)

문서의 첫 줄을 읽던 오픈이의 눈이 한 곳에서 멈췄습니다.
> "인그레스 컨트롤러가 있어야 인그레스를 충족할 수 있다. 리소스만 생성한다면 효과가 없다."

> **'인그레스 리소스 말고 컨트롤러라는 게 또 따로 있어야 한다고?'**

그렇습니다. Ingress는 라우팅 규칙을 적은 **리소스(YAML)**와 그 규칙을 실제로 읽고 처리하는 **컨트롤러(Pod)**가 한 팀이 되어 움직여야 합니다.

> **[그림 5-12. Ingress 리소스(선언)와 Ingress Controller(집행) — 선언과 집행의 분리]**

| 구성 요소 | 역할 | 비유 | 쿠버네티스 철학 |
| :--- | :--- | :--- | :--- |
| **Ingress 리소스** | 라우팅 규칙을 정의한 문서 (YAML) | 콜센터의 연결 규칙 | **선언** |
| **Ingress Controller** | 외부 요청을 처리하는 소프트웨어 | 규칙을 실행하는 콜센터 본체 | **집행** |

오픈이는 우선 미니큐브에서 Ingress 컨트롤러부터 활성화했습니다.

```bash
minikube addons enable ingress # 인그레스 애드온 활성화
```

> **[그림 5-14. Ingress Controller Pod가 Running 상태임을 확인]**

> **💡 실제 운영 환경에서의 Ingress Controller**
> 실제로는 `ingress-nginx` 같은 컨트롤러를 직접 설치하거나 클라우드 서비스가 제공하는 전용 컨트롤러를 사용합니다. 방식의 차이일 뿐, 클러스터 내부에서 컨트롤러 파드가 요청을 분산하는 구조는 동일합니다.

#### ① 두 서비스 준비
이제 테스트를 위해 `/order`와 `/stores` 요청을 처리할 백엔드 파드와 서비스를 준비하겠습니다.

| 파일 | 종류 | 역할 |
| :--- | :--- | :--- |
| `order-deploy.yml` | Deployment | 주문 응답 파드 ("주문 접수 완료") |
| `order-service.yml` | Service | 주문 파드를 묶는 내부 창구 (Port 5678) |
| `stores-deploy.yml` | Deployment | 매장 응답 파드 ("매장 선택") |
| `stores-service.yml` | Service | 매장 파드를 묶는 내부 창구 (Port 5678) |

#### ② 규칙 문서 작성
이제 컨트롤러에게 어떤 경로를 어디로 보낼지 정의한 지침서를 전달할 차례입니다.

**ex12/ingress-ex01.yml**
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ex12-ingress
spec:
  ingressClassName: nginx
  rules:
  - http:
      paths:
      - path: /order
        pathType: Prefix
        backend:
          service:
            name: order-service
            port:
              number: 5678
      - path: /stores
        pathType: Prefix
        backend:
          service:
            name: stores-service
            port:
              number: 5678
```

오픈이는 `ex12/` 폴더의 모든 파일을 적용해 두 서비스와 Ingress 규칙을 한 번에 등록했습니다.

```bash

kubectl get ingress # 등록된 인그레스 확인
```

## 5.3 브라우저에서 Pod까지의 경로

요청이 실제 파드에 닿기까지, 쿠버네티스 내부에서는 어떤 일들이 벌어질까요? 오픈이는 이 복잡한 과정을 크게 3단계로 정리해 보았습니다.

> **[그림 5-19. 요청이 Pod까지 닿는 전체 흐름 — 세 단계가 일어나는 위치]**

### 1단계 — 진입: 본사 대표번호로 전화 연결
외부 손님이 대표번호(`NodePort`)로 전화를 걸면, 자동 안내 시스템(`kube-proxy`)이 전화를 받아 콜센터(`Ingress Controller`)로 연결해 줍니다.

| 비유 | IT 용어 | 설명 |
| :--- | :--- | :--- |
| 외부 손님 | 외부 호스트 | 브라우저를 통해 요청을 보내는 사용자 |
| 본사 대표번호 | **NodePort** | 클러스터로 진입하기 위한 공개된 문 |
| 자동 안내 | **kube-proxy (1차)** | 요청을 인그레스 컨트롤러 파드로 배달 |
| 콜센터 | **Ingress Controller** | URL 내용을 읽고 판단을 내리는 파드 |

> **[그림 5-20. 손님이 본사 대표번호로 전화, 자동 안내가 콜센터로 연결]**
> **[그림 5-21. NodePort + kube-proxy(1차)가 외부 요청을 Ingress Controller로 전달]**

### 2단계 — 부서 결정: 상담원이 담당 부서 확인
콜센터 상담원(`Ingress Controller`)이 요청을 분석합니다. "주문 부서로 가야겠군"이라고 판단하면, 부서 명부(`CoreDNS`)를 확인해 해당 부서의 내선 번호(`ClusterIP`)를 알아낸 뒤 전화를 돌려줍니다.

| 비유 | IT 용어 | 설명 |
| :--- | :--- | :--- |
| 상담원 | **Ingress Controller** | URL 경로를 읽고 목적지 서비스를 결정 |
| 부서 명부 | **CoreDNS** | 서비스 이름을 가상 IP(ClusterIP)로 풀이 |
| 대표 내선 | **ClusterIP** | 서비스가 가진 고유의 내부 진입 주소 |

> **[그림 5-22. 콜센터 상담원이 부서 명부에서 주문 부서의 대표 내선을 조회]**
> **[그림 5-23. Ingress Controller가 URL을 읽어 Service 선택, DNS가 이름을 ClusterIP로 변환]**

### 3단계 — 직원 응답: 가용 직원에게 자동 분배
부서 내선으로 전화가 연결되면, 자동 분배기(`kube-proxy`)가 현재 응대가 가능한 직원(`Pod`)에게 최종적으로 전화를 연결합니다. 이때 부서 매니저(`Endpoint Controller`)는 누가 일하고 있는지(파드의 생사 여부)를 실시간으로 파악해 분배기에 명단을 제공합니다.

| 비유 | IT 용어 | 설명 |
| :--- | :--- | :--- |
| 자동 분배기 | **kube-proxy (2차)** | ClusterIP를 실제 Pod IP로 변환 (L4 로드밸런싱) |
| 응대 직원 명단 | **EndpointSlice** | 현재 살아있는 파드들의 IP 목록 |
| 부서 매니저 | **Endpoint Controller** | 파드 변화를 감지해 목록을 최신화 |
| 응대 직원 | **Pod** | 실제 비즈니스 로직을 처리하는 앱 |

> **[그림 5-24. 매니저가 명단 갱신, 자동 분배가 살아있는 직원에게 전화 연결]**
> **[그림 5-25. ClusterIP → kube-proxy(2차)가 Pod IP로 변환 → Pod 응답]**

---

### 정보의 깊이 (L4 vs L7)
이 세 단계는 각각 패킷의 어느 부분까지 읽느냐에 따라 계층이 나뉩니다.

* **L4 (전송 계층)**: 봉투 겉면(IP, Port) 정보만 보고 빠르게 배달합니다. (`NodePort`, `kube-proxy`, `ClusterIP`)
* **L7 (응용 계층)**: 봉투 안쪽의 구체적인 용건(URL, Host)까지 읽고 정교하게 판단합니다. (**`Ingress Controller`**)

| 단계 | 역할 주체 | 하는 일 | 판단 기준 |
| :--- | :--- | :--- | :--- |
| **1. 진입** | NodePort + kube-proxy | 입구 통과 및 인그레스로 전달 | IP 및 포트 |
| **2. 결정** | Ingress Controller + DNS | URL 분석 및 서비스 결정 | **URL 경로, Host** |
| **3. 응답** | kube-proxy + Pod | 실제 파드 IP로 변환 및 처리 | 실시간 IP 목록 |

---

## 이것만은 기억하자

* **Service는 Pod의 고정 전화번호입니다.** 파드 IP가 수시로 바뀌어도 변하지 않는 주소를 제공하며 트래픽을 골고루 분산합니다.
* **삼박자의 조력자**: 이름을 풀이하는 **DNS**, 실시간 목록을 관리하는 **Endpoint Controller**, 그리고 실제 길을 닦는 **kube-proxy**가 협력하여 패킷을 배달합니다.
* **Ingress는 지능형 안내 데스크입니다.** 도메인과 URL 경로를 읽어 수많은 서비스를 하나의 진입점으로 묶어 관리합니다. **선언(리소스)**과 **집행(컨트롤러)**이 한 팀이라는 사실을 잊지 마세요.

네트워크라는 뼈대는 이제 완벽히 갖춰졌습니다. 하지만 실제 서비스를 운영하려면 DB 비밀번호 같은 보안 정보와, 파드가 사라져도 데이터가 보존되는 영속성 처리가 필수적입니다.

다음 챕터에서는 설정값(**ConfigMap**), 보안 비밀(**Secret**), 그리고 데이터의 영속성(**Volume**)에 대해 알아보겠습니다.
