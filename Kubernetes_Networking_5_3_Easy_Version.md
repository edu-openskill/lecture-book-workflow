# 5.3 브라우저에서 Pod까지의 경로

오후 늦게 옥상에 잠깐 올라갔습니다. 바람을 맞으며 머릿속 그림을 다시 한번 정리하고 싶었기 때문입니다. NodePort, kube-proxy, Ingress Controller, ClusterIP... 이름은 다 알겠는데, 막상 "사용자가 클릭했을 때 누가 어디서 무얼 하지?"라고 물으면 흐름이 꼬였기 때문입니다.

자리로 돌아와 흰 종이를 한 장 꺼냈습니다. 브라우저부터 Pod까지의 경로를 죽 그어 봤습니다. 복잡한 네트워크 장비 대신, **'본사 콜센터의 업무 분담'**으로 생각하니 흐름이 선명해졌습니다. 일을 세 단계로 나누어 보겠습니다.

---

### 5.3.1 1단계 진입 - 본사 대표번호로 전화 연결

외부 손님이 본사 대표번호(**NodePort**)로 전화를 걸면, 자동 안내 시스템(**kube-proxy**)이 전화를 받아 실제 상담원이 있는 콜센터(**Ingress Controller**)로 연결해 주는 단계입니다.

**[비유 설명: 그림 5-20 관련]**
* **외부 손님**이 본사 대표번호로 전화를 겁니다.
* **ARS**가 "콜센터로 연결합니다..."라며 전화를 받습니다.
* 전화는 상담원이 대기 중인 **본사 콜센터**로 넘어갑니다.

| 비유 | IT 용어 | 한 줄 설명 |
| :--- | :--- | :--- |
| **외부 손님** | **외부 호스트** | 브라우저로 접속을 시도하는 사용자 |
| **본사 대표번호** | **NodePort** | 클러스터 외벽에 뚫린 공식 입구(포트) |
| **ARS** | **kube-proxy (1차)** | 들어온 요청을 콜센터(Ingress)로 전달하는 안내원 |
| **본사 콜센터** | **Ingress Controller** | 요청 내용을 읽고 판단을 내릴 주인공 파드(Pod) |

**[기술 설명: 그림 5-21 관련]**
사용자가 주소를 입력하면, 요청은 **NodePort**라는 입구를 통과합니다. 이때 노드에 상주하는 **kube-proxy**가 이 요청을 낚아채서, 실제 길을 찾아줄 **Ingress Controller** 파드로 정확히 배달합니다.

---

### 5.3.2 2단계 부서 결정 - 상담원이 담당 부서 확인

이제 콜센터 상담원(**Ingress Controller**)이 고객의 용건을 확인합니다. "주문하러 오셨나요?" 상담원은 미리 들고 있는 **부서 명부(Ingress 규칙)**를 보고, 해당 부서의 **내선 번호(ClusterIP)**로 전화를 돌려줍니다.

**[비유 설명: 그림 5-22 관련]**
* 상담원이 **부서 명부**를 확인합니다. "주문 부서는 내선 1234번이네!"
* 상담원이 용건에 맞는 **부서 대표 내선**으로 전화를 연결합니다.

| 비유 | IT 용어 | 한 줄 설명 |
| :--- | :--- | : :--- |
| **콜센터 상담원** | **Ingress Controller** | URL 주소를 읽고 어느 서비스로 보낼지 결정함 |
| **부서 명부** | **내부 명단** | 어떤 주소가 어떤 서비스인지 적힌 안내 표 (API Server 구독) |
| **부서 대표 내선** | **ClusterIP** | 특정 서비스(부서)로 연결되는 내부 전용 가상 주소 |

**[기술 설명: 그림 5-23 관련]**
**Ingress Controller**는 사용자가 보낸 URL 경로(예: `/order`)를 읽습니다. 자신이 관리하는 규칙에 따라 "이건 `order-service`용이네!"라고 판단한 뒤, 해당 서비스의 가상 주소인 **ClusterIP**로 요청을 보냅니다.

---

### 5.3.3 3단계 직원 응답 - ARS가 응대 직원에게 연결

부서 대표 번호로 연결되면, 다시 한번 **내부 안내 시스템(kube-proxy)**이 작동합니다. 이번에는 부서 안에서 실제로 일하고 있는 **직원(Pod)** 중 한 명에게 전화를 넘깁니다. 이때 **부서 매니저(Endpoint Controller)**는 누가 출근했는지(Pod의 상태)를 실시간으로 체크해 명단을 최신으로 관리합니다.

**[비유 설명: 그림 5-24 관련]**
* **부서 매니저**가 직원들의 출근 상태를 실시간으로 확인해 명단을 만듭니다.
* **ARS**가 그 명단을 보고, 지금 전화를 받을 수 있는 **직원**에게 최종 연결합니다.

| 비유 | IT 용어 | 한 줄 설명 |
| :--- | :--- | :--- |
| **부서 매니저** | **Endpoint Controller** | 살아있는 파드(Pod) 목록을 실시간으로 관리함 |
| **ARS** | **kube-proxy (2차)** | 서비스 주소를 실제 파드(Pod)의 IP로 바꿔주는 배달원 |
| **응대 직원** | **Pod** | 실제 비즈니스 요청을 처리하고 응답을 주는 일꾼 |

**[기술 설명: 그림 5-25 관련]**
**ClusterIP**로 향한 요청은 **kube-proxy**에 의해 실제 **Pod의 IP**로 변환됩니다. **Endpoint Controller**가 "지금 1번 파드가 살아있다"고 알려준 정보를 바탕으로, 요청이 안전하게 도착해 응답이 돌아옵니다.

---

### 5.3.4 정보의 깊이 (L4 vs L7)

이 과정의 핵심 차이는 **"봉투를 직접 뜯어보는가?"**에 있습니다.

* **L4 (1, 3단계):** 봉투 겉면의 주소와 포트만 보고 빠르게 배달합니다. 내용은 궁금해하지 않습니다. (**kube-proxy**)
* **L7 (2단계):** 봉투를 직접 열어 안의 편지 내용(URL 경로, 도메인)을 읽고 정교하게 판단합니다. (**Ingress Controller**)

| 단계 | 주체 | 판단 기준 | 비유 |
| :--- | :--- | :--- | :--- |
| **1. 진입** | **NodePort** | IP와 포트 번호 | "몇 번 문으로 들어왔나?" |
| **2. 결정** | **Ingress** | URL 경로, Host | "무슨 용건으로 왔나? (주문/매장)" |
| **3. 응답** | **Service** | 파드(Pod) 상태 | "지금 누가 일할 수 있나?" |
