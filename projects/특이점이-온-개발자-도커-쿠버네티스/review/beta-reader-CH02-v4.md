## 강점 (다수 또는 전원 언급)

- 푸드코트 비유(식당↔Namespace, 창구↔veth, 홀↔docker0, 키오스크↔iptables) — **9명 전원 호평**
- "엔진=프런트, 커널=관리인" 한 줄 멘탈모델 — 5명 명시
- 2.7.2 ubuntu가 `-d`로 즉시 종료되는 장면 — Try/Fail 학습효과로 5명 호평 (특히 SI·CS)
- attach vs exec PID 1 비교 — CS 학생·DevOps 호평
- bind/volume USB 비유 — 임베디드·신입·SI 호평

## 심각도별 이슈

### 높음

1. **cgroups 부재 (3명)**
   - 위치: §2.1.2 컨테이너 가상화 정의
   - 증상: 격리(namespace)만 설명되고 자원 제한(cgroups)이 0줄
   - 영향: DevOps·임베디드는 "한 컨테이너가 CPU/메모리 독점하면?", CS 학생은 "OS 수업의 cgroup 어디 갔지?"
   - 제안: "격리 = namespace, 자원 제한 = cgroups. 한 컨테이너가 CPU·메모리를 독점하지 못하게 커널이 한도를 건다." 한 단락 + `--memory`, `--cpus` 옵션 1줄

2. **캐릭터 부재 (4명)**
   - 위치: §2.1 이후 ~ §2.9 종료까지 ~750라인
   - 증상: "선배가 던져준 단어" 1회 외 팀장·동료·선배 대사 0건
   - 영향: 오픈이 독백 950라인이 길고 단조로움. 비유가 대화에서 촉발되지 않고 서술자가 직접 꺼냄
   - 제안: 2.5 도입(네트워크 시작) + 2.7 도입(생명주기) + 2.9 도입(마운트)에 팀장 또는 동료 한 줄 대사 삽입(비유 촉발 역할)

3. **951라인 분량 — 조건부 (2명, 입문 페르소나)**
   - 위치: 챕터 전체
   - 증상: PM·프론트엔드에게 "Docker 한 챕터"가 아니라 "리눅스+네트워크+CLI 입문서"로 느껴짐
   - 영향: 2명 조건부 통과
   - 제안: §2.6 컨테이너 안의 Linux를 부록 또는 별도 챕터로 분리 → 본 챕터 ~600라인 (이미 outline-v2 v5 목표 850~950과 비교 시 상한)

### 중간

4. **iptables 양면 깊이 — 6명**

   - 입문(P3, P5, P6): 깊이 과함 → 본문에서 제거하고 참고 박스로 격하 권장

## 페르소나별 상세 (요약)

### 1. 신입 백엔드 — 4/4/3
주방·푸드코트 비유는 강력. 그러나 환경 구축(WSL2 활성화, Docker Desktop Server 미시작) 디테일 부족. `<컨테이너ID>` 단축형이 갑자기 등장. 캐릭터 부재로 950라인이 길게 느껴짐.

### 2. 데이터 엔지니어 — 4/4/4
docker0/iptables/Network Namespace 비유→정의 처리 호평. SNAT 누락이 Airflow 워커→외부 API 시나리오에서 의문. Windows `/c/...` 경로 안내 부족.

### 3. 프론트엔드 — 4/3/2 (조건부)
Vercel 자동화 경험자에게 951라인은 "Docker 한 챕터"보다 "리눅스+네트워크+CLI 입문서". 2.5는 비유→정식용어 전환에서 비유가 끊김. 캐릭터 사라짐.

### 4. SI 개발자 — 4/4/3 (조건부)
"VM 갈아엎고 Docker 쓸 만큼인가" 결정타가 부족. 부팅 시간·이미지 크기 실측 한 줄, SI 운영 시나리오(데이터 보존 권한·uid/gid) 보강 필요.

### 5. PM/기획자 — 4/3/2 (조건부)
주방·푸드코트 큰 그림은 회의에서 인용 가능. 하지만 §2.5.3 iptables/DNAT, §2.6 vim 모드는 비개발자 좌절 포인트. vim → nano 또는 호스트 편집기+바인드 마운트 권장.

### 6. 비전공 전직자 — 4/3/3
컨테이너~이미지 비유는 최고. 2.5부터 용어 폭탄(커널·Namespace·veth·docker0·iptables·DNAT·WSL2). "외우지 않아도 됩니다" 안전장치 한 줄 필요.

### 7. DevOps 엔지니어 — 4/3/3 (조건부)
보안·운영 함정 3종(루트 컨테이너 기본값, latest 태그, commit 운영) 모두 누락. 그대로 현업 가면 사고. cgroups·MASQUERADE 보강 필요.

### 8. 임베디드 개발자 — 4/3/4
namespace는 다루지만 cgroups 0줄. mount의 실체(bind = mount namespace에 호스트 inode 노출, volume = `/var/lib/docker/volumes/`) 명시 필요. 부팅 시간 수치 한 줄.

### 9. CS 학생 — 4/4/4
OS 수업 키워드와 정확히 매핑. 정밀도 보강 — namespace 6~8종 + cgroup 한 쌍 명시, NAT PREROUTING/SNAT 양방향, default bridge DNS 미지원 이유, runc/containerd 각주.

### 10. CTO/리드 — 4/4/3 (조건부 추천)
951라인 적정 — 분량보다 밀도. 합격선이지만 commit 안티패턴이 미진해 "이번 주 commit 금지" 가드레일 없으면 단독 자습 부담. `--rm`/`prune`/보안 한 줄 보강.

## 결론

- 비유 체계와 Try/Fail 흐름은 강력. 대부분 통과.
- 결정타는 **(1) cgroups 한 단락 추가 + (2) 캐릭터 1회씩 등장 + (3) §2.6 분리**. 이 셋이 입문~숙련 전 페르소나 만족도를 끌어올림.
- 숙련 페르소나(DevOps·CTO·CS)의 정밀도 지적은 참고 박스로 처리 가능. 본문 톤은 유지.
- 이전 v3 대비 비유 체계가 단단해진 점은 명확. v4에서는 분량·캐릭터 두 축이 다음 단계의 결정타.
