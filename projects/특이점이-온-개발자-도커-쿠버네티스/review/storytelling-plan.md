## 0. 공통 설계 원칙

### 0.1 서술자 목소리
- 오픈이 **어깨 너머 카메라**. 오픈이가 친 명령, 본 화면, 느낀 감각만 서술.
- 학습 상태 메타 해설 금지 (audit §3.1 금지 표현표 참조).

### 0.2 캐릭터 배치 전략

| 캐릭터 | 역할 | 등장 빈도 |
|--------|------|----------|
| **오픈이** | 주인공 / 실행자 | 전 섹션 |
| **선배** | 힌트 제공자 / 첫 실마리 던지는 사람 | 각 챕터 오프닝 1회 + 중반 1회 |
| **팀장** | 문제 제기자 / 큰 방향 잡아줌 | 챕터당 1~2회 (주요 전환점) |
| **동료** | 실사용자 / 불만 제기 | K8s 섹션 후반 (CH04~CH06) |

**2챕터 연속 캐릭터 부재 금지** — 회상 대사라도 한 줄 삽입.

### 0.3 오픈이의 시간·공간 축

| 챕터 | 시간 | 장소 | 감각 키 |
|------|------|------|---------|
| CH01 | 낮→퇴근 후 저녁 | 사무실 자리 → 집 책상 | 에러 로그 화면, 버스 창밖 풍경, 밤의 노트북 팬 소리 |
| CH02 | 밤 | 집 책상 | 커피 향, 키보드 소리, 터미널 깜박임 |
| CH03 | 다음 날 저녁~밤 | 집 책상 | 진행 바, 여러 터미널 창, 브라우저 탭 |
| CH04 | 며칠 뒤 저녁 | 집 책상 | kubectl 긴 출력, Pod 상태 변화 |
| CH05 | 이어지는 밤 | 집 책상 | 명령어 반복 타이핑, 표·다이어그램 노트 |
| CH06 | 주말/여러 저녁 | 집 책상 | 전체 스택 구동, F5 연타, 지도 정리 |

### 0.4 Try/Fail 배치 원칙
- **예제 코드(code/ex*, k8s yaml)에 실제로 존재하는 실패 케이스만** 사용
- 억지로 지어내지 않음. 배우는 과정에서 **자연히 마주치는** 실수(nginx 포그라운드, ubuntu `-d` 즉시 종료, CrashLoop, ImagePullBackOff 등)

## 1. CH01 — 왜 컨테이너인가

**챕터 목표**: 환경 불일치 문제 → 컨테이너 해법의 필요성 납득
**주요 캐릭터**: 선배 (1회 대사)

### 1.1 내 PC에선 됐는데 서버에선 안 됐다

| 항목 | 내용 |
|------|------|
| 목적 | 독자 공감대 형성 — "내 PC에선 된다" 고전 상황 |
| 오픈이 상황 | 낮. 배포 스크립트 돌렸는데 서버에서 실패. 에러 로그 창 앞 |
| 흐름 | (1) 배포 명령 → 실패 로그 (2) 로컬 재현 시도 → 로컬은 돌아감 (3) 선배 한마디 촉발 |
| 필수 요소 | **대사** 선배 "그 문제, Docker 쓰면 끝나." / **독백** `*'같은 코드인데 왜 저기선 안 되지.'*` / **감각** 화면 가득한 빨간 글자 |
| 다음으로 | Docker? 생소하다 → 배경부터 보자 |

### 1.2 항구에서 빌려온 이름

#### 1.2.1 하역 인력이 배 한 척을 묶던 시절
| 항목 | 내용 |
|------|------|
| 목적 | "컨테이너" 어원 — 물류 혁신 이전 상황 |
| 서술 방식 | 오픈이가 퇴근길 지하철에서 검색해 본 영상 한 장면. "왜 컨테이너가 등장했나" 역사 |
| 주의 | **역사 브리핑만 나열하지 말 것.** 오픈이가 영상 보는 장면을 감싸는 틀로 제시 |

#### 1.2.2 표준 상자가 항구를 바꾼 날
| 항목 | 내용 |
|------|------|
| 목적 | 맥린의 표준 컨테이너 — 규격화가 가져온 변화 |
| 흐름 | 영상 속 설명 → 오픈이가 "아 이거랑 지금 내 문제 비슷한데" 느낌 |
| 필수 요소 | **독백** 연결 지점 |

#### 1.2.3 IT로 건너온 컨테이너
| 항목 | 내용 |
|------|------|
| 목적 | 현대 IT에서 "컨테이너" 용어 사용 — 같은 발상 |
| 흐름 | 검색 결과에 Docker 공식 페이지 뜨는 장면. 첫 단어가 "컨테이너" |
| 다음으로 | Docker와 Kubernetes가 뭔지 어렴풋이 알아두자 |

### 1.3 Docker와 Kubernetes — 공장과 관제 센터

| 항목 | 내용 |
|------|------|
| 목적 | 두 도구의 성격 차이를 비유로 먼저 잡음 |
| 오픈이 상황 | 집 도착. 노트북 열기 직전 |
| 흐름 | 해운 비유 연장 — Docker=선적 규격, Kubernetes=항만 관제 |
| 주의 | **프랜차이즈 비유 금지** (이미 제거됨). 해운 일관 유지 |

### 1.4 이 책의 학습 지도

| 항목 | 내용 |
|------|------|
| 목적 | 6챕터 여정 지도 제시 |
| 주의 | **현재 브리핑형 → 오픈이의 "오늘 밤부터 뭘 하기로 했나" 계획** 형태로 전환 |
| 분량 | 짧게. 교과서 로드맵이 아니라 오픈이의 메모 느낌 |
