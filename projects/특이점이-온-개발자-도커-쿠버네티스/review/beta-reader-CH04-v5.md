## 심각도별 이슈

### 높음

1. **YAML 친절도 — 5명**
   - 위치: §4.3.3 첫 YAML (L215~225), §4.4.2 Deployment YAML (L312~331)
   - 증상: `apiVersion: v1` vs `apps/v1` 변경 설명 부재. 들여쓰기 2칸·탭 금지·콜론 뒤 공백 1개 가이드 부재
   - 영향: 비전공·신입·프론트·PM 모두 첫 YAML 진입 장벽
   - 제안:
     - §4.3.3 첫 YAML 위에 들여쓰기 가이드 박스 (3~4줄)
     - apiVersion 표 1개 — Pod=v1, Deployment=apps/v1, Service=v1. "코어 리소스 vs 워크로드 리소스" 그룹 분류
     - `kind`/`metadata`/`spec` 5초 가이드

2. **Deployment → ReplicaSet → Pod 3단 위임 다이어그램 — 3명**
   - 위치: §4.4.3 ReplicaSet (그림 4-16)
   - 증상: 그림 4-13(selector-labels) + 그림 4-16(ReplicaSet-Pod) 분리. Deployment→RS→Pod 통합 위임이 본문에만 있고 시각화 부재
   - 영향: 데이터 엔지니어는 "Airflow Scheduler→Executor→Worker"와 비교 시 확신 안 섬, 임베디드는 평면 구조 익숙해 위임 한 단계도 헷갈림, CS 학생은 "Deployment가 직접 Pod를 만드는가, RS를 만드는가" 정확한 멘탈 모델 못 잡음
   - 제안:
     - §4.4.3 시작에 통합 위임 다이어그램 1장 — "Deployment(매뉴얼) → ReplicaSet(개수 관리자) → Pod(실제 일꾼)" 3단 박스
     - 비유: "본사 지침서 → 지역 본부(개수 맞춤) → 가맹점"
     - 검증 명령어 한 줄 — `kubectl get deploy,rs,pod`로 세 리소스 동시 노출 ("실제로 클러스터에 객체로 존재")

3. **Minikube 환경구축 — 3명 (비전공·신입 첫 실습 진입 장벽)**
   - 위치: §4.2.2 (L135~155)
   - 증상: "Chocolatey 미리 설치돼 있어야 합니다" 한 줄. Chocolatey 설치 링크·관리자 PowerShell 정의·`--driver=docker` 옵션·Docker Desktop 실행 확인·Hyper-V/WSL2/VT-x 트러블슈팅 모두 부재
   - 영향: 비전공·신입은 첫 실습에서 좌초할 위험 매우 높음
   - 제안: §4.2.2를 "사전 준비 → 설치 → 첫 시작 → 트러블슈팅" 4단계로 확장
     - 사전 준비: Docker Desktop 실행 (`docker info`), Chocolatey 설치(chocolatey.org/install) + `choco -v` 검증
     - 설치: 기존 `choco install minikube` 유지
     - 첫 시작: `minikube start --driver=docker` (옵션 한 줄 풀이)
     - 트러블슈팅: 흔한 에러 3개 박스 — DRV_NOT_HEALTHY → Docker Desktop 실행 / VT-x not available → BIOS 가상화 활성화 / 권한 부족 → 관리자 권한 터미널

6. **자동복구 메커니즘 probe→kubelet→RS — 1명 (DevOps 시야)**
   - 위치: §4.4.3 (L370~381)
   - 증상: ReplicaSet이 "개수만" 본다는 톤. 실제 자동복구 트리거(컨테이너 죽음/응답 실패 → kubelet → RS) 누락. probe·kubelet 언급 0건
   - 영향: 운영 시야 절반만 열림. 챕터의 핵심 약속("새벽 2시 죽으면 누가 살리나")의 기술적 답변이 "Deployment가 알아서" 수준에 머묾
   - 제안:
     - 4.4.3 끝 또는 새 절(4.4.4 자동복구 신호의 길) — `livenessProbe` 한 줄 등장. "kubelet이 컨테이너를 주기적으로 두드려보고 응답 없으면 종료 → 그 종료를 RS가 감지해 새 Pod" 3단 흐름. `livenessProbe.httpGet` 4~5줄 예시
     - L368 직후 "그러면 컨테이너가 멈췄을 땐 누가 알아챌까?" 한 문장으로 다음 절 연결

7. **컨트롤플레인 박스 6개 한꺼번에 — 2명**
   - 위치: §4.1.4 참고 박스 (L99~109)
   - 증상: 본문은 3단계로 압축됨 (개선). 박스 펼치면 6개 (API Server·etcd·Scheduler·Controller Manager·kubelet·kube-proxy) 한꺼번에 — v4 그대로
   - 제안: 박스 분할 — 컨트롤 플레인 4개(API Server·etcd·Scheduler·Controller Manager)는 §4.1.4에 두고, 워커 노드 2개(kubelet·kube-proxy)는 §4.2 도입 직전 또는 첫 본문 등장 시점으로 분산. "본사 먼저 → 상가 건물 짓기" 흐름과 정렬

## 결론

**CH04는 가장 안정적인 챕터.** v4에서 즉시투입이었던 CTO가 v5에서도 즉시투입 유지, 비통과 0명, 평균 점수 유지. 새벽 2시 도입·프랜차이즈 비유·IP cliffhanger 3대 강점이 흔들리지 않았고, SI 운영 시야 매핑(maxSurge·rollout undo)으로 SI 비통과·조건부 페르소나가 회복.

**우선순위 권고**: 위 표 1~3번(Minikube 환경 구축·YAML 들여쓰기/apiVersion·Deployment→RS→Pod 통합 다이어그램)만 적용해도 비전공·신입 입문 페르소나의 첫 실습 좌초 위험이 사라지고 평균 +0.3~0.5점 상승 예상. 모두 단일 다이어그램·박스 추가로 가능합니다.

**핵심 진단**: CH04는 v4 호평 강점을 그대로 유지하면서 v4 지적 6개 중 2개를 부분 개선한 안정 라운드. v6에서 환경 구축·YAML·위임 다이어그램 3가지만 보강하면 입문 페르소나에서도 통과 등급으로 전환 가능합니다.
