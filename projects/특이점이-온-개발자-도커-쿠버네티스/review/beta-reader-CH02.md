## 심각도별 이슈

### 높음 (독자가 막힘)

**[이야기 파트 완전 부재]**
챕터 2 전체에 이야기 파트가 없다. 재성 캐릭터는 등장하지 않는다. 챕터 1 마지막에 재성이 컨테이너와 처음 만났는데, 챕터 2에서 그 다음 이야기가 이어지지 않는다. 2.1에서 "밀키트" 비유를 한 문장으로 쓰고 즉시 `Dockerfile` 설명으로 넘어간다. 비유가 이야기 파트가 아닌 기술 파트 도입부에 인라인으로 끼워져 있어, 스타일 규칙(이야기 먼저 → 기술 뒤에서)을 위반한다. 이야기 파트에 공감한 독자가 챕터 2에서 갑작스러운 기술 서적 분위기에 이탈할 위험이 있다.

**[Spring Boot / Gradle — 의도 밖 범위]**
2.6.3 (Spring Boot 백엔드 서버 만들기)에서 `entrypoint.sh`의 `./gradlew build` 명령이 직접 등장하고 `UserController.java` 코드가 노출된다. 씨앗에서 "Spring Boot/Flask/Gradle 심화"를 의도 밖 범위로 명시했다. 실습 실행 시 컨테이너 내부에서 Gradle 빌드가 돌아가며 수 분이 소요되는데, 독자는 이것이 왜 오래 걸리는지, 빌드 실패 시 어떻게 해야 하는지 알 수 없다.

**[학습 목표 이후 이야기 전환 없음]**
1행 `# 챕터 2. Docker 다루기` 바로 아래에 학습 목표 bullet 목록이 나오고, 곧바로 `## 2.1 프로비저닝` 기술 내용이 시작된다. 챕터 들어가기 전 "재성은 오늘도 컨테이너를 켰다" 같은 짧은 도입 이야기가 없다.

### 중간 (혼란스럽지만 진행 가능)

**[`---` 수평선 규칙 위반]**
1406행에 `---` 수평선이 있다. 글쓰기 규칙에서 수평선 사용을 전면 금지하고 있다.

**[depends_on 글-코드 불일치]**
2.5.1에서 `depends_on`을 설명하지만 2.6.5의 실제 EX07 `docker-compose.yml`에는 `depends_on`이 없다. 독자가 직접 따라 치면 DB보다 백엔드가 먼저 올라와 연결 실패가 날 수 있다. (2.6.6에 "잠시 기다린 후 새로고침"이라고 언급되어 있어 우회는 가능하지만 원인을 모른다.)

**[NGINX nginx.conf 진입 장벽]**
EX01에서 처음 등장하는 nginx.conf는 `upstream`, `server`, `listen`, `server_name`, `location`, `proxy_pass` 6개 지시어가 한꺼번에 제시된다. 각 지시어를 처음 접하는 독자를 위한 줄별 설명은 주석에만 의존한다. 핵심 지시어 한두 개를 먼저 소개하고 나머지를 점진적으로 추가하는 구조가 더 적합하다.

**[2.1.2 섹션 번호 중복]**
`### 2.1.2 빌드 시점과 실행 시점` 과 `### 2.1.2 Dockerfile : 스크립트 작성` 두 개가 존재한다. 뒤의 것이 `### 2.1.3`이 되어야 한다. 이후 2.1.3, 2.1.4 번호도 한 칸씩 밀려야 한다.

### 낮음 (개선하면 좋음)

**[챕터 2 목표 "챕터 2에서 배운 컨테이너(NGINX, Redis, MySQL)가 Docker Compose로 연결됨"과 실제 구성 불일치]**
챕터 2 목표 설명에는 "2장에서 배운 컨테이너가 Docker Compose로 연결됨"이라고 했지만, 실제 챕터 2에서 처음으로 NGINX, Redis, MySQL을 배운다. 챕터 1에서는 이들을 간단히 실행만 했다. 표현을 "챕터 2에서 각 컨테이너를 익히고, 마지막에 Docker Compose로 연결한다"로 수정하면 흐름이 더 명확하다.

**[Flask 프레임워크 안내 부족]**
EX03, EX04에서 Flask가 사용된다. 각 파일 상단 주석에 "스프링의 @RestController와 유사한 역할"이라는 힌트가 있지만, Flask를 전혀 모르는 독자(프론트, PM 등)는 왜 Python인지, 왜 이 코드가 이런 형태인지 이해하기 어렵다. 코드 전에 "Flask는 Python의 경량 웹 서버 라이브러리입니다. 실습 목적으로만 사용합니다"라는 한 줄 설명이 도움이 된다.

**[컨테이너 종료 명령 반복]**
`docker stop $(docker ps -q)` / `docker rm $(docker ps -aq)` 조합이 챕터 전반에 걸쳐 5회 반복된다. 첫 등장 시 충분히 설명하고 이후에는 "이전과 동일하게 컨테이너를 정리합니다" 한 줄로 대체할 수 있다.

### 3. 박지우 (프론트엔드 개발자, React 2년, 서버 경험 없음)

**[이야기]** 이야기 파트가 없어 챕터 시작부터 어렵다. 밀키트 비유는 와닿지만 너무 짧게 끝난다. 재성이 "이 설정 매번 반복하기 귀찮아" 같은 상황에 공감하는 이야기가 있었으면 챕터 진입이 쉬웠을 것이다.

**[기술]** NGINX는 "CDN 같은 거구나"로 이해한다. 그런데 nginx.conf의 문법이 낯설다. `upstream`, `location`, `proxy_pass` 세 개가 동시에 나오면 어느 줄이 "경로를 앱으로 보내는 줄"인지 바로 파악하기 어렵다. 코드 한 줄씩 읽어주는 파트가 더 있으면 좋겠다.

**[실습]** EX01~EX04는 따라갈 수 있다. EX07에서 Spring Boot가 나오는 순간 낯선 용어(`@GetMapping`, `ResponseEntity`, Gradle)가 한꺼번에 등장해 잠깐 멈추게 된다. 서버를 다루는 사람을 위한 코드라는 안내가 있으면 덜 부담스럽다.

**[총평]** 전반부는 따라갈 수 있지만 이야기 파트 부재 + 종합 실습에서의 Spring Boot 노출이 부담. 조건부 통과.

**[점수]** 이야기 2/5 | 기술 3/5 | 실습 3/5

**[총평]** 데이터 분석가에게는 서버 사이드 개념이 낯설어 중간에 이탈한다. 이야기 파트가 있었다면 더 오래 붙잡을 수 있었을 것.

### 7. 윤성호 (CTO / 리드 엔지니어, K8s 운영 경험 있음)

**[이야기]** 이야기 파트가 없는 것은 이 독자에게 오히려 빠른 진행으로 편하다. 다만 이 책이 이야기 기반 개념서라면 챕터 2에서도 이야기 파트가 있어야 일관성이 유지된다.

**[기술]** 내용 수준은 적절하다. EX07 docker-compose.yml에 `depends_on`이 없는 것은 운영 관점에서 놓친 부분이다. MySQL 컨테이너가 완전히 준비되기 전에 Spring Boot가 DB 연결을 시도하면 실제로 에러가 발생할 수 있다.

**[실습]** EX07 entrypoint.sh가 컨테이너 시작마다 `git clone`과 `./gradlew build`를 실행하는 방식은 프로덕션 안티패턴이다. "실습용 설계이며, 실제 운영에서는 이미지 레이어에 빌드 산출물을 포함시킨다"는 주석이 있으면 좋겠다.

**[총평]** 기술 내용 자체는 무난. 이야기 파트 일관성과 종합 실습의 설계 주석 보완이 필요.

**[점수]** 이야기 3/5 | 기술 4/5 | 실습 4/5

---

### 8. 한지민 (시니어 풀스택, Docker Compose 사용 경험 있음)

**[이야기]** 경험자 입장에서 이야기 파트 없이 바로 기술 들어가는 게 편하다. 하지만 책 전체 콘셉트가 이야기 기반이라면 일관성이 깨진다.

**[기술]** `---` 수평선이 "이것만은 기억하자" 바로 앞에 있다(1406행). 글쓰기 규칙 위반. 섹션 번호 2.1.2가 두 개 있는 것도 편집 오류다.

**[실습]** 다 알고 있는 내용이라 빠르게 확인. EX07 docker-compose.yml의 `depends_on` 누락이 눈에 띈다. `healthcheck`와 `condition: service_healthy` 조합이 더 안정적인데, 그 수준까지 다루기 어렵다면 최소한 `depends_on: db`는 있어야 한다.

**[총평]** 내용 자체는 깔끔하다. 편집 오류 2건 수정 필요.

**[점수]** 이야기 3/5 | 기술 4/5 | 실습 4/5

### 9. 오태양 (부트캠프 수료생, HTML/JS/SQL 기초, 명령어 첫 경험)

**[이야기]** 이야기 파트가 없어 챕터 1보다 훨씬 어렵게 느껴진다. 재성이 "매번 같은 설정 반복… 귀찮다"는 감정을 공유하는 장면이 있었으면 더 쉽게 진입했을 것이다.

**[기술]** Dockerfile은 한 줄씩 설명이 있어 따라갈 수 있다. NGINX가 나오는 2.2부터 체감 난이도가 급상승한다. `upstream`, `proxy_pass` 등이 한 번에 나오고, `host.docker.internal`이 왜 필요한지 처음 접하는 독자에게 설명이 짧다.

**[실습]** EX01~EX02는 명령어 복사로 실행 가능. EX03에서 Flask 서버 코드가 나오는데 백엔드 경험이 없어서 "이게 왜 여기 있지?" 혼란. EX07 Spring Boot + Gradle에서 완전 이탈. 빌드에 수 분이 걸리는데 화면에 아무것도 안 나오면 오류인지 진행 중인지 모른다.

**[총평]** 전반부는 조건부 통과. 이야기 파트 + EX07 Spring Boot 범위 조정 필요.

**[점수]** 이야기 2/5 | 기술 3/5 | 실습 2/5

### 10. 송유나 (CS 전공 대학생, OS/네트워크 이론, 실무 경험 없음)

**[이야기]** 이야기 파트가 없는 것은 아쉽지만, 이론 배경이 있어서 기술 내용 자체는 소화 가능하다. 밀키트 비유, 놀이공원 매표소, 오케스트라 악보 모두 자연스럽게 이해된다.

**[기술]** OS 수업에서 배운 프로세스, 네트워크 수업의 포트 개념이 Docker 컨테이너에 매핑되어 잘 이해된다. `CMD` vs `ENTRYPOINT` 차이도 실행 흐름으로 설명이 명확하다. `---` 수평선이 규칙 위반이라는 것은 편집 지식이 있어 발견했다.

**[실습]** EX01~EX06 모두 무리 없이 진행 가능. EX07에서 Gradle 빌드가 수 분 소요되는데, 이것이 정상인지 터미널에 표시되지 않아 긴장한다. 빌드 로그를 실시간으로 확인하는 방법(`docker compose logs -f backend`)이 안내되면 좋겠다.

**[총평]** 이론 배경 덕분에 전반적으로 이해 가능. 이야기 파트 부재가 유일한 아쉬움.

**[점수]** 이야기 3/5 | 기술 4/5 | 실습 4/5
