## 심각도별 이슈

### 높음 (독자가 막힘)

없음. 인트로 챕터이고 코드 실습이 없어서 독자가 물리적으로 막히는 지점은 없습니다.

### 중간 (혼란스럽지만 진행 가능)

1. **`---` 수평선 반복 사용**
   - 9행(학습 목표 아래), 67행(1.1 끝), 85행(1.2 끝), 110행(1.3 끝)에 `---` 수평선이 등장합니다. style.md와 structure.md의 "수평선 사용 금지" 규칙 위반입니다. 섹션 구분은 `##` 헤더로 충분하며 `---`는 제거해야 합니다.

2. **1.3 학습 흐름의 아키텍처 이미지 — 용어 선행 노출 문제**
   - 그림 1-6(챕터 3 아키텍처)과 그림 1-7(챕터 4 아키텍처)이 아직 배우지 않은 기술 용어(Pod, Deployment, Service, ConfigMap, Ingress 등)를 포함하고 있을 가능성이 높습니다. 인트로 챕터에서 이런 이미지를 보여주면, 입문자 페르소나(한재원, 강민서, 최민준)는 "이걸 다 알아야 하나?"라는 압도감을 느낍니다. 이미지 없이 텍스트 설명만으로 로드맵을 보여주거나, 이미지 속 용어를 최소화하는 것이 적절합니다.

3. **1.1.3 이미지/컨테이너 혼용**
   - "애플리케이션이 실행되는 데 필요한 라이브러리와 설정 파일, 의존성을 전부 하나의 컨테이너에 담아 버립니다"라고 했는데, 엄밀히 말하면 컨테이너가 아니라 **이미지**에 담는 것입니다. 1.2에서는 "애플리케이션과 실행 환경을 하나의 이미지로 묶고, 그 이미지를 컨테이너로 실행합니다"라고 정확하게 썼는데, 1.1.3은 부정확합니다. 같은 챕터 안에서 표현이 불일치합니다.

### 낮음 (개선하면 좋음)

4. **1.1.3 전환 문장의 교과서적 톤**
   - "그렇다면 이 이야기가 IT와 무슨 관련이 있을까요?"는 전형적인 교과서 전환 문장입니다. "말콤 맥린이 해결한 것과 똑같은 문제가 소프트웨어 세계에도 있습니다"처럼 서술형으로 이어가는 것이 이야기 톤에 더 맞습니다.

5. **"학습 목표" 블록의 위치와 스타일**
   - 챕터 제목 바로 아래에 "## 학습 목표" 블록이 있는데, structure.md의 챕터 헤더 규칙("챕터 제목 바로 아래에 `>` blockquote로 버전/요약/핵심개념 작성")과 맞지 않습니다.

## 페르소나별 상세

### 1. 이준혁 (백엔드 3년, Java/Spring)

**[이야기]** 말콤 맥린 이야기는 배경지식 없이도 술술 읽힙니다. 해상 물류 비효율 -> 표준 컨테이너 도입 -> 전 세계 물류 변혁이라는 서사가 깔끔합니다. "내 컴퓨터에서 됐는데 서버에서 안 돼"라는 문제를 실무에서 직접 겪어봤기 때문에 IT 컨테이너 비유가 즉각 와닿습니다. 다만 1.1.3에서 IT로 전환할 때 "그렇다면 이 이야기가 IT와 무슨 관련이 있을까요?"라는 문장이 교과서 냄새가 납니다.

**[기술]** 기술 파트가 1.2에 테이블 하나로 요약되어 있어서 첫 챕터로서는 적절합니다. Docker = 만들기, Kubernetes = 운영이라는 구분이 명확합니다.

**[총평]** 인트로 챕터로서 역할을 잘 합니다. 짧고 핵심만 전달합니다.

**[점수]** 이야기 4/5 | 기술 4/5

### 2. 박서연 (Python 개발자 2년, 배포 경험 약간)

**[이야기]** 물류 컨테이너 비유가 명쾌합니다. 맥린이 트럭 운전사에서 시작해 전용 선박을 만들기까지의 스토리가 짧지만 인상적입니다. "환경 차이 문제"는 Python 개발에서도 pip 의존성 충돌로 직접 겪어봤기 때문에 공감됩니다.

**[기술]** 1.2에서 Docker와 Kubernetes를 각각 한 문단으로 소개하고 표로 정리한 구조가 깔끔합니다. 그림 1-6, 1-7의 기술 용어가 약간 낯설지만 로드맵 텍스트가 충분히 보완합니다.

**[총평]** 인트로로서 필요한 것만 담고 있습니다. 다음 챕터로 자연스럽게 이어지는 마무리도 적절합니다.

**[점수]** 이야기 4/5 | 기술 4/5

### 3. 김지윤 (프론트엔드 3년, React)

**[이야기]** 말콤 맥린 이야기는 흥미롭게 읽었습니다. 서버나 인프라를 잘 몰라도 이해할 수 있는 비유입니다. 그림 1-3("개발 환경 OK, 운영 환경 ERROR")이 공감됩니다 — 프론트엔드에서도 로컬에선 되는데 배포하면 빌드가 깨지는 경험이 있으니까요.

**[기술]** Docker와 Kubernetes 소개가 짧아서 부담 없이 읽힙니다. 다만 Kubernetes 설명에서 "수십, 수백 개로 늘어납니다"라는 문장에 프론트엔드 개발자로서 거리감이 생깁니다.

**[총평]** 첫 챕터로서 진입 장벽이 낮습니다. 코드 없이 개념과 비유만으로 구성된 것이 서버 경험 없는 독자에게 유리합니다. 1.3의 아키텍처 이미지만 좀 부담스럽습니다.

**[점수]** 이야기 4/5 | 기술 3/5

### 4. 최민준 (데이터 분석가, 배포 안 해봄)

**[이야기]** 컨테이너 물류 비유는 명확합니다. 말콤 맥린이라는 실존 인물의 이야기라서 더 흥미롭습니다. 데이터 분석 환경에서도 "로컬 Jupyter에서는 되는데 서버에서는 패키지가 없다"는 상황이 있기 때문에 그림 1-3에 공감합니다.

**[기술]** Docker와 Kubernetes의 역할 차이가 표 하나로 깔끔하게 정리되어 있습니다. "각 챕터는 이전 챕터의 한계를 느끼는 것에서 시작합니다"라는 문장이 안심을 줍니다.

**[총평]** 인트로 챕터답게 가볍습니다. 배포를 안 해본 사람도 읽을 수 있습니다.

**[점수]** 이야기 4/5 | 기술 3/5

### 5. 오새봄 (PM 3년차, 코드 읽기만 가능)

**[기술]** Docker/Kubernetes 비교 표가 한눈에 들어옵니다. PM이 개발팀과 소통할 때 이 정도 깊이면 충분합니다.

**[총평]** 코드가 하나도 없어서 PM도 편하게 읽을 수 있는 챕터입니다.

**[점수]** 이야기 4/5 | 기술 4/5

**[기술]** "프론트엔드, 백엔드, 데이터베이스, 캐시 서버까지 수십, 수백 개"라는 표현이 와닿지 않습니다. 1.3의 그림 1-6, 1-7에서 용어가 생소해서 부담이 생겼습니다.

**[총평]** 인트로로서 부담이 적습니다. 코드 없이 비유와 그림으로 시작하는 것이 비전공자에게 좋습니다.
