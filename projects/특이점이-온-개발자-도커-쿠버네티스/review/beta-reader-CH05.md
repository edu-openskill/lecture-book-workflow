## 심각도별 이슈

### 🔴 높음 (학생이 막힘 / 핵심 학습 손상)

**H1. kube-proxy(1차)/(2차) 동일성 안내 누락 — L863, L1109**

같은 kube-proxy 프로그램이 노드 안에서 두 시점에 개입한다는 사실이 첫 등장 시점에 명시되지 않음. 비유에서는 "ARS 자동안내"와 "자동 전화 분배"로 완전히 다른 역할로 묘사돼, 비전공자(PM·부트캠프)도 시니어(DevOps·CTO)도 똑같이 "별도 인스턴스인가?"라고 오해. 5.3.1 도입에 한 줄("같은 kube-proxy가 노드 안에서 두 번 — NodePort 진입 직후, ClusterIP 처리 시점 — 개입합니다") 추가 권장.

**H2. 그림 5-17·5-18 실제 스크린샷 미교체 — L698, L705**

`/order` "주문 접수 완료" / `/stores` "매장 선택" 페이지가 mock 상태. 실습 클라이맥스(브라우저로 두 경로 분기 확인)의 시각적 보상이 빈 화면. 출판 전 필수 교체. (이미지 옆 HTML 주석 `<!-- MOCK -->`은 의도적 표시이나 5명이 이를 발견함 → 실제 캡처 필요한 상태)

### 🟠 중간 (혼란하지만 진행 가능 / 정확성 보강)

**M1. NodePort 포트 불일치 — L747 그림 5-19 vs 실습 30080**

5.1.4 실습은 `nodePort: 30080`. 5.3.1 도입 큰 그림(5-19)은 `:9000`으로 표기. CS 학생·DevOps·CTO 모두 "왜 다르지?" 질문. ingress-nginx의 NodePort Service가 별도라는 점을 한 줄 캡션이나 도입 다리 문장으로 안내하거나, 같은 30080 또는 일반화된 `:NodePort`로 통일.

**M2. ex11 → ex12 전환 시 리소스 정리 누락 — L595~608 부근**

ex11에서 만든 NodePort Service(nginx-service:30080)와 ex12의 Ingress Controller Service의 NodePort가 충돌할 가능성. 초보자(SI·백엔드)가 가장 많이 막히는 지점. `kubectl delete -f ex11/` 한 줄 안내.

**M3. kube-proxy(1차)의 정확한 동작 — L863~L866**

현재 본문 "iptables 규칙을 따라 들어온 요청을 Ingress Controller로 자동 연결하는 라우터"는 kube-proxy가 Ingress를 인식하는 듯한 오해 유발. **실제는 NodePort Service의 backend endpoints로 DNAT**하고, 그 backend가 우연히 Ingress Controller Pod일 뿐. 정확한 표현: "들어온 요청을 해당 NodePort Service의 backend Pod(이 경우 Ingress Controller Pod)로 DNAT". 5.3 학습 목표상 단순화 가치는 있지만, "kube-proxy가 두 번 같은 메커니즘(DNAT)을 적용한다"는 점을 짚으면 H1과도 자연스럽게 연결됨.

**M4. Ingress Controller의 DNS 조회 흐름 정확성 — L962**

본문 "이름이 정해진 직후 ... 클러스터 DNS가 곧바로 `order-service` 를 ClusterIP로 풀어 줍니다". 실제 nginx-ingress는 **API Server를 watch해 Service/Endpoint 정보를 직접 캐시**하고 기본적으로 **Pod IP로 upstream 구성**. DNS 조회는 nginx-ingress의 기본 경로가 아님. 개념서 수준의 단순화는 OK이지만 :::note로 "실제 nginx-ingress는 API watch 방식도 사용"한 줄 권장.

### 🟡 낮음 (개선하면 좋음)

**L1. ClusterIP가 가상 IP라는 실체 보충 — L131~133 (5.1.3 ClusterIP 첫 등장)**

"클러스터 내부의 Pod끼리만 서로를 부를 때 사용"만 있고, ClusterIP가 어느 NIC에도 바인드되지 않는 가상 IP(iptables 체인에만 존재)라는 한 줄 없음. 임베디드/네트워크 전공자가 "ping하면 응답 오나?" 의문 (임베디드 8). 5.3.1 3단계에서 DNAT 언급은 있으나 ClusterIP 첫 등장 시점에 힌트가 빠짐.

**L2. LoadBalancer 다이어그램 — 그림 5-5 (L240~261)**

Node 1·2·3 안에 각각 "Service" 박스가 그려져 있어 "Service가 노드마다 하나씩?" 오해 (DevOps·CTO 지적). Service는 클러스터 레벨 논리 리소스, 각 노드의 kube-proxy가 iptables 규칙으로 구현. 캡션 한 줄 보충.

**L3. order/stores 이미지 정보 부족 — L595~608**

`-text=주문 접수 완료` 같은 옵션이 어떤 이미지(hashicorp/http-echo)인지 명시 없음. 프론트엔드 페르소나가 "이게 뭔데 이 옵션을 쓰는 거지?" (프론트엔드 3).

**L4. minikube service --url 통로 유지 비유 약함 — L385~389**

"커서가 멈춰 섰다 → 통로가 유지되는 방식"이 짧게 지나감. PM이 "통로 유지가 무슨 뜻이지?" 잠시 막힘 (PM 5).

**L5. 5.2.3 어색한 문장 — L592 "직접 하거나"**

"실제 운영에서는 `ingress-nginx` 같은 컨트롤러를 직접 하거나"는 동사 누락. "직접 설치하거나" 또는 "직접 배포하거나" (CTO 10).

**L6. 5.3.1 캐릭터 부재**

5.3.1 도입(L713~717)이 강의 톤. 이전 절(5.1·5.2)에는 캐릭터 시행착오 있는데 5.3은 거의 강의. 부트캠프 수료생이 "이야기 느낌이 급격히 줄어든다" 지적 (6번).

## 페르소나별 핵심 인사이트

### 비전공자(PM·부트캠프)
비유 체계는 강하게 작동. "Ingress 설정 바꿔야 해요"라고 들으면 뭔지 설명 가능한 수준. **kube-proxy 1·2차만 풀어주면 거의 완성**.

### 입문자(백엔드·프론트·SI)
Docker→K8s 전환의 네트워크 사고 변화가 잘 잡힘. 다만 **5.3.1의 정보 밀도(전체 흐름 + 단계별 비유 + IT + 표)**가 압도적이라 두 번 읽어야 한다는 의견. 분량 자체보다 **첫 1단계에서 kube-proxy 1·2차 안내가 빠진 게** 누적 부담의 시작.

### 시니어(DevOps·CTO)
**기술적 부정확성 2건 일관 지적**: (a) kube-proxy(1차)가 Ingress를 직접 인식하는 듯한 묘사, (b) Ingress Controller의 DNS 조회 흐름이 nginx-ingress 실제 동작과 다름. 개념서 수준의 단순화는 인정하나, :::note 한두 줄로 보강 가능.

### 깊이 독자(임베디드·CS 학생)
TCP/IP·OSI를 아는 독자에게는 충분한 깊이. 다만 **ClusterIP의 물리적 실체(가상 IP)** 한 줄, **iptables 규칙 예시** 정도가 있으면 만족도 더 올라감.

## 종합 평가

CH05 v2는 개념서로서 **완성도가 높은 수준**입니다. Service → Ingress → 전체 흐름의 3단 구성이 논리적으로 잘 쌓이고, 비유 일관성도 강합니다. 출판 가능 상태에 매우 가까우나 아래 두 가지만 처리하면 한 단계 더 올라갑니다.

**필수 처리 2건**:
1. kube-proxy 1차/2차 동일성 안내 (9/10 페르소나가 막힘 → 학습 흐름 중간에서 멈추는 빈도)
2. 그림 5-17·5-18 실제 캡처 교체 (실습 클라이맥스의 시각 보상)

**권장 처리 4건** (정확성·완결성):
- kube-proxy(1차) 정확화, Ingress Controller DNS 조회 흐름 보강 note, NodePort 포트 일관성, ex11→ex12 리소스 정리

**선택 처리 6건** (만족도 향상):
- ClusterIP 가상 IP 한 줄, LoadBalancer 다이어그램 보강, 어색한 문장 다듬기, 캐릭터 보완 등
