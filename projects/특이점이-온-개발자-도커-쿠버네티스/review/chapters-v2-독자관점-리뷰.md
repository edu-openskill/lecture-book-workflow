## 1. 전제

### 1.1 독자 페르소나

아래 조건을 엄격하게 고수한 상태로 v2 전체를 통독했다.

- 3년차 Java/Spring 백엔드 개발자
- Spring Boot로 JAR 만들어 서버에 `java -jar`로 띄워 본 경험만 있음
- **Docker 한 번도 써본 적 없음.** 이미지/컨테이너가 뭔지 개념도 모름
- **Kubernetes는 이름만 들어봄.** 어떤 역할인지 모름
- **네트워크 지식 전무.** TCP/UDP는 이름만, iptables/DNAT/bridge/namespace는 아예 모름
- **NGINX/Redis 써본 적 없음.** 리버스 프록시, 로드밸런싱 개념도 모름
- MySQL은 SQL 쿼리만 조금. 설정/운영 경험 없음
- WSL2 모름. 리눅스 명령어는 `ls`, `cd` 정도
- YAML 문법은 Spring의 application.yml 수준만
- 책의 목적: "큰 그림" 이해

### 2.4 가장 큰 약점 5가지

1. **용어 정의가 코드/YAML보다 늦게 등장** — selector/label, Endpoint Controller, ImagePullBackOff, reverse proxy 등
2. **OS/네트워크 저수준 개념의 암묵적 가정** — iptables, namespace, PID, IP, port가 설명 없이 쓰임
3. **Java/Spring 배경의 활용 기회 다수 누락** — HttpSession, application.yml, JDBC, @Value, Gradle, java -jar의 연결 부족
4. **비유 → 기술 전환 시 대응 관계 명시 부족** — "주방이 Pod", "픽업 창구가 veth pair" 같은 매핑이 독자 스스로 끌어오게 됨
5. **챕터 간 다리가 "문제 제시"만 있고 "문제 체감"이 없음** — 특히 CH04→CH05: "Pod IP가 바뀐다"는 말은 있는데, 오픈이가 그 문제로 삽질하는 장면이 없음

## 3. 공통 패턴 — 모든 챕터에서 반복되는 이슈

6개 챕터 전체에서 반복 관찰된 패턴이다. 각 패턴은 여러 구체 이슈의 근본 원인이다.

### P1. 용어가 정의되기 전에 코드/YAML에 먼저 등장

**예시**
- `selector`/`label` 이 deploy-ex01.yml 에 먼저 나오고 설명은 뒤 (CH04-v2:408)
- `Endpoint Controller`가 등장한 후에야 "Endpoint"라는 용어가 뭔지 설명됨 (CH05-v2:152)
- `ImagePullBackOff` 용어가 증상으로 먼저 나옴 (CH06-v2:1049)
- `reverse proxy` 가 이름 없이 역할만 묘사 (CH03-v2:234)

**영향**: 독자가 YAML을 보고 "이게 뭐지?"에서 멈춤. 그 아래 줄부터 읽어 답을 찾아야 함.

**처방**: YAML/코드 블록 **직전**에 등장하는 새 용어를 "한 줄 사전 정의" (참고 박스 짧게)

---

### P2. 비유 → 기술 전환 시 "대응 관계" 명시 부족

**예시**
- 푸드코트 비유에서 "픽업 창구 = veth pair, 홀 = docker0" 표는 있는데, 이어지는 본문에서 비유 용어와 기술 용어가 섞여 독자가 매핑을 스스로 끌어옴 (CH02-v2:214)
- 프랜차이즈 "본사/가맹점"과 K8s 컴포넌트(API Server/etcd/kubelet)의 대응이 표 하나로 처리되고 본문에서 재활용 부족 (CH04-v2:167)
- 밀키트 비유(CH03-v2:19)와 Dockerfile의 실제 동작이 1:1 대응 아님

**영향**: 비유는 읽을 때 직관적이지만, 기술 섹션에 오면 비유가 머릿속에서 흐려진다.

**처방**: 비유 테이블을 제시한 후 다음 문단 첫 줄부터 **비유 용어와 기술 용어를 괄호 병기**. 예: "픽업 창구(veth pair)의 한쪽 끝은..."

**미활용 접점 목록**

|------------|------------------------|---------|
| `HttpSession` | Redis 세션 저장소 | CH03-v2:675 |
| `application.yml` | ConfigMap | CH06-v2:52 |

**영향**: 이미 알고 있는 개념으로 새 개념을 이해할 수 있는 "인지 지름길"을 놓친다.

**처방**: 각 접점에 1~2문장 다리. "Spring의 @EnableRedisHttpSession을 써봤다면, 이게 내부적으로 그 원리입니다" 수준.

### P4. OS/네트워크 저수준 개념의 암묵적 가정

**가정된 선행 지식**

- iptables가 뭔지, "규칙"이 어떻게 커널에 심어지는지 (CH02, CH05)
- Namespace가 뭔지 (Network/PID/Mount 등 각각) (CH02, CH04)
- "PID 1"이 왜 특별한지 (CH02)
- 포트의 계층 (호스트 포트 vs 컨테이너 포트 vs Service 포트 vs Pod 포트) (CH02, CH05)
- IP 주소의 종류 (실제 vs 가상/ClusterIP) (CH05)
- 환경 변수가 프로세스 시작 시에만 주입된다는 사실 (CH06)

**영향**: 비유로 이해했다고 생각하지만, 실제로는 "블랙박스를 다른 말로 부르고 있을 뿐" 상태로 넘어감. CH05 kube-proxy에서 이 누적이 터진다.

**처방**: 각 개념 첫 등장 시 "참고 박스"로 2~3줄 선행 정의. 깊이는 제한(이 책 목적은 큰 그림).

---

### P5. 플랫폼 차이(WSL2/macOS VM)가 텍스트 중심, 계층도 부재

**예시**
- CH02-v2:112 WSL2 한 문단으로 설명. Windows 사용자는 "Windows 위에 리눅스 커널을 얹는다"가 뭘 의미하는지 시각화 없이 이해 어려움
- CH02-v2:236 Linux vs Windows/macOS에서 docker0의 위치 차이 — 말로만 설명. 그림이 없음

**영향**: Windows 사용자는 `localhost`로 접근이 되는데 `컨테이너 IP`로는 안 되는 이유를 끝까지 블랙박스로 둔다.

**처방**: 각 OS별 레이어 다이어그램 1장 (GEMINI PROMPT). 예: `Linux: 호스트 → 커널 → docker0 → 컨테이너 / Windows: 호스트 → WSL2 → 리눅스 커널 → docker0 → 컨테이너`

---

### P6. 코드 블록 앞뒤에서 "왜"와 "어떻게"의 연결이 약함

**예시**
- MySQL Dockerfile의 `COPY init.sql /docker-entrypoint-initdb.d` — 이 경로가 왜 특수한지 본문 설명 없음 (CH03-v2:828)
- `SPRING_DATASOURCE_URL`의 `?useSSL=false&serverTimezone=UTC&...` — 각 옵션의 의미 설명 없음 (CH03-v2:1269)
- `nginx.conf`의 `events {}`, `http {}` 블록 — 왜 이 구조인지 설명 없음 (CH03-v2:1218)

**영향**: 복사-붙여넣기 수준으로 따라갈 수는 있지만, 설정을 고쳐야 하는 상황이 오면 멈춘다.

**처방**: 주석으로 핵심만 인라인 설명. 심화는 참고 박스로.

---

### P7. 챕터 내 실습(EX01, EX02...) 목적 차이를 비교표로 안 보여줌

**예시**
- CH03의 EX01(경로 라우팅) / EX02(로드밸런싱) / EX03(캐싱) — 각 nginx.conf 차이점 비교표 없음 (CH03-v2:291)
- CH06.3의 backend-deploy / db-deploy / frontend-deploy / redis-deploy — 구조가 거의 같은데 차이점 부각 안 됨

**영향**: 각 실습을 **같은 패턴의 변주**로 인식하지 못하고, 매번 처음 보는 것처럼 읽게 된다.

**처방**: 실습 시작 전 "이번에 달라지는 것" 표 한 장. `| 실습 | 목표 | 변화 요소 |`

## 4. 챕터별 상세 이슈 (줄 순서)

각 이슈는 우선순위(HIGH/MEDIUM/LOW) 표시. HIGH는 **독자가 책을 놓을 가능성이 있는 지점**, MEDIUM은 **따라가긴 하지만 뭉개진 채 지나가서 나중에 혼란**, LOW는 **다듬으면 좋지만 이해에는 영향 적음**.

### 4.1 CH01-v2 (9건)

#### [CH01-v2:36] 주방 비유 전환의 어색함
- 원문: "배포 얘기 하다가 왜 갑자기 물류가 나오는지 몰랐습니다."
- 카테고리: 흐름어색
- 독자가 멈칫하는 이유: 선배 대사 "물류 역사에 이미 있었어"가 갑자기 나왔고 오픈이가 "왜 갑자기"라고 반응하는데, 독자도 여기서 멈춘다. 비유 시작 신호가 약함
- 제안: 선배 대사 직전에 한 문장 추가 — "선배는 화면을 끄고 일어섰습니다. '너 물류 역사 들어봤어?'"
- 우선순위: LOW

- 카테고리: 기타
- 독자가 멈칫하는 이유: 역사적 사실이 짧게 지나가 실재감이 없다
- 제안: "1950년대 미국 뉴욕의 트럭 운전사 말콤 맥린은 항구에서 화물을 기다리며 하루의 절반을 보내고 있었습니다" 수준의 디테일

## 9. 책 전체 마무리 리뷰

### 9.1 CH06 회고 섹션 평가

**강점**
- CH01 배포 지옥 회수 → 자동화 달성까지의 완결된 여정
- 각 챕터의 역할을 한 줄로 정리
- 감정과 개념의 결합 ("오픈이의 손가락이 떨렸던 순간" → "이제는 자동화")

**약점**
1. "책을 덮은 뒤"의 행동 가이드 부족 — "실무 첫 단계는?" 없음
2. 쿠버네티스 학습 연속성 표시 부족 — 이 책은 기초, StatefulSet/HPA/RBAC 등은 다음 단계임이 암묵적

### 9.2 "마치며" 섹션 추가 권장

CH06 말미에 별도 섹션:

```markdown
### 마치며

이 책은 쿠버네티스의 **기초**를 다룹니다.

**이 책을 마치면 할 수 있는 것**
- Pod, Deployment, Service, Ingress로 서비스 배포
- ConfigMap/Secret/PV/PVC로 설정과 데이터 관리
- Namespace로 리소스 분리
- 기본 디버깅(logs/describe/events/endpoints) 순서 적용

**다음에 배울 수 있는 것 (제목만)**
- StatefulSet: 상태를 가진 Pod (DB 클러스터 등)
- DaemonSet: 모든 노드에 배포되는 Pod (로그 수집 등)
- Job/CronJob: 일회성 · 정기 작업
- HPA(HorizontalPodAutoscaler): 자동 스케일링
- RBAC: 역할 기반 접근 제어
- NetworkPolicy: Pod 간 통신 제한
- Helm: 패키지 관리

**실무로 나갈 때**
팀이 이미 쓰는 쿠버네티스 배포 방식을 먼저 이해하고,
필요한 개념을 하나씩 깊이 학습하는 것이 효율적입니다.
"모든 걸 알고 시작"할 수는 없습니다.
오픈이가 그랬듯, 문제를 만나며 성장합니다.
```
