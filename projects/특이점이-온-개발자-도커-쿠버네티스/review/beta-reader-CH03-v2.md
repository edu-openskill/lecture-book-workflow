## 심각도별 이슈

### 높음 (독자가 막힘)

1. **이야기 파트-기술 파트 미분리**
   - 규칙은 "이야기 먼저 -> 기술 뒤에서"인데, 현재는 오픈이/선배 대화 2~3줄 -> 곧바로 기술/코드가 반복되는 구조입니다. 밀키트, 공용 사물함, 오케스트라 악보 등 좋은 비유가 있으나 기술 설명 사이에 인라인으로 끼워져 있어 독립적인 이야기 파트가 존재하지 않습니다.

2. **캐릭터 규칙 위반**
   - 저자 메모리에 "오픈이/팀장 대화는 프롤로그에만"이 명시되어 있으나, CH03에 오픈이/선배 대화가 약 10회 등장합니다 (13행, 98~100행, 250~252행, 373~375행, 462~466행, 555행, 770~776행, 1082~1084행, 1563행 등).

3. **`---` 수평선 8개소**
   - 11행, 254행, 383행, 401행, 863행, 900행, 1202행, 1234행에 수평선이 있습니다. 네트워크 돋보기 박스 구분선으로 사용 중이나 style.md에서 전면 금지입니다.

### 중간 (혼란스럽지만 진행 가능)

4. **이모지 사용**
   - 돋보기, 전구, 체크/엑스 이모지가 네트워크 돋보기 시리즈에 포함되어 있습니다. style.md에서 전면 금지합니다.

5. **EX07 depends_on 누락**
   - 3.5.1에서 `depends_on`을 설명했지만 종합 실습 docker-compose.yml(1477~1510행)에는 적용되지 않았습니다. 글과 코드가 불일치합니다. v1에서 지적된 이슈이며 미개선입니다.

6. **Spring Boot 코드 잔존**
   - 면책 안내(1372행)는 추가되었으나 `UserController.java` 6줄(1399~1404행)이 여전히 노출됩니다. 의도 밖 범위(Spring Boot 심화)에 해당합니다.

7. **nginx.conf 진입 장벽**
   - 코드가 먼저, 설명이 뒤따르는 순서입니다. "핵심은 location과 upstream"이라는 정리(357행)가 코드 뒤에 위치하여, 초보 독자는 코드를 읽는 동안 어디에 집중해야 할지 모릅니다.

### 낮음 (개선하면 좋음)

8. **네트워크 돋보기 흐름 차단**
   - 네트워크 돋보기 심화 콘텐츠가 실습 흐름 중간에 삽입되어 있으나 "건너뛰어도 됩니다" 안내가 없습니다.

9. **entrypoint.sh 안티패턴 주석 부재**
   - 컨테이너 시작마다 `git clone`과 `./gradlew build`를 실행하는 방식은 프로덕션 안티패턴입니다. "실습용 설계"라는 주석이 필요합니다.

## 페르소나별 상세

### 1. 이준혁 (백엔드 3년, Java/Spring)

**[이야기]** 밀키트 비유, 오케스트라 악보 비유가 적절합니다. 다만 캐릭터 대화와 기술 설명이 섞여 있어 이야기 파트의 독립성이 없습니다.

**[기술]** Dockerfile, docker-compose.yml 설명이 단계적입니다. Spring Boot 코드는 배경지식이 있어 자연스럽습니다. depends_on 누락이 눈에 띕니다.

**[실습]** EX01~EX07 전체 실행 가능합니다. Gradle 빌드 시간 안내가 있으면 좋겠습니다.

**[총평]** 기술적으로 탄탄하지만, 이야기/기술 파트 분리와 캐릭터 규칙 해결이 필요합니다.

**[점수]** 이야기 2/5 | 기술 4/5 | 실습 4/5

### 2. 박서연 (Python 개발자 2년)

**[이야기]** 대화체가 읽기 편합니다. Flask 코드가 익숙합니다.

**[기술]** EX03, EX04 Flask 실습이 자연스럽습니다. NGINX 설정 설명도 단계별 그림 추가 덕분에 v1 대비 개선되었습니다.

**[실습]** depends_on 누락으로 DB 연결 실패 가능성이 걸립니다.

**[총평]** 전반적으로 무리 없이 진행 가능합니다.

**[점수]** 이야기 3/5 | 기술 4/5 | 실습 4/5

### 3. 김지윤 (프론트엔드 3년, React)

**[이야기]** 비유가 와닿지만 독립된 이야기 파트 없이 기술 사이에 삽입되어 효과가 약합니다.

**[기술]** NGINX conf가 여전히 부담스럽지만 v1 대비 단계별 그림이 추가되어 개선되었습니다. EX07에서 Spring Boot 등장 시 낯선 용어가 한꺼번에 나옵니다.

**[실습]** EX01~EX04는 따라갈 수 있습니다. EX07에서 속도가 떨어집니다.

**[총평]** 조건부 통과. 이야기 파트 분리와 Spring Boot 범위 조정이 되면 통과.

**[점수]** 이야기 2/5 | 기술 3/5 | 실습 3/5

### 4. 최민준 (데이터 분석가)

**[이야기]** 밀키트 비유는 좋지만 이야기가 너무 짧습니다.

**[기술]** Flask 코드는 읽을 수 있지만, NGINX conf + Spring Boot가 겹치면 이해 불가입니다.

**[실습]** EX07에서 이탈합니다.

**[총평]** 이탈. Spring Boot 범위 조정 필요.

**[점수]** 이야기 2/5 | 기술 2/5 | 실습 2/5

### 5. 오새봄 (PM 3년차)

**[이야기]** 대화체는 읽기 편하지만 기술 용어가 빠르게 쌓입니다.

**[기술]** Docker Compose의 "한 번에 띄운다"는 개념은 이해하지만, YAML 문법이 어렵습니다.

**[실습]** 독자 상수에 해당하지 않아 이탈합니다.

**[총평]** PM에게는 개념 요약이면 충분합니다.

**[점수]** 이야기 2/5 | 기술 1/5 | 실습 1/5

### 6. 한재원 (부트캠프 비전공자)

**[이야기]** 대화체 덕분에 진입은 쉽지만 기술 용어가 빠르게 쌓여 중반에 이탈합니다.

**[기술]** Dockerfile까지는 따라가지만, NGINX, Redis, Docker Compose가 한 챕터에 다 나오면 과부하입니다.

**[실습]** EX01~EX02까지 가능, 이후 이탈.

**[총평]** 이탈. 챕터가 너무 많은 기술을 한 번에 다룹니다.

**[점수]** 이야기 2/5 | 기술 2/5 | 실습 2/5

### 7. 정유나 (CTO 12년차)

**[이야기]** 편집 관점에서 캐릭터 규칙 위반이 가장 큰 문제입니다.

**[기술]** 기술적으로 정확합니다. `---` 수평선 8곳, 이모지 사용이 규칙 위반입니다.

**[실습]** depends_on 누락과 entrypoint.sh 안티패턴 주석 부재가 아쉽습니다.

**[총평]** 편집 규칙 정비가 필요합니다.

**[점수]** 이야기 2/5 | 기술 4/5 | 실습 4/5

### 8. 임성현 (시니어 인프라 5년)

**[이야기]** 캐릭터 규칙 위반이 눈에 띕니다. 대화체가 기술 깊이를 약화시키는 부분이 있습니다.

**[기술]** 네트워크 돋보기 시리즈의 기술 정확도가 높습니다. entrypoint.sh의 git clone + gradle build 패턴에 "실습용" 주석이 필요합니다.

**[실습]** depends_on 누락은 운영 관점에서 명확한 누락입니다.

**[총평]** 기술 정확도는 좋으나 편집/구조 이슈가 있습니다.

**[점수]** 이야기 2/5 | 기술 4/5 | 실습 4/5

### 9. 강민서 (부트캠프 1개월차)

**[이야기]** 대화체가 친근합니다. 비유가 이해를 돕습니다.

**[기술]** Dockerfile은 한 줄씩 주석이 있어 따라갈 수 있습니다. NGINX부터 어려워지고, Docker Compose YAML은 처음 보는 문법입니다.

**[실습]** EX01~EX02는 가능합니다. EX07은 빌드 시간이 길어 멈춘 것인지 진행 중인지 모릅니다.

**[총평]** 조건부 통과. 전반부는 가능, 후반부에서 속도 급감.

**[점수]** 이야기 3/5 | 기술 2/5 | 실습 2/5

**[실습]** 전체적으로 무리 없이 진행 가능합니다.

**[총평]** 이론 배경 덕분에 전반적으로 이해 가능합니다.
