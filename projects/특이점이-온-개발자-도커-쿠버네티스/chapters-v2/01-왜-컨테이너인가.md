# Ch.1 왜 컨테이너인가

> 한 줄 요약: 컨테이너는 환경을 통째로 포장해서 어디서든 똑같이 실행하는 기술이다
> 핵심 개념: 컨테이너, Docker, Kubernetes

## 1.1 말콤 맥린 : 컨테이너의 아버지

### 1.1.1 해상 운송의 초기

Docker를 이해하려면 '컨테이너'라는 개념이 어디서 왔는지부터 살펴야 합니다. 그 시작은 의외로 바다 위에 있습니다.

컨테이너가 등장하기 전, 해상 물류는 온전히 사람의 힘에 의존했습니다. 항구에 배가 도착하면 수백 명의 노동자가 화물을 하나씩 내리고 분류하고 다시 트럭에 옮겨 실어야 했습니다. 길게는 몇 주씩 걸리는 작업이었습니다.

### 1.1.2 컨테이너의 도입

이 비효율을 가장 가까이에서 체감한 사람이 있었습니다. 바로 트럭 운전사 말콤 맥린입니다.

항구에서 화물이 하나씩 옮겨지는 걸 반복적으로 지켜보던 그는 단순하지만 강력한 의문을 품습니다. **"왜 짐을 통째로 옮기지 않는 걸까?"**

이 질문에서 컨테이너라는 개념이 탄생했습니다. 화물을 낱개로 다루는 대신, 표준 규격의 컨테이너 하나에 담아서 통째로 옮기자는 아이디어였습니다.

크레인으로 컨테이너를 그대로 배에 올리고, 도착지에서도 같은 방식으로 내리면 됩니다. 운송 속도와 효율은 이전과는 비교할 수 없을 정도로 올라갔습니다.

맥린은 여기서 멈추지 않았습니다. 아예 컨테이너 전용 선박을 직접 만들어 버립니다.

그 상상은 결국 현실이 되었습니다. 컨테이너는 해상 운송의 표준으로 자리 잡았고 전 세계 물류 체계를 바꿔 놓았습니다.

### 1.1.3 IT에서의 컨테이너

그렇다면 이 이야기가 IT와 무슨 관련이 있을까요? IT에서 말하는 '컨테이너'도 해상 운송의 컨테이너와 본질은 같습니다.

> **컨테이너(Container)** 는 어디서든 동일하게 실행되는 표준화된 실행 환경을 의미합니다.

컨테이너는 바로 이 문제를 해결합니다. 애플리케이션이 실행되는 데 필요한 라이브러리와 설정 파일, 의존성을 전부 하나의 컨테이너에 담아 버립니다. 말콤 맥린이 화물을 표준 컨테이너에 넣어 어디서든 똑같이 옮길 수 있게 한 것과 같은 원리입니다.

## 1.2 Docker와 Kubernetes

컨테이너라는 개념은 이해했습니다. 하지만 개념만으로는 아무것도 실행할 수 없습니다.

말콤 맥린의 경우도 마찬가지였습니다. "짐을 통째로 옮기자"는 아이디어만으로는 화물이 움직이지 않았습니다. 표준 규격에 맞는 컨테이너를 찍어낼 공장이 필요했고, 컨테이너 전용 선박과 크레인을 갖춘 항구가 필요했습니다. 나아가 컨테이너가 수백, 수천 개로 늘어나자, 어떤 배에 무엇을 실을지, 어떤 순서로 내릴지를 자동으로 관리할 시스템까지 갖춰야 했습니다.

IT 세계도 똑같은 과정을 밟았습니다. 맥린에게 공장이 필요했듯, IT에는 **Docker** 가 그 역할을 합니다. 애플리케이션과 실행 환경을 하나의 '이미지'로 묶어 표준화된 컨테이너로 찍어내는 도구입니다.

컨테이너가 폭발적으로 늘어나자 맥린에게 자동화된 항구 시스템이 필요했듯, IT에는 **Kubernetes** 가 등장했습니다. 컨테이너가 꺼지면 자동으로 되살리고, 트래픽이 몰리면 숫자를 늘리며, 새 버전을 중단 없이 배포하는 거대한 운영 플랫폼입니다.

| 구분 | Docker | Kubernetes |
|------|--------|------------|
| 역할 | 컨테이너를 만들고 실행 | 컨테이너 운영 자동화 |
| 비유 | 화물을 컨테이너에 담는 기술 | 컨테이너를 관리하는 항구 시스템 |
| 핵심 기능 | 이미지 빌드, 컨테이너 실행 | 자동 복구, 스케일링, 무중단 배포 |

## 1.3 이 책의 학습 흐름

이 책은 컨테이너 하나를 띄우는 것에서 시작해, Kubernetes 환경에서 여러 컨테이너를 연결하고 관리하는 것까지 단계별로 경험하는 여정입니다. 각 챕터는 이전 챕터의 한계를 느끼는 것에서 시작합니다.

![](../assets/CH01/chap01-roadmap.png)
*이 책의 학습 흐름*

**챕터 2 — Docker 이해하기.** Docker의 작동 원리를 이해하고, 직접 설치해서 사용해 봅니다. 컨테이너 하나를 띄우는 것부터 시작해 리눅스 명령어로 내부를 탐색하고, 이미지를 직접 만들어 Docker Hub에 공유합니다.

**챕터 3 — Docker 다루기.** 챕터 2에서는 컨테이너를 만들 때마다 같은 패키지를 반복 설치해야 했습니다. Dockerfile로 이 과정을 자동화하고, 여러 종류의 컨테이너를 만들어 Docker Compose로 한 번에 띄우고 서로 연결해 봅니다.

<!--![](../assets/CH01/chap01-ch3-arch.png)
--*챕터 3에서 다루는 컨테이너 구성* -->

**챕터 4 — Kubernetes 시작하기.** 여러 컨테이너를 띄울 수 있게 되었지만, 컨테이너가 죽으면 누가 살려주고, 새 버전은 어떻게 중단 없이 배포할까요? 로컬에 Kubernetes 환경을 구성하고, Pod와 Deployment로 자동 복구와 무중단 배포를 직접 경험합니다.

<!--![](../assets/CH01/chap01-ch4-arch.png)
--*챕터 4에서 다루는 Kubernetes 환경*-->


**챕터 5 — Kubernetes 네트워킹.** Pod를 띄울 수 있게 되었지만, 외부에서 어떻게 접속하고 Pod끼리는 어떻게 통신할까요? Service로 고정 주소를 만들고, Ingress로 외부 요청을 라우팅하며, Docker에서 배운 네트워크 개념이 Kubernetes에서 어떻게 확장되는지 전체 흐름을 조립합니다.

**챕터 6 — Kubernetes 운영하기.** 네트워킹까지 갖춘 상태에서 실제 서비스를 운영하는 데 필요한 것들을 다룹니다. ConfigMap과 Secret으로 설정을 외부에서 주입하고, Volume으로 데이터를 보존하며, 챕터 3에서 만든 웹사이트를 Kubernetes 위에 배포합니다.

## 이것만은 기억하자

- **컨테이너에 담으면 어디서든 똑같다.** 말콤 맥린이 화물을 표준 컨테이너에 넣어 전 세계 어디든 보낸 것처럼, Docker는 애플리케이션을 컨테이너에 담아 어디서든 동일하게 실행합니다.
- **Docker는 컨테이너를 만들고, Kubernetes는 컨테이너를 운영한다.** Docker가 컨테이너를 만드는 기술이라면, Kubernetes는 수백 개의 컨테이너를 자동으로 관리하는 항구 시스템입니다.
- **이 책은 컨테이너 하나에서 시작해 Kubernetes까지 도달하는 여정이다.** 각 챕터는 이전 챕터의 한계를 해결하며 한 단계씩 성장합니다.

이제 직접 Docker를 배워 보겠습니다. 챕터 2에서는 Docker가 컨테이너를 어떻게 만드는지 원리를 이해하고, 설치부터 이미지 생성까지 직접 따라 합니다.
