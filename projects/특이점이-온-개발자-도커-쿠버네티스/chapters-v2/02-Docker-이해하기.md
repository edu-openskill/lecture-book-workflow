# Ch.2 Docker 이해하기

> 한 줄 요약: Docker는 이미지로 환경을 찍고 컨테이너로 실행한다
> 핵심 개념: 가상화, 컨테이너, 이미지, Docker CLI, 마운트

## 2.1 Docker 작동 원리

Docker가 환경을 통째로 옮길 수 있는 이유를 이해하려면 **가상화** 부터 알아야 합니다.

### 2.1.1 가상화 : 하나의 컴퓨터를 여러 개로 사용하는 법

> **가상화(Virtualization)** 는 하나의 물리 서버를 논리적으로 여러 대처럼 나누어 쓰는 기술입니다. 가상화를 도입하면 서버를 효율적으로 쓸 수 있고 서로 다른 환경을 안전하게 격리할 수 있습니다.

하나의 주방에 네 명의 요리사가 일하고 있습니다. 하지만 이 주방에는 냉장고와 가스레인지처럼 핵심 설비가 각각 하나뿐이라, 한 번에 한 명의 요리사만 사용할 수 있는 구조입니다.

![](../assets/CH02/chap01-5.png)
*하나의 주방을 공유하는 네 명의 요리사*

네 명의 요리사가 동시에 요리를 하려면, 주방 환경을 어떻게 구성해야 할까요?

**첫 번째 방법** 은 요리사 수에 맞춰 냉장고, 가스레인지 등 핵심 주방 설비를 각각 추가로 구매해 각 요리사가 독립적으로 사용할 수 있도록 주방을 확장하는 것입니다.

이 방식은 요리사마다 필요한 도구와 공간을 따로 제공하기 때문에, 각자가 완전히 독립된 환경에서 조리할 수 있다는 장점이 있습니다.

그러나 인원 수에 맞춰 설비를 모두 개별적으로 갖춰야 하므로 상대적으로 많은 비용이 발생하는 단점도 있습니다.


![](../assets/CH02/chap01-6.png)
*요리사별 독립 주방 — 주방 전체를 복제하는 방식*


**두 번째 방법** 은 냉장고나 칼처럼 여러 요리사가 함께 사용할 수 있는 기본 설비는 공유하고, 각 요리사에게 필요한 조리대만 개별적으로 마련하는 것입니다.


![](../assets/CH02/chap01-7.png)
*기본 설비 공유 + 개별 조리대 — 컨테이너 가상화 비유*

이 방식은 주방의 핵심 도구를 효율적으로 활용하면서도, 요리사마다 독립적인 작업 공간을 확보할 수 있다는 장점이 있습니다.

다만 주방 설비를 함께 사용하는 구조이기 때문에, 서로 방해하지 않도록 일정한 사용 규칙이 필요합니다.

### 2.1.2 컨테이너 가상화 : OS 없이 격리하다

가상화의 두 번째 방법을 컨테이너 가상화라고 합니다.

> **컨테이너 가상화(Container Virtualization)** 는 하나의 운영 체제를 함께 쓰면서 애플리케이션이 실행되는 환경만 따로 분리하는 방식입니다.

컨테이너 가상화는 애플리케이션 하나를 독립된 컨테이너 안에 넣어 실행하는 방식입니다.

이 컨테이너 안에는 그 앱이 실행되는 데 필요한 라이브러리, 설정, 파일시스템 환경만 들어있고, 운영체제는 호스트의 OS를 공유해 사용합니다.

![](../assets/CH02/chap02-container.png)
*컨테이너 가상화 구조*

컨테이너마다 파일시스템, 네트워크, 프로세스 공간이 분리되어 있어서 컨테이너 1에서 설치한 라이브러리가 컨테이너 2에 영향을 주지 않습니다.

### 2.1.3 Docker : 이미지에서 컨테이너까지

Docker는 컨테이너를 어떻게 만드는 걸까요? 명령어 하나를 따라가 보면 흐름이 보입니다.

컨테이너 생성 명령을 실행하면 **Docker 엔진** 이 요청을 받습니다. 이때 Docker 엔진이 직접 컨테이너를 만드는 건 아닙니다. Docker 엔진은 호스트 OS의 **커널** 에게 **'격리된 프로세스를 하나 만들어 달라'** 고 요청합니다.

커널은 건물 관리인과 같습니다. 입주자(프로세스)가 들어오면 방을 배정하고, 입주자들끼리 충돌을 일으키지 않도록 관리합니다. Docker 엔진이 "방 하나 내주세요"라고 요청하면, 커널이 실제로 격리된 공간을 만들어주는 역할을 합니다.

> **커널(Kernel)** 은 운영체제(OS)의 핵심 구성 요소로, 프로세스를 만들고 실행 순서를 정하며, 프로세스끼리 서로 간섭하지 않도록 메모리를 보호합니다.

Docker 엔진의 요청을 받은 커널은 격리된 환경에서 새 프로세스를 만듭니다. 이 **격리된 환경에서 실행되는 프로세스** 가 **컨테이너** 입니다.

![](../assets/CH02/chap01-9.png)
*docker run 실행 흐름*

### 2.1.4 Docker 이미지

컨테이너가 격리된 환경에서 실행되는 프로세스라면, 컨테이너는 뭘 기반으로 만들어지는 걸까요? 컨테이너를 만들려면 설계도가 필요합니다. 그 설계도가 **이미지** 입니다.

이미지는 컨테이너를 만들기 위한 패키지입니다. 안에는 OS의 **기본 라이브러리(커널 제외)** 와 **애플리케이션**, **설정 파일**이 레이어(층)로 겹쳐 쌓인 구조입니다. 이미지는 붕어빵 틀과 같습니다. 틀(이미지) 하나로 붕어빵(컨테이너)을 여러 개 찍어낼 수 있고, 틀 자체는 변하지 않습니다.

![](../assets/CH02/chap01-13.png)
*하나의 이미지로 여러 컨테이너를 만들 수 있다*

이미지는 **Docker Hub** 라는 저장소에서 가져다 쓸 수 있습니다. 직접 이미지를 만들어 Docker Hub에 올릴 수도 있습니다.

![](../assets/CH02/fig-1-bp-0.png)
*Docker의 흐름*

전체 흐름을 정리하면 이렇습니다. Docker Hub에서 이미지를 내려받아 컨테이너를 실행하고, 컨테이너 안에서 작업한 뒤 수정한 내용을 새 이미지로 저장합니다. 이 이미지를 Docker Hub에 올리면 다른 사람도 가져다 쓸 수 있습니다.

이미지 받고, 컨테이너 만들고, 수정하고, 다시 이미지로 저장하는 흐름입니다. 이제 직접 설치해 보겠습니다.

## 2.2 Docker Desktop : 설치

1. [Docker 공식 사이트](https://www.docker.com/products/docker-desktop/)에 접속하여 자신의 OS에 맞는 **Docker Desktop** 을 다운로드합니다.
2. 다운로드한 설치 파일을 실행하고, 안내에 따라 설치합니다.
3. 설치가 끝나면 Docker Desktop을 실행합니다.

> **Windows** 의 경우 Docker Desktop 설치 전에 **WSL2(Windows Subsystem for Linux 2)** 를 먼저 설치해야 합니다. Docker는 Linux 커널 기능을 사용해 컨테이너를 실행하는데 Windows에는 Linux 커널이 없으므로, WSL2에서 Linux 커널을 제공받아야 합니다.

설치가 끝났다면 터미널에서 다음 명령어로 정상 설치 여부를 확인합니다.

**[실습]** Docker 설치 상태를 확인하는 명령어입니다.

```bash
docker version   # Docker 버전 확인
```

Client와 Server 정보가 모두 출력되면 Docker가 정상적으로 설치된 것입니다.

## 2.3 Docker CLI : 기본 명령어

Docker가 설치되었으니 이제 직접 사용해 보겠습니다. 이미지를 내려받는 것부터 시작합니다.

### 2.3.1 docker pull : 이미지 다운로드

`docker pull` 명령어는 Docker Hub에서 이미지를 내려받습니다.

**[실습]** nginx 이미지를 다운로드하는 명령어입니다.

```bash
docker pull nginx   # nginx 이미지 다운로드
```

![](../assets/CH02/chap01-15.png)
*nginx 이미지 다운로드*

### 2.3.2 docker run : 컨테이너 실행

이미지를 내려받았으니 이제 실행해 보겠습니다.

**[실습]** nginx 이미지를 기반으로 컨테이너를 실행하는 명령어입니다. 

```bash
docker run nginx   # nginx 컨테이너 실행
```

![](../assets/CH02/chap01-17.png)
*nginx 컨테이너 실행*

컨테이너를 실행하자마자 터미널 입력이 막히는 것은 현재 프로세스가 **포그라운드(Foreground)** 상태로 실행되고 있기 때문입니다. 

전화 통화가 연결된 동안에는 다른 전화를 받을 수 없는 것처럼, 컨테이너가 터미널의 제어권을 꽉 잡고 있는 상태입니다.

> **포그라운드(Foreground) :**  프로세스가 터미널을 독점하고 있는 상태입니다. 작업이 완료되거나 종료되기 전까지 터미널에 입력할 수 없습니다.

**CTRL + C** 를 누르면 터미널을 빠져나올 수 있습니다.

### 2.3.3 docker run -d : 백그라운드 실행

> **백그라운드(Background) :** 프로세스가 터미널을 점유하지 않고 뒤에서 독립적으로 실행되는 상태입니다.

컨테이너를 띄워놓고 터미널도 계속 쓰고 싶다면 **-d** 옵션을 붙이면 됩니다.

**[실습]** `-d` 옵션을 사용하여 컨테이너를 백그라운드에서 실행하는 명령어입니다.

```bash
docker run -d nginx   # -d : detached 모드
```

**컨테이너 ID** 가 출력되고 터미널은 입력 가능한 상태로 돌아옵니다. 컨테이너 ID는 실행할 때마다 달라집니다.

![](../assets/CH02/chap01-18.png)
*백그라운드 실행 결과*

### 2.3.4 docker ps : 컨테이너 목록 출력

백그라운드에서 실행 중인 컨테이너가 정말 돌아가고 있는지 확인하려면 `docker ps`를 사용합니다.

**[실습]** 실행 중인 컨테이너 목록을 출력하는 명령어입니다.

```bash
docker ps   # 실행 중인 컨테이너 출력
```

### 2.3.5 자주 사용하는 Docker 명령어

| 명령어 | 설명 | 예시 |
|--------|------|------|
| `docker pull <이미지명>` | Docker Hub에서 이미지 다운로드 | `docker pull nginx` |

## 2.4 컨테이너의 통신

컨테이너는 격리된 환경에서 실행되는 프로세스입니다. 이때 격리되어 있으면 외부와 어떻게 통신할까요?

### 2.4.1 네트워크 격리 — 푸드코트의 식당들

푸드코트를 떠올려 보겠습니다. A, B, C 식당은 나란히 붙어 있지만 벽으로 완전히 분리되어 있어 서로의 내부를 전혀 볼 수 없습니다. 하지만 각 식당마다 홀을 향해 열린 '픽업 창구'가 하나씩 있어서, 이 작은 창구를 통해 주문을 받고 완성된 음식을 내보냅니다. 주방은 독립된 공간으로 격리되어 있지만, 창구를 통해 외부와 필요한 통신을 이어가는 구조입니다.

![](../assets/CH02/ch2-step1-namespace1.png)
*푸드코트 네트워크 격리 비유*

도커의 네트워크 구조도 이와 같습니다. 컨테이너를 실행하면 가장 먼저 **독립된 네트워크 공간(Network Namespace)** 이 생성되어 외부와 완전히 격리됩니다.

그다음 **가상 케이블(veth pair)** 의 한쪽 끝을 컨테이너에, 반대쪽 끝을 호스트의 **가상 스위치(docker0)** 에 연결합니다. 이 케이블이 일종의 다리 역할을 하여, 격리되어 있던 컨테이너가 비로소 외부와 데이터를 주고받을 수 있는 통신 통로를 확보하게 됩니다.

![](../assets/CH02/diagram/ch2-step1-namespace.png)
*컨테이너 네트워크 격리 구조*

| 구성 요소 | 역할 | 비유 |
|----------|------|------|
| **Network Namespace** | 컨테이너별 독립 네트워크 공간 | 식당 |
| **veth pair** | 컨테이너와 호스트를 잇는 가상 케이블 | 픽업 창구 |
| **docker0** | 모든 컨테이너의 연결이 모이는 가상 스위치 | 홀 |

이 단계가 끝나면 컨테이너는 호스트와 통신이 가능합니다. 리눅스에서는 docker0가 호스트 PC에 직접 존재하기 때문에, 컨테이너의 내부 IP로 바로 접근할 수 있습니다. 

하지만 Windows와 Mac에서는 도커가 내부에 **별도의 리눅스 환경을** 만들고 그 안에서 컨테이너를 실행합니다. **docker0 브릿지는** 이 리눅스 환경 안에 있기 때문에 Windows와 Mac 같은 환경에서는 컨테이너의 내부 IP를 찾을 수 없습니다. 그래서 컨테이너와 연결을 위한 다른 방법이 필요합니다.

### 2.4.2 포트포워딩 — 키오스크

이번에는 반대 상황입니다. 외부에서 손님이 푸드코트를 찾아왔습니다. 보안상 손님은 주방 안으로 직접 들어갈 수 없으므로, 입구에 설치된 키오스크를 이용해야 합니다. 대신 입구에 키오스크가 있습니다. 손님이 키오스크에서 짜장면을 누르면 A식당으로, 돈까스를 누르면 B식당으로 주문이 전달됩니다. 

![](../assets/CH02/ch2-step2-dnat1.png)
*푸드코트 키오스크 포트포워딩 비유*

키오스크처럼 도커에서 외부의 요청을 내부 컨테이너로 전달하는 것을 **포트포워딩(Port Forwarding)** 이라고 합니다. 외부에서 호스트 PC의 특정 포트로 요청이 들어오면, 도커는 미리 설정된 규칙에 따라 해당 포트와 연결된 컨테이너로 요청을 보냅니다.

이때 도커 시스템이 내부적으로 요청의 도착지 주소를 실제 컨테이너 주소로 바꿔서 전달하는데, 이 기술적인 과정을 **DNAT(목적지 주소 변환)** 라고 부릅니다. 우리가 포트포워딩이라는 '연결 규칙'을 만들면, 도커가 DNAT이라는 '배달 기술'을 사용해 데이터를 전달하는 구조입니다.

![](../assets/CH02/diagram/ch2-step2-dnat.png)
*iptables DNAT 포트포워딩*

> **참고: iptables**
> iptables는 리눅스 커널에 내장된 **데이터 길잡이** 입니다. 복잡한 길목에서 **"특정 포트로 들어온 요청은 저쪽 컨테이너로 보내라"** 와 같이 데이터의 경로를 지시하는 이정표 역할을 합니다. 도커는 이 시스템을 뒤에서 조종하여, 우리가 직접 하기 어려운 복잡한 네트워크 길 찾기 설정을 자동으로 수행합니다.

이 단계가 끝나면 **외부에서 컨테이너에 접근할 수 있는 상태** 가 됩니다.

### 2.4.3 DNS — 안내 지도

푸드코트에 온 손님이 ‘B식당’을 찾는 상황을 가정해 보겠습니다. 식당 이름은 알지만 정확한 위치를 모를 때, 홀에 붙어 있는 안내 지도를 확인하면 **B식당은 2구역** 이라는 정보를 금방 얻을 수 있습니다. 지도를 통해 이름만으로 목적지를 곧장 찾아가는 것처럼, 도커 내부에도 이런 길잡이 역할을 하는 안내 지도가 존재합니다.

![](../assets/CH02/ch2-step3-dns1.png)
*푸드코트 안내 지도 DNS 비유*

도커 내부적으로 **DNS 서버**를 운영합니다. 새로운 컨테이너가 생성될 때마다 그 이름과 부여된 IP 주소를 이 배치표에 자동으로 적어 넣습니다.

컨테이너가 다른 컨테이너의 이름으로 요청을 보내면, 도커 DNS가 배치표를 확인해 이름을 실제 IP 주소로 변환하여 연결해 줍니다. 덕분에  개발자가 매번 바뀌는 IP 주소를 외우지 않아도, 컨테이너 이름만 알면 통신할 수 있습니다.

> **참고: DNS(Domain Name System)**
> 이름을 IP 주소로 바꿔주는 시스템입니다. 브라우저에서 google.com을 입력하면 DNS가 IP로 바꿔주는 것과 같은 원리입니다.

![](../assets/CH02/diagram/ch2-step3-dns.png)
*Docker DNS 이름 변환*

다만, 이 이름을 통해 컨테이너를 찾는 기능은 도커가 기본으로 제공하는 네트워크에서는 작동하지 않습니다. 사용자가 만든 **사용자 정의 네트워크** 안에서만 이 기능을 활용할 수 있습니다. 네트워크를 만들고  사용하는 구체적인 방법은 챕터 3에서 다루겠습니다.

## 2.5 Linux : 컨테이너 안에서 쓰는 명령어

컨테이너 안은 리눅스 환경이므로 기본적인 리눅스 명령어를 알아야 사용할 수 있습니다. Docker 작업에서 리눅스 명령어는 계속 나오므로 여기서 한 번 정리하고 넘어가겠습니다.

### 2.5.1 Ubuntu : 환경 세팅

리눅스 환경을 만들기 위해 Ubuntu 환경을 준비합니다.

**[실습]** Ubuntu 컨테이너를 실행하는 명령어입니다.  `-p 80:80`은 포트포워딩 설정으로, 2.4에서 배운 키오스크와 같은 원리입니다.


```bash
docker run -it -p 80:80 ubuntu   # -it 옵션은 -i(입력 가능), -t(터미널 환경 제공)를 조합한 옵션
```

![](../assets/CH02/chap01-43.png)
*Ubuntu 컨테이너 실행*

포트 포워딩은 -p [호스트 포트]:[컨테이너 포트] 형식으로 사용합니다. 예를 들어 -p 80:80은 호스트의 80번 포트로 들어온 요청을 컨테이너의 80번 포트로 전달한다는 뜻입니다. 콜론(:)을 기준으로 왼쪽은 **외부 통로** , 오른쪽은 **내부 통로** 라고 이해하시면 명확합니다.


### 2.5.2 탐색 명령어 : pwd, cd, ls, clear

현재 위치를 확인하고 원하는 디렉토리로 이동하며 파일 목록을 조회하는 기본 명령어입니다.

| 명령어 | 설명 | 예시 |
|--------|------|------|
| `pwd` | 현재 위치 경로 출력 | `/root` |
| `cd <경로>` | 해당 폴더로 이동 | `cd home` |
| `cd ..` | 상위 폴더로 이동 | |
| `ls` | 현재 폴더의 파일/폴더 목록 출력 | |
| `ls -l` | 상세 정보와 함께 목록 출력 | |
| `ls -a` | 숨김 파일 포함 전체 출력 | |
| `ls -la` | 숨김 파일 포함 상세 출력 | |
| `clear` | 터미널 화면 비우기 | |

#### 절대 경로와 상대 경로

> 최상위 폴더(/)부터 경로를 표기하는 방식을 **절대 경로**, 현재 위치를 기준으로 경로를 표기하는 방식을 **상대 경로** 라고 합니다.

예를 들어 /home/ubuntu 경로에서 /bin 폴더로 이동하려면 상대 경로 `cd bin`으로는 이동할 수 없고, 절대 경로 `cd /bin`으로 이동해야 합니다.

먼저 `cd /home/ubuntu`로 이동한 뒤, 상대 경로와 절대 경로의 차이를 확인해 보겠습니다.

![](../assets/CH02/chap01-45.png)
*절대 경로로 이동*

실습 후 `cd /`로 루트 경로로 돌아갑니다.

### 2.5.3 파일/폴더 관리 : mkdir, touch, rm, cp, mv

폴더를 만들고, 파일을 생성하거나 복사·이동·삭제하는 명령어입니다.

| 명령어 | 설명 | 예시 |
|--------|------|------|
| `mkdir <폴더명>` | 폴더 생성 | `mkdir hello` |
| `touch <파일명>` | 빈 파일 생성 | `touch a.txt` |
| `rm <파일명>` | 파일 삭제 | `rm a.txt` |
| `rm -r <폴더명>` | 폴더 삭제 (하위 포함) | `rm -r hello` |
| `cp <원본> <사본>` | 파일 복사 | `cp a.txt b.txt` |
| `mv <원본> <대상>` | 파일 이동 또는 이름 변경 | `mv a.txt /tmp` |

> `mv`는 같은 경로에서 파일명만 변경하는 용도로도 사용할 수 있습니다. 예: `mv b.txt c.txt`

### 2.5.4 패키지 관리 : apt

컨테이너에는 기본적으로 최소한의 프로그램만 설치되어 있습니다. 필요한 도구를 설치할 때는 **apt** 명령어를 사용합니다. apt 명령어는 필요한 도구(nginx, vim 등)를 설치할 때 사용하는 패키지 관리 명령어입니다.

| 명령어 | 설명 |
|--------|------|
| `apt update` | 설치 가능한 패키지 목록을 최신 상태로 갱신 |
| `apt list \| grep <키워드>` | 패키지 검색 |
| `apt install -y <패키지명>` | 패키지 설치 (-y: 자동 승인) |

**[실습]** 패키지 목록 갱신 후 nginx를 설치하고 실행하는 명령어입니다.

```bash
apt update           # 패키지 목록 갱신
apt install -y nginx # 패키지 설치
nginx                # 패키지 실행
```

**[실습]** net-tools를 설치하고 포트 상태를 확인하는 명령어입니다.

### 2.5.6 프로세스 관리 : ps, kill

실행 중인 프로세스를 확인하고 불필요한 프로세스를 종료할 때 사용하는 명령어입니다.

| 명령어 | 설명 | 예시 |

### 2.5.7 파일 검색과 로그 확인 : find, tail

| 명령어 | 설명 | 예시 |

| `tail -n <숫자> <파일>` | 마지막 N줄 출력 | `tail -n 50 access.log` |

**[실습]** 파일 이름으로 위치를 검색하는 명령어입니다.

## 2.6 컨테이너 생명주기

빠져나오는 방법에 따라 컨테이너가 꺼지기도 하고 살아있기도 합니다.

### 2.6.1 exit vs detach

실행 중인 컨테이너에서 빠져나오는 방법에 따라 컨테이너가 종료될 수도 있고 계속 살아있을 수도 있습니다.

**[실습]** exit로 빠져나오면 컨테이너가 종료됩니다.

```bash
docker run -it --name dead ubuntu   # dead라는 이름의 ubuntu 컨테이너 실행
exit                                 # 컨테이너 종료
docker ps                            # 실행 중인 컨테이너 확인 → 없음
```

![](../assets/CH02/chap01-28.png)
*컨테이너 실행*

![](../assets/CH02/chap01-29.png)
*exit 후 컨테이너 종료*

컨테이너가 종료되었습니다. 이번에는 다른 방법으로 빠져나와 보겠습니다.

**[실습]** `CTRL + P` -> `CTRL + Q`로 빠져나오면 컨테이너가 살아있습니다.

```bash
docker run -it --name alive ubuntu   # alive라는 이름의 ubuntu 컨테이너 실행
# CTRL + P → CTRL + Q 입력
docker ps                            # alive 컨테이너가 그대로 실행 중
```

![](../assets/CH02/chap01-30.png)
*alive 컨테이너 실행*

*프로세스 유지 확인*

정리하면, **exit** 는 프로세스를 끄는 것이고 **CTRL + P, Q** 는 연결만 끊는 것입니다.

### 2.6.2 -dit : 백그라운드 인터랙티브 실행

`docker run` 사용 시에 `-dit` 옵션은 다음과 같습니다.

| 옵션 | 역할 |
|------|------|
| `-d` | 백그라운드 실행 (detached) |
| `-i` | 입력 가능한 상태 유지 (interactive) |
| `-t` | 터미널 환경 제공 (TTY) |
| `-dit` | 위 세 옵션을 조합하여 백그라운드에서 터미널 입력이 가능한 상태로 실행 |

#### -d 옵션

아까 nginx는 `-d`만으로 백그라운드 실행이 됐습니다. ubuntu도 같은 방식으로 될까요? 하나씩 실행해 보겠습니다. 먼저 nginx 컨테이너를 실행합니다.

**[실습]** -d 옵션으로 nginx를 백그라운드에서 실행하는 명령어입니다.

```bash
docker run -d nginx   # nginx 백그라운드 실행
docker ps             # 실행 중인 컨테이너 확인
```

![](../assets/CH02/chap01-32.png)
*nginx 백그라운드 실행*

`-d` 옵션으로 nginx 컨테이너를 실행하면 백그라운드에서 프로세스가 실행됩니다.

이번에는 ubuntu 컨테이너를 실행합니다.

**[실습]** -d 옵션으로 ubuntu를 실행하는 명령어입니다.

```bash
docker run -d ubuntu   # ubuntu 백그라운드 실행
docker ps              # 실행 중인 컨테이너 확인
```

![](../assets/CH02/chap01-33.png)
*ubuntu -d 실행 후 종료*

실행 중인 컨테이너 목록에 ubuntu가 없습니다. 분명 방금 실행했는데 사라졌습니다. ubuntu 컨테이너는 `-d` 옵션만 쓰면 백그라운드에서 실행되었다가 즉시 종료됩니다.

> Docker 컨테이너는 **메인 프로세스가 살아있는** 동안만 유지됩니다. nginx는 웹 서버이므로 백그라운드에서 요청을 기다리며 실행되도록 설계되어 있습니다. 반면 ubuntu의 메인 프로세스는 `bash`(사용자 명령을 받아 실행하는 프로그램)입니다. bash는 사용자가 명령을 입력해야 동작하기 때문에 터미널이 연결되어 있지 않으면 할 일이 없어 즉시 종료됩니다.

#### -dit 옵션

`-dit` 옵션을 쓰면 어떻게 될까요?

**[실습]** -dit 옵션으로 백그라운드에서 실행하는 명령어입니다.

```bash
docker run -dit ubuntu   # ubuntu 백그라운드 + 인터랙티브 실행
docker ps                # 실행 중인 컨테이너 확인
```

![](../assets/CH02/chap01-34.png)
*ubuntu -dit 실행*

이번에는 ubuntu 컨테이너가 백그라운드에서 정상적으로 실행됩니다.

### 2.6.3 CMD : 컨테이너 시작 명령어 덮어쓰기

> **CMD(COMMAND)** 는 컨테이너가 시작될 때 실행되는 기본 프로세스를 정의하는 명령입니다. `docker run <이미지명> <CMD명령>`으로 직접 명령을 주면 `<CMD명령>` 옵션이 실행되며, 명령을 주지 않으면 이미지에 설정된 기본 CMD가 실행됩니다.

`sleep 1000`은 프로세스가 1000초 동안 대기하도록 만드는 명령입니다.

**[실습]** CMD 명령을 지정하여 ubuntu를 실행하는 명령어입니다.

```bash
docker run -d ubuntu sleep 1000   # sleep 명령으로 ubuntu 컨테이너 유지
docker ps                         # 실행 중인 컨테이너 확인
```

`sleep 1000`이 메인 프로세스가 되어 1000초 동안 대기하므로, ubuntu 컨테이너는 `-d` 옵션만으로도 종료되지 않고 유지됩니다.

![](../assets/CH02/chap01-35.png)
*CMD로 프로세스 유지*

아까 `-d`만으로는 종료됐는데 이번에는 살아있습니다. bash 대신 **sleep** 이 메인 프로세스가 되었기 때문입니다. 이렇게 뒤에 명령어를 붙이면 기본 프로세스를 덮어쓸 수 있습니다.

### 2.6.4 attach : 실행 중인 컨테이너에 접근

백그라운드에서 돌고 있는 컨테이너에 다시 들어가고 싶을 때는 **attach** 와 **exec** 명령어를 사용합니다.

> **attach 명령어** 는 현재 실행 중인 **메인 프로세스(PID 1)** 에 직접 접근하는 명령어입니다. `docker attach <컨테이너ID>` 명령어로 실행합니다.

![](../assets/CH02/chap01-36.png)
*attach 명령어의 동작 방식*

ubuntu 컨테이너를 백그라운드로 실행합니다.

`attach` 명령어로 접근하면 ubuntu의 입력 터미널 창이 나타납니다.

**[실습]** attach 명령어로 실행 중인 컨테이너의 메인 프로세스에 접근하는 명령어입니다.

```bash
docker run -dit ubuntu  # ubuntu 백그라운드 실행
docker ps               # 실행중인 컨테이너 확인
docker attach d2b1      # attach로 접근
```

![](../assets/CH02/chap01-38.png)
*attach로 접근*

`attach` 명령어는 **메인 프로세스(PID 1)** 에 직접 연결되므로, 여러 터미널에서 접속해도 모두 같은 화면과 같은 프로세스를 공유합니다.

다만 이 상태에서 잘못 빠져나와 메인 프로세스를 종료하면 컨테이너 전체가 종료되니 주의해야 합니다.

`CTRL+P`, `CTRL+Q`를 입력해 프로세스를 빠져나옵니다.

### 2.6.5 exec : 컨테이너에 새 프로세스로 접근

`attach`는 메인 프로세스에 직접 붙기 때문에 잘못 빠져나오면 컨테이너가 종료될 위험이 있습니다. `exec`는 새 프로세스를 따로 만들어서 들어가므로 메인 프로세스를 건드리지 않습니다.

> **exec 명령어** 는 현재 실행 중인 메인 프로세스가 아닌, 메인 프로세스와 동일한 환경의 **새로운 프로세스** 를 생성하여 접근하는 방식입니다. `docker exec <옵션> <컨테이너ID> <CMD>` 명령어로 실행합니다.

![](../assets/CH02/chap01-39.png)
*exec 명령어의 동작 방식*

다음 명령어로 프로세스 내부에 접근합니다.

**[실습]** exec 명령어로 컨테이너 내부에 새 프로세스를 생성하여 접근하는 명령어입니다.

```bash
docker exec -it d2b1 bash   # 실행 중인 컨테이너에 새 bash 접속
```

![](../assets/CH02/chap01-40.png)
*exec로 접근*

`exec` 명령어로 접근한 상태에서, 호스트 PC에서 **터미널을 하나 더 열어** 다음 명령어를 실행합니다.

![](../assets/CH02/chap01-41.png)
*호스트에서 새 터미널 창 실행*

새 터미널에서 다음 명령어를 실행합니다. 컨테이너 내부에서 실행 중인 프로세스 정보를 출력합니다.

**[실습]** 컨테이너 내부에서 실행 중인 프로세스 목록을 출력하는 명령어입니다.

```bash
docker exec d2b1 ps aux   # 컨테이너 내부 프로세스 목록 확인
```

조회 결과에서 PID 1이 메인 프로세스이고 **PID 12** 가 `exec` 명령어로 접근할 때 생성된 프로세스입니다.

![](../assets/CH02/chap01-42.png)
*프로세스 목록 확인*

`exec` 명령어로 실행된 프로세스는 **메인 프로세스(PID 1)** 에 영향을 주지 않고 독립적으로 명령을 실행합니다.

메인 프로세스(PID 1)의 세션에서 임시로 선언한 변수는 해당 세션에만 존재하므로, exec로 새로 접속한 프로세스에서는 보이지 않습니다.두 프로세스는 환경을 공유하지만 서로 다른 터미널 세션을 사용하기 때문입니다.

## 2.7 이미지 만들기

컨테이너 안에서 이것저것 설치했는데, 이 상태를 저장할 수 있을까요? 컨테이너를 이미지로 구우면 됩니다.

### 2.7.1 Tomcat : 이미지 내려받기 & 실행하기

**[실습]** Tomcat 컨테이너를 실행하는 명령어입니다.

```bash
docker run -d -p 8080:8080 tomcat   # Tomcat 컨테이너 백그라운드 실행
```

![](../assets/CH02/chap01-56.png)
*Tomcat 컨테이너 실행*

`docker ps`로 실행 중인 Tomcat 컨테이너를 확인합니다. **컨테이너 ID** 는 `5fcd`입니다.

브라우저에서 `localhost:8080`으로 접속하면 404 에러가 뜹니다.

![](../assets/CH02/chap01-58.png)
*Tomcat 404 에러*

별도 컨트롤러가 없으면 슬래시(/)로 들어온 요청은 서버 안의 index.html로 응답합니다. Tomcat 이미지에 webapps/ROOT/index.html이 없으니 응답할 페이지가 없어서 404가 뜨는 것입니다.

### 2.7.2 Tomcat 컨테이너 수정 : index.html 만들기

직접 index.html을 만들어 보겠습니다.

Tomcat의 index.html 파일은 webapps/ROOT 폴더에 위치합니다. Tomcat 프로세스 내부로 접근해서 그 경로로 이동해 보겠습니다.

**[실습]** Tomcat 컨테이너 내부로 접근하는 명령어입니다.

```bash
docker exec -it 5fcd bash  # Tomcat 프로세스 연결
cd /usr/local/tomcat/webapps # webapps 경로 이동
```

> webapps 경로는 `find / -name webapps` 등으로 검색해 찾을 수 있습니다.

webapps 폴더에서 `ls` 명령어로 확인하면 내부가 비어 있습니다.

![](../assets/CH02/chap01-61.png)
*webapps 폴더 비어있음*

webapps 폴더에 ROOT 폴더를 만들고 그 안에 index.html 파일을 작성합니다.

**[실습]** vim 패키지를 설치하고 index.html 파일을 생성하는 명령어입니다.

```bash
mkdir ROOT          # ROOT 폴더 생성
cd ROOT             # ROOT 폴더로 이동

apt update          # 패키지 최신화
apt install -y vim  # vim 패키지 설치
vim index.html      # index.html 파일 생성
```

vim은 터미널 기반 텍스트 편집기입니다. 처음 열리면 **일반 모드**라서 바로 타이핑이 안 됩니다. `i` 키를 누르면 **입력 모드**로 전환되어 내용을 입력할 수 있고, 입력이 끝나면 `ESC`를 눌러 일반 모드로 돌아온 뒤 `:wq`를 입력하고 `ENTER`를 누르면 저장 후 종료됩니다. 실수로 잘못 입력했다면 `:q!`로 저장하지 않고 나갈 수 있습니다.

![](../assets/CH02/chap01-64.png)
*index.html 작성*

브라우저에서 `localhost:8080`으로 접속하면 방금 만든 index.html 페이지가 응답합니다.

![](../assets/CH02/chap01-65.png)
*index.html 응답 확인*

### 2.7.3 docker commit : 이미지 굽기

이제 이 상태를 이미지로 저장해 보겠습니다.

먼저 컨테이너에서 빠져나옵니다. `exec`로 만든 셸이므로 `exit`를 입력해도 컨테이너는 종료되지 않습니다.

> **docker commit <컨테이너ID> <dockerhub아이디/이미지명:태그>** 형태로 명령어를 작성합니다. Docker Hub에서 계정과 리포지토리를 찾는 데 사용되므로, 본인의 Docker Hub 아이디로 변경해야 합니다.

**[실습]** 컨테이너의 현재 상태를 새 이미지로 저장하는 명령어입니다.

```bash
docker commit 5fcd <본인-dockerhub-id>/tomcat   # 컨테이너 현재 상태를 이미지로 저장
```

![](../assets/CH02/chap01-67.png)
*이미지 커밋 완료*

### 2.7.4 docker push : Docker Hub에 저장

이미지가 만들어졌으니 Docker Hub에 올려 보겠습니다.

`docker login` 명령어로 Docker Hub에 로그인합니다. Docker Hub 계정이 없다면 [hub.docker.com](https://hub.docker.com/)에서 먼저 회원가입을 합니다.

**[실습]** Docker Hub에 로그인하는 명령어입니다.

```bash
docker login   # Docker Hub 로그인
```

Username과 Password를 입력한 뒤 ENTER를 누릅니다.

`docker push <이미지명>` 형태로 명령어를 실행하여 내 Docker Hub에 저장합니다.

**[실습]** 이미지를 Docker Hub에 업로드하는 명령어입니다.

```bash
docker push <본인-dockerhub-id>/tomcat   # 이미지를 Docker Hub에 업로드
```

![](../assets/CH02/chap01-69.png)
*이미지 푸시 완료*

Docker Hub의 Repositories 탭에서 올린 이미지를 확인할 수 있습니다.

![](../assets/CH02/chap01-70.png)
*Docker Hub 저장소 확인*

이미지 하나만 있으면 누구든 같은 환경을 그대로 쓸 수 있습니다.

## 2.8 마운트 : 데이터 보존

실습을 마무리하며 이전에 만든 컨테이너를 삭제하면 안에 저장해 둔 데이터까지 전부 사라집니다. 컨테이너는 휘발성이기 때문입니다. 중요한 데이터나 로그는 **마운트** 로 외부에 빼놔야 합니다.

> **마운트(Mount)** 는 컨테이너 내부의 경로를 외부 저장소에 연결하는 기능입니다. 마운트를 사용하면 컨테이너가 삭제되어도 데이터는 외부에 그대로 남아 있습니다.

Docker는 두 가지 마운트 방식을 제공합니다. 로컬 PC의 폴더에 직접 연결하는 **바인드 마운트(Bind Mount)** 와 Docker가 관리하는 저장소를 사용하는 **볼륨 마운트(Volume Mount)** 입니다.

### 2.8.1 바인드 마운트 : 호스트 폴더를 직접 연결

> **바인드 마운트(Bind Mount)** 는 호스트 PC의 실제 폴더를 그대로 컨테이너 내부에 연결하는 방식입니다.

![](../assets/CH02/bind-mount.png)
*호스트 PC의 폴더와 컨테이너 내부 폴더가 직접 연결*

**[실습]** 바인드 마운트를 설정하여 ubuntu 컨테이너를 실행하는 명령어입니다.

먼저 호스트에 마운트할 폴더를 생성합니다. **-mount** 옵션은 호스트에 연결할 폴더가 미리 만들어져 있어야만 에러 없이 동작하므로 주의해야 합니다.

```bash
# Windows
mkdir -p /c/app/bind

# Mac / Linux
mkdir -p ~/app/bind
```

```bash
# Windows
docker run -it --mount type=bind,src=C:/app/bind,dst=/app/bind ubuntu

# Mac / Linux
docker run -it --mount type=bind,src=$HOME/app/bind,dst=/app/bind ubuntu
```

![](../assets/CH02/chap01-73.png)
*바인드 마운트 실행*

명령어를 실행하면 호스트의 `bind` 폴더와 컨테이너 내부의 `/app/bind` 폴더가 연결됩니다.

**[실습]** app 폴더의 내부를 확인하는 명령어입니다.

```bash
ls /app
```

![](../assets/CH02/chap01-74.png)
*bind 폴더 확인*

`/app/bind` 폴더에 `a.txt` 파일을 생성합니다.

**[실습]** 마운트된 폴더에 파일을 생성하는 명령어입니다.

```bash
touch /app/bind/a.txt
```

![](../assets/CH02/chap01-76.png)
*파일 생성 확인*

생성한 파일이 PC의 하드디스크에 저장됩니다.

![](../assets/CH02/chap01-77.png)
*호스트 PC에 저장 확인*

컨테이너 안에서 만든 파일이 호스트 PC에도 그대로 있습니다. 실습 후 **exit** 명령어로 컨테이너를 빠져나옵니다.

### 2.8.2 볼륨 마운트 : Docker가 관리하는 저장소

> **볼륨 마운트(Volume Mount)** 는 Docker 엔진 내부의 전용 저장 공간(Volume)을 컨테이너 폴더와 연결하는 방식입니다.

![](../assets/CH02/volume-mount.png)
*Docker 엔진이 관리하는 내부 저장 공간에 데이터 저장*

다음 명령어로 현재 연결된 볼륨 정보를 확인합니다.

**[실습]** 현재 볼륨 목록을 확인하는 명령어입니다.

```bash
docker volume ls   # 볼륨 목록 확인
```

![](../assets/CH02/chap01-80.png)
*볼륨 목록 조회*

연결된 볼륨이 없습니다. 다음 명령어로 볼륨이 연결된 컨테이너를 생성합니다.

`docker run -it --mount type=volume,src=<볼륨명>,dst=<컨테이너경로> <이미지>` 형태로 명령어를 실행합니다.

**[실습]** 볼륨 마운트를 설정하여 ubuntu 컨테이너를 실행하는 명령어입니다.

```bash
docker run -it --mount type=volume,src=metacoding-volume,dst=/app/volume ubuntu   # 볼륨 마운트로 ubuntu 실행
```

![](../assets/CH02/chap01-81.png)
*볼륨 마운트 실행*

`/app/volume` 폴더가 자동으로 생성됩니다. 

> 이때 컨테이너 내부 폴더에 이미 파일들이 들어있다면, 볼륨 마운트는 그 파일들을 새 볼륨으로 복사해와서 보존해 줍니다.
> 반면, 바인드 마운트는 호스트의 폴더 내용으로 내부를 완전히 덮어버리므로 기존 파일이 사라질 수 있습니다.

![](../assets/CH02/chap01-82.png)
*volume 폴더 생성*

`/app/volume` 폴더 내부에 `b.txt` 파일을 생성합니다.

**[실습]** 볼륨 마운트된 폴더에 파일을 생성하는 명령어입니다.

```bash
touch /app/volume/b.txt
```

![](../assets/CH02/chap01-83.png)
*b.txt 파일 생성*

`b.txt` 파일이 생성되었다면 볼륨에 저장된 것입니다.

`exit` 명령어로 컨테이너를 종료한 뒤 볼륨 정보를 확인합니다.

**[실습]** 컨테이너 종료 후 볼륨이 유지되는지 확인하는 명령어입니다.

```bash
exit
docker volume ls
```

![](../assets/CH02/chap01-84.png)
*볼륨 유지 확인*

컨테이너는 종료되었지만 볼륨은 유지되고 있습니다. 새 컨테이너에 같은 볼륨을 연결하면 이전 데이터를 그대로 사용할 수 있습니다.

**[실습]** 새로운 컨테이너에서 기존 볼륨의 데이터를 확인하는 명령어입니다.

```bash
docker run -it --mount type=volume,src=metacoding-volume,dst=/app/volume ubuntu   # 볼륨 마운트로 ubuntu 재실행
ls /app/volume
```

![](../assets/CH02/chap01-85.png)
*볼륨 데이터 재사용*

컨테이너를 새로 만들었는데 이전 데이터가 그대로 있습니다. 새로 만든 컨테이너에서도 이전 컨테이너가 쓰던 볼륨의 데이터를 그대로 볼 수 있습니다. 볼륨이 컨테이너와 독립적으로 유지되기 때문입니다.

| 구분 | 바인드 마운트 | 볼륨 마운트 |
|------|------------|-----------|
| 저장 위치 | 호스트 PC의 지정 폴더 | Docker 엔진 내부 저장 공간 |
| 관리 주체 | 사용자 직접 관리 | Docker 엔진이 관리 |
| 경로 지정 | 절대 경로 필요 | 볼륨 이름만 지정 |
| 사용 사례 | 개발 중 소스 코드 공유, 설정 파일 연결 | DB 데이터 보존, 컨테이너 간 데이터 공유 |
| 성능/보완 | 호스트 환경에 의존적 (직접 노출) | 도커가 최적화하여 관리 (더 안전함) |
| 기존 데이터 처리 | 호스트 폴더가 컨테이너 내부를 덮어씀 | 컨테이너 내부 파일을 볼륨으로 가져옴 |

## 이것만은 기억하자

- **컨테이너**는 격리된 환경에서 실행되는 프로세스입니다. 가볍고 빠르게 생성/삭제할 수 있고, 이미지만 있으면 어디서든 동일한 환경을 재현할 수 있습니다.
- **이미지**는 컨테이너를 만드는 틀입니다. 하나의 이미지로 여러 컨테이너를 만들 수 있고, `commit`으로 새 이미지를 만들어 Docker Hub에 공유할 수 있습니다.
- **포트 포워딩**은 호스트의 포트로 들어온 요청을 컨테이너의 포트로 전달합니다.
- **마운트**는 컨테이너 외부에 데이터를 저장하는 방법입니다. 컨테이너가 삭제되어도 데이터는 남아있습니다.

이제 컨테이너 하나를 만들 수 있게 되었습니다. 이미지도 만들고, Docker Hub에 공유하고, 데이터도 보존할 수 있습니다. 하지만 실제 서비스는 컨테이너 하나로 끝나지 않습니다. 웹 서버, 백엔드 API, 데이터베이스를 각각 컨테이너로 띄워야 하고, 사용자가 늘면 같은 서버를 여러 대 복제해야 합니다. 컨테이너를 하나씩 따로 실행하고 관리하는 건 한계가 있습니다. 다음 챕터에서는 여러 컨테이너를 한꺼번에 관리하는 방법을 알아보겠습니다.
