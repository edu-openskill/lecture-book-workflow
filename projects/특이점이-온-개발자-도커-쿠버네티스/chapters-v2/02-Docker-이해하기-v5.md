# Ch.2 Docker 이해하기

오픈이는 퇴근하고 집으로 돌아와 커피 한 잔을 옆에 두고 노트북을 열었습니다. 오늘 저녁은 Docker에 쓰기로 마음먹었습니다. 목표는 하나로 잡았습니다.

**낮에 서버에 올리다 실패한 그 빌드 결과물을, Docker 컨테이너에 담아 다시 실행해 본다.**

선배가 한 마디 던져준 이름 하나에 목적을 걸어둔 셈이었습니다. 어제까지 Docker라는 단어조차 모르던 상태에서 오늘 저녁에 바로 될 일 같지는 않았지만, 여기까지 가봐야 선배가 말한 이 도구가 정말 자기 문제를 풀어주는지 확인할 수 있을 것 같았습니다. 설치부터 들어가고 싶은 손을 잠깐 멈췄습니다. *'명령어를 치기 전에 이 도구가 환경을 어떻게 다루는지부터 이해하는 편이 나을 텐데.'* 오후에 겪은 문제가 환경 불일치라면, Docker의 원리를 먼저 봐야 뒤에 나올 명령어들이 손에 잡힐 것 같았습니다.

## 2.1 컨테이너는 무엇인가

### 2.1.1 주방 네 개 대신 주방 하나

오픈이가 맡은 앱을 담을 **상자**가 컨테이너라면, 그 상자가 어떤 원리로 환경을 따로 가지는지부터 이해할 필요가 있었습니다. 자료를 몇 개 훑어보니 같은 비유가 반복되고 있었습니다. 주방과 요리사 그림이었습니다.

한 주방에 요리사 네 명이 있습니다. 네 명이 동시에 요리를 하려면 주방을 어떻게 꾸며야 할까요.

![](../assets/CH02/chap01-5.png)

*그림 2-1 한 주방에 네 명의 요리사가 있는 상황*

첫째는 요리사마다 냉장고, 가스레인지, 조리대를 각자 한 세트씩 사주는 길입니다. 네 사람이 완전히 독립된 환경에서 요리합니다. 대신 같은 설비를 네 번 갖춰야 하니 비용이 많이 듭니다.

![](../assets/CH02/chap01-6.png)

*그림 2-2 주방 설비를 통째로 복제하는 첫 번째 길*

둘째는 냉장고와 가스레인지 같은 공용 설비는 함께 쓰고, 칼·도마·조리대처럼 개인이 쓸 도구만 각자 챙기는 길입니다. 장비 수는 그대로인데 공간은 네 명이 동시에 쓸 수 있습니다.

![](../assets/CH02/chap01-7.png)

*그림 2-3 공용 설비는 공유하고 개별 공간만 쪼개는 두 번째 길*

IT에 바로 대응되는 두 가상화 방식이 있었습니다. 첫 번째가 **하이퍼바이저 가상화**로 결과물이 **가상 머신(VM)**, 두 번째가 **컨테이너 가상화**로 결과물이 **컨테이너**입니다. 둘을 묶어서 **가상화**라고 부릅니다. 오픈이가 오늘 목표로 잡은 것이 두 번째, 바로 이 **컨테이너**입니다. 공용 설비는 공유하면서 각자 자기 조리대만 따로 가지는 구조라, **같은 OS 위에서 여러 앱이 서로 충돌하지 않고 각자 돌 수 있게** 해 주는 방식입니다. 오후에 겪은 "로컬과 서버 환경이 달라서 안 됐다"는 상황이 이 안에 들어오면 자리가 잡혔습니다. 내 앱을 이 컨테이너에 담으면, 컨테이너 안의 조리대는 내 앱 기준으로 세팅된 상태로 서버로 건너가는 셈입니다.

> **참고: 하이퍼바이저 가상화**
> 하이퍼바이저(가상화 소프트웨어)가 하드웨어 위에 OS를 통째로 따로 올려 독립된 컴퓨터처럼 쓰게 해 주는 방식입니다. 이 방식으로 만들어진 결과물을 **가상 머신(VM)**이라 부릅니다. 격리가 완전한 대신 OS가 매번 통째로 뜨므로 무겁고 기동이 느립니다.
>
> **참고: 컨테이너 가상화**
> 호스트 OS의 커널(운영체제의 핵심)을 여러 컨테이너가 공유하고, 파일시스템·네트워크·프로세스 공간만 컨테이너별로 분리하는 방식입니다. 이 방식으로 만들어진 결과물을 **컨테이너**라 부릅니다. 가볍고 빠르게 뜹니다.
>
> **참고: 가상화(Virtualization)**
> 하나의 물리 서버를 논리적으로 여러 대처럼 나누어 쓰는 기술입니다. 자원을 효율적으로 쓰면서 서로 다른 환경을 안전하게 격리하는 것이 목적입니다.

### 2.1.2 IT로 옮긴 구조

비유만 머리에 두면 나중에 실제 용어와 헷갈릴 수 있어서, 오픈이는 이 주방 그림을 IT 구조로 한 번 그려보기로 했습니다.

![](../assets/CH02/chap02-container.png)

*그림 2-4 컨테이너 가상화의 전체 구조*

맨 아래에 **하드웨어**, 그 위에 **호스트 OS**, 그 위에 **Docker 엔진**이 올라갑니다. 엔진 위에 컨테이너들이 나란히 떠 있고, 각 컨테이너 안에는 애플리케이션(App)과 라이브러리(Library)가 같이 들어 있습니다. 주방 비유의 공용 설비가 호스트 OS, 각자의 조리대가 컨테이너입니다.

과제 쪽으로 끌어다 놓으면 그림이 분명해졌습니다. 오픈이의 앱과 앱이 쓰는 라이브러리가 한 컨테이너 안에 들어가고, 그 컨테이너가 호스트 OS 위에 얹히는 식입니다. 다른 컨테이너는 다른 앱과 다른 라이브러리를 가져도 충돌하지 않습니다. 내 앱이 런타임을 업그레이드해도 옆 컨테이너는 영향을 받지 않는다는 얘기입니다.

> **참고: 컨테이너의 격리**
> 컨테이너마다 **파일시스템**(`/bin`, `/lib` 같은 폴더), **네트워크**(IP와 포트), **프로세스 공간**(PID 번호)이 따로 존재합니다. 한 컨테이너가 라이브러리를 업그레이드해도 다른 컨테이너는 영향이 없고, 두 컨테이너가 같은 포트 번호를 써도 충돌하지 않습니다. 이 세 가지 격리는 리눅스 커널이 뒤에서 처리합니다.

### 2.1.3 docker run이 하는 일

컨테이너를 알았으니, 이 컨테이너가 실제로 어떻게 **만들어지는지** 한 번 짚어두고 가는 편이 나을 것 같았습니다. 뒤에서 `docker run` 명령을 손으로 치게 될 텐데, 그 한 줄이 내부에서 무슨 일을 하는지 알고 치는 것과 모르고 치는 건 다를 테니까요.

![](../assets/CH02/chap01-9.png)

*그림 2-5 docker run 명령이 커널의 격리 기능을 타고 컨테이너가 되는 흐름*

1. 사용자가 `docker run <컨테이너명>`명령어를 친다.
2. **Docker 엔진**이 이 명령을 받고, 로컬에 있는 이미지를 읽는다(없으면 Docker Hub에서 내려받는다).
3. Docker 엔진이 **커널**에게 "이 이미지 사양으로 격리된 프로세스 하나 만들어 줘"라고 요청한다.
4. 커널이 격리 공간(파일시스템/네트워크/프로세스)을 만들고, 그 안에서 새 프로세스를 띄운다.
5. 이 격리된 프로세스가 곧 **컨테이너**다.

컨테이너를 실제로 만드는 건 Docker 엔진이 아니라 커널입니다. 커널이 건물 관리인이라면, Docker 엔진은 프런트 직원에 가깝습니다. 입주자(프로세스)가 들어오면 프런트가 관리인에게 방 배정을 요청합니다. 실제로 방을 비우고 열쇠를 건네는 일은 관리인이 합니다.

> **참고: 커널(Kernel)**
> 운영체제(OS)의 핵심입니다. 프로세스 생성, 메모리 관리, 격리 공간 생성을 담당합니다. Docker의 모든 컨테이너는 호스트 PC의 **같은 리눅스 커널**을 공유합니다. 그 하나의 커널이 각 컨테이너에 격리 공간을 할당합니다.

여기까지 오면 오늘 목표에서 한 걸음 나간 셈이었습니다. 내 앱을 담을 컨테이너의 정체와 만들어지는 과정이 그림으로 잡혔습니다. 다음은 그 컨테이너를 실제로 **찍어내는 원본**이 뭔지였습니다.

## 2.2 이미지: 컨테이너의 설계도

컨테이너가 격리된 프로세스라는 건 알았는데, 그 프로세스가 무엇을 실행하는지를 결정하는 원본이 따로 있어야 했습니다. 원본이 없으면 오픈이의 앱도 어디에 어떻게 들어가야 할지가 없으니까요. 그 원본이 **이미지**입니다.

![](../assets/CH02/chap01-13.png)

*그림 2-6 하나의 이미지로 여러 컨테이너를 찍어내는 구조*

이미지는 붕어빵 틀에 가깝습니다. 틀(이미지) 하나로 붕어빵(컨테이너)을 여러 개 찍어낼 수 있고, 틀 자체는 변하지 않습니다. 같은 이미지에서 나온 컨테이너는 어디서 띄우든 똑같은 환경을 가집니다. 이 덕에 "내 PC와 서버 환경이 같다"가 가능합니다. 과제 쪽으로 옮기면 이렇게 됩니다. 오픈이가 로컬에서 만든 **이미지**를 서버에서 그대로 꺼내 실행하면, 서버의 런타임 버전이 뭐든 내 앱은 자기 이미지 안의 런타임으로 돕니다.

이미지는 **Docker Hub**라는 공용 저장소에서 내려받을 수도 있고, 직접 만들어 올릴 수도 있습니다. 내 앱을 담은 이미지를 Hub에 올려두고 서버에서 받아 쓰는 그림이 자연스럽게 그려졌습니다.

![](../assets/CH02/fig-1-bp-0.png)

*그림 2-7 Docker의 전체 흐름*

전체 흐름은 한 문단으로 정리됐습니다. Docker Hub에서 이미지를 내려받아 컨테이너를 실행하고, 컨테이너 안에서 작업한 뒤 수정한 내용을 새 이미지로 저장하고, 다시 Hub에 올리면 다른 사람이 가져다 씁니다. **이미지 받고, 컨테이너 만들고, 수정하고, 이미지로 저장하는** 순환. 이게 Docker가 돌아가는 모양이고, 오픈이가 오늘 저녁에 따라갈 여정 전체의 틀이기도 합니다.

> **참고: 이미지 레이어**
> 이미지 안에는 OS 기본 라이브러리, 런타임, 애플리케이션, 설정 파일이 **레이어**로 겹쳐 쌓여 있습니다. 아래 레이어는 재사용되고 캐싱되기 때문에, 앱만 바뀌면 맨 위 레이어만 새로 받으면 됩니다. 이미지 전체를 매번 통째로 내려받지 않아도 되는 이유입니다. 커널은 호스트와 공유하므로 이미지에 들어가지 않습니다. 이게 이미지가 가벼운 이유입니다.

개념은 여기까지 잡고 도구를 깔 차례였습니다.

## 2.3 Docker 설치

Docker Desktop 설치는 공식 사이트(https://www.docker.com/products/docker-desktop/)에서 OS에 맞는 설치 파일을 받아 기본 옵션대로 설치하면 끝입니다. 오픈이는 설치가 끝난 뒤 터미널에 버전을 찍어봤습니다.

```bash
docker version   # Docker 버전 확인
```

Client와 Server 정보가 둘 다 뜨면 설치가 완료된 겁니다. 오픈이 화면에도 두 블록이 나란히 찍혔습니다. 도구 준비는 끝났습니다.

> **참고: Windows와 macOS의 내부 리눅스**
> Docker는 리눅스 커널의 기능을 쓰기 때문에, 리눅스 커널이 없는 Windows와 macOS에서는 Docker Desktop이 내부에 리눅스 환경을 한 겹 더 띄웁니다. Windows는 **WSL2**(Windows Subsystem for Linux 2)를, macOS는 경량 Linux VM을 씁니다. 사용자 입장에서는 신경 쓸 일이 없지만, 뒤에 나올 네트워크 얘기에서 이 차이가 중요해집니다.

## 2.4 Docker CLI 첫 실습

오픈이의 앱을 이미지로 만들어 보는 건 챕터 후반 과제이고, 그 전에 기본 명령을 몇 개 손에 익혀두는 편이 나았습니다. 남이 만든 이미지부터 받아 띄워보며 `docker pull`/`run`/`ps`/`stop`/`rm`의 감을 잡는 단계였습니다. 연습 대상으로는 입문 자료에 공통으로 등장하는 **nginx**가 적당해 보였습니다.

### 2.4.1 docker pull: 이미지 가져오기

```bash
docker pull nginx   # nginx 이미지 다운로드
```

![](../assets/CH02/chap01-15.png)

*그림 2-8 nginx 이미지 다운로드 결과*

진행 바가 줄줄이 올라가더니 몇 초 뒤 프롬프트가 돌아왔습니다. 이미지가 로컬 저장소에 내려앉은 겁니다. 아직 실행된 건 아니고, 설계도만 들어온 상태였습니다.

### 2.4.2 docker run: 컨테이너 띄우기

받아둔 이미지로 컨테이너를 한 번 띄워 봤습니다.

```bash
docker run nginx   # nginx 컨테이너 실행
```

![](../assets/CH02/chap01-9.png)

*그림 2-9 nginx 컨테이너 실행*

엔터를 누르자마자 터미널이 뚝 멈췄습니다. 커서가 깜빡이지도 않고, 아무 입력도 받지 않았습니다.

*'뭐가 잘못된 건가.'*

nginx가 **포그라운드** 상태로 터미널을 붙잡고 있었던 겁니다. 전화 통화가 연결된 동안 다른 전화를 받지 못하는 것과 같은 상황입니다. `CTRL + C`로 빠져나왔습니다.

실제 앱은 서버에서 돌아가는 동안 터미널을 계속 잡고 있으면 곤란하니, **백그라운드로 돌리는 옵션**을 알아둬야겠다는 생각이 자연스럽게 붙었습니다.

> **참고: 포그라운드(Foreground)**
> 프로세스가 터미널을 독점하는 상태입니다. 프로세스가 끝나거나 강제 종료될 때까지 그 터미널로 다른 명령을 내릴 수 없습니다.

### 2.4.3 docker run -d: 백그라운드로 띄우기

컨테이너는 띄우고 터미널도 같이 쓰려면 `-d` 옵션이 필요했습니다.

```bash
docker run -d nginx   # -d: detached, 백그라운드 실행
```

![](../assets/CH02/chap01-17.png)

*그림 2-10 백그라운드 실행 결과*

## 2.5 컨테이너의 통신

내 앱이 컨테이너에서 돌아도 결국 브라우저로 접속해야 의미가 있었습니다. 격리된 프로세스라는 개념은 익혔지만, 이 격리된 것이 바깥과 어떻게 연결되는지를 모르면 nginx 화면을 브라우저로 볼 수조차 없을 테니까요. 뒤의 Kubernetes 네트워크에서 이름만 바뀌어 다시 나올 부품들이라고도 해서, 여기서 큰 그림을 잡아두기로 했습니다.

### 2.5.1 네트워크 한눈에 보기

오픈이는 머릿속에 **푸드코트** 그림을 하나 그려봤습니다. 여러 식당(컨테이너)이 같은 홀에 모여 있고, 가운데에 가상 스위치(**docker0**)가 놓여 있어 각 식당이 거기 연결됩니다. 홀 입구에는 외부 손님을 받는 **키오스크**(포트포워딩)가 있고, 홀 한쪽에는 식당 이름을 가리키는 **안내 지도**(Docker DNS)가 걸려 있는 그림이었습니다.

세 가지 통신 경로가 이 한 장면에 전부 들어 있습니다.

| 통신 방향 | 푸드코트에서 | Docker에서 |
|----------|--------------|-----------|
| 식당 ↔ 식당 | 홀의 가상 스위치를 거쳐 연결 | docker0 + veth |
| 외부 손님 → 식당 | 입구 키오스크가 주문을 넘김 | 포트포워딩 |
| 이름으로 식당 찾기 | 홀의 안내 지도가 이름→위치로 | Docker DNS |

하나씩 열어봅니다.

### 2.5.2 컨테이너끼리 — 푸드코트 주방과 창구

![](../assets/CH02/ch2-step1-namespace1.png)

*그림 2-12 푸드코트의 식당, 픽업 창구, 홀이 컨테이너 네트워크의 비유*

A, B, C 식당이 벽으로 완전히 분리되어 있습니다. 서로의 내부는 볼 수 없습니다. 대신 각 식당마다 홀 쪽으로 **픽업 창구**가 하나씩 있어서, 음식이 그 창구를 통해 홀로 나갑니다. 주방은 독립인데 창구 덕분에 외부와 연결됩니다.

Docker의 네트워크도 이 구조입니다. 컨테이너가 뜰 때 가장 먼저 **독립된 네트워크 공간(주방)** 이 생깁니다. 그다음 그 공간과 호스트를 잇는 **가상 케이블(창구)** 이 생성됩니다. 케이블의 반대쪽 끝은 호스트의 **가상 스위치(홀)** 에 꽂힙니다. 이 구조 덕분에 격리된 컨테이너가 외부와 데이터를 주고받을 수 있습니다.

![](../assets/CH02/diagram/ch2-step1-namespace.png)

*그림 2-13 컨테이너 네트워크 격리 구조*

| 구성 요소 | 푸드코트 비유 |
|----------|---------------|
| **Network Namespace** | 식당 주방 (컨테이너별 독립 네트워크 공간) |
| **veth pair** | 픽업 창구 (컨테이너와 호스트를 잇는 가상 케이블) |
| **docker0** | 푸드코트 홀 (모든 컨테이너 연결이 모이는 가상 스위치) |

> **참고: Network Namespace / veth pair / docker0**
> - **Network Namespace**: 리눅스 커널이 제공하는 '독립된 네트워크 공간'입니다. 컨테이너가 뜰 때 하나 생성되며, 그 안은 자기만의 IP와 포트 공간을 가집니다.
> - **veth pair**: 'virtual ethernet(가상 이더넷)'의 약자입니다. 한쪽 끝은 컨테이너 안(`eth0`)에, 반대쪽 끝은 호스트(`veth...`)에 꽂히는 가상 케이블입니다.
> - **docker0**: Docker가 호스트에 자동으로 만드는 가상 스위치(브릿지)입니다. 모든 컨테이너의 veth가 여기로 모입니다.

식당끼리 이야기는 끝났지만, 바깥 손님이 이 홀 안의 식당에 어떻게 닿는지는 아직 그림이 없었습니다. 오픈이의 앱도 결국 브라우저라는 바깥 손님을 받아야 하니, 그쪽 문부터 열어둘 필요가 있었습니다.

### 2.5.3 외부에서 들어오기 — 입구 키오스크

방금 본 docker0는 컨테이너끼리 연결되는 **내부 홀**입니다. 그런데 외부 사용자(브라우저)가 이 홀 안의 특정 컨테이너에 바로 닿는 건 쉽지 않습니다. 이유가 두 가지입니다.

- 외부 사용자는 컨테이너의 내부 IP를 모릅니다. 컨테이너가 뜰 때마다 바뀌기도 하고, 바깥으로 공개된 주소도 아닙니다.
- **Linux에서는** docker0가 호스트 PC 안에 바로 존재하니까 호스트에서 컨테이너 IP로 접근이 됩니다. **하지만 Windows와 macOS에서는** docker0가 2.3에서 본 내부 리눅스 VM 안에 있어서, 호스트에서 컨테이너 IP로 바로 접근이 되지 않습니다. 오픈이처럼 Windows나 macOS에서 실습 중이라면 이 경로 자체가 막혀 있는 셈입니다.

외부에서 컨테이너로 들어오는 **별도의 통로**가 필요했습니다.

![](../assets/CH02/ch2-step2-dnat1.png)

*그림 2-14 입구 키오스크가 손님의 주문을 각 식당으로 넘겨주는 비유*

푸드코트에 비유하면, 외부 손님은 주방에 직접 들어가지 못합니다. 대신 입구의 **키오스크**에 주문을 넣으면, 키오스크가 해당 식당으로 그 주문을 넘겨줍니다. 짜장면은 A식당으로, 돈까스는 B식당으로 가는 식입니다.

Docker에서 이 키오스크 역할이 **포트포워딩**입니다. 호스트 PC의 특정 포트를 바깥으로 공개해 두고, 그 포트로 들어온 요청을 미리 지정해둔 컨테이너의 포트로 넘기는 방식입니다. `docker run -p <호스트 포트>:<컨테이너 포트>` 형태로 지정합니다.

![](../assets/CH02/diagram/ch2-step2-dnat.png)

*그림 2-15 호스트 포트가 컨테이너 포트로 연결되는 구조*

브라우저에서 호스트 포트로 요청을 보내면, Docker가 그 요청을 가로채 도착지를 컨테이너 내부 주소로 바꿉니다. 사용자 입장에서는 이 변환이 보이지 않고, "localhost에 접속했더니 컨테이너 안의 페이지가 나왔네" 정도로만 경험됩니다. 오픈이의 앱도 내부 기본 포트에서 돌 텐데, 그 포트를 `-p`로 바깥에 열어주면 브라우저에서 접속할 수 있다는 얘기였습니다.

> **참고: 포트포워딩 / DNAT / iptables**
> - **포트포워딩**: "외부 포트 → 컨테이너 포트"로 요청을 넘기는 규칙입니다. `docker run -p 8080:80` 식으로 지정합니다.
> - **DNAT(Destination NAT)**: 요청의 도착지 주소를 바꿔치기하는 기술입니다. 포트포워딩이 "규칙"이라면 DNAT은 "그 규칙을 실제로 집행하는 손"에 해당합니다.
> - **iptables**: 리눅스 커널에 네트워크 규칙을 심는 도구입니다. Docker는 포트포워딩을 iptables 규칙으로 구현합니다. 이 이름은 뒤에 Kubernetes의 `kube-proxy`에서 다시 나옵니다.

### 2.5.4 이름으로 찾기 — 홀의 안내 지도

![](../assets/CH02/ch2-step3-dns1.png)

*그림 2-16 안내 지도가 이름을 위치로 바꿔주는 비유*

손님이 "B식당"을 찾고 있습니다. 이름은 아는데 몇 구역인지는 모릅니다. 홀에 걸린 **안내 지도**가 "B식당 = 2구역"을 알려줍니다. 지도만 보면 됩니다.

Docker 내부에도 이 안내 지도가 있습니다. 이름이 **Docker DNS**입니다. 새 컨테이너가 뜨면 이름과 IP가 자동으로 DNS에 등록되고, 다른 컨테이너가 그 이름으로 요청을 보내면 Docker DNS가 이름을 IP로 변경해 줍니다. 매번 바뀌는 IP 주소를 외울 필요가 없습니다. 오픈이의 앱이 나중에 DB 컨테이너에 연결해야 할 때 이 구조가 그대로 쓰일 터였습니다.

![](../assets/CH02/diagram/ch2-step3-dns.png)

*그림 2-17 Docker DNS가 이름을 IP로 변환*

다만 여기에 함정이 하나 있습니다. 이 이름 기반 통신은 **기본 bridge 네트워크에서는 작동하지 않습니다**. **사용자 정의 네트워크** 안에서만 자동 DNS가 켜집니다. 이 차이는 다음 챕터에서 `docker network create`로 직접 만들어보며 확인하게 됩니다.

> **참고: DNS / 사용자 정의 네트워크**
> - **DNS(Domain Name System)**: 이름(`google.com`)을 IP로 바꿔주는 시스템입니다. 브라우저 도메인 해석과 같은 원리입니다.
> - **사용자 정의 네트워크**: `docker network create`로 사용자가 직접 만드는 bridge 네트워크입니다. 기본 bridge와 달리 Docker 내장 DNS가 자동 활성화되어 컨테이너 이름으로 서로를 부를 수 있습니다.

### 2.5.5 지금까지의 지도

푸드코트 한 장면에 세 가지 통신 방식이 다 들어 있습니다. 표로 정리하면 이렇습니다.

| 통신 방향 | 비유 | 기술 이름 |
|----------|------|---------|
| 컨테이너 ↔ 컨테이너 | 주방 + 창구 + 홀 | Network Namespace + veth + docker0 |
| 외부 ↔ 컨테이너 | 입구 키오스크 | 포트포워딩 (iptables DNAT) |
| 이름으로 컨테이너 찾기 | 홀의 안내 지도 | Docker DNS (사용자 정의 네트워크에서) |

이 표는 뒤의 Kubernetes 네트워킹 챕터에서 **K8s 버전**으로 다시 등장합니다. docker0 → Pod 네트워크, iptables DNAT → `kube-proxy`, Docker DNS → `CoreDNS`로 이름만 바뀌고 큰 그림은 그대로입니다.

오늘의 목표에 비추어 정리하면 이 정도 진전이었습니다. 내 앱의 포트를 `-p`로 열어주면 브라우저로 접속이 된다는 그림이 생겼고, DB 같은 동료 컨테이너와의 통신은 사용자 정의 네트워크를 쓸 때 이름 기반으로 가능하다는 사실도 미리 알아뒀습니다. 이제 컨테이너 **안쪽**을 한 번 들여다볼 차례였습니다. 내 앱의 JAR를 어디에 어떻게 넣을지 감을 잡으려면 안이 어떻게 생겼는지부터 봐야 했습니다.

## 2.6 컨테이너 안의 리눅스

오픈이는 Ubuntu 컨테이너를 포트까지 열어둔 채로 띄우고 안에 들어가 봤습니다.

```bash
# -it: 표준입력 유지(-i) + TTY 연결(-t), -p 80:80: 포트포워딩
docker run -it -p 80:80 ubuntu
```

![](../assets/CH02/chap01-43.png)

*그림 2-18 Ubuntu 컨테이너 안으로 들어간 상태*

프롬프트가 `root@...#` 모양으로 바뀌었습니다. 방금까지 호스트의 터미널이었는데 이제 컨테이너 안의 쉘이었습니다. `ls`로 디렉터리를 찍어보니 익숙한 리눅스 폴더들이 쭉 나왔습니다. 호스트와는 완전히 다른 공간에 서 있는 느낌이었습니다.

> **참고: 포트(Port)**
> 서버에서 통신을 기다리는 번호 붙은 창구입니다. 포트가 다르면 같은 서버 안에서 여러 애플리케이션이 독립적으로 요청을 받습니다.

### 2.6.1 자주 쓰는 리눅스 명령어

내부를 들여다볼 때 쓸 명령을 한 표로 모아뒀습니다. 뒤에 내 앱 이미지를 만들 때 이 명령들이 손에 익어 있어야 안에서 파일을 만들고 옮기는 작업이 매끄러워집니다.

| 용도 | 명령 | 설명 |
|------|------|------|
| 위치 확인 | `pwd` | 현재 위치 |
| 이동 | `cd <경로>` | 폴더 이동 |
| 목록 | `ls`, `ls -la` | 파일/폴더 목록 (숨김 포함) |
| 폴더 생성 | `mkdir <이름>` | 폴더 만들기 |
| 파일 생성 | `touch <이름>` | 빈 파일 |
| 복사 | `cp <원본> <사본>` | 파일 복사 |
| 이동/이름변경 | `mv <원본> <대상>` | 이동 또는 rename |
| 삭제 | `rm <파일>`, `rm -r <폴더>` | 삭제 |
| 패키지 설치 | `apt update && apt install -y <이름>` | 패키지 설치 |
| 프로세스 | `ps -ef`, `kill <PID>` | 실행 중 프로세스 / 종료 |
| 검색 | `find <경로> -name <이름>` | 파일 검색 |
| 로그 | `tail -n <수> <파일>` | 마지막 N줄 |

`/bin`처럼 `/`로 시작하는 **절대 경로**와 `bin`처럼 현재 위치 기준인 **상대 경로**를 혼동하지 않는 게 처음에 중요합니다. `/home/ubuntu`에서 `/bin`으로 가려면 `cd /bin`이라고 써야지 `cd bin`으로는 안 됩니다.

### 2.6.2 실습: 컨테이너 안에 nginx 깔아보기

컨테이너는 최소한의 프로그램만 들어 있다 보니, 필요한 도구는 `apt`로 직접 깔아야 했습니다. 이 감도 한 번 손으로 잡아두면 내 앱 이미지에 필요한 도구를 넣을 때 망설임이 줄 것 같아서 가볍게 실습해 봤습니다.

```bash
apt update           # 패키지 목록 갱신
apt install -y nginx # nginx 설치
nginx                # nginx 실행
```

![](../assets/CH02/chap01-46.png)

*그림 2-19 nginx 설치와 실행 결과*

설치 로그가 한참 흐른 뒤 nginx가 떴습니다. 브라우저에서 `localhost:80`에 접속했더니 익숙한 nginx 환영 페이지가 응답했습니다. 앞서 `-p 80:80`으로 포트를 열어둔 덕분이었습니다.

![](../assets/CH02/chap01-48.png)

*그림 2-20 nginx 환영 페이지 응답*

### 2.6.3 vim으로 파일 편집

설정 파일을 만질 때 쓰는 편집기가 **vim**입니다.

| 단계 | 동작 | 키 |
|------|------|----|
| 1 | 파일 열기/생성 | `vim <파일명>` |
| 2 | 입력 모드 | `i` |
| 3 | 일반 모드로 | `ESC` |
| 4 | 저장 후 종료 | `:wq` |

컨테이너에서 빠져나올 때는 `exit`입니다. 그런데 `exit`를 치고 `docker ps`를 쳐보니, 방금까지 돌던 컨테이너가 목록에서 사라져 있었습니다. 안쪽 구조는 익혔는데 **언제 살아 있고 언제 죽는지**는 아직 감이 없다는 뜻이었습니다. 내 앱 컨테이너가 뜻밖에 꺼지면 곤란하니, 이 부분을 다음에 짚고 가기로 했습니다.

## 2.7 컨테이너 생명주기

### 2.7.1 exit vs detach

오픈이는 같은 상황을 한 번 더 만들어 차분히 비교했습니다. 이름을 붙여 띄운 뒤 `exit`로 나와봤습니다.

```bash
docker run -it --name dead ubuntu   # 실행 후 exit로 나옴
exit
docker ps                            # 목록에 없음 (종료됨)
```

![](../assets/CH02/chap01-29.png)

*그림 2-21 exit로 빠져나오면 컨테이너가 종료*

예상대로 dead는 목록에 없었습니다. 같은 방식으로 한 번 더 띄우되 이번엔 `CTRL+P → CTRL+Q`로 빠져나왔습니다.

```bash
docker run -it --name alive ubuntu   # CTRL+P → CTRL+Q로 나옴
docker ps                            # 살아 있음
```

![](../assets/CH02/chap01-31.png)

*그림 2-22 CTRL+P → CTRL+Q로는 컨테이너가 유지*

이번에는 alive가 살아 있었습니다. 같은 `docker run`이었는데 결과가 갈린 이유는 나오는 방식이 하는 일이 달라서였습니다.

| 방식 | 하는 일 | 결과 |
|------|---------|------|
| `exit` | 컨테이너의 메인 프로세스(bash)를 **종료** | 컨테이너가 종료 |
| `CTRL+P → CTRL+Q` | 터미널과 컨테이너의 **연결만 끊음** | 컨테이너는 살아 있음 |

결론은 한 줄로 수렴됐습니다. **메인 프로세스가 살아 있어야 컨테이너가 살아 있다.** 내 앱 컨테이너라면 메인 프로세스가 내 앱 자체일 테니 앱이 끝나지 않는 한 컨테이너도 살아 있을 거라는 감이 잡혔습니다.

### 2.7.2 -dit와 CMD

이 규칙을 잡고 나니 앞에서 본 장면 하나가 말이 됐습니다. nginx는 `-d`만으로도 백그라운드에서 잘 돌았는데, ubuntu는 `-d`만으로는 바로 꺼졌습니다.

```bash
docker run -d nginx   # 살아 있음
docker run -d ubuntu   # 바로 꺼짐
```

![](../assets/CH02/chap01-33.png)

*그림 2-23 ubuntu는 -d만으로는 띄우자마자 종료*

두 이미지의 메인 프로세스가 다르기 때문입니다.

| 이미지 | 메인 프로세스 | `-d`만으로 유지? |
|--------|------------|-----------------|
| nginx | nginx 데몬 (항상 요청을 기다림) | 유지됨 |
| ubuntu | `bash` (사용자 입력을 기다림) | 터미널 없으면 즉시 종료 |

bash는 사용자 입력을 받기 위한 셸이라 터미널이 없으면 "받을 사람이 없네" 하고 스스로 종료됩니다. ubuntu를 백그라운드로 살리려면 `-dit`(백그라운드(`-d`) + 표준입력 유지(`-i`) + TTY 연결(`-t`))을 쓰거나 메인 프로세스를 다른 걸로 바꿔야 합니다.

```bash
docker run -d ubuntu sleep 1000   # bash 대신 sleep을 메인으로
docker ps                         # 1000초 동안 살아 있음
```

![](../assets/CH02/chap01-35.png)

*그림 2-24 CMD를 sleep으로 바꾸면 유지*

sleep은 터미널 없이도 1000초 동안 할 일이 있는 프로세스입니다. 이렇게 기본 실행 명령을 덮어쓰는 것이 **CMD**입니다. 내 앱 이미지를 만들 때 CMD로 "앱 실행 명령"을 박아두면, 컨테이너가 뜰 때 자동으로 앱이 돈다는 그림이 그려졌습니다.

> **참고: CMD**
> 컨테이너가 시작될 때 실행되는 **기본 프로세스**를 정의하는 설정입니다. `docker run <이미지> <CMD>` 형태로 직접 주면 이미지 기본값을 덮어씁니다.

### 2.7.3 attach vs exec

백그라운드로 떠 있는 컨테이너에 다시 들어가는 방식은 두 가지입니다.

![](../assets/CH02/chap01-39.png)

*그림 2-25 attach는 메인 프로세스에 직접 붙고, exec는 새 프로세스를 만들어 들어간다*

| 방식 | 동작 | 위험성 |
|------|------|--------|
| `docker attach <ID>` | **메인 프로세스(PID 1)** 에 직접 연결 | 잘못 빠져나오면 메인이 종료되고 컨테이너도 꺼짐 |
| `docker exec -it <ID> bash` | **새 프로세스**를 만들어 접근 | 안전. 빠져나와도 메인에 영향 없음 |

실제로 쓰는 건 거의 항상 `exec`입니다. 돌고 있는 내 앱 컨테이너에 들어가 로그를 확인하거나 파일을 만져야 할 때 `attach`로 들어갔다가 실수로 메인을 꺼버리면 서비스가 내려가니까요.

```bash
docker exec -it <컨테이너ID> bash   # 새 bash 세션으로 진입
```

여기까지 오면 컨테이너의 **살고 죽는 규칙**이 머리에 정리됩니다. 마지막 과제로, 이제 내 앱을 실제로 이미지에 담는 시도를 할 차례였습니다.

## 2.8 내 이미지 만들기 — 오늘의 과제로 돌아오기

챕터 도입에서 오픈이가 잡은 목표를 다시 꺼내면 "낮에 실패한 그 빌드 결과물을 Docker로 담아 실행해 보기"였습니다. 여기까지 쌓아둔 부품으로 이 일을 맛보기 수준에서 한 번 해볼 차례였습니다. 정석은 다음 챕터의 Dockerfile인데, 지금은 수동 방식인 `docker commit`으로도 같은 그림이 가능합니다. 먼저 Tomcat으로 commit/push 흐름을 익힌 뒤, 같은 방식으로 내 앱 이미지를 만들어 보는 순서였습니다.

### 2.8.1 Tomcat으로 commit 감 잡기

```bash
docker run -d -p 8080:8080 tomcat
```

브라우저에서 `localhost:8080`으로 접속했더니 **404**가 떴습니다.

![](../assets/CH02/chap01-58.png)

*그림 2-26 Tomcat 404 에러 화면*

공식 Tomcat 이미지의 `webapps/` 폴더가 비어 있어서입니다. 여기에 `ROOT/index.html`을 하나 넣은 뒤 그 상태를 이미지로 굳히면, 다음에는 바로 쓸 수 있는 "페이지가 들어 있는 Tomcat 이미지"가 되는 셈입니다.

오픈이는 `exec`로 컨테이너 안에 들어가 필요한 경로를 만들고 index.html을 작성했습니다.

```bash
docker exec -it <컨테이너ID> bash
cd /usr/local/tomcat/webapps
mkdir ROOT && cd ROOT
apt update && apt install -y vim
vim index.html    # 간단한 HTML 작성 후 저장
```

저장을 마치고 브라우저에서 다시 접속했더니 방금 만든 페이지가 응답했습니다.

![](../assets/CH02/chap01-65.png)

*그림 2-27 index.html 응답 확인*

이 상태를 이미지로 굳혔습니다.

```bash
docker commit <컨테이너ID> <본인-dockerhub-id>/tomcat
```

![](../assets/CH02/chap01-67.png)

*그림 2-28 이미지 커밋 완료 화면*

로컬 이미지 목록을 보니 새 이미지가 들어와 있었습니다. 새 컨테이너를 띄우면 webapps/ROOT/index.html이 처음부터 들어 있는 Tomcat이 뜨는 셈입니다. **내가 세팅한 상태를 그대로 복제할 수 있는 이미지가 생겼다**는 경험이 이 절의 핵심이었습니다.

### 2.8.2 Docker Hub에 올려 어디서든 꺼내 쓰기

Hub에 올려두면 다른 PC나 서버에서 `docker pull`로 받아 쓸 수 있습니다. 과제의 마지막 조각이 이것이었습니다. 집 노트북에서 구운 이미지를 회사 서버에서 바로 받을 수 있는지가 오늘의 진짜 확인 지점이었습니다.

```bash
docker login
docker push <본인-dockerhub-id>/tomcat
```

![](../assets/CH02/chap01-70.png)

*그림 2-29 Docker Hub 저장소 확인*

Hub의 Repositories 탭에 이미지가 올라와 있었습니다.

오픈이는 방금 거친 흐름을 되짚어 보았습니다. 베이스 이미지에서 컨테이너를 하나 띄워 필요한 설정을 얹고, 그 상태를 `commit`으로 이미지로 굳혀 Hub에 `push`한 흐름이었습니다. **어제 서버에서 안 돌던 런타임·라이브러리·경로 문제가 이 컨테이너 안에서는 내가 세팅한 대로 고정된다**는 감만큼은 분명히 들어왔습니다. "내 PC에서는 되는데 서버에서는 안 된다"의 이유가 여기서 사라집니다.

실제 운영에서 이 반복을 버티려면 **Dockerfile** 자동화가 필요하다는 얘기가 이 절 도입부에서 잠깐 지나갔습니다. 수동으로 감을 잡은 오늘의 진전을 다음 챕터에서 그 자동화로 이어갑니다.

## 2.9 데이터 보존: 마운트

마지막으로 남은 과제가 하나 있었습니다. 내 앱이 DB를 쓰는 경우, DB도 컨테이너로 띄우면 어떻게 되는가. 컨테이너는 지워지면 안에 있던 파일도 그대로 같이 사라집니다. 지금까지 실습해 본 텍스트 몇 줄 수준이라면 괜찮았지만, DB 컨테이너를 그렇게 지우면 **진짜 데이터가 날아가는** 사고가 됩니다.

데이터를 컨테이너 바깥으로 빼두는 방법이 **마운트**입니다.

> **참고: 마운트(Mount)**
> 컨테이너 내부의 폴더를 외부 저장소에 **연결**하는 기능입니다. USB를 PC에 꽂으면 PC의 특정 폴더에서 USB 내용이 보이는 것과 같은 원리입니다. 양쪽이 같은 곳을 가리키므로 한쪽의 변경이 반대쪽에도 보이고, 컨테이너를 삭제해도 외부 데이터는 남습니다.

Docker는 두 가지 마운트를 지원합니다.

### 2.9.1 바인드 마운트: 호스트 폴더와 직접 연결

![](../assets/CH02/bind-mount.png)

*그림 2-30 호스트 PC의 폴더와 컨테이너 내부 폴더를 직접 연결*

오픈이는 호스트에 폴더를 하나 만들고, 컨테이너 안의 경로와 직접 연결했습니다.

```bash
# macOS / Linux
mkdir -p ~/app/bind
docker run -it --mount type=bind,src=$HOME/app/bind,dst=/app/bind ubuntu
```

Windows(Git Bash/WSL)에서는 `/c/app/bind` 같은 경로를 `src`로 써주면 됩니다.

컨테이너 안에서 `/app/bind/a.txt`를 `touch`로 만들고, 호스트의 `~/app/bind`를 봤습니다.

![](../assets/CH02/chap01-77.png)

*그림 2-31 호스트 PC에서 같은 파일이 보이는 것을 확인*

같은 파일이 양쪽에 있었습니다. 개발 중 소스 코드를 호스트에서 고치고 컨테이너에서 바로 반영해야 하는 상황에 딱 맞는 방식입니다.

### 2.9.2 볼륨 마운트: Docker가 관리하는 저장소

![](../assets/CH02/volume-mount.png)

*그림 2-32 Docker 엔진이 관리하는 내부 저장 공간에 데이터가 저장*

두 번째는 Docker 엔진이 관리하는 별도 저장 공간을 붙이는 방식입니다. 이름만 지정하면 실제 저장 위치는 Docker가 알아서 관리합니다.

```bash
docker run -it --mount type=volume,src=metacoding-volume,dst=/app/volume ubuntu
touch /app/volume/b.txt
exit
docker volume ls
```

컨테이너를 `exit`로 내린 뒤 볼륨 목록을 찍어보니, 컨테이너는 사라졌는데 볼륨은 그대로 남아 있었습니다.

![](../assets/CH02/chap01-84.png)

*그림 2-33 컨테이너가 사라져도 볼륨은 유지*

새 컨테이너에 같은 볼륨을 다시 붙이면 앞서 만든 `b.txt`가 그대로 있습니다. DB 컨테이너의 실제 데이터를 보존할 때 이 성질이 결정적으로 쓰입니다.

### 2.9.3 둘의 선택 기준

| 구분 | 바인드 마운트 | 볼륨 마운트 |
|------|------------|-----------|
| 저장 위치 | 호스트 PC 지정 폴더 | Docker 엔진 내부 |
| 관리 주체 | 사용자 직접 | Docker 엔진 |
| 주 사용처 | 개발 중 소스 코드 연결 | DB 데이터 보존 |
| 기존 파일 처리 | 호스트 폴더 내용으로 덮어씀 | 컨테이너 내부 파일을 볼륨으로 복사 |

내 앱 개발 중에는 **바인드**, DB 데이터 같은 영구 보존 대상은 **볼륨**입니다.

## 이것만은 기억하자

- **컨테이너는 격리된 프로세스다.** 파일시스템·네트워크·프로세스 공간만 따로 쪼개고, 커널은 호스트와 공유한다. VM보다 가볍고 빨리 뜬다.
- **이미지는 설계도, 컨테이너는 찍혀 나온 결과다.** 붕어빵 틀 하나로 여러 붕어빵을 찍듯, 이미지 하나로 여러 컨테이너를 찍는다.
- **네트워크의 큰 그림은 푸드코트다.** 식당(Namespace) + 창구(veth) + 홀(docker0)이 컨테이너 간 통신, 입구 키오스크가 포트포워딩, 홀의 안내 지도가 Docker DNS.
- **메인 프로세스가 살아 있어야 컨테이너가 살아 있다.** 내 앱 이미지의 CMD에 앱 실행 명령을 박아두는 이유가 여기에 있다.
- **데이터 보존은 마운트로.** 개발 중 파일은 바인드, 영구 데이터는 볼륨.

오늘 저녁의 목표였던 "내 앱을 Docker로 담아 실행해 보기"는 수동 방식으로 감까지 잡았습니다. 내 PC와 서버 환경이 달라서 안 되던 그 문제의 정체와, 이 도구가 어떤 방식으로 그걸 풀어주는지가 머리에 들어왔습니다. 오픈이는 오늘 밤 여기서 멈추기로 했습니다. 내 앱을 실제로 담는 일은 수동 commit으로 한 번 더 반복하기보다, 다음에 Dockerfile로 제대로 자동화해서 붙잡는 편이 낫겠다는 판단이 섰습니다.

남은 것도 분명해졌습니다. 수동 commit은 반복 작업에 약하고, 앱이 늘어나면 `docker run` 여러 번을 손으로 치는 것도 지치는 일이 됩니다. 다음 챕터에서는 이 작업을 **Dockerfile로 자동화**하고, 여러 컨테이너를 **Docker Compose로 한 번에** 묶어 띄워봅니다. 2.5.4에서 예고한 **사용자 정의 네트워크**도 여기서 직접 만들어, 컨테이너끼리 이름으로 통신하는 모습을 눈으로 보게 됩니다.
