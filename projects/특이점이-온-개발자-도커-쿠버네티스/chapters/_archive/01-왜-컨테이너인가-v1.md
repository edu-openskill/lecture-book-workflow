### 1.1.2 컨테이너의 도입

이 비효율을 가장 가까이에서 체감한 사람이 있었습니다. 바로 트럭 운전사 말콤 맥린입니다.

항구에서 화물이 하나씩 옮겨지는 걸 반복적으로 지켜보던 그는 단순하지만 강력한 의문을 품습니다. **"왜 짐을 박스째로 통째로 옮기지 않는 걸까?"**

이 질문에서 컨테이너라는 개념이 탄생했습니다. 화물을 낱개로 다루는 대신, 표준 규격의 큰 상자 하나에 담아서 통째로 옮기자는 아이디어였습니다.

크레인으로 컨테이너를 그대로 배에 올리고, 도착지에서도 같은 방식으로 내리면 됩니다. 운송 속도와 효율은 이전과는 비교할 수 없을 정도로 올라갔습니다.

## 1.2 Docker : 컨테이너를 쉽게 쓰는 도구

### 1.2.1 가상화 : 하나의 컴퓨터를 여러 개로 사용하는 법

말콤 맥린은 "상자에 넣으면 어디서든 똑같이 옮길 수 있다"는 걸 증명했습니다. Docker도 같은 원리입니다. 그렇다면 Docker는 이 "상자"를 어떻게 만들까요? 그 출발점은 **가상화** 입니다.

> **가상화(Virtualization)** 는 하나의 물리 서버를 논리적으로 여러 대처럼 나누어 쓰는 기술입니다. 가상화를 도입하면 서버를 효율적으로 쓸 수 있고 서로 다른 환경을 안전하게 격리할 수 있습니다.

주방에 비유해 보겠습니다.

하나의 주방에 네 명의 요리사가 일하고 있습니다. 그런데 냉장고와 가스레인지가 각각 하나뿐이라 한 번에 한 명만 쓸 수 있는 구조입니다.

첫 번째 방식은 격리가 확실하지만 비용이 크고, 두 번째 방식은 자원을 아끼면서도 독립적인 작업 공간을 유지합니다. Docker의 컨테이너 가상화가 바로 두 번째 방식입니다.

### 1.2.2 컨테이너 가상화 : OS 없이 격리하다

> **컨테이너 가상화(Container Virtualization)** 는 하나의 운영 체제를 함께 쓰면서 애플리케이션이 실행되는 환경만 따로 분리하는 방식입니다.

하나의 서버에는 OS가 설치되어 있고, 그 핵심에는 **커널** 이 있습니다. 컨테이너 가상화는 이 커널을 여러 애플리케이션이 함께 공유하되 각 애플리케이션의 실행 공간만 독립적으로 분리하는 방식입니다. 분리된 실행 공간 하나하나가 바로 **컨테이너** 입니다.

컨테이너 안에는 앱 실행에 필요한 라이브러리, 설정, 파일시스템만 들어 있습니다. OS 전체를 복제하지 않고, 커널은 호스트의 것을 함께 쓰면서 실행 환경만 격리합니다.

컨테이너마다 파일시스템, 네트워크, 프로세스 공간이 분리되어 있어서 컨테이너 A에서 설치한 라이브러리가 컨테이너 B에 영향을 주지 않습니다.

컨테이너 가상화에서는 **컨테이너 엔진(Container Engine)** 이 컨테이너의 생성, 실행, 중지, 삭제를 맡습니다. Docker가 대표적인 컨테이너 엔진으로, 이미지 관리부터 컨테이너 실행까지 운영에 필요한 기능을 두루 갖추고 있습니다.

### 1.2.3 Docker : 이미지에서 컨테이너까지

Docker는 어떻게 컨테이너를 만들까요? 명령어 하나를 따라가며 흐름을 살펴보겠습니다.

지금은 명령어를 실행하지 않아도 됩니다. 동작 흐름만 이해하면 충분합니다.

**[참고]** nginx 컨테이너를 실행하는 명령어입니다.

```bash
docker run nginx   # nginx 컨테이너 실행
```

이 명령어를 실행하면 **Docker 엔진**이 요청을 받습니다. Docker 엔진이 직접 컨테이너를 만들지는 않습니다. OS **커널**에 "격리된 프로세스를 하나 만들어 줘"라고 요청할 뿐입니다.

커널은 건물의 관리사무소 같은 존재입니다. 전기, 수도, 엘리베이터를 관리사무소가 관리하듯 커널은 컴퓨터의 CPU, 메모리, 프로세스를 관리합니다. 프로세스를 만들고 격리하고 자원을 나눠주는 것은 오직 커널만 할 수 있습니다.

> **커널(Kernel)** 은 운영체제(OS)의 핵심 구성 요소로, 프로세스를 만들고 실행 순서를 정하며, 프로세스끼리 서로 간섭하지 않도록 메모리를 보호합니다.

Docker 엔진의 요청을 받은 커널은 격리된 환경에서 새 프로세스를 만듭니다. 이것이 컨테이너입니다.

### 1.2.4 Docker 이미지

컨테이너는 격리된 프로세스이지만 빈 깡통이 아닙니다. 무언가를 담아야 실행할 수 있습니다. 그 "무언가"가 바로 **이미지**입니다.

이미지는 컨테이너를 만들기 위한 패키지입니다. 안에는 OS, 애플리케이션, 설정 파일이 레이어(층)로 겹쳐 쌓인 구조입니다. 예를 들어 nginx 이미지 안에는 Linux + nginx + 기본 설정이 들어 있습니다. 레이어 구조의 자세한 동작 방식은 챕터 3 Dockerfile에서 다룹니다.

비유하면 이미지는 붕어빵 틀입니다. 틀(이미지) 하나로 붕어빵(컨테이너)을 여러 개 찍어낼 수 있습니다. 틀 자체는 변하지 않고 찍혀 나온 붕어빵은 각각 독립적입니다. 이미지 하나로 똑같은 컨테이너를 몇 개든 만들 수 있습니다.

## 1.3 Kubernetes : 컨테이너가 많아지면

Docker로 컨테이너를 만들고 실행하는 방법을 살펴보았습니다. 그런데 컨테이너가 하나일 때는 괜찮지만, 운영 환경에서는 이야기가 달라집니다.

서비스가 성장하면 프론트엔드, 백엔드, 데이터베이스, 캐시 서버까지 컨테이너가 수십 개로 늘어납니다. 이때 생기는 문제를 살펴보겠습니다.

**컨테이너가 죽으면 누가 살려주나요?** 새벽에 백엔드 컨테이너가 메모리 부족으로 종료되었습니다. 개발자가 알림을 확인하고 직접 다시 띄울 때까지 서비스는 멈춰 있습니다.

**트래픽이 몰리면 어떻게 하나요?** 이벤트 기간에 사용자가 열 배로 늘었습니다. 컨테이너를 수동으로 하나씩 추가하고, 이벤트가 끝나면 다시 줄여야 합니다. 매번 사람이 해야 합니다.

**새 버전을 배포할 때 서비스가 끊기나요?** 기존 컨테이너를 멈추고 새 컨테이너를 띄우는 그 짧은 순간, 사용자는 오류 화면을 보게 됩니다.

말콤 맥린의 비유로 돌아가면, Docker가 "화물을 표준 상자에 담는 기술"이라면 이제 필요한 것은 "수백 개의 상자를 자동으로 싣고 내리고 관리하는 항구 시스템"입니다.

> **쿠버네티스(Kubernetes)** 는 컨테이너를 자동으로 배포, 확장, 복구, 관리하는 운영 플랫폼입니다. "컨테이너 3개를 항상 유지해라"라고 선언하면, 하나가 죽어도 자동으로 새 컨테이너를 띄워 3개를 맞춥니다.

Docker Compose와 Kubernetes는 역할이 다릅니다. Docker Compose는 여러 컨테이너를 한 번에 실행하는 도구입니다. "이 컨테이너들을 같이 띄워라"를 정의합니다. 반면 Kubernetes는 컨테이너를 운영하는 플랫폼입니다. 자동 복구, 스케일링, 무중단 배포처럼 "운영 중에 일어나는 일"을 자동으로 처리합니다.

| 구분 | Docker Compose | Kubernetes |
|------|---------------|------------|
| 역할 | 여러 컨테이너를 한 번에 실행 | 컨테이너 운영 자동화 |
| 자동 복구 | 컨테이너가 죽으면 수동 재시작 | 자동으로 새 컨테이너 생성 |
| 스케일링 | 수동으로 설정 변경 후 재실행 | 선언만 하면 자동 확장/축소 |
| 무중단 배포 | 지원하지 않음 | 롤링 업데이트로 중단 없이 배포 |
| 적합한 환경 | 개발, 테스트 | 운영, 프로덕션 |

이 책에서는 챕터 2~3에서 Docker와 Docker Compose를 익힌 뒤, 챕터 4에서 Kubernetes로 넘어갑니다. Docker Compose의 한계를 직접 느낀 뒤에 Kubernetes를 배우면, 왜 필요한지가 자연스럽게 체감됩니다.

**챕터 2 — Docker 이해하기.** Docker를 설치하고 컨테이너를 실행합니다. 리눅스 명령어로 컨테이너 내부를 탐색하고, 이미지를 직접 만들어 Docker Hub에 공유합니다. 마운트로 컨테이너 외부에 데이터를 보존하는 방법도 배웁니다.

**챕터 3 — Docker 다루기.** Dockerfile로 환경 구성을 자동화합니다. NGINX로 트래픽을 분산하고, Redis로 세션을 공유하고, MySQL로 데이터를 저장합니다. Docker Compose로 이 모든 컨테이너를 한 번에 띄우는 풀스택 웹사이트를 만듭니다.

**챕터 4 — Kubernetes 시작하기.** Minikube로 로컬에 Kubernetes 클러스터를 구성합니다. Pod, Deployment, Service로 컨테이너를 관리하고, ConfigMap과 Secret으로 설정을 외부에서 주입합니다. 챕터 3에서 Docker Compose로 만든 풀스택 웹사이트를 Kubernetes 위에 배포합니다.


## 이것만은 기억하자

- **상자에 담으면 어디서든 똑같다.** 말콤 맥린이 화물을 표준 컨테이너에 넣어 전 세계 어디든 보낸 것처럼, Docker는 애플리케이션을 컨테이너에 담아 어디서든 동일하게 실행합니다.
- **Docker는 컨테이너를 만들고 실행하는 도구다.** 이미지에서 컨테이너를 만들고, 수정한 컨테이너를 다시 이미지로 저장할 수 있습니다.
- **컨테이너가 많아지면 Kubernetes가 필요하다.** 자동 복구, 스케일링, 무중단 배포를 사람 대신 Kubernetes가 처리합니다.
- **이 책은 컨테이너 하나에서 시작해 Kubernetes 운영까지 도달하는 여정이다.** 챕터 2에서 Docker를 익히고, 챕터 3에서 여러 컨테이너를 다루고, 챕터 4에서 Kubernetes로 자동화된 운영 환경을 구성합니다.

이제 직접 Docker를 설치하고 컨테이너를 실행해 보겠습니다. 챕터 2에서는 Docker Desktop을 설치하고, 명령어 하나로 컨테이너를 띄우는 것부터 시작합니다.
