# 챕터별 큰 그림 추가 계획

각 챕터 도입부에 해당 기술의 **구성 요소 간 관계**를 다이어그램으로 보여준다. 단순 흐름(A→B→C)이 아니라, "누가 누구를 관리/참조/연결하는가"를 구조적으로 보여주는 것이 핵심.

---

## 챕터 1: Docker의 큰 그림

### 현재 상태
- 1.2.3에서 Docker 엔진 내부 동작만 설명
- Docker를 구성하는 요소들(이미지, 컨테이너, 볼륨, 레지스트리)의 **관계**를 보여주지 않음

### 추가 위치
- **1.2 Docker: 작동 원리** 도입부

### 보여줄 관계
- Docker Hub에서 이미지를 가져온다 (pull)
- 이미지로부터 컨테이너가 생성된다
- 컨테이너는 격리된 프로세스로, 내부에 파일시스템을 가진다
- 컨테이너의 파일시스템은 휘발성이다 → 볼륨을 연결해야 데이터가 유지된다
- Dockerfile로 커스텀 이미지를 만들 수 있다
- 만든 이미지를 Docker Hub에 올릴 수 있다

### D2 다이어그램

```d2
direction: down

docker-host: Docker Host (로컬 PC) {
  docker-engine: Docker Engine {
    style.font-size: 12

    images: 이미지 저장소 {
      nginx: nginx:1.20 {shape: package}
      mysql: mysql:8.0 {shape: package}
      custom: my-app:v1 {shape: package}
    }

    containers: 실행 중인 컨테이너 {
      c1: 컨테이너 A {
        shape: rectangle
        process: 프로세스 (PID 1)
        fs: 파일시스템 (휘발성)
      }
      c2: 컨테이너 B {
        shape: rectangle
        process: 프로세스 (PID 1)
        fs: 파일시스템 (휘발성)
      }
    }

    images -> containers: "run (이미지 → 컨테이너 생성)"
  }

  volume: 볼륨 / 바인드 마운트 {
    shape: cylinder
    style.font-size: 13
  }

  dockerfile: Dockerfile {
    shape: page
    style.font-size: 13
  }

  docker-engine.containers.c1 -> volume: "마운트\n(데이터 영구 보관)"
  docker-engine.containers.c2 -> volume: "마운트"
  dockerfile -> docker-engine.images.custom: "build\n(이미지 생성)"
}

registry: Docker Hub (레지스트리) {
  shape: cloud
  style.font-size: 14
}

registry -> docker-host.docker-engine.images: "pull (다운로드)"
docker-host.docker-engine.images -> registry: "push (업로드)"
```

### 설명 문구 (예시)
> Docker는 위와 같은 구조로 이루어져 있습니다. Docker Engine이 이미지를 관리하고, 이미지로부터 컨테이너를 생성합니다. 컨테이너 내부의 파일은 컨테이너가 삭제되면 함께 사라지기 때문에, 데이터를 보존하려면 볼륨을 연결해야 합니다. Dockerfile을 작성하면 나만의 이미지를 만들 수 있고, Docker Hub를 통해 다른 사람과 공유할 수 있습니다.

## 챕터 2: 웹 서비스 아키텍처의 큰 그림

### 현재 상태
- 2.6.1에서 아키텍처가 나오지만 챕터 마지막
- NGINX, Redis, MySQL을 배울 때 이것들이 **서로 어떻게 연결되는지** 모른 채 시작함

### 추가 위치
- **챕터 2 도입부** (학습 목표 아래, 2.1 시작 전)

### 보여줄 관계
- 브라우저가 NGINX에 요청한다
- NGINX는 정적 파일(HTML/CSS/JS)은 직접 응답하고, API 요청은 백엔드로 전달한다 (리버스 프록시)
- NGINX가 여러 백엔드 서버에 요청을 분산한다 (로드밸런싱)
- 백엔드는 MySQL에서 데이터를 읽고 쓴다
- 백엔드는 Redis에 세션을 저장한다 (서버가 여러 대여도 세션 공유)
- 이 모든 컨테이너를 Docker Compose가 한 번에 관리한다

### D2 다이어그램

```d2
direction: right

browser: 브라우저 {
  shape: rectangle
  style.font-size: 14
}

docker-compose: Docker Compose (컨테이너 통합 관리) {
  style.font-size: 13
  style.stroke-dash: 3

  nginx: NGINX 컨테이너 {
    style.font-size: 13
    style.bold: true

    static: 정적 파일 서빙\n(HTML, CSS, JS) {
      style.font-size: 11
    }
    proxy: 리버스 프록시\n(/api → 백엔드 전달) {
      style.font-size: 11
    }
    lb: 로드밸런싱\n(요청 분산) {
      style.font-size: 11
    }
  }

  backend1: Backend 컨테이너 1 {
    shape: rectangle
    style.font-size: 12
    app: Spring Boot
  }

  backend2: Backend 컨테이너 2 {
    shape: rectangle
    style.font-size: 12
    app: Spring Boot
  }

  redis: Redis 컨테이너 {
    shape: cylinder
    style.font-size: 12
    desc: 세션 저장소\n(서버 간 세션 공유)
  }

  mysql: MySQL 컨테이너 {
    shape: cylinder
    style.font-size: 12
    desc: 영구 데이터 저장소
  }

  volume: 볼륨 {
    shape: cylinder
    style.font-size: 11
  }

  nginx -> backend1: "API 요청\n(라운드 로빈)"
  nginx -> backend2: "API 요청\n(라운드 로빈)"
  backend1 -> mysql: "데이터 조회/저장"
  backend2 -> mysql: "데이터 조회/저장"
  backend1 -> redis: "세션 저장/조회"
  backend2 -> redis: "세션 저장/조회"
  mysql -> volume: "데이터 영구 보관"
}

browser -> docker-compose.nginx: "HTTP 요청"
```

### 설명 문구 (예시)
> 이 챕터에서 최종적으로 만들 웹 서비스의 전체 구조입니다. 브라우저의 요청은 NGINX가 받습니다. 정적 파일은 NGINX가 직접 응답하고, API 요청은 백엔드 서버로 전달합니다. 백엔드가 여러 대일 때는 로드밸런싱으로 요청을 분산하고, 세션은 Redis에 저장하여 어떤 서버에 연결되든 같은 세션을 유지합니다. 데이터는 MySQL에 영구 저장합니다. 이 구조를 하나씩 배워가며 완성해보겠습니다.

## 챕터 3: 쿠버네티스의 큰 그림

### 현재 상태
- 3.1.2에서 리소스 표 + image-prompt가 있지만, 실제 다이어그램이 없음
- 리소스 간의 **소유/참조/주입 관계**가 시각적으로 보이지 않음

### 추가 위치
- **3.1.2 쿠버네티스의 핵심 리소스** — 기존 image-prompt를 실제 다이어그램으로 교체

### 보여줄 관계
- Deployment가 ReplicaSet을 생성하고, ReplicaSet이 Pod를 원하는 개수만큼 유지한다
- Service가 selector로 특정 label의 Pod를 찾아 트래픽을 연결한다
- Ingress가 URL 경로에 따라 적절한 Service로 라우팅한다
- ConfigMap과 Secret이 Pod에 환경변수로 주입된다
- Pod가 PVC를 참조하고, PVC가 PV와 바인딩되어 영구 저장소를 사용한다

### D2 다이어그램

```d2
direction: down

user: 외부 요청 {
  shape: person
  style.font-size: 14
}

cluster: 쿠버네티스 클러스터 {
  style.font-size: 14

  ingress: Ingress {
    shape: hexagon
    style.font-size: 13
    style.bold: true
    desc: URL 경로별 라우팅 규칙
  }

  service: Service {
    shape: hexagon
    style.font-size: 13
    style.bold: true
    desc: 고정 IP + selector로\nPod 자동 연결
  }

  deployment: Deployment {
    style.font-size: 13
    style.bold: true

    rs: ReplicaSet {
      style.font-size: 12
      desc: Pod 개수 유지\n(replicas)

      pod1: Pod 1 {
        shape: rectangle
        container: 컨테이너
      }
      pod2: Pod 2 {
        shape: rectangle
        container: 컨테이너
      }
      pod3: Pod 3 {
        shape: rectangle
        container: 컨테이너
      }
    }

    strategy: 배포 전략\n(RollingUpdate) {
      shape: page
      style.font-size: 11
    }

    strategy -> rs: "관리"
  }

  configmap: ConfigMap {
    shape: page
    style.font-size: 12
    desc: 일반 설정값\n(DB 주소, URL 등)
  }

  secret: Secret {
    shape: page
    style.font-size: 12
    desc: 민감 정보\n(비밀번호, API 키)
  }

  pvc: PVC {
    shape: rectangle
    style.font-size: 12
    desc: 저장소 요청서\n(용량, 권한)
  }
}

pv: PV (영구 저장소) {
  shape: cylinder
  style.font-size: 13
}

# 외부 → 내부 접근 경로
user -> cluster.ingress: "HTTP 요청"
cluster.ingress -> cluster.service: "경로별 라우팅\n(/api → backend-svc)"
cluster.service -> cluster.deployment.rs: "selector 매칭\n(app: nginx)"

# Deployment가 ReplicaSet과 Pod를 관리
cluster.deployment -> cluster.deployment.rs: "생성/관리"

# 설정 주입
cluster.configmap -> cluster.deployment.rs.pod1: "envFrom\n(환경변수 주입)" {style.stroke-dash: 3}
cluster.secret -> cluster.deployment.rs.pod1: "envFrom\n(비밀 정보 주입)" {style.stroke-dash: 3}
cluster.configmap -> cluster.deployment.rs.pod2: "" {style.stroke-dash: 3}
cluster.secret -> cluster.deployment.rs.pod2: "" {style.stroke-dash: 3}

# 볼륨 연결
cluster.deployment.rs.pod1 -> cluster.pvc: "volumeMount"
cluster.pvc -> pv: "바인딩\n(accessMode, capacity 매칭)"
```

### 설명 문구 (예시)
> 쿠버네티스의 리소스들은 위와 같은 관계로 연결되어 있습니다. `Deployment`는 `ReplicaSet`을 만들고, `ReplicaSet`은 설정된 개수만큼 `Pod`를 유지합니다. `Service`는 selector를 통해 특정 라벨의 Pod를 찾아 트래픽을 전달하고, `Ingress`는 URL 경로에 따라 적절한 Service로 요청을 라우팅합니다. `ConfigMap`과 `Secret`은 환경변수 형태로 Pod에 설정을 주입하고, `PVC`는 `PV`와 바인딩되어 Pod에 영구 저장소를 제공합니다. 이 챕터에서는 이 리소스들을 하나씩 배워보겠습니다.
