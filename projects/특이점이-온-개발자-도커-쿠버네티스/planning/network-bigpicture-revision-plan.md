## A. Pod 네트워크 정의 보강 🔴 최우선

### 대상
`chapters-v2/04-Kubernetes-시작하기-v5.md` 줄 176~177 (참고 박스: Pod의 네트워크)

### 이유
CH05.3.2 대응표에서 `docker0 (bridge) → Pod 네트워크` 매핑을 제시하지만, CH04.3의 현재 Pod 네트워크 박스는 **"한 Pod 안 localhost"** 만 설명함. "클러스터 전체에 펼쳐진 Pod 네트워크"라는 핵심 개념이 빠져 있어 독자가 대응표에서 "Pod 네트워크가 뭐더라?"로 멈춘다.

### Before
```
> **참고: Pod의 네트워크**
> 한 Pod 안의 컨테이너들은 **같은 IP를 공유**합니다. `localhost`로 서로 직접 통신할 수 있습니다. Docker에서는 컨테이너가 네트워크 단위였다면, Kubernetes에서는 **Pod가 네트워크 단위**입니다.
```

### After
```
> **참고: Pod의 네트워크 (두 레벨)**
>
> **1. Pod 안쪽** — 한 Pod 안의 컨테이너들은 **같은 IP를 공유**합니다. `localhost`로 서로 직접 통신합니다.
>
> **2. Pod 사이** — 클러스터에 떠 있는 **모든 Pod는 하나의 공용 네트워크**를 공유합니다. 각 Pod에 고유한 IP가 할당되고, Pod끼리는 이 네트워크를 통해 서로를 부릅니다. 챕터 2의 **docker0**(푸드코트 홀)가 노트북 한 대 안의 홀이었다면, Pod 네트워크는 클러스터 전체에 펼쳐진 더 큰 홀이라고 생각하면 됩니다. 이 네트워크의 정식 이름과 동작은 챕터 5에서 다시 등장합니다.
>
> Docker에서는 컨테이너가 네트워크 단위였다면, Kubernetes에서는 **Pod가 네트워크 단위**입니다.
```

## D. Service 비유 통일 (가맹점 대표 전화번호)

### 대상
- `chapters-v2/05-Kubernetes-네트워킹-v5.md` 줄 29 (콜센터 비유)

### 이유
CH04 L56 매핑 표와 CH04 L452 예고에서 **"가맹점 대표 전화번호"** 로 잡아둔 비유가 CH05.1.1에서 **"콜센터 대표 번호"** 로 바뀜. 독자는 "Pod가 가맹점인가 상담사인가"로 한 번 멈춘다. 프랜차이즈 세계관 유지를 위해 **콜센터 → 가맹점 대표 전화번호**로 통일.

### Before (CH05 L29)
```
필요한 건 "**Pod IP가 어떻게 바뀌든 항상 같은 주소**"였습니다. 콜센터 대표 번호처럼. 상담사 개인 번호는 교대로 바뀌어도 대표 번호로 걸면 당번 상담사가 받는 방식입니다. 상담사가 여럿일 때는 대표 번호가 요청을 돌려가며 한 명씩 연결해 줍니다.
```

### After
```
필요한 건 "**Pod IP가 어떻게 바뀌든 항상 같은 주소**"였습니다. **가맹점 대표 전화번호**처럼요. 매장 안 직원 내선 번호는 교대 때마다 바뀌어도, 손님은 늘 가맹점 대표 번호로 걸면 됩니다. 한 매장에 직원이 여럿이면 대표 번호가 요청을 돌려가며 한 명씩 연결해 줍니다.
```

## E. CH06.3 502/500 설명 분리

### 대상
`chapters-v2/06-Kubernetes-운영하기-v5.md` 줄 463 부근 (1단계 502), 줄 520 부근 (2단계 500)

### 이유
1단계에서 backend-service가 없어 502가 떨어지는 상황과, 2단계에서 DB가 없어 500이 떨어지는 상황이 텍스트에서 섞여 있음. "Nginx 설정에 따라 기동 실패로 보이기도"라는 보강이 오히려 두 단계를 혼동시킴.

### Before (1단계 L463 주변)
```
`/api/users` 호출은 개발자 도구 네트워크 탭에서 502로 떨어지고 있었습니다.
이 자체가 좋은 신호였습니다. ... 최소 경로가 동작하고 있다는 뜻이었습니다.
```

### After (1단계)
```
`/api/users` 호출은 개발자 도구 네트워크 탭에서 502로 떨어지고 있었습니다. 프론트 Nginx의 `proxy_pass`가 향하는 `backend-service`가 아직 없기 때문입니다.

이 자체가 좋은 신호였습니다. ... 최소 경로가 동작하고 있다는 뜻이었습니다.
```

### Before (2단계 L520 주변)
```
`/api/users` 호출이 이번엔 502 대신 500으로 바뀌었습니다. 개발자 도구의 응답을 열어보니 **DB 연결 실패**였습니다. 앱 설정에 따라 백엔드 Pod 자체가 DB 연결 실패로 기동에 실패할 수도 있습니다.
```

### After (2단계)
```
`/api/users` 호출이 이번엔 502 대신 500으로 바뀌었습니다. 1단계의 502는 "프록시 대상이 없다"는 뜻이었고, 지금 500은 "프록시는 성공했는데 백엔드가 DB를 못 붙는다"는 뜻입니다. 개발자 도구 응답에도 **DB 연결 실패** 메시지가 찍혀 있었습니다.

> **참고**: Spring Boot 설정에 따라 백엔드 Pod 자체가 DB 연결 실패로 기동에 실패해 CrashLoopBackOff 상태가 될 수도 있습니다. 그 경우 `/api/users` 호출은 502로 돌아오므로, 반대로 502가 보인다고 해서 항상 1단계 상태인 건 아닙니다.
```

CH05에서는 `minikube service --url`로 접근했는데, CH06에서는 `minikube tunnel` 등장. 둘 다 "Minikube 한 겹을 뚫는 도구"인데 차이가 설명되지 않음.

### 삽입 위치 (CH06 `minikube tunnel` 첫 등장 근처)

### After (한 줄 참고 박스 추가)
```
> **참고: `minikube service` vs `minikube tunnel`**
> - `minikube service <서비스명> --url` — 특정 Service 하나에 대해 임시 접근 경로를 뚫습니다. 빠른 확인용.
> - `minikube tunnel` — 클러스터 전체의 LoadBalancer/Ingress 트래픽을 호스트로 끌어옵니다. 지속 실행 상태로 두고, 이 책처럼 Ingress가 있는 구성에서는 이쪽이 맞습니다.
```

## G. CH05.3.1 전체 경로 표에 구체 예시 컬럼 추가 (선택)

### 대상
`chapters-v2/05-Kubernetes-네트워킹-v5.md` 줄 263~267 (5.3.1 요청의 여정 표)

### 이유
현재 표는 "단계 / 컴포넌트 / 하는 일 / 확인하는 것" 4열. 추상적 분류만 있어 독자가 실제 값을 머릿속에 그리기 어려움. **구체 예시 컬럼** 하나 추가하면 추상→구체 변환이 즉시 일어남.

### Before
```
| 단계 | 컴포넌트 | 하는 일 | 확인하는 것 |
|------|---------|--------|-----------|
| 1 | **브라우저** | 요청 전송 | - |
| 2 | **Ingress Controller** | URL 경로 확인 → 적절한 Service로 라우팅 | URL, Host |
| 3 | **Service** | Label-Selector로 매칭된 Pod 그룹에 요청 전달 | Label |
| 4 | **kube-proxy** | iptables 규칙으로 실제 Pod IP로 변환 | IP, Port |
| 5 | **Pod** | 애플리케이션이 요청을 처리 | JSON, 비즈니스 로직 |
```

### After
```
| 단계 | 컴포넌트 | 하는 일 | 확인하는 것 | 구체 예 |
|------|---------|--------|-----------|--------|
| 1 | **브라우저** | 요청 전송 | - | `https://my-shop.com/api/orders` |
| 2 | **Ingress Controller** | URL 경로 확인 → 적절한 Service로 라우팅 | URL, Host | `/api/*` → `backend-service` |
| 3 | **Service** | Label-Selector로 매칭된 Pod 그룹에 요청 전달 | Label | `app: backend` |
| 4 | **kube-proxy** | iptables 규칙으로 실제 Pod IP로 변환 | IP, Port | `10.96.0.50:80 → 10.244.0.5:8080` |
| 5 | **Pod** | 애플리케이션이 요청을 처리 | JSON, 비즈니스 로직 | `{ "orders": [...] }` |
```
