# v5 공통 설정

chapters-v2 재집필 시 모든 챕터가 따를 기반. 기존 chapters-v2의 "말이 안 되는 설정" 문제를 해결하기 위한 세계관·원칙 문서.

## 1. 캐릭터 재정의

### 오픈이 (주인공, 독자 대리인)
- **직급**: 입사 3~6개월차 주니어 개발자
- **배경**: Spring Boot로 사이드 프로젝트/사내 작은 도구를 혼자 만들어본 경험. 배포는 JAR 업로드나 로컬 실행 수준. 운영 서버 경험 없음
- **역할**: 선임이 설계한 구조를 **로컬/스테이징에서 재현하며 학습**. 운영 권한 없음. 장애 대응·운영 의사결정은 하지 않음
- **학습 동기**: 팀이 도입했거나 도입 검토 중인 기술을 주니어가 따라잡아야 하는 상황. 선임이 "이거 다음 주까지 따라해보고 공유해줘" 같은 학습 과제를 던짐

### 선임 (선배, 멘토)
- **역할**: 오픈이에게 학습 과제를 던지고 방향만 잡아줌
- **등장 빈도**: 챕터당 1~2회. 아무 때나 지나가며 정답을 주지 않음
- **대사 톤**: 짧고, 트리거 수준. 정확한 기술 용어를 괄호로 매칭하는 교과서 대사 금지

### 팀장
- **역할**: 배경으로만 존재. 과제 지시나 학습 방향 회고 시점에만 등장
- **등장 빈도**: 챕터당 최대 1회. CH01, CH04, CH06 정도에만 의미 있게 등장
- **대사 톤**: 업무적. 힌트 제공 금지

### 바꾸는 것
- 팀장/선배가 **비유를 즉답**하는 장면 제거 (주방, 배달앱, 콜센터 등은 오픈이 내면 독백으로 이동)
- 선임이 대사 안에서 **K8s 리소스명을 괄호로 매칭**하는 장면 제거

## 2. 시간·공간 설정

### 시간 축
- chapters-v2의 "수요일 새벽 3시 / 목요일 새벽 3시 / 금요일 타임세일 / 월요일 긴급패치" 같은 **1주일 내 극적 압축** 버리기
- v5는 **몇 주~두 달** 단위 느슨한 학습 기간. "지난주 회고에서 나온 얘기", "이번 분기 도입 검토 과제", "사내 스터디"
- 새벽 알람·무중단 긴급 패치 등 **운영 장애 장면 전부 제거**

### 공간
- 오픈이의 주 작업 환경은 **로컬 노트북** (개발용 Docker Desktop, Minikube)
- "운영 서버 직접 조작" 장면 없음
- 스테이징 환경은 언급만 (실제 조작은 선임 담당)

## 4. 대사 정책

- **대사 비중 전체 15% 이하** (기존 20%에서 축소)
- 팀장/선배 대사는 **트리거**만 (한 문장, 두 문장 이내)
- 비유를 펼치는 대사 금지. 비유는 **오픈이의 내면 독백**으로 이동
- 정답·핵심 용어를 대사가 먼저 주지 않음. 오픈이가 **공식 문서/로그/실험**을 거쳐 도달
- 선배가 기술 용어를 괄호로 매칭하는 대사 금지 (예: `선배: "메뉴판(ConfigMap)이랑 금고(Secret)"` 형태 전면 금지)

## 5. 정답에 도달하는 경로

각 챕터에서 오픈이가 핵심 개념을 알게 되는 방식을 다양화.

| 경로 | 비중 | 예시 |
|------|-----|------|
| 공식 문서·레퍼런스 | 30% | Docker / K8s 공식 문서에서 해당 절 발견 |
| 실습 로그·에러 메시지 | 30% | kubectl describe의 Events 메시지로 원인 발견 |
| 직접 실험 (A/B 비교) | 20% | `docker run` vs Deployment Pod 비교 |
| 선임의 짧은 한 마디 | 15% | "그건 사용자 정의 네트워크 얘기야" 수준 |
| 팀 회고/스터디 언급 | 5% | "지난주 회고에서 이 얘기 나왔어" |

## 6. 메타 파탄 금지

- 캐릭터가 **챕터 번호**("CH03에서 봤잖아")를 언급하지 않음. 필요하면 "전에 배운 iptables" 같은 기억 언급으로
- 저자 서술 ("이 책 전체에 쓸 비유를 심어둡니다")이 본문에 노출되지 않음. 내러티브 안에서 자연스러운 연결로
- 오픈이가 "지난 3주 챕터별로" 회상하는 장면 금지

## 7. 서사 장치 제한

- **판서/A4/화이트보드** 장면은 전체 책에서 **2회 이내**
- 선배가 **노트북에 준비된 그림**을 띄우는 장면 금지
- 커피/퇴근 같은 분위기 묘사는 감각 묘사 수준으로만 (장면 전환 장치로 남용 금지)

## 8. 실습 동기 원칙

"오픈이가 왜 지금 이걸 실행하는가"가 반드시 있어야 함.
- docker commit은 "학습 목적으로 방금 수정한 상태 저장해보는" 동기
- Docker Hub push는 "공식 튜토리얼 마지막 단계" 혹은 "팀 공유"
- Tomcat 띄우기는 "이미지 만들기 연습 대상"으로 명확히 학습 맥락화

## 12. 참고 박스(팁박스) 문체

모든 참고 박스의 문장은 **존댓말**로 작성합니다.

- "~이다 / ~한다 / ~된다" 같은 평어체 금지
- "~입니다 / ~합니다 / ~됩니다 / ~씁니다 / ~있습니다" 존댓말 사용
- 명사 종결문("~ 실행 단위.")도 "~ 실행 단위입니다."로
- 본문 내러티브와 톤을 일치시켜 독자가 참고 박스를 읽을 때 이질감이 없도록

## 13. 문장 품질

**모든 문장은 일단 말이 되어야 한다.** 비유·스토리텔링을 위해 의미를 희생하는 문장을 쓰지 않는다.

- **주어-서술어 호응 검수**: "장비는 많이 듭니다" 같은 미스매치 금지. "장비가 많이 필요합니다" 또는 "비용이 많이 듭니다"처럼 서술어에 맞는 주어를 쓴다
- **조사/수식관계 점검**: 꾸며주는 말과 꾸밈 받는 말의 관계가 모호하지 않게
- **축약으로 인한 의미 붕괴 금지**: 짧은 문장을 선호하되, 뜻이 잘려나가면 안 됨
- **작성 후 자체 검수**: 초안 작성 후 각 문장을 **독립적으로** 읽어보고 어색한 부분은 즉시 교정. 전체 흐름이 좋다고 개별 문장의 어색함을 넘기지 않는다
- **의성어·감각어 남용 금지**: 수사를 위해 넣은 단어가 실제 의미와 어긋나면 빼기

## 14. 보존 사항 (v5에서도 유지)

- 프랜차이즈 매핑 (본사=K8s / 매장=Pod / 대표전화=Service / 공용금고=Secret 등) — 단, 저자 메타 선언 없이
- 말콤 맥린 해상 컨테이너 역사 — CH01 유지, 단 "선배가 강의하는" 형태가 아니라 오픈이가 사내 자료/검색에서 접하는 형태
- 주방 비유, 푸드코트 비유, 콜센터 대표번호 비유 — 비유 자체는 유지, 전달 방식만 내면 독백으로
- Docker↔K8s 대응표 (CH05) — 그대로 유지
- 전체 이미지/다이어그램 placeholder — 재사용 가능한 것은 그대로 유지
