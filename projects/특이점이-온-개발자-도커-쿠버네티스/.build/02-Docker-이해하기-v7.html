2.1 가상화 - 가상머신과 컨테이너

2.1.1 한 주방, 네 명의 요리사

먼저 컨테이너의 원리부터 정리하기로 했습니다. 자료를 한참 읽다 떠오른 장면은 한 주방에 요리사 여러 명이 들어가는 모습이었습니다.

한 주방에 요리사 네 명이 있습니다. 네 명이 동시에 요리를 하려면 주방을 어떻게 꾸며야 할까요.

그림 2-1. 한 주방에 네 명의 요리사가 있는 상황

첫 번째 방식은 요리사마다 냉장고, 가스레인지, 조리대를 각자 한 세트씩 사주는 것입니다. 이렇게 하면 네 사람이 완전히 독립된 환경에서 요리하지만, 같은 설비를 네 번 갖춰야 하니 비용이 많이 듭니다.

그림 2-2. 주방 설비를 통째로 복제하는 첫 번째 방식

두 번째 방식은 냉장고와 가스레인지 같은 공용 설비는 함께 쓰고, 칼·도마·조리대처럼 개인이 쓸 도구만 각자 챙기는 것입니다. 장비 수는 그대로인데 공간은 네 명이 동시에 쓸 수 있습니다.

그림 2-3. 공용 설비는 공유하고 개별 공간만 쪼개는 두 번째 방식

이렇게 하나의 주방을 여러 명이 사용하는 것처럼, 하나의 컴퓨터를 여러 대처럼 나눠 쓰는 IT 기술을 가상화(Virtualization) 라고 부릅니다.

가상화(Virtualization): 하나의 물리 서버를 논리적으로 여러 대처럼 나누어 쓰는 기술입니다. 자원을 효율적으로 쓰면서 서로 다른 환경을 안전하게 격리하는 것이 목적입니다.

가상화에도 두 가지 방식이 있습니다. 설비를 통째로 복제하는 쪽이 하이퍼바이저 가상화, 공유하는 쪽이 컨테이너 가상화입니다. Docker는 두 번째 방식의 도구입니다. 같은 OS 위에서 여러 앱이 서로 충돌하지 않고 각자 돌게 해 줍니다.

하이퍼바이저 가상화 vs 컨테이너 가상화

하이퍼바이저 가상화(Hypervisor Virtualization) 는 하드웨어 위에 OS 자체를 따로 띄우는 방식이고, 결과물은 가상 머신(VM) 입니다. 격리는 완전하지만 OS 전체가 새로 떠야 해서 무겁고 기동이 느립니다. 반면 컨테이너 가상화(Container Virtualization) 는 호스트 OS의 커널을 공유하면서 파일시스템·네트워크·프로세스 공간만 따로 두는 방식이고, 결과물은 컨테이너입니다. 커널을 공유하므로 가볍고 빠르게 뜹니다.

'아, 도커는 내 컴퓨터 OS 하나를 여러 컨테이너가 나눠 쓰면서, 각자 격리된 공간에서 따로 돌게 하는 거구나.'

2.1.2 IT로 옮긴 구조

비유로 컨테이너의 원리는 잡혔습니다. 같은 구조를 IT 용어로 옮기면 다음과 같습니다.

맨 아래에 하드웨어가 있고, 그 위에 호스트 OS와 Docker 엔진이 한 층(OS 계층)을 이룹니다. 그 위에 컨테이너가 나란히 떠 있고, 각 컨테이너 안에는 애플리케이션(App)과 라이브러리(Library)가 같이 들어갑니다.

주방 비유의 공용 설비가 호스트 OS, 각자의 조리대가 컨테이너입니다.

그림 2-4. 컨테이너 가상화의 전체 구조 (Linux 호스트 기준. Windows·macOS는 2.3에서 다룸)

이 구조에서 주방장에 해당하는 것이 OS의 핵심, 곧 커널입니다. 요리사들이 주방장의 명령에 따라 움직이고 어떤 위치에서 어떤 역할을 할지도 주방장이 정해 주듯, 컨테이너에 격리 공간을 만들어 주는 일은 커널이 합니다.

커널(Kernel): 운영체제(OS)의 핵심입니다. 프로세스 생성, 메모리 관리, 격리 공간 생성을 담당합니다. Docker의 모든 컨테이너는 호스트 PC의 같은 리눅스 커널을 공유하고, 그 하나의 커널이 각 컨테이너에 격리 공간을 할당합니다.

그림 2-5. 커널은 OS의 핵심으로, 앱과 하드웨어 사이에서 시스템 자원을 관리합니다

이 구조의 핵심은 호스트 OS의 커널 하나를 모든 컨테이너가 공유한다는 점입니다. VM처럼 OS를 통째로 따로 올리지 않으니 가볍고 빠르게 뜹니다. Docker 엔진이 그 커널에 "새 격리 공간을 만들어 달라"고 요청하면, 실제로 프로세스를 격리하는 일은 커널이 합니다. 그래서 컨테이너마다 다른 앱과 다른 라이브러리를 가져도 충돌하지 않고, 한쪽이 런타임을 업그레이드해도 옆 컨테이너는 영향을 받지 않습니다.

컨테이너의 격리

사실 컨테이너 안에서 도는 것은 처음부터 격리된 무언가가 아니라, 호스트 OS 위에서 다른 것들과 함께 도는 평범한 프로세스입니다. 그대로 두면 한 컨테이너의 프로세스가 다른 컨테이너의 파일을 들여다보거나, 두 컨테이너가 같은 포트를 두고 충돌할 수 있습니다. 그래서 리눅스 커널은 컨테이너마다 파일시스템(/bin, /lib 같은 폴더), 네트워크(IP와 포트), 프로세스(PID 번호) 세 가지 공간을 따로 만들어 줍니다. 이렇게 컨테이너마다 세 공간을 따로 갖는 상태가 바로 격리입니다.

2.1.3 컨테이너가 만들어지는 과정

컨테이너가 무엇인지는 잡혔습니다. 그런데 이 컨테이너가 어떻게 만들어지는지는 아직 명확하지 않았습니다. 명령 한 줄을 친 그 순간 무엇이 어떤 순서로 움직이는지를 한 번은 그려 둬야, 나중에 명령어를 칠 때 흐름이 더 명확할 것 같았습니다.

그림 2-6. 컨테이너 실행 명령이 커널의 격리 기능을 타고 컨테이너가 되는 흐름
  1. 사용자가 컨테이너를 실행하라는 명령을 친다.
  2. Docker 엔진이 이 명령을 받고, 로컬에 있는 이미지를 읽는다(없으면 Docker Hub에서 내려받는다).
  3. Docker 엔진이 커널에게 "이 이미지 사양으로 격리된 프로세스 하나 만들어 줘"라고 요청한다.
  4. 커널이 격리 공간(파일시스템/네트워크/프로세스)을 만들고, 그 안에서 새 프로세스를 띄운다.
  5. 이 격리된 프로세스가 곧 컨테이너다.

여기서 짚어 둘 점이 하나 있습니다. 컨테이너를 실제로 만드는 건 Docker 엔진이 아니라 커널입니다. 커널이 주방장이라면, Docker 엔진은 홀의 프런트 직원에 가깝습니다. 새 주문이 들어오면 프런트가 주방장에게 그대로 넘기고, 실제로 새 요리사(프로세스)를 불러 작업을 시작시키는 일은 주방장이 합니다.

엔진과 커널의 역할 분담이 정리되자 다음 의문이 따라왔습니다.

'그럼 이 안에서 뭘 실행할지는 어디서 정해지지.'

2.2 이미지 - 컨테이너의 설계도

컨테이너가 격리된 프로세스라는 건 알았습니다. 그런데 그 프로세스가 무엇을 실행할지를 결정하는 원본이 따로 있어야 합니다. 그 원본이 이미지입니다.

그림 2-7. 하나의 이미지로 여러 컨테이너를 찍어내는 구조

이미지는 붕어빵 틀에 가깝습니다. 틀(이미지) 하나로 붕어빵(컨테이너)을 여러 개 찍어낼 수 있고, 틀 자체는 변하지 않습니다. 그래서 같은 이미지에서 나온 컨테이너는 어디서 띄우든 똑같은 환경을 가집니다. 이 덕분에 "내 PC와 서버 환경이 같다"가 가능합니다. 로컬에서 만든 이미지를 서버에서 그대로 꺼내 실행하면 서버의 런타임 버전이 무엇이든 컨테이너는 자기 이미지 안의 런타임으로 돕니다.

이미지(Image): 컨테이너를 만드는 데 필요한 파일시스템과 실행 정보가 한 덩어리로 묶인 읽기 전용 패키지입니다. 애플리케이션 코드, 라이브러리, 설정, 실행 명령이 함께 들어 있어 어디서 띄우든 동일한 환경이 만들어집니다.

이미지는 Docker Hub라는 공용 저장소에서 내려받을 수도 있고, 직접 만들어 올릴 수도 있습니다.

그림 2-8. Docker의 전체 흐름

Docker Hub를 활용하면 내 PC에서 세팅한 환경을 서버에서 재현하는 일도, 팀원끼리 환경을 공유하는 일도 가능합니다.

이미지 레이어

이미지는 한 덩어리가 아니라 여러 레이어가 층층이 쌓여 있습니다. 아래쪽에는 OS 기본 라이브러리나 런타임처럼 잘 안 바뀌는 부분이 있고, 위쪽에는 애플리케이션 코드처럼 자주 바뀌는 부분이 있습니다. 한 번 받은 아래 레이어는 저장해 두고 다시 쓰기 때문에, 코드만 바뀐 새 버전이 나오면 맨 위 레이어만 새로 받으면 됩니다. 여기에 커널은 호스트와 공유하므로 이미지에 담지 않습니다. 이런 구조 덕에 도커 이미지는 OS 전체를 담는 VM 이미지에 비해 훨씬 가볍습니다.

원리를 짚었으니 이제 Docker를 설치하고 직접 띄워 보겠습니다.

Docker는 리눅스 커널의 기능을 쓰기 때문에 리눅스 기반이 아닌 Windows와 macOS에서는 Docker Desktop이 내부에 리눅스 환경을 한 겹 더 띄웁니다. Windows는 WSL2(Windows Subsystem for Linux 2)를, macOS는 경량 Linux VM을 씁니다. 사용자 입장에서는 신경 쓸 일이 없지만 뒤에 나올 네트워크 얘기에서 이 차이가 중요해집니다.

2.5 컨테이너 통신

왜 접속이 안 되는지 알려면 컨테이너가 외부와 어떻게 닿는지부터 짚어야 했습니다.

2.5.1 네트워크 한눈에 보기

오픈이가 자료를 펼치니 컨테이너 네트워크가 크게 세 가지로 나뉘어 있었습니다. 컨테이너끼리 연결하는 방법, 외부 요청이 컨테이너로 들어오는 길, 이름으로 컨테이너를 찾는 방법입니다.

주방 비유로 가상화를 잡았던 것처럼, 이번에는 식당 여러 개가 모인 푸드코트로 그려 봤습니다.

2.5.2 컨테이너끼리 - 주방 모니터와 주문 전광판

A, B, C 식당이 푸드코트에 입점해 있습니다. 식당끼리는 벽으로 완전히 분리되어 있어 서로의 주방을 들여다볼 수 없습니다.

각 식당의 주방에는 주방 모니터가 한 대씩 있고, 푸드코트 한가운데에는 모두가 보는 주문 전광판이 걸려 있습니다. 주방 모니터와 전광판은 한 가닥의 전용 통신 케이블로 이어져 있고, 모든 식당의 케이블이 전광판 본체에 꽂혀 있습니다. 식당끼리 벽이 막혀 있어도 이 전광판을 거치면 서로의 주문 현황을 알 수 있습니다.

그림 2-13. 주방 모니터·전용 통신 케이블·주문 전광판이 컨테이너 네트워크의 비유

이 비유를 IT로 옮기면 세 가지가 짝지어집니다.

푸드코트 IT 용어 한 줄 설명
벽으로 분리된 주방 Namespace 컨테이너마다 새로 생기는 격리된 네트워크 공간입니다
전용 통신 케이블 veth pair 격리된 두 공간을 잇는 한 쌍짜리 가상 인터페이스입니다
주문 전광판 docker0 같은 브리지에 묶인 컨테이너 사이로 패킷(데이터 묶음)을 중계하는 가상 브리지입니다

컨테이너가 뜨는 순간 자기만의 Namespace가 새로 생기고 veth pair 한 쌍이 만들어집니다. 한 끝은 컨테이너 안에, 다른 끝은 호스트의 docker0에 자동으로 꽂히면서 docker0가 빈 IP(172.17.0.x)를 컨테이너에 할당합니다. 이때 A 컨테이너가 B의 IP로 호출하면 데이터는 A → docker0 → B의 경로로 흐릅니다. docker0는 어떤 케이블이 어느 IP에 묶여 있는지 표를 들고 있어서 들어온 데이터를 정확히 그 IP의 케이블로만 흘려보냅니다.

그림 2-14. 컨테이너 네트워크 격리 구조

'각자 격리되어 있는데 케이블 한 가닥으로 다 통하네. 그럼 푸드코트 바깥의 손님은 어떻게 주문하지.'

다음으로 알아본 건 외부 손님이 식당에 어떻게 주문을 넣느냐였습니다.

2.5.3 외부에서 들어오기 - 입구 키오스크

docker0는 도커가 호스트에 기본으로 만들어 두는 가상 스위치입니다. 이 위에서 동작하는 네트워크를 기본 네트워크라고 부릅니다. 공유기가 집안 기기를 한 네트워크로 묶어 주듯 docker0도 같은 호스트의 컨테이너들을 이 기본 네트워크로 묶어 둡니다.

그래서 docker run으로 컨테이너를 띄우면 별도 옵션을 지정하지 않는 한 자동으로 이 기본 네트워크에 들어갑니다. 같은 네트워크 안에 있으니 IP만 알면 서로 통신할 수 있습니다.

문제는 공유기 너머, 즉 호스트 PC 바깥에 있는 외부 사용자입니다. 브라우저처럼 외부에서 컨테이너로 바로 접근하기는 어렵습니다. 이유는 두 가지입니다.

  • Windows·macOS에서는 Docker가 가상 리눅스 안에서 돌기 때문에 docker0와 컨테이너 IP가 호스트 터미널에서는 보이지 않습니다.
  • 설령 리눅스 환경이라 해도 외부 사용자는 컨테이너의 내부 IP를 알 수 없습니다. 컨테이너가 뜰 때마다 IP가 바뀌기도 하고 바깥으로 공개된 주소도 아니기 때문입니다.

따라서 외부에서 컨테이너로 들어오는 별도의 통로가 필요합니다.

푸드코트에 비유하면 외부 손님은 주방에 직접 들어가지 못합니다. 대신 입구의 키오스크에 주문을 넣으면 키오스크가 해당 식당으로 그 주문을 넘겨줍니다. 짜장면은 A 식당으로, 돈까스는 B 식당으로 주문이 전달됩니다. 어떤 메뉴를 누르느냐가 곧 어느 식당으로 갈지를 정하는 분기 기준입니다.

그림 2-15. 입구 키오스크가 손님의 주문을 각 식당으로 넘겨주는 비유

이 비유를 IT로 옮기면 세 가지가 짝지어집니다.

푸드코트 IT 용어 한 줄 설명
입구 키오스크 포트포워딩 호스트 PC의 외부 공개 포트로 들어온 요청을 컨테이너 포트로 자동으로 넘기는 메커니즘입니다
키오스크 화면의 매핑 iptables 규칙(DNAT) 호스트 포트로 온 요청의 도착지를 컨테이너 포트로 바꿔치기하는 한 줄짜리 변환 규칙입니다
식당 주방 컨테이너 포트 컨테이너 안에서 앱이 듣고 있는 포트(:80 등)입니다

docker run -p 8080:80을 실행하면 Docker는 iptables에 "호스트의 8080 포트로 들어온 요청을 컨테이너의 80 포트로 전달"이라는 규칙을 한 줄 심어 둡니다. 이제 브라우저가 localhost:8080을 호출하면 iptables가 이 규칙대로 요청의 도착지를 컨테이너 80 포트로 바꿔치기하고 docker0를 거쳐 해당 컨테이너로 전달합니다. 사용자는 이 변환을 모르고 "localhost에 접속했더니 컨테이너 안의 페이지가 나왔네" 정도로만 경험합니다.

그림 2-16. 호스트 포트가 컨테이너 포트로 연결되는 구조

'외부에서 들어올 땐 결국 호스트 포트로만 들어오는구나. iptables가 그 안에서 컨테이너 포트로 길을 바꿔주고, 컨테이너 IP는 외부에 노출할 필요가 없네.'

2.5.4 이름으로 찾기 - 홀의 안내 지도

컨테이너끼리 통신하려면 결국 상대 IP를 알아야 합니다. 한두 개라면 IP를 적어 두고 쓸 수 있지만, 수가 늘어나면 일일이 외우는 게 번거로워집니다.

'인터넷은 naver.com 같은 도메인 주소만 알면 알아서 연결되잖아. 컨테이너도 이름으로 찾을 수는 없나.'

푸드코트로 옮겨 보면, 손님은 식당 이름은 알지만 푸드코트 안에서 식당이 어디 있는지는 알 수 없습니다. 그럴 때 홀에 걸린 안내 지도에 "B식당 = 2구역"이라고 표시되어 있으면, 이름을 보고 위치를 쉽게 찾을 수 있습니다.

그림 2-17. 안내 지도가 이름을 위치로 바꿔주는 비유

이 비유를 IT로 옮기면 두 가지가 짝지어집니다.

푸드코트 IT 용어 한 줄 설명
안내 지도 Docker DNS 컨테이너 이름을 IP로 자동 변환해 주는 내장 DNS(Domain Name System) 서버입니다
같은 푸드코트 안 사용자 정의 네트워크 같은 네트워크에 묶인 컨테이너끼리만 이름으로 통신할 수 있습니다

새 컨테이너가 뜰 때마다 이름과 IP가 자동으로 Docker DNS에 등록됩니다. 그다음 app 컨테이너가 db라는 이름으로 요청을 보내면 Docker DNS가 먼저 받아 db를 172.17.0.3으로 변환해 알려 주고, app은 그 IP로 다시 요청을 보냅니다. 컨테이너가 다시 떠서 IP가 바뀌어도 이름은 그대로라 코드를 수정할 필요가 없습니다.

그림 2-18. Docker DNS가 이름을 IP로 변환 (사용자 정의 네트워크 한정)

다만 이 자동 DNS는 사용자 정의 네트워크(User-defined Network) 안에서만 켜집니다. 지금까지 본 기본 네트워크(docker0)에서는 작동하지 않습니다. 그래서 사용자가 직접 네트워크를 만들어 컨테이너를 묶어 둬야 이름으로 통신할 수 있습니다. 이 차이는 다음 챕터에서 직접 만들어 보며 확인합니다.

'naver.com처럼 컨테이너도 이름으로 부르면 되는구나. IP는 일일이 외울 필요 없겠네.'

오픈이는 여기까지 컨테이너의 통신을 정리했습니다. 이번엔 컨테이너 안에 들어가 어떻게 생겼는지 직접 보기로 했습니다.

2.6 컨테이너 내부 - 격리된 작은 리눅스

리눅스를 연습하려면 리눅스 환경이 필요합니다. Ubuntu는 가장 널리 쓰이는 배포판 중 하나라 자료가 풍부하고, Docker Hub에 공식 이미지가 올라와 있어서 바로 받아 띄울 수 있습니다. 오픈이는 포트포워딩을 사용해 ubuntu 컨테이너 내부로 들어갔습니다.

터미널 ubuntu 컨테이너 안으로 들어가기
# -i 입력 받기, -t 터미널 화면, -p 80:80 포트포워딩 docker run -it -p 80:80 ubuntu
실행결과
$ docker run -it -p 80:80 ubuntu
root@c0b6dd5261d8:/#
그림 2-19. Ubuntu 컨테이너 안으로 들어간 상태

프롬프트가 root@...# 모양으로 바뀌었습니다. 방금까지는 호스트의 터미널이었는데, 이제는 컨테이너 안의 쉘입니다. 호스트와는 분리된 공간에 들어와 있다는 감각이 손끝에 옵니다.

  • 상대 경로는 현재 디렉터리를 기준으로 목적지까지의 경로를 나타냅니다. 현재 위치에 따라 같은 표기가 다른 곳을 가리킬 수 있습니다. 예: bin(현재 디렉터리 아래 bin), ../etc(상위 디렉터리의 etc).

터미널 exit로 빠져나오는 시나리오

실행결과

그림 2-28. ubuntu는 -d만으로는 바로 종료

'어 왜 실행 목록에 없지?'

실행 중인 프로세스 목록에 ubuntu가 없었습니다. 컨테이너가 실행되자마자 바로 종료됐기 때문입니다. 반면 이전에 실습했던 nginx는 똑같은 -d 옵션으로도 잘 돌아갔습니다.

원인을 찾다 메인 프로세스의 개념을 알게 되었습니다. 도커 컨테이너는 시작과 동시에 실행되는 메인 프로세스가 종료되면 컨테이너도 함께 종료됩니다. 이 관점에서 두 이미지를 비교해 보니 차이가 분명했습니다.

먼저 nginx는 '웹 서버로서 요청을 대기하는 것' 이 목적입니다. 그래서 메인 프로세스인 nginx는 요청이 올 때까지 계속 깨어 있는 데몬으로 동작합니다. 백그라운드에서도 '대기'라는 할 일이 분명히 있기 때문에 프로세스가 죽지 않고 컨테이너가 계속 살아 있습니다.

데몬(Daemon): 백그라운드에서 요청을 기다리며 계속 실행되는 프로그램입니다. nginx나 데이터베이스처럼 '서버' 역할을 수행하며, 요청을 기다리는 것 자체가 일이라 터미널이 없어도 종료되지 않습니다.

반면 ubuntu는 '사용자에게 명령어를 입력받는 것' 이 목적입니다. 그래서 메인 프로세스인 bash는 사용자의 입력을 받는 터미널이 있어야 합니다. -d 옵션을 줘서 터미널이 연결되지 않으면, bash는 더 이상 할 일이 없다고 판단해 종료됩니다.

bash(셸): 사용자의 명령어를 운영체제에 전달하는 창구입니다. 입력받을 터미널이 없으면 할 일이 사라져 프로세스가 종료됩니다.

ubuntu를 백그라운드로 살리려면 -dit(-d 백그라운드, -i 입력 받기, -t 터미널 화면)을 쓰거나 메인 프로세스를 다른 것으로 바꿔야 합니다.

2.7.4 attach vs exec

오픈이는 백그라운드로 띄운 컨테이너에 다시 들어가는 방법이 궁금해졌습니다. 자료를 살펴보니 두 가지가 있었습니다. 메인 프로세스에 직접 연결하는 attach와, 새 프로세스를 하나 더 띄우는 exec입니다. 결과는 비슷해 보이지만 내부에서 일어나는 일이 다르다고 해서, 두 방법을 직접 비교해 봤습니다.

먼저 attach는 컨테이너 안에서 이미 돌고 있는 메인 프로세스를 연결합니다. 같은 프로세스를 공유하기 때문에 여기서 잘못 빠져나와 프로세스를 종료하면 컨테이너 전체가 같이 꺼집니다.

그림 2-30. attach는 이미 떠 있는 PID 1 프로세스에 터미널을 직접 꽂는 방식

컨테이너의 메인 프로세스는 PID 1로 식별됩니다.

먼저 ubuntu를 백그라운드로 띄우고, 그 컨테이너에 docker attach로 접속했습니다.

터미널 -dit 조합으로 ubuntu 살려 두기
docker run -dit ubuntu # ubuntu 백그라운드 실행 docker ps # 실행중인 컨테이너 확인 docker attach d2b1 # attach로 접근

ubuntu의 입력 터미널이 나타났습니다. attach는 메인 프로세스에 직접 연결되므로, 여러 터미널에서 붙어도 모두 같은 화면을 공유합니다. 다만 이 상태에서 잘못 빠져나와 메인 프로세스를 종료하면 컨테이너 전체가 꺼집니다. 그래서 CTRL+P → CTRL+Q로 안전하게 빠져나왔습니다.

이번에는 같은 컨테이너에 exec로 다시 들어가 봤습니다.

그림 2-32. exec는 컨테이너 안에 새 bash 프로세스를 띄우고 내 터미널에 연결하는 방식

그림처럼 exec는 메인 프로세스에 직접 연결하는 대신, 컨테이너 안에 별도의 bash 프로세스를 새로 하나 띄우고 거기에 내 터미널을 연결합니다. 메인 프로세스와 분리되어 있어 이쪽에서 exit로 나와도 메인 프로세스는 종료되지 않습니다.

터미널 실행 중 컨테이너에 새 bash 접속
docker exec -it d2b1 bash # 실행 중인 컨테이너에 새 bash 접속
실행결과

그림 2-35. 프로세스 목록 확인

PID 1이 컨테이너의 메인 프로세스이고, PID 12가 방금 exec로 만들어진 프로세스입니다. 두 프로세스는 같은 컨테이너 안에서 환경을 공유하지만 터미널 세션은 서로 독립입니다. 그래서 exec에서 선언한 임시 변수가 메인 프로세스 세션에서는 보이지 않고, 반대도 마찬가지입니다.

방식 동작 위험성
docker attach <ID> 메인 프로세스(PID 1) 에 직접 연결 잘못 빠져나오면 메인이 종료되고 컨테이너도 꺼짐
docker exec -it <ID> bash 새 프로세스를 만들어 접근 안전. 빠져나와도 메인에 영향 없음

'attach는 메인 프로세스에 직접 붙는 방식이고, exec는 새 프로세스를 만들어 접근하는 방식이구나. 앞으로는 exec만 써야겠다.'

여기까지 공부하고 나니 컨테이너의 생명 주기는 더는 헷갈리지 않았습니다. 오픈이는 의자에서 일어나 잠깐 어깨를 풀고 다시 노트북 앞에 앉았습니다. 그러다 한 가지가 마음에 걸렸습니다. 앞서 Ubuntu 컨테이너 안에 패키지를 깔고 파일도 몇 개 만들어 두었습니다. 컨테이너를 내리는 순간 그 작업이 다 사라진다면 다음에 똑같은 걸 또 깔아야 합니다.

'내가 수정한 이 상태를 기록해 두려면 어떻게 해야 하지.'

필요한 건 지금 컨테이너 상태를 그대로 기록해 두는 방법입니다.

2.8 commit - 실행 중인 컨테이너를 이미지로

그 일을 하는 명령이 docker commit입니다. 이 명령어는 컨테이너의 현재 상태를 그대로 이미지로 저장합니다.

지금까지 오픈이는 남이 올려둔 이미지(ubuntu, nginx, tomcat)를 Docker Hub에서 받아 쓰기만 했습니다. 반대로, 직접 수정한 컨테이너를 이미지로 만들어 Docker Hub에 올릴 수도 있습니다. 그러면 다른 컴퓨터에서도 이 이미지를 받으면 같은 상태로 쓸 수 있습니다.

2.8.1 Tomcat으로 commit 실습하기

자기만의 이미지를 만들기 위해 Tomcat 이미지를 받아 컨테이너를 실행했습니다.

터미널 Tomcat 컨테이너 실행

실행 후 브라우저에서 localhost:8080 으로 접속했더니 404가 떴습니다.

2.9 마운트로 데이터 보존

한숨 돌리고 계속 공부하던 오픈이는 한 가지를 발견했습니다. 방금 commit으로 만든 이미지에 담기는 건 컨테이너의 초기 상태뿐이라는 점입니다.

컨테이너가 실행된 뒤 새로 쌓이는 데이터는 이미지에 저장되지 않습니다. 그래서 컨테이너가 삭제되는 순간 함께 사라져 버립니다. 데이터베이스처럼 실시간으로 데이터를 쌓는 컨테이너가 삭제되면, 그동안 저장한 데이터도 통째로 날아갑니다.

'컨테이너를 내려도 데이터는 남아 있어야 하는데.'

결국 데이터를 잃지 않으려면 컨테이너 외부에 보관해야 합니다. 이를 위해 컨테이너와 외부 저장소를 연결해 주는 기능이 바로 마운트입니다.

Docker의 마운트는 두 가지 방식이 있습니다.

2.9.1 바인드 마운트: 호스트 폴더와 직접 연결

먼저 해 본 방식은 호스트 PC의 특정 폴더를 컨테이너 내부의 경로와 연결하는 바인드 마운트입니다. USB를 컴퓨터에 꽂으면 그 안의 파일을 그대로 볼 수 있듯, 컨테이너도 호스트 PC의 폴더를 꽂아서 쓰는 방식입니다.

한쪽에서 파일을 수정하면 다른 쪽에서도 즉시 확인됩니다. USB를 뽑아도 안에 든 파일이 사라지지 않듯, 컨테이너를 삭제해도 파일은 호스트 PC에 그대로 남습니다.

바인드 마운트(Bind Mount): 호스트 PC의 특정 폴더를 컨테이너 안 경로와 직접 연결하는 마운트 방식입니다. 사용자가 지정한 폴더를 컨테이너가 그대로 빌려 쓰기 때문에, 한쪽의 변경 사항이 실시간으로 양쪽에 반영됩니다. 어느 폴더를 쓸지는 사용자가 직접 지정합니다.

docker run -it --mount type=bind,src=<호스트 경로>,dst=<컨테이너 경로> <이미지> 형태로 명령어를 작성합니다.

그림 2-42. 호스트 폴더와 컨테이너 경로가 같은 데이터를 바라보도록 묶인 상태

오픈이는 호스트의 /c/app/bind 경로에 폴더를 하나 만들고, 컨테이너 안의 경로와 직접 연결했습니다.

터미널 bind mount 폴더 준비와 마운트 실행
# 호스트 PC에 공유할 폴더 생성 mkdir -p /c/app/bind # 호스트 폴더(src)와 컨테이너 경로(dst)를 연결 docker run -it --mount type=bind,src=/c/app/bind,dst=/app/bind ubuntu

macOS·Linux 사용자는 호스트 경로를 ~/app/bind로 바꿔 사용합니다.

단계명령
폴더 생성mkdir -p ~/app/bind
마운트 실행docker run -it --mount type=bind,src=$HOME/app/bind,dst=/app/bind ubuntu
확인ls ~/app/bind

컨테이너 안에서 a.txt를 만들고, 호스트 쪽 폴더를 확인해 봤습니다.

터미널 bind mount 양방향 동기화 확인
touch /app/bind/a.txt # 컨테이너 안에서 파일 생성 ls /c/app/bind # 호스트 터미널에서 확인
실행결과
$ ls /c/app/bind
a.txt
그림 2-43. 호스트 PC에서 같은 파일이 보이는 것을 확인

같은 파일이 양쪽에 있었습니다. 호스트에서 소스 코드를 고치면 컨테이너에 바로 반영되니까, 개발할 때 쓰기 딱 맞는 방식입니다.

2.9.2 볼륨 마운트: Docker가 관리하는 저장소

바인드 마운트를 써 보니 신경 쓰이는 점이 하나 있었습니다. 호스트의 어느 폴더를 쓸지, 경로를 어떻게 정리해 둘지 사용자가 직접 관리해야 한다는 점입니다. 이 일을 Docker에 맡길 수는 없을까.

그림 2-44. Docker 엔진이 관리하는 내부 저장 공간에 데이터가 저장되는 구조

볼륨 마운트가 그 일을 합니다. Docker 엔진이 자기 안에 별도 저장 공간을 두고, 사용자는 볼륨에 이름만 붙여 컨테이너와 연결합니다. 그래서 실제 저장 위치는 Docker가 맡아서 관리합니다.

볼륨 마운트(Volume Mount): Docker 엔진이 자체적으로 관리하는 저장 공간(볼륨)을 컨테이너에 연결하는 마운트 방식입니다. 사용자는 볼륨 이름만 정하면 되고, 실제 저장 위치는 Docker가 알아서 관리합니다. 컨테이너가 사라져도 볼륨에 담긴 데이터는 그대로 남아 있어, 다른 컨테이너에서 같은 이름으로 다시 연결할 수 있습니다.

docker run -it --mount type=volume,src=<볼륨 이름>,dst=<컨테이너 경로> <이미지> 형태로 명령어를 작성합니다.

오픈이는 metacoding-volume이라는 이름으로 볼륨을 하나 만들면서 ubuntu 컨테이너를 띄웠습니다. 존재하지 않는 볼륨 이름을 주면 Docker가 자동으로 만들어 줍니다.

터미널 named volume 마운트로 ubuntu 실행
# 볼륨 마운트로 ubuntu 실행 docker run -it --mount type=volume,src=metacoding-volume,dst=/app/volume ubuntu
실행결과
$ docker run -it --mount type=volume,src=metacoding-volume,dst=/app/volume ubuntu
root@d6f096d57b3e:/#
그림 2-45. 볼륨 마운트로 ubuntu 실행

컨테이너 안에 들어가자 /app/volume 폴더가 자동으로 만들어져 있었습니다. 오픈이는 이 폴더에 빈 파일을 하나 만들고, 컨테이너를 빠져나와 볼륨이 그대로 남아 있는지 확인했습니다.

터미널 볼륨에 파일 생성
touch /app/volume/b.txt # 볼륨 안에 빈 파일 생성 exit # 컨테이너 종료 docker volume ls # 볼륨 목록 재확인
실행결과

컨테이너는 사라졌지만 볼륨은 목록에 그대로 남아 있었습니다. 이어서 같은 볼륨을 연결해 새 컨테이너를 다시 띄웠습니다.

터미널 같은 볼륨으로 새 컨테이너에서 데이터 확인

그림 2-47. 볼륨 데이터 재사용

방금 전에 만들었던 b.txt가 새 컨테이너 안에서도 그대로 보였습니다. 컨테이너를 내려도 데이터는 사라지지 않았습니다. 처음 던졌던 걱정 한 줄이 풀렸습니다.

오픈이는 하루 사이에 익힌 키워드를 정리했습니다. 격리, 이미지, 포트포워딩, CMD, 볼륨이 머릿속에 정리됐습니다. 어제 두 시간을 흘려보냈던 그 에러도 이제는 어디에서 막혔던 건지 짚을 수 있을 것 같았습니다.

다만 한 가지가 남았습니다. 오늘 띄운 건 컨테이너 한 대였습니다. 실제 회사에서 다루는 사내 도구는 화면 하나에도 프론트엔드, 백엔드, 데이터베이스가 같이 굴러갑니다. docker run 명령을 컨테이너마다 따로 친다고 생각하니 명령줄이 길어지는 그림이 먼저 떠올랐습니다.

'한 대는 됐다. 그런데 세 대를 함께 띄우려면 이 명령줄을 매번 칠 수는 없을 텐데.'

식은 커피 잔을 옆으로 밀어두고 노트북 뚜껑을 닫았습니다. 창밖이 어느새 어두워져 있었습니다.

이것만은 기억하자

  • 컨테이너는 격리된 프로세스입니다. 파일시스템·네트워크·프로세스 공간만 따로 쪼개고, 커널은 호스트와 공유합니다. VM보다 가볍고 빨리 뜹니다.
  • 이미지는 설계도, 컨테이너는 찍혀 나온 결과입니다. 붕어빵 틀 하나로 여러 붕어빵을 찍듯, 이미지 하나로 여러 컨테이너를 찍어냅니다.
  • 컨테이너 통신은 docker0가 중심입니다. 컨테이너끼리는 docker0로 묶이고, 외부 요청은 포트포워딩으로 들어옵니다.
  • 메인 프로세스가 살아 있어야 컨테이너가 살아 있습니다. 이미지의 CMD에 앱 실행 명령을 지정해 두는 이유가 여기에 있습니다.
  • 마운트는 컨테이너와 외부 저장소를 잇습니다. 바인드는 호스트의 특정 폴더를, 볼륨은 Docker 엔진이 관리하는 저장소를 컨테이너에 연결합니다.

다음 챕터에서는 한 번에 컨테이너 한 대가 아니라, 프론트엔드·백엔드·DB가 함께 뜨는 실제 서비스 구성을 다룹니다. Dockerfile로 환경을 자동으로 만들고 Docker Compose로 여러 컨테이너를 한 번에 묶어 올려 보겠습니다.
