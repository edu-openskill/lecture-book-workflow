샘플 미리보기 v9 — 4.1.4를 컨트롤 플레인 명령 흐름에만 집중. 컨테이너 엔진·kube-proxy 박스 제거(kube-proxy는 5장 네트워킹에서 다룸). 워커 노드는 kubelet과 Pod 두 박스만. 챕터 파일은 아직 수정되지 않았습니다.

4.1.4 Kubernetes의 동작 원리 (샘플)

신규 가맹점 하나가 문을 여는 과정은 크게 신청 → 판단 → 오픈 세 단계로 흘러갑니다. 쿠버네티스도 똑같은 세 단계로 Pod 하나를 띄웁니다. 하나씩 들여다보겠습니다.

1. 신청 — 개발자가 본사 안내데스크에 명령을 낸다
개발자가 kubectl로 명령을 입력하면 그 요청은 가맹 희망자가 신청서를 들고 본사 안내데스크에 들르는 것과 같습니다. 외부에서 들어오는 모든 요청은 가장 먼저 Kube API Server(안내데스크)를 통과합니다.

그림 4-4. 신청 단계 — 개발자의 명령이 본사 안내데스크에 도착하는 길

2. 판단 — 본사 부서들이 어디에 낼지 정하고 현황을 기록한다
안내데스크가 신청을 접수하면 그 내용을 etcd(기록실)에 보관합니다. 기록된 정보를 보고 Scheduler(출점 담당팀)는 새 가맹점을 어느 워커 노드에 낼지 정하고, Controller Manager(운영점검팀)는 클러스터가 원하는 상태대로 굴러가는지 끊임없이 비교합니다. 세 부서는 안내데스크를 통해 서로 상태를 주고받으며 결론을 좁힙니다.

그림 4-5. 판단 단계 — 안내데스크와 세 부서가 양방향으로 결론을 좁히는 모습

3. 오픈 — 현장 매니저가 받아 매장을 연다
결론이 정해지면 그 지시는 다시 Kube API Server를 거쳐 워커 노드의 kubelet에 도착합니다. kubelet은 현장 매니저처럼 본사의 지시를 받아 노드 안에서 Pod를 직접 만들고, 컨테이너의 생사·상태가 흐트러지지 않게 계속 챙깁니다. 가맹점이 실제로 문을 여는 단계입니다.

그림 4-6. 오픈 단계 — kubelet이 본사의 지시를 받아 Pod를 실제로 띄우는 모습

구성 요소비유역할
Kube API Server본사 안내데스크모든 요청이 가장 먼저 도달하는 입구입니다. 외부 명령과 내부 부서 간의 소통이 모두 이 통로를 거칩니다.
etcd본사 기록실클러스터의 원하는 상태와 현재 상태를 모두 기록·조회하는 데이터베이스입니다.
Scheduler출점 담당팀새 Pod를 어느 워커 노드에 배치할지 결정합니다.
Controller Manager운영점검팀원하는 상태와 실제 상태를 끊임없이 비교하다가 차이가 생기면 복구 작업을 지시합니다.
kubelet현장 매니저본사의 지시를 받아 노드 안에서 Pod를 만들고, 컨테이너가 살아 있는지·정의된 상태와 같은지 계속 점검해 유지합니다.

v8 → v9 변경 요약

  • 그림 4-6 워커 노드: kubelet + 컨테이너 엔진 + Pod + kube-proxy 4박스 → kubelet + Pod 2박스
  • kubelet 박스 확장 — 기능 2줄(본사 지시 받아 Pod 생성 / 컨테이너 생사·상태 점검) 노출
  • 그림 4-4·4-5의 워커 노드 흐림 박스도 kubelet/엔진/Pod/kube-proxy → kubelet/Pod만 흐림
  • 본문 3단계에서 "컨테이너 엔진(주방 시설)을 호출해…"·"입구의 kube-proxy가 들어오는 요청을 라우팅…" 문장 제거
  • 용어 테이블 6행 → 5행 (kube-proxy 제외 — 5장 네트워킹 챕터로 양보)
  • 그림 번호 시프트(4-5 이후 한 자리 더 밀림) 동일

5렌즈 자가 점검 (v9)

렌즈결과근거
① 연결PASS 4.1.4가 "명령 흐름"에만 집중 → 5장(네트워킹)이 "트래픽 흐름"을 받음. 책 전체 호 분담이 깔끔해짐. 단계 1·2·3이 신청→판단→오픈으로 가맹점 라이프사이클 유지. 단락 cold start "개발자가" / "안내데스크가" / "결론이"로 진입점 분산.
② 코드 정합PASS kube-proxy를 Pod 생성 흐름에서 분리한 것이 실제 동작에 더 충실. kubelet의 두 기능(Pod 생성 + 상태 점검)도 공식 동작 부합. ex 코드 변경 없음.
③ 라벨/AI투PASS 단계 도입은 본문 볼드 한 줄. "셈입니다·것입니다·이것이 바로" 없음. 5장 직접 참조 삭제 유지(샘플 배너 외 본문엔 챕터 번호 직접 언급 없음).
④ 과거형PASS 본문 모든 종결 ~합니다/~입니다. 분위기 보강 과거형 없음.
⑤ 의인화PASS "안내데스크·기록실·출점 담당팀·운영점검팀·현장 매니저"는 비유 캐릭터 범위. "kubelet이 Pod를 만들고 상태를 점검"은 시스템 동작 사실. 강한 인격 부여 없음.

결정 요청

  1. 그림 4-6 워커 노드 = kubelet + Pod 두 박스 구성 OK인지
  2. kubelet 박스의 기능 두 줄 표현(본사 지시 받아 Pod 생성 / 컨테이너 생사·상태 점검) 톤 OK인지
  3. kube-proxy를 4.1.4에서 완전히 빼는 결정 OK인지 (5장 네트워킹에서 본격 등장)
