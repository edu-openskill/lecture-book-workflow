4.1.4 Kubernetes의 동작 원리 (샘플)

개발자가 명령을 내릴 때 쿠버네티스 내부에서 어떻게 처리되는지 흐름을 살펴보겠습니다.

본사에 가맹 신청이 들어오면 안내데스크가 받아 내부 부서들이 검토하고, 결정된 내용이 현장 운영팀을 거쳐 새 매장으로 내려갑니다.

그림 4-4. 본사가 가맹 신청을 받아 현장에서 새 매장을 여는 길

쿠버네티스에서는 가맹 신청을 넣는 사람이 개발자, 안내데스크가 Kube API Server, 본사 부서들이 etcd·Scheduler·Controller Manager, 현장 운영팀이 워커 노드에 해당합니다. 같은 흐름에 실제 이름을 붙여 보겠습니다.

v1 → v2 변경 요약

  • 오른쪽 가맹점 1·2·3 박스 3개 → 현장 운영팀 A·B 박스 2개 + 각 안에 "신규 가맹점" (이미 있는 매장이 아니라 "이제 생기는" 모습)
  • 안내데스크 비유 유지
  • 그림 4-3(워커 노드 2대)과 현장 운영팀 2곳이 1:1 매칭
  • 화살표 라벨: "가맹 신청" / "내부 검토" / "개점 지시" 유지
  • 그림 아래 매핑 문장에 "현장 운영팀 = 워커 노드" 추가
