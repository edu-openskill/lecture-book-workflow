샘플 미리보기 v15 — 비유를 추상적으로 정리. "가맹 희망자가 신청서를 들고 안내데스크에 들른다" 같은 구체적 장면 묘사 제거하고 "프랜차이즈 본사에 가맹점을 신청한다" 같은 추상 행위만 노출. 도입·1단계·3단계 마무리 정리. 챕터 파일은 아직 수정되지 않았습니다.

4.1.4 Kubernetes의 동작 원리 (샘플)

개발자가 명령을 내려 Pod 하나가 만들어지는 일은 프랜차이즈 본사가 새 가맹점을 여는 흐름과 비슷합니다. 크게 신청 → 판단 → 지시 세 단계로 일어납니다. 하나씩 들여다보겠습니다.

1. 신청 — 개발자가 클러스터에 명령을 보낸다
개발자가 클러스터에 명령을 보내는 일은 프랜차이즈 본사에 가맹점을 신청하는 것과 같습니다. 외부에서 들어오는 모든 요청은 가장 먼저 Kube API Server(안내데스크)를 통과합니다.

그림 4-4. 신청 단계 — 개발자의 명령이 Kube API Server에 도착하는 길

2. 판단 — 컨트롤 플레인 부서들이 협의해 결론을 좁힌다
Kube API Server(안내데스크)가 신청을 접수하면 그 내용을 etcd(기록실)에 보관합니다. 기록된 정보를 보고 Scheduler(출점 담당팀)는 새 Pod를 어느 워커 노드에 띄울지 정하고, Controller Manager(운영점검팀)는 클러스터 상태가 원하는 모습으로 유지되는지 끊임없이 비교합니다. 세 부서는 Kube API Server를 통해 서로 상태를 주고받으며 결론을 좁힙니다.

그림 4-5. 판단 단계 — Kube API Server와 세 부서가 양방향으로 결론을 좁히는 모습

3. 지시 — 컨트롤 플레인이 워커 노드 kubelet에 지시한다
결론이 정해지면 그 지시는 다시 Kube API Server를 거쳐 워커 노드의 kubelet(현장 매니저)에 도착합니다. kubelet은 노드 안에서 Pod를 직접 만들고, 컨테이너의 상태가 정의된 모습에서 벗어나지 않게 계속 점검합니다. 여기까지 오면 Pod가 실제로 생성됩니다.

그림 4-6. 지시 단계 — kubelet이 컨트롤 플레인의 지시를 받아 Pod를 띄우는 모습

구성 요소비유역할
Kube API Server본사 안내데스크모든 요청이 가장 먼저 도달하는 입구입니다. 외부 명령과 내부 부서 간의 소통이 모두 이 통로를 거칩니다.
etcd본사 기록실클러스터의 원하는 상태와 현재 상태를 모두 기록·조회하는 데이터베이스입니다.
Scheduler출점 담당팀새 Pod를 어느 워커 노드에 배치할지 결정합니다.
Controller Manager운영점검팀원하는 상태와 실제 상태를 끊임없이 비교하다가 차이가 생기면 복구 작업을 지시합니다.
kubelet현장 매니저본사의 지시를 받아 노드 안에서 Pod를 만들고, 컨테이너가 살아 있는지·정의된 상태와 같은지 계속 점검해 유지합니다.
