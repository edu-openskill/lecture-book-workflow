현재 · v8

4.3.4 Pod 조회

생성 명령을 내렸다면 이제 상태를 확인할 차례입니다.

실행결과
$ kubectl get pod
NAME READY STATUS RESTARTS AGE
hello-pod1 1/1 Running 0 8m39s
hello-pod2 1/1 Running 0 23s
그림 4-12. Pod 목록 조회 결과

'Pod 두 개가 떴다.'

그림 4-13. Pod 상세 조회

노트북에 두 개의 Pod가 나란히 떴습니다. 그런데 팀장이 던졌던 첫 번째 질문이 다시 마음에 걸렸습니다.

'근데 이것도 죽어 버리면 도커랑 차이가 없는데?'

4.4.1 Pod 하나를 직접 만들면 생기는 문제

방금 만든 Pod로 작은 실험을 해 봤습니다. 운영 중에 누군가의 실수로 Pod가 삭제되거나, 프로그램 오류로 종료된다면 어떤 일이 벌어질까요.

실행결과
$ kubectl delete pod hello-pod1
pod "hello-pod1" deleted
$ kubectl get pod
NAME READY STATUS RESTARTS AGE
hello-pod2 1/1 Running 0 19m
그림 4-14. Pod 삭제 후 목록을 조회하면 hello-pod1이 사라져 있습니다

hello-pod1은 목록에서 사라졌고, 시간이 지나도 다시 나타나지 않았습니다.

'이게 팀장님이 말씀하신 상황이구나.'

제안

4.3.4 Pod 조회

생성 명령을 내렸다면 이제 상태를 확인할 차례입니다. 이 화면은 잠시 뒤 실패를 확인할 때 다시 꺼내 비교할 기준입니다.

실행결과
$ kubectl get pod
NAME READY STATUS RESTARTS AGE
hello-pod1 1/1 Running 0 8m39s
hello-pod2 1/1 Running 0 23s
그림 4-12. 정상 상태 - hello-pod1·hello-pod2가 모두 Running

STATUS 칸에 두 Pod 모두 Running이 찍혀 있습니다.

'Pod 두 개가 떴다. 그런데 이것도 죽어 버리면 도커랑 뭐가 다르지?'

왜 바꾸나 · describe는 정상 Pod 메타데이터 덤프라 실패 흐름에 기여하지 않습니다. 이 자리에서는 표 한 줄 소개로 줄이고, describe는 4.4.1에서 "정말 사라졌다"를 증명하는 도구로 옮깁니다.

4.4.1 Pod 하나를 직접 만들면 생기는 문제

팀장이 던진 새벽 장애 질문이 계속 걸렸습니다. 직접 만든 Pod 하나를 강제로 지워 보고, 누가 살려 주는지 직접 확인하기로 했습니다.

실행결과
$ kubectl delete pod hello-pod1
pod "hello-pod1" deleted
$ kubectl get pod
NAME READY STATUS RESTARTS AGE
hello-pod2 1/1 Running 0 19m
그림 4-13. hello-pod1을 지우자 목록에서 사라졌습니다 (그림 4-12와 비교)

조금 전 그림 4-12에서는 hello-pod1과 hello-pod2가 나란히 Running이었습니다. 지금 목록에는 hello-pod2만 남았습니다.

'혹시 잠깐 사이에 다시 올라오지 않을까?'

1분을 기다린 뒤 다시 조회했습니다. describe로 hello-pod1을 직접 찾아봐도 결과는 같았습니다.

실행결과
$ kubectl get pod
NAME READY STATUS RESTARTS AGE
hello-pod2 1/1 Running 0 20m
$ kubectl describe pod hello-pod1
Error from server (NotFound): pods "hello-pod1" not found
그림 4-14. 1분 뒤에도 hello-pod1은 없고, describe조차 NotFound입니다

hello-pod2의 AGE만 19m에서 20m으로 늘었을 뿐, hello-pod1 자리는 비어 있습니다. describe로 직접 찾아도 서버는 "그런 Pod 없음"이라고 답합니다. 아무도 다시 띄워 주지 않았습니다.

'이게 팀장님이 말씀하신 상황이구나. 직접 만든 Pod는 죽으면 그걸로 끝이다.'

가맹점 하나가 폐업했는데도 본사가 손을 쓰지 않는 것과 같습니다. 직접 만든 Pod는 일회성이라, 사라져도 다시 채워 줄 책임자가 없습니다.

달라진 점 · ① 그림 4-12를 "비교 기준"으로 명시 · ② 1분 뒤 재조회로 "복구 안 됨"을 Tell 대신 Show · ③ describe를 실패 진단(NotFound)에 재배치 · ④ 캡션이 사실 나열에서 전환 문장으로 · ⑤ ex08(nginx)에 없는 가짜 크래시·Events 조작 없음, 모두 실제 재현되는 출력.
