# 3장. Redis 세션 공유

3장에서는 2개 이상의 서버가 Redis를 통해 같은 세션을 공유하는 구조를 알아봅니다. 해당 실습은 실행 단계에서 제공되는 깃 주소를 git clone 하고 docker compose up 서버를 실행합니다.

## 3.1 준비

### 전체 아키텍처 개요

본 프로젝트는 클라이언트의 요청이 경로(`Path`)에 따라 서로 다른 서버로 전달되지만, 세션은 공통된 Redis 저장소를 바라보게 하여 데이터 일관성을 유지하는 구조를 학습합니다.

### 요청 흐름

아래는 Client → Nginx → Spring → Redis 요청 흐름을 그려놓은 개념도입니다.

![image.png](3%EC%9E%A5%20Redis%20%EC%84%B8%EC%85%98%20%EA%B3%B5%EC%9C%A0/image.png)

요청 흐름은 클라이언트가 Nginx에 요청하면 경로에 따라 app1 또는 app2로 전달되고, 두 서버 모두 Redis에 세션을 저장/조회하는 구조입니다.

## 3.2 스프링 서버 준비

![image.png](3%EC%9E%A5%20Redis%20%EC%84%B8%EC%85%98%20%EA%B3%B5%EC%9C%A0/image%201.png)

서버 A와 서버 B는 로직이 동일하며, 요청이 들어오면 동일 세션의 count 값을 읽고 1 증가시킨 뒤 다시 세션에 저장합니다.

### **서버 A / 서버 B 역할**

- **Spring Server A(app1)**: /app1 경로로 들어오는 요청을 처리하며, 세션에서 count 값을 읽어 없으면 생성하고 있으면 +1 증가시킨 뒤 Redis에 저장합니다.
- **Spring Server B(app2)**: /app2 경로로 들어오는 요청을 처리하며, 동일한 세션 ID로 Redis에 저장된 count 값을 읽어 +1 증가시킨 뒤 다시 Redis에 저장합니다.

따라서 같은 브라우저에서 /app1 → /app2 → /app1 → /app2로 이동해도 count가 끊기지 않고 계속 증가하면 두 서버가 Redis를 통해 같은 세션을 공유하고 있다는 것을 확인할 수 있습니다.

---

### Redis 설정 (공통 설정)

두 서버는 동일한 Redis 호스트에 접속해야 하며 설정은 **application.properties,build.gradle**를 확인 합니다.

 **application.properties**

```java
# Spring Session의 세션 저장소를 Redis로 사용합니다(세션을 메모리 대신 Redis에 저장/공유).
spring.session.store-type=redis

# Redis 서버 접속 정보입니다. 환경변수(SPRING_REDIS_HOST)가 있으면 그 값을 쓰고, 없으면 기본값 redis를 사용합니다.
spring.data.redis.host=${SPRING_REDIS_HOST:redis}

# Redis 서버 포트입니다. 환경변수(SPRING_REDIS_PORT)가 있으면 그 값을 쓰고, 없으면 기본값 6379를 사용합니다.
spring.data.redis.port=${SPRING_REDIS_PORT:6379}
```

**build.gradle**

```java
// Redis를 Spring Data 방식(Template/Repository 등)으로 연결하고 Redis에 접근하기 위한 스타터
implementation 'org.springframework.boot:spring-boot-starter-data-redis'

// Spring Session이 HTTP 세션을 Redis에 저장/공유할 수 있도록 해주는 모듈(여러 서버에서 세션 공유 가능)
implementation 'org.springframework.session:spring-session-data-redis'
```

## 3.3 Nginx 라우터 준비

Nginx는 URL 경로에 따라 요청을 적절한 컨테이너로 분기합니다.

![image.png](3%EC%9E%A5%20Redis%20%EC%84%B8%EC%85%98%20%EA%B3%B5%EC%9C%A0/image%202.png)

---

### nginx 설정 (Path 기반 라우팅)

`nginx.conf` 파일의 핵심 라우팅 로직입니다.

```dart
events {
    worker_connections 1024;
}

http {
  upstream app1_upstream { server app1:8080; }   # app1 연결 대상
  upstream app2_upstream { server app2:8080; }   # app2 연결 대상

  server {
    listen 80;                                   # 80 포트 수신

    location /app1 { proxy_pass http://app1_upstream/; }
    location /app2 { proxy_pass http://app2_upstream/; }

    location = / { return 302 /app1/; }    
    
    proxy_redirect off;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;      # 기본 진입 경로는 app1으로
  }
}
```

경로가 `/app1/`이면 app1으로, `/app2/`이면 app2로 전달됩니다. 기본 접속(`/`)은 `/app1/`로 리다이렉트되어 있습니다.

## 3.4 Redis 공유 예제

이 예제는 **서버가 2대(Spring A, Spring B)로 나뉘어 있어도 세션이 끊기지 않는지**를 확인하는 데 목적이 있습니다. 요청이 Spring A로 가든 Spring B로 가든 Session ID가 같으면 Redis에서 동일 세션을 읽어 count가 이어집니다.

 어느 서버로 요청이 가더라도 같은 세션 상태를 읽어올 수 있게 만드는 구조입니다. 

![image.png](3%EC%9E%A5%20Redis%20%EC%84%B8%EC%85%98%20%EA%B3%B5%EC%9C%A0/image%203.png)

아래 코드는 세션 값을 읽고 저장하는 부분입니다.

**(확인) 경로: src/main/java/com/metacoding/spring_session_share_app1/HomeController.java**

```java
HttpSession session = request.getSession();

Integer count = (Integer) session.getAttribute("count");
count = (count == null) ? 1 : count + 1;
session.setAttribute("count", count);
```

세션 공유는 "같은 세션 ID로 다른 서버에 접근했을 때 상태가 유지되는가"로 확인할 수 있습니다.

---

응답 HTML에는 서버 이름과 방문 횟수가 출력되며 아래 템플릿에서 `{{server}}`, `{{count}}`가 어떻게 표시되는지 확인할 수 있습니다.

(확인)**경로:  src/main/resources/templates/index.mustache**

```html
<h1>{{server}}</h1>
<p>현재 세션 방문 횟수: {{count}}</p>
```

app1과 app2를 오가도 `count`가 계속 증가하면 Redis 세션 공유가 제대로 동작하고 있다는 뜻입니다.

## 3.6 정리

### 왜 서버 2대가 Redis를 공유해야 하는가?

![image.png](3%EC%9E%A5%20Redis%20%EC%84%B8%EC%85%98%20%EA%B3%B5%EC%9C%A0/image%208.png)

서버가 여러 대일 때 세션을 각 서버 메모리에만 저장하면 사용자가 다른 서버로 이동하는 순간 로그인 상태가 끊기게 됩니다.

- **문제점**: 유저가 서버 A에서 로그인했는데, 다음 요청이 서버 B로 전달되면 서버 B에는 로그인 정보가 없어 다시 로그인을 요구하게 됩니다.
- **해결책**: 서버 외부에 독립적인 **Redis**와 같은 저장소를 두어 세션 정보를 공유하면, 유저가 어떤 서버로 접속하더라도 중단 없는 서비스를 이용할 수 있습니다.
