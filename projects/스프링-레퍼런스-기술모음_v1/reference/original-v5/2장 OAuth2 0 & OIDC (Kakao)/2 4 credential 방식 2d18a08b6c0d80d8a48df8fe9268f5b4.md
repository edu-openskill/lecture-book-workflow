# 2.4 credential 방식

아래 내용은 코드를 그대로 따라 치는 실습이 아니라, Credential 방식이 실제로 어떤 구성요소로 이루어져 있고 로그인부터 토큰 검증까지 메시지가 어떻게 이동하는지를 흐름을 이해하는 데 목적이 있습니다. 

## **1) Credential이란?**

Credential(크리덴셜) 방식은 모바일 앱에서 로그인할 때 주로 사용하는 흐름입니다. Flutter, React Native, Android, iOS 같은 앱 환경에서 공식 SDK가 로그인 과정을 처리하고, 앱은 로그인 결과로 토큰을 받아서 사용합니다.

웹(브라우저)에서 흔히 보는 “인가코드(code) 받고 → 리다이렉트 받고 → 토큰으로 교환” 같은 과정은 앱에서는 대부분 SDK 내부에서 처리됩니다. 앱 개발자는 최종 결과(토큰)만 받아서 쓰면 됩니다.

---

## 2) **Credential 구조**

Credential 방식에서는 SDK가 OAuth/OIDC의 복잡한 중간 과정을 모두 처리하기 때문에, 앱은 토큰과 사용자 정보를 바로 받고 서버는 토큰의 유효성만 검증하면 됩니다.

즉, 웹에서 사용하는 **Authorization Code → Redirect → 교환 과정이** 앱 내부에서는 **SDK 안에서 자동으로 처리됩니다.**

![image.png](2%204%20credential%20%EB%B0%A9%EC%8B%9D/image.png)

Credential 방식은 아래 5단계로 이해하면 됩니다.

1. 앱이 SDK로 로그인 요청
2. 카카오/구글이 로그인 성공 후 토큰 발급
3. 앱이 토큰을 안전한 저장소에 저장
4. 앱이 서버 API 호출 시 토큰을 함께 전달
5. 서버가 토큰이 유효한지 확인하고 우리 서비스 로그인 완료 처리(세션/JWT 발급)

핵심은 “앱은 토큰을 받는다”, “서버는 토큰이 진짜인지 확인한다” 입니다.

---

## **3) 토큰 저장 방식**

카카오 로그인이 성공하면 앱은 access_token(그리고 상황에 따라 refresh_token, id_token)을 받습니다. 이 토큰들은 로그인 상태를 증명하는 값이기 때문에, 어디에 어떻게 저장하느냐가 중요합니다.

- **안전한 저장소에 저장합니다.** iOS는 Keychain, Android는 Keystore 기반 저장소를 사용합니다. Flutter에서는 보통 flutter_secure_storage 같은 패키지를 사용합니다.

로그인 성공 후 토큰을 보안 저장소에 저장하고, 앱 시작 시 저장된 토큰을 읽어 로그인 상태를 복원하며, 로그아웃 시 저장된 토큰을 삭제하는 흐름을 처리하는 코드 구현이 필요합니다.

---

## 4) **앱 로그인**

Flutter SDK에서는 카카오톡이 설치돼 있으면 카카오톡(loginWithKakaoTalk)으로 로그인하고, 실패하거나 설치돼 있지 않으면 카카오계정(loginWithKakaoAccount) 로그인으로 전환하여 로그인을 요청합니다.

로그인 성공 응답에서 토큰(access_token, 필요 시 id_token)을 추출하며, 검증 방식에 맞는 토큰만 서버로 전달해 서버 인증 처리를 완료하는 흐름의 코드 구현이 필요합니다.

여기서 받은 token 중에서 무엇을 서버에 보낼지는 “검증 방식”에 따라 달라집니다.

---

## **5) me()는 언제 쓰나?**

me()는 “앱에서 화면에 보여줄 프로필이 필요할 때” 사용하는 용도입니다. 즉, 앱 UI를 채우거나 로컬 상태에 저장하려는 목적입니다.

```dart
final me = await [UserApi.instance.me](http://userapi.instance.me/)();
// 화면 표시용: me.kakaoAccount?.profile?.nickname 등
```

중요한 점은, me() 호출 결과가 “우리 서버 로그인 성공의 최종 근거”가 되면 안 된다는 것입니다. 최종 확정은 서버가 토큰을 확인한 뒤에 합니다.

---

## **6) 토큰 검증 방식 2가지**

Credential의 토큰 발급에도 code방식과 마찬가지로 Access Token과 OIDC(OpenID connect) 두가지를 지원합니다.

---

### 1. **Access Token 검증 방식**

![image.png](2%204%20credential%20%EB%B0%A9%EC%8B%9D/image%201.png)

클라이언트(Flutter)가 앱에서 로그인 후 **me()로 사용자 정보를 조회**해 앱에서 사용합니다. 이후 로그인 완료 처리(우리 서비스 세션/JWT 발급 등)를 위해 **access_token만 우리 서버로 전달**하고, 서버가 카카오 API에 토큰 유효성 조회를 호출해서 검증합니다.

- **Flutter(앱) 사용자 조회(앱에서 사용 목적)**
    - 예: UserApi.instance.me()로 프로필/ID 등을 받아 화면 표시, 로컬 상태 저장 등에 사용
    - 이 단계는 “앱에서 사용자 정보 활용”이고, 서버 인증의 최종 근거는 아닙니다.
- **서버 검증**
    - **토큰 정보 보기(유효성/만료/앱 일치 확인)**
        
        GET https://kapi.kakao.com/v1/user/access_token_info
        
        요청 헤더: Authorization: Bearer {ACCESS_TOKEN}
        
        응답 주요 필드:
        
        - id: 카카오 회원번호(서비스 사용자 식별자)
        - expires_in: 남은 만료 시간(초)
        - app_id: 이 토큰이 발급된 카카오 앱 ID

---

### **2. ID Token(OIDC) 서명 검증 방식**

![image.png](2%204%20credential%20%EB%B0%A9%EC%8B%9D/image%202.png)

클라이언트(Flutter)가 앱에서 로그인 후 **me()로 사용자 정보를 조회**해 앱에서 사용합니다. OIDC가 활성화된 경우 로그인 응답 토큰에 **idToken(JWT)** 이 포함될 수 있고, 로그인 완료 처리(우리 서비스 세션/JWT 발급 등)를 위해 **id_token만 우리 서버로 전달**한 뒤, 서버가 **OIDC 공개키(JWKS)로 서명 검증 + 클레임 검증**을 수행해 최종 인증을 확정합니다.

- **Flutter: 사용자 정보 조회(앱에서 쓰는 용도)**
    - UserApi.instance.me()는 프로필/닉네임 같은 걸 받아서 **화면에 보여주거나 로컬 상태에 저장**하는 단계입니다.
- **Flutter → 서버: id_token만 전달**
    - 로그인 성공 후 받은 id_token을 우리 서버로 보냅니다.
    - 서버는 이 토큰을 가지고 “카카오가 진짜 발급한 토큰인지” 확인합니다.
- **서버: id_token 검증**
    
    **서명 검증: “카카오가 발급한 게 맞나?”**
    
    1. 카카오 설정 문서(Discovery)에서 jwks_uri 주소를 확인
    2. jwks_uri에서 공개키 목록(JWKS)을 가져옴
    3. id_token 헤더의 kid에 맞는 공개키를 골라 **RS256 서명 검증**
        
        → 여기까지 통과하면 “카카오가 발급한 토큰”임이 거의 확정됩니다.
        

---

## **7) 정리**

Credential 방식은 앱에서 SDK로 로그인을 처리하고, 앱은 토큰을 받아 저장합니다. 서버는 앱이 전달한 토큰이 유효한지 확인한 뒤 우리 서비스 세션/JWT를 발급합니다.

- me()는 앱 UI용으로 쓰는 호출이고, 서버 인증의 최종 근거가 아닙니다.
- 서버 검증은 Access Token 방식 또는 ID Token(OIDC) 방식으로 나뉩니다.
- 토큰은 안전한 저장소에 저장하는 것이 기본입니다.
