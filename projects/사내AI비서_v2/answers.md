## STEP 1: 씨앗

- **주제**: RAG 기반 사내 AI 비서 만들기 — LLM + 벡터DB + FastAPI로 사내 문서 검색 AI 구축
- **집필 동기**: 기존 RAG 자료가 너무 이론적이다 — 실전 프로젝트로 처음부터 끝까지 만들어보는 자료가 부족
- **저자 경험**: AI가 범용적으로 작성 — "처음 접하는 사람의 전형적 경험"으로 작성
- **핵심 메시지**: "직접 만들어봐야 이해한다" — 환각 체험 → 코드 작성 → 튜닝 실험까지 직접 해보는 여정
- **책의 의도**: RAG 전체 파이프라인 이해 — 문서 파싱 → 청킹 → 임베딩 → 검색 → 답변 생성의 흐름을 개념적으로 이해
- **의도 밖 범위**: Docker/PostgreSQL 세팅 상세, Langfuse/모니터링 상세, CSS/Jinja2 프론트엔드 UI 구현

## STEP 5: 챕터 집필

### CH01 변수 결정
- **LLM 비유**: 외부인 비유 — "회사 사정을 모르는 외부인에게 질문하는 것"
- **한 줄 기억**: 오픈북 시험 비유 — "RAG는 오픈북 시험이다"

### 챕터 번호 결정
- **시작 번호**: CH01부터 시작 (CH03이 아닌 CH01~CH10)

### CH02 변수 결정
- **이번 버전**: v0.2 — FastAPI로 직원/연차/매출 CRUD 시스템
- **이전 버전에서 변경**: v0.1(더미 문서 RAG) → v0.2(실제 PostgreSQL DB 기반 사내 시스템)
- **왜 해야 하나**: RAG로 문서 검색은 되지만, "김대리 연차 몇 개?"처럼 실시간 DB 데이터를 물어보면 답을 못 한다. AI 비서가 조회할 사내 시스템을 먼저 만들어야 한다.
- **비유**: 식당 비유 — API는 식당 웨이터, DB는 주방, 요청(주문)을 받아 데이터(음식)를 가져다준다
- **핵심 개념**: REST API, CRUD 패턴
- **챕터 성격**: 코드 없는 소개 챕터 — 완성 시스템을 실행하고 Swagger UI로 확인만
- **한 줄 기억**: "AI 비서가 조회할 사내 시스템을 만들었다. API는 웨이터처럼 요청을 받아 DB에서 데이터를 가져다준다."

### CH03 변수 결정
- **이번 버전**: v0.3 — 사내 문서 수집 전략과 표준 (코드 없는 개념 챕터)
- **도입**: 사내 문서를 VectorDB에 넣으려는데 PDF/DOCX/엑셀 형식이 제각각이고, 어떤 문서를 넣어야 할지도 모르겠다
- **비유**: 도서관 정리 — 책을 서가에 꽂는데 분류 기준이 없으면 나중에 찾을 수 없다. 메타데이터가 라벨이다
- **핵심 개념**: 문서 품질, 메타데이터, 청킹 전략 사전 설계, 재인덱싱 전략
- **한 줄 기억**: "AI에게 좋은 답을 원하면 좋은 문서를 넣어야 한다. 도서관처럼 분류하고 라벨을 붙이자."

### CH04 변수 결정
- **이번 버전**: v0.4 — VectorDB 구축 (파싱 → 청킹 → 임베딩 → ChromaDB 저장)
- **도입**: CH03에서 정리한 문서를 벡터DB에 넣으려는데, 통째로 넣으면 검색이 안 됨
- **비유**: 요리 비유 — 재료(문서) → 손질(파싱) → 다지기(청킹) → 양념(임베딩) → 냉장고 정리(벡터DB)
- **핵심 개념**: 문서 파싱, 청킹, 임베딩, 벡터 저장/검색, 임베딩 모델 선택 기준
- **한 줄 기억**: "문서는 조각내야 찾는다. 손질(파싱), 다지기(청킹), 양념(임베딩), 냉장고(벡터DB)."

### CH05 변수 결정
- **이번 버전**: v0.5 — RAG Q&A 엔진 (LCEL 파이프라인 + 출처 강제 + 멀티턴)
- **도입**: 동료의 반복 질문 공세 — 벡터 검색 결과를 줘도 "그냥 답을 알려줘"라고 함
- **비유**: 도서관 사서 — 질문받고 책 찾고 읽어서 답변하고 출처 알려주는 사서. 멀티턴은 메모장 가진 사서
- **핵심 개념**: LCEL 파이프라인, 출처 강제(Source Grounding), WindowMemory 멀티턴
- **한 줄 기억**: "검색은 재료, 답변은 요리다. LCEL 파이프라인이 이 레시피다."

### CH06 변수 결정
- **이번 버전**: v0.6 — 통합 에이전트 (QueryRouter + MCP 도구 + ReAct 에이전트)
- **도입**: 동료 복합 질문 — "진아 연차 얼마 남았어요? 연차 신청 절차도요?" RAG가 DB 데이터 답 못 함
- **비유**: 안내데스크 + 담당자 — QueryRouter는 안내데스크(질문 보고 어느 담당자로 연결), @tool은 각 담당자, 에이전트는 안내데스크처럼 일 분배
- **핵심 개념**: QueryRouter (3단계 라우팅), MCP 도구 (@tool 데코레이터), ReAct 에이전트 (AgentExecutor)
- **한 줄 기억**: "AI 비서는 안내데스크다. 질문을 듣고, 맞는 담당자를 찾아 정보를 가져온다."

### CH07 변수 결정
- **이번 버전**: v0.7 — 운영 안정화 (ResponseCache + TokenTracker + Retry + ConnectHRAgent)
- **도입**: 동료의 불만 — 같은 질문에 또 20초, 비용 추적 불가, 에러 시 멈춤
- **비유**: 사서의 메모장 + 업무 일지 — 답변 메모장(ResponseCache), 임베딩 메모장(EmbeddingCache), 업무 일지(TokenTracker), 재시도 매뉴얼
- **핵심 개념**: ResponseCache(TTL), EmbeddingCache, TokenTracker, ConnectHRAgent
- **한 줄 기억**: "같은 질문엔 캐시로 답하고, 운영은 기록에서 시작한다."

### CH08 변수 결정
- **이번 버전**: v0.8 — 검색 품질 튜닝 (청킹/Retriever/리랭킹/하이브리드 실험 4개)
- **이전 버전에서 변경**: ex07 기본 RAG → 검색 품질 최적화
- **왜 해야 하나**: 정확도 문제 — '연차 규정' 물어봤는데 '출장 규정'을 가져온다. 문서가 많아지면 엉뚱한 결과가 섞인다.
- **비유**: 도서관 사서의 검색 실력 — 처음엔 제목만 보고 찾던 사서가, 목차도 보고, 내용도 훑고, 최종 확인까지 하게 된다.
- **핵심 개념**: 청킹 최적화, 리랭킹, 하이브리드 검색
- **실패담**: 규정 혼동 — '병가 규정' 물어봤는데 '출장 규정'을 가져와서 답변이 완전 엉뚱. 키워드가 비슷해서 생긴 문제.
- **한 줄 기억**: "검색이 바뀌면 답변이 바뀐다. RAG의 품질은 LLM이 아니라 Retriever가 결정한다."
- **추가 설명**: Semantic Chunking 원리 (임베딩 기반 청킹이 왜 Fixed-size보다 나은지)

### CH09 변수 결정
- **이번 버전**: v0.9 — 질문+답변 고급 튜닝 (Parent Retriever, HyDE, Multi-Query, Evidence)
- **이전 버전에서 변경**: ex08 검색 튜닝 → 질문 이해 + 답변 품질
- **왜 해야 하나**: 질문 해석 불량 — '연차 규정'이라고 물었는데 '연차'라는 단어만 매칭해서 한계가 있다. 의도를 이해해야 한다.
- **비유**: 사서의 언어 능력 — 처음엔 단어만 알아듣던 사서가, 문장의 의도를 파악하고, 근거를 대며 답변하게 된다.
- **핵심 개념**: Parent Document Retriever, HyDE, 답변 근거(Evidence)
- **실패담**: 동의어 못 찾음 — '휴가'라고 물어봤는데 문서에는 '연차'라고 써져있어서 못 찾음. 같은 뜻인데도.
- **한 줄 기억**: "질문을 바꿔야 답변이 달라진다. RAG에서 진짜 병목은 LLM이 아니라, 질문을 어떻게 해석하느냐다."

### CH10 변수 결정
- **이번 버전**: v1.0 — 고급 문서 처리 + 평가 프레임워크 (Vision LLM, OCR, Precision@k)
- **이전 버전에서 변경**: ex09 텍스트만 → 이미지까지 + 정량 평가
- **왜 해야 하나**: 이미지 속 정보 손실 — PDF 조직도, 표가 텍스트로 변환되면 정보가 날아간다. 이미지 자체를 이해해야 한다.
- **비유**: 사서의 눈 + 성적표 — 글만 읽던 사서가 그림도 볼 수 있게 됐다. 그리고 성적표로 실력을 수치화한다.
- **핵심 개념**: OCR, Vision LLM, Precision@k
- **실패담**: PDF 표가 깨짐 — PDF에서 표를 추출했는데 셀이 다 합쳐져서 의미 없는 텍스트가 됨. 이미지로 보면 한눈에 답인데.
- **한 줄 기억**: "측정해야 개선할 수 있다. 느낌이 아니라 숫자로 품질을 측정해야 진짜 개선할 수 있다."
- **추가 설명**: 전체 프로젝트 회고 (CH01~CH10 여정 돌아보기)
