9.1 "휴가"라고 물었는데 "연차유급휴가"를 못 찾는다

그림 9-1. 질문 해석 튜닝. 질문을 바꾸고 결과를 가공해 검색의 앞뒤를 매만집니다

챕터 8에서 검색 엔진 자체를 튜닝했습니다. 청킹, 리랭커, 하이브리드. 검색 정확도가 꽤 올랐어요. 동료가 다시 테스트합니다.

동료: "휴가 규정 알려줘."

답변이 이상했습니다. "복리후생 안내" 문서의 단편이 나왔고, 정작 찾고 싶던 취업규칙 제15조 연차유급휴가는 빠졌어요. 문서에는 "연차유급휴가"라고 적혀 있고 동료는 "휴가"라고 물었습니다. 같은 뜻인데 단어가 다른 상황입니다.

동료: "WFH 정책이 뭐야?"

"관련 문서를 찾지 못했습니다."

문서에는 "재택근무"라고 적혀 있습니다. WFH → 재택근무를 연결하지 못한 것이죠. 챕터 8에서 푼 건 "어떤 문서를 가져오느냐" 였습니다. 지금 문제는 다릅니다. 검색 엔진이 잘 돌아도 질문을 이해 못 하면 소용이 없어요.

9.2 사서의 언어 능력을 키운다

챕터 8의 사서는 서가 정리 법을 배웠습니다. 이번 챕터의 사서는 언어 능력을 키웁니다.

첫째, 단어를 번역합니다. "쉬는 날" → "연차유급휴가". 약어, 동의어 확장입니다.

둘째, 답을 먼저 상상합니다. "쉬는 날 규정이라면… 1년 근속 시 15일 유급휴가 같은 내용이겠지." 그 상상한 답과 비슷한 실제 문서를 찾습니다. HyDE(Hypothetical Document Embeddings). 질문이 아니라 상상 답변으로 검색.

셋째, 질문을 여러 갈래로 바꿉니다. "휴가 규정" → "연차 사용 방법", "유급휴가 일수", "휴가 신청 절차". 각각 검색 후 합집합. Multi-Query.

넷째, 찾은 조각의 원본 전체를 꺼냅니다. "연차 신청은 3일 전 서면" 같은 짧은 청크가 걸리면, 그 청크가 속한 원 문서(제15~16조) 전체를 반환. Parent Document Retriever.

다섯째, 긴 결과를 압축합니다. 검색된 문서에서 질문과 관련된 문장만 추려 LLM에 넘깁니다. Contextual Compression.

그림 9-2. 챕터 8이 서가 정리였다면, 챕터 9는 사서의 언어, 요약 능력입니다

9.4 실습 1: Parent Document Retriever (step1)

작은 청크로 검색하되, LLM에게는 그 청크가 속한 부모 문서 전체를 넘깁니다. 정확도(작은 청크)와 맥락(큰 문서)의 장점을 동시에 취합니다.

실습 1 tuning/step1_advanced_retriever/retrievers.py. ParentDocumentRetriever
from langchain.retrievers import ParentDocumentRetriever from langchain_text_splitters import RecursiveCharacterTextSplitter def build_parent_retriever(vectorstore, docstore) -> ParentDocumentRetriever: # 작은 청크(검색용): 400자 child_splitter = RecursiveCharacterTextSplitter(chunk_size=400, chunk_overlap=50) # 큰 청크(반환용): 2000자 parent_splitter = RecursiveCharacterTextSplitter(chunk_size=2000, chunk_overlap=200) return ParentDocumentRetriever( vectorstore=vectorstore, docstore=docstore, child_splitter=child_splitter, parent_splitter=parent_splitter, )

docstore는 부모 문서를 id로 저장해 두는 in-memory 저장소입니다. 벡터 DB는 자식 청크를 가지고 있고, 검색 시 자식의 parent_id로 원본을 꺼내 반환합니다.

터미널 실험 1 실행
python -m tuning.step1_advanced_retriever
그림 9-4. Parent Doc. 작은 조각이 매칭돼도 반환은 큰 맥락 단위로 돌아옵니다

이어지는 Contextual Compression은 검색된 큰 문서에서 질문과 무관한 부분을 LLM이 잘라내 줍니다. 토큰 비용이 줄고 LLM이 핵심에 집중합니다.

그림 9-5. Compression 후. 관련 문장만 남아 프롬프트가 가벼워집니다

9.5 실습 2: HyDE / Multi-Query / 약어 확장 (step2)

HyDE: 상상 답변으로 검색

질문 대신 "이 질문에 어울릴 법한 답변" 을 LLM에 먼저 생성시키고, 그 가상 답변을 임베딩해 검색합니다. 답변과 답변 간 거리가 질문-답변 거리보다 가까운 경우가 많기 때문입니다.

실습 2 tuning/step2_query_rewrite/rewriters.py. HyDE
from langchain.prompts import PromptTemplate HYDE_PROMPT = PromptTemplate.from_template( "다음 질문에 대한 가상의 회사 규정 조항을 3~4문장으로 작성하세요.\n질문: {question}" ) def hyde_retrieve(question: str, llm, vectorstore, k: int = 5): hypothetical = llm.invoke(HYDE_PROMPT.format(question=question)).content # 실제 검색은 원 질문이 아니라 가상 답변으로 진행 return vectorstore.similarity_search(hypothetical, k=k)
그림 9-6. HyDE 적용 전/후. 상상 답변과의 유사도가 원 질문보다 높은 청크를 찾아옵니다

Multi-Query: 질문을 여러 갈래로

LLM에게 "이 질문을 다른 표현으로 3가지 만들어 줘"라고 시킨 뒤, 각 표현으로 따로 검색하고 합집합을 만듭니다. 단일 질문의 어휘 다양성 한계를 흡수합니다.

tuning/step2_query_rewrite/rewriters.py. MultiQuery
from langchain.retrievers.multi_query import MultiQueryRetriever multi_q = MultiQueryRetriever.from_llm(retriever=base_retriever, llm=llm) docs = multi_q.get_relevant_documents("휴가 규정 알려줘")
그림 9-7. 하나의 질문이 3개의 재작성으로 확장되어 각각 검색됩니다

약어, 동의어 확장

사내 용어 사전을 하나 만들어 두고, 질문에 약어가 포함되면 원 표현으로 치환합니다.

tuning/step2_query_rewrite/rewriters.py. 약어 확장
COMPANY_GLOSSARY = { "WFH": "재택근무", "PTO": "연차유급휴가", "OT": "초과근무", } def expand_abbreviations(question: str) -> str: for abbr, full in COMPANY_GLOSSARY.items(): question = question.replace(abbr, f"{abbr}({full})") return question
그림 9-8. "WFH" → "WFH(재택근무)"로 확장된 뒤 관련 문서가 잡힙니다

9.6 Self-Query Retriever: 메타데이터 필터를 LLM이 채우기

"2025년 인사팀 문서만" 같은 조건이 섞인 질문에서 LLM이 메타데이터 필터(department=인사팀, year=2025)를 자동으로 추출하게 할 수 있습니다. Self-Query Retriever가 LLM을 호출해 쿼리에서 조건을 뽑아 벡터 검색의 where 절에 심어 줍니다.

그림 9-9. Self-Query. LLM이 "2025년 인사팀"이라는 조건을 메타데이터 필터로 변환합니다

용어 정리

본문 속 표현 진짜 용어 정식 정의
"단어를 번역" 약어·동의어 확장 사내 용어 사전을 이용해 약어를 원 표현으로 치환
"답을 먼저 상상" HyDE 질문에 대한 가상 답변을 LLM이 먼저 생성하고, 그 답변으로 검색
"여러 갈래로 물어보기" Multi-Query 한 질문을 LLM이 여러 표현으로 재작성해 각각 검색 후 합집합
"원본 문서 전체 꺼내기" Parent Document Retriever 작은 청크로 검색하되 반환은 큰 부모 문서 단위
"핵심 문장만 추리기" Contextual Compression 검색된 문서에서 질문과 무관한 부분을 LLM이 제거
"조건을 필터로 변환" Self-Query Retriever LLM이 질문에서 메타데이터 조건을 뽑아 where 필터로 심음

이것만은 기억하자

  • 검색의 앞뒤를 다듬으면 같은 엔진이 더 똑똑해집니다. 챕터 8이 엔진 자체를 바꿨다면 챕터 9는 입력(쿼리)과 출력(결과)을 가공합니다. 인프라 변경 없이 체감 품질이 가장 크게 오르는 구간입니다.
  • 약어, 동의어 사전은 무조건 만드세요. 사내 용어는 공식 문서와 일상 표현이 다릅니다. "WFH ↔ 재택근무" 같은 매핑 한 줄이 실패 질문을 크게 줄여 줍니다.
  • Parent Doc + Compression 조합이 안정적입니다. 정확도는 작은 청크, 맥락은 부모 문서, 프롬프트 비용은 압축으로 잡는 전략입니다. 챕터 10에서는 텍스트로 못 잡는 이미지, 표 영역을 다룹니다.
