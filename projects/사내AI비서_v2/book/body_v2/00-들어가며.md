# 들어가며

어느 날 팀장이 이런 말을 합니다.

"AI로 사내 문서 검색 시스템 만들어봐."

일단 ChatGPT에 물어봅니다. "우리 회사 연차 규정이 어떻게 되나요?" AI는 자신 있게 답합니다. 연차는 15일이고, 3일 전까지 신청하면 된다고. 그런데 우리 회사 규정을 열어보면 전부 틀렸습니다. AI가 그럴듯하게 지어낸 겁니다.

이것이 LLM 환각(Hallucination)의 현실입니다.

> 환각(Hallucination): LLM이 학습 데이터에 없는 내용을 사실처럼 생성하는 현상입니다.

RAG(Retrieval-Augmented Generation)는 바로 이 문제를 풀기 위해 등장했습니다.

> RAG(Retrieval-Augmented Generation): 질문이 들어오면 관련 문서를 먼저 검색하고, 그 문서를 근거로 답변을 생성하는 방식입니다.

하지만 RAG를 처음 접하면 개념들이 낯섭니다. 임베딩, 벡터DB, 청킹, 리랭킹 — 읽을 때는 이해한 것 같아도, 막상 사내 문서를 넣고 질문하면 엉뚱한 답이 나옵니다. 직접 만들어봐야 비로소 왜 이렇게 구성하는지 체감할 수 있습니다.

그래서 이 책은 사내 AI 비서 **ConnectHR** 을 함께 만들면서 RAG의 전체 그림을 익힙니다. 처음에는 가장 단순한 프롬프트 방식으로 시작합니다. 그 한계를 직접 느끼고 나서, 문서를 파싱하고, 벡터DB에 넣고, RAG 파이프라인을 구성하고, 에이전트로 통합하고, 마지막으로 검색 품질을 튜닝하고 성적표로 측정합니다. 하나의 프로젝트가 챕터를 거듭하며 진화하는 여정입니다.

---

## 이 책이 맞는 분

- Python 기초 문법은 아는 분 (함수, 클래스, 패키지 설치)
- LLM이나 RAG는 처음이거나, 이론만 읽어본 분
- "조각난 예제 말고 하나의 프로젝트를 처음부터 끝까지 만들어보고 싶은" 분

## 한눈에 보기

| 챕터 | 핵심 질문 | 끝나면 뭐가 보이나 |
|------|----------|-----------------|
| CH01 | AI는 왜 거짓말을 할까? | 환각 vs RAG 응답 비교 |
| CH02 | AI가 조회할 DB는 어떻게 만들까? | CRUD API 동작 |
| CH03 | 어떤 문서를, 어떤 기준으로 넣을까? | 수집 규칙 확립 (개념) |
| CH04 | 문서를 어떻게 검색 가능한 형태로 바꿀까? | 벡터 검색 + 유사도 |
| CH05 | 검색 결과를 어떻게 답변으로 만들까? | 질문 → 답변 + 출처 |
| CH06 | DB 조회와 문서 검색을 한 번에 하려면? | **ConnectHR 탄생** |
| CH07 | 캐시, 비용 추적, 에러 처리는? | 운영 안정화 |
| CH08 | 왜 엉뚱한 문서를 가져올까? | 튜닝 전후 비교 |
| CH09 | 같은 뜻인데 왜 못 찾을까? | 쿼리 변환 + 근거 |
| CH10 | PDF 이미지는? 품질은 어떻게 측정할까? | **성적표 + v1.0 완성** |

세 가지 전환점이 이야기를 이끕니다.

1. **CH01** — LLM 환각을 보고, RAG가 필요하다는 걸 깨닫습니다.
2. **CH06** — RAG만으로는 부족하고, DB와 문서를 합쳐야 한다는 걸 깨닫습니다.
3. **CH08** — 기본 RAG로는 한계가 있고, 튜닝이 필요하다는 걸 깨닫습니다.

모든 챕터는 **실패에서 시작**합니다. 에러를 만나고, 원인을 찾고, 해결하는 과정이 곧 학습입니다.

각 챕터는 "이야기 파트"와 "기술 파트"로 나뉩니다. 이야기 파트에서 왜 이 기술이 필요한지를 상황과 비유로 먼저 보여주고, 기술 파트에서 코드로 구현합니다. 이야기 파트만 읽어도 전체 흐름을 이해할 수 있습니다.

## 권장 사양

이 책의 예제를 따라하려면 아래 환경이 필요합니다.

| 항목 | 권장 사양 | 비고 |
|------|----------|------|
| **Python** | 3.10 ~ 3.12 | 3.13 이상은 일부 호환성 문제 가능 |
| **RAM** | 16GB 이상 | 임베딩 모델 로딩 시 8GB 이상 사용 |
| **디스크** | 10GB 여유 | 모델 파일 + Docker 이미지 |
| **Docker** | Docker Desktop 설치 | PostgreSQL 실행에 필요 |
| **LLM** | Ollama + llama3.1:8b | 로컬에서 무료로 실행 |

CH06 이후 에이전트 기능(Tool Calling)을 사용하려면 Ollama 기준 `llama3.1:8b` 이상 모델이 필요합니다.

**로컬 환경이 어려운 경우**, OpenAI API(gpt-4o-mini)를 대안으로 사용할 수 있습니다. 각 예제의 `.env` 파일에서 `LLM_PROVIDER=openai` 로 변경하고 API 키만 설정하면 동일하게 동작합니다. Ollama 설치나 GPU 없이도 모든 챕터를 진행할 수 있습니다.

자세한 환경 설정 방법은 **부록 A. 환경 설정** 을 참고하세요.
