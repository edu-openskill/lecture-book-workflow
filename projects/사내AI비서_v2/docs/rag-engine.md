# ex11 RAG 엔진 — 교육용 해설

`src/pipeline.py`의 `RagPipeline`이 조립하는 질의 파이프라인 4단계를 데이터 흐름 관점에서 설명합니다. 각 단계가 **어떤 약점을 보완하러 들어갔는지**를 중심으로 읽어 주세요.

---

## 전체 그림 — 도서관 비유

```
사용자 질문
   │
   ▼
[1] QueryExpander        ← 사서가 "WFH 뜻 모를 수도 있으니 풀어 적자" (질문을 고쳐 씀)
   │
   ▼
[2] EnsembleRetriever    ← 의미 사서 + 단어 사서 두 명이 같이 찾아옴
   │
   ▼
[3] CrossEncoderReranker ← 전문 평론가가 후보를 꼼꼼히 다시 읽고 순위 매김
   │
   ▼
[4] ParentDocumentRetriever ← 자식 청크의 parent_id로 부모 페이지 원문 복원
   │
   ▼
 상위 k개 → LLM이 답변 생성
```

네 단계는 각자 **다른 약점**을 보완합니다. 하나씩 들여다봅니다.

---

## 1. QueryExpander — 질문을 다듬는 사서

### 이게 없으면 무슨 일이 생기나

문서에는 `"재택근무는 입사 6개월 이상 정규직에 한해 신청 가능"`이라고 적혀 있습니다. 그런데 사용자는 이렇게 물어요.

> `"WFH 정책 알려줘"`

벡터 검색은 단어 의미를 어느 정도 파악하지만, `"WFH"` 같은 약어는 학습 데이터에서 흔치 않으면 `"재택근무"`와의 유사도가 낮게 나옵니다. 검색 1위가 엉뚱한 문서가 될 수 있는 거죠.

### QueryExpander가 하는 일

`DEFAULT_ABBREVIATIONS` 사전을 훑어서 질문에 약어가 보이면 **괄호로 풀이를 덧붙입니다**. 원래 단어를 지우지 않고 **둘 다 남기는** 게 핵심이에요.

```
입력:  "WFH 정책 알려줘"
출력:  "WFH(재택근무) 정책 알려줘"
```

왜 둘 다 남기냐면, 문서에 `"WFH"`라고 적힌 곳도 있고 `"재택근무"`라고 적힌 곳도 있기 때문입니다. 확장된 질문은 **양쪽 표현을 동시에** 품게 됩니다.

### 실제 구현 (핵심 3줄)

```python
for abbr, full in dictionary.items():
    if abbr in query and full not in query:
        query = query.replace(abbr, f"{abbr}({full})")
```

사전을 생성자로 받기 때문에 부서별(법무팀·개발팀)·언어별로 다른 사전을 꽂을 수 있어요. 이게 도메인 객체로 분리된 이유입니다.

## 2. EnsembleRetriever — 두 사서가 함께 뽑기

### 벡터 검색 하나만 쓰면 생기는 구멍

벡터 검색(ChromaDB)은 **의미**를 잡는 데 강합니다. `"휴가"`와 `"쉼"`을 비슷한 걸로 봐요. 반대로 **정확한 기호·숫자·조항 번호**엔 약합니다. `"제4조"` 같은 검색은 `"제4조"`가 있는 문서보다 `"네 번째 항목"` 같은 해석이 먼저 나올 수 있어요.

BM25는 그 반대예요. **단어가 얼마나 특별하게 겹치는가**로 점수를 매깁니다. 일반 단어(조사·보편어)는 무시하고 특별한 단어(제4조·WFH·USB)를 정확히 본 문서에 높은 점수를 줘요. 대신 의미 유사성에는 둔감합니다.

두 방법은 **서로의 약점을 정확히 메꿉니다**. 그래서 같이 씁니다.

### ChromaDB vs BM25Retriever — 원리 자체가 다름

같은 "검색"이라 묶여 있지만 알고리즘이 완전히 다릅니다.

| | ChromaDB | BM25Retriever |
|---|----------|---------------|
| 뭐로 찾나 | **의미**로 (벡터 유사도) | **단어**로 (단어 빈도·희귀도) |
| 강점 | 다른 표현·동의어 이해 | 정확한 키워드·숫자·고유명사 |
| 약점 | 정확한 기호·약어에 둔감 | `"휴가"`와 `"쉼"`을 남남으로 봄 |
| 내부 수식 | 임베딩 모델 → 코사인 유사도 | TF × IDF |
| 라이브러리 | `chromadb` | `rank-bm25` |
| 저장소 | SQLite 파일 (디스크 영구) | 메모리만 (프로세스 켤 때마다 재빌드) |
| 추가 비용 | 임베딩 모델 ~500MB 다운로드 | 모델 없음, 순수 파이썬 계산 |

같은 질문 `"WFH 정책"`을 두 사서에게 던졌을 때 반응이 정반대예요.

| 문서 | ChromaDB | BM25 |
|------|---------|------|
| `"재택근무는 입사 6개월 이상..."` | **찾음** (재택근무 ≈ WFH) | 못 찾음 (WFH 단어 없음) |
| `"WFH 정책 v2.0 변경 안내"` | 짧아서 모호할 수 있음 | **강하게 찾음** (WFH·정책 정확 매칭) |

하나는 의미로, 다른 하나는 글자로 찾기 때문에 **한쪽이 놓치는 걸 다른 쪽이 줍습니다**. 그래서 Ensemble이 필요해요.

### 데이터 흐름 따라가기

질문이 `"WFH(재택근무) 정책 알려줘"`로 들어오면:

**(a) Vector 사서 (ChromaDB)**

```python
res = collection.query(query_texts=["WFH(재택근무) 정책 알려줘"], n_results=9)
```

ChromaDB가 모든 청크를 벡터로 이미 가지고 있기 때문에 **수 ms** 안에 9개 후보를 뱉습니다. `distance`를 `1 - distance`로 뒤집어 유사도로 바꿉니다.

> **왜 `1 - distance`인가**
>
> ChromaDB는 결과를 "거리(distance)"로 돌려줍니다. 낮을수록 가까운(좋은) 결과예요.
> 반면 BM25는 높을수록 좋은 결과로 점수를 냅니다. 두 점수를 섞어 쓰려면 **방향을 통일**해야 하니, ChromaDB 쪽을 `1 - distance`로 뒤집어 "높을수록 좋음"으로 맞춥니다. 코사인 거리를 쓰는 경우 `1 - cosine distance = cosine similarity`라 원래의 유사도를 복원하는 셈입니다.
>
> | distance | 1 - distance (유사도) |
> |---|---|
> | 0.0 (완전히 같음) | 1.0 (최고) |
> | 0.5 | 0.5 |
> | 1.0 (완전히 다름) | 0.0 (최저) |

```
[("재택근무는 입사 6개월 이상...", score=0.78),
 ("연차 신청 절차는...", score=0.41),
 ...]
```

**(b) BM25 사서 (rank-bm25)**

```python
bm25 = self._get_bm25()           # 최초 1회만 인덱스 빌드
hits = bm25.search(query, top_k=9)
```

`rank-bm25` 라이브러리가 **단어 빈도·문서 길이·역문서빈도(IDF)** 를 계산해 점수를 냅니다.

```
[("재택근무는 입사 6개월 이상...", score=5.2),
 ("퇴직금은 근속 1년 이상...",    score=0.3),
 ...]
```

두 사서의 점수 스케일이 달라요(0~1 vs 0~10+). 그래서 **정규화**합니다.

```python
normalized = (score - min) / (max - min)   # 0~1로 맞춤
```

**(c) 합치기**

같은 `content`를 키로 두 점수를 포개 놓고 가중 합산:

```
hybrid_score = alpha * vector_score + (1 - alpha) * bm25_score
```

`alpha=0.5`는 반반. 벡터를 더 믿고 싶으면 0.7, 키워드를 더 믿고 싶으면 0.3으로 조정합니다.

## 3. CrossEncoderReranker — 꼼꼼한 평론가의 재채점

### 왜 2단계만으론 부족한가

Ensemble이 돌려준 상위 9개는 **"질문 단어와 겹치거나 의미가 비슷한 문서"** 입니다. 그런데 정말로 **질문에 답을 주는지**는 다른 문제예요.

예시: `"병가 며칠까지 가능해?"` 질문에 Ensemble이 9개 돌려주면 그 안엔 이런 것들이 섞여 있을 수 있습니다.

| 후보 | Ensemble 점수 | 진짜 답 되나? |
|------|-------------|------------|
| `"연간 30일의 유급 병가를 사용할 수 있으며..."` | 0.78 | **O** |
| `"보안 규정 위반 시 징계 절차가 진행되며..."` | 0.65 | X (병가랑 무관) |
| `"휴일근무는 2배가 적용됩니다"` | 0.52 | X (병가랑 무관, 휴가 단어만 매칭) |
| `"육아휴직은 만 8세 이하 자녀가..."` | 0.48 | X |

`"병가"`·`"30일"`이 들어간 건 1위가 맞는데, **점수 차이가 작아** 벡터 노이즈만으로 순위가 뒤집힐 수 있어요.

### Cross-Encoder가 달라지는 이유

보통 임베딩 모델은 **문장을 벡터 하나로 압축**해서 비교합니다(Bi-Encoder). 빠르지만 정보 손실이 생겨요.

Cross-Encoder는 다릅니다. `(질문, 문서)` 쌍을 **한 시퀀스로 연결해 통째로 입력**합니다.

```
입력: [CLS] 병가 며칠까지 가능해? [SEP] 연간 30일의 유급 병가... [SEP]
출력: 0.91  (관련성 점수)
```

트랜스포머의 **self-attention이 질문의 각 단어와 문서의 각 단어를 모두 짝지어 봅니다**. 그래서 "병가"와 "30일"의 연결관계, "휴일근무"와 "병가"의 비연결관계까지 섬세히 잡아냅니다.

결과는 대개 이렇게 바뀝니다.

| 후보 | CrossEncoder 점수 | 순위 |
|------|----------------|-----|
| `"연간 30일의 유급 병가를 사용할 수 있으며..."` | 0.91 | **1위로 고정** |
| `"보안 규정 위반 시 징계 절차..."` | 0.08 | 3위→밀림 |
| `"휴일근무는 2배..."` | 0.11 | 4위→밀림 |
| `"육아휴직은 만 8세 이하..."` | 0.15 | 5위→밀림 |

### 왜 모든 문서에 안 쓰나 (핵심 감각)

Cross-Encoder는 문서마다 **모델을 한 번씩 호출**합니다. 벡터DB의 문서가 수천 개면 수천 번 호출 = 수십 초~수 분. 실시간 검색엔 못 써요.

그래서 **2단계(Ensemble)로 먼저 수를 확 줄이고 → 3단계(CE)로 좁혀진 후보에만 모델을 씀**. 이게 "빠름 + 정확함"을 둘 다 얻는 실전 RAG의 기본기입니다.

우리 구현에서는 `top_k=k*2` (보통 6개)만 CE에 통과시켜 1초 이내에 끝나도록 조정했어요.

## 4. ParentDocumentRetriever — 자식으로 찾고 부모로 돌려주기

### 작게 쪼갠 대가

문서를 벡터DB에 넣을 때 보통 **작게(200~300자) 청킹**합니다. 짧을수록 벡터 유사도 정확도가 높거든요. 문제는 LLM한테 넘겨 답변을 만들 때.

`"연간 30일의 유급 병가를 사용할 수 있으며,"` 이 청크만 받으면 LLM은 뒤에 이어지는 **조건**(3일 이상 연속 시 진단서 필요)을 모릅니다. 답변이 반쪽짜리가 돼요.

### 해결: 인덱싱을 두 갈래로 나누기

`ex11/data/docs`를 파싱할 때 **자식·부모를 동시에** 만들어 따로 저장합니다.

- **부모** = 페이지 단위 원문 전체 → `ex11/data/parent_docs.json`에 딕셔너리로 저장
- **자식** = 220자 내외 청크. ChromaDB 컬렉션에 `parent_id` 메타데이터와 함께 저장

검색 시엔 자식 쪽 벡터 유사도를 이용해 정확히 짚어 내고, 최종 LLM에 넘길 땐 자식의 `parent_id`를 키로 부모 페이지 원문을 꺼냅니다.

```
ChromaDB(자식)        parent_docs.json(부모)
 ┌──────────────┐      ┌──────────────────────────┐
 │ parent_id    │      │ "취업규칙_v1.0__p2": {    │
 │  = 취업규칙__p2 │ ───▶ │   "content": "…병가…진단서…", │
 │ content      │      │   "source": "취업규칙_v1.0", │
 │  = "연 30일…" │      │   "page": 2               │
 └──────────────┘      │ }                         │
                       └──────────────────────────┘
```

검색 결과 중 같은 `parent_id`가 여러 자식에서 나오면 **한 번만** 부모로 복원해 중복을 거릅니다. 그래서 상위 k개가 "**서로 다른 부모 페이지** k개"가 돼요.

### 자식·부모로 나누는 이점 세 가지

1. **검색 정밀도**: 벡터 유사도는 자식(작은 단위)에서 계산. `"병가 며칠"`이라는 짧은 질문이 200자 청크와 매칭될 때가 페이지 전체와 매칭될 때보다 훨씬 정확.
2. **답변 맥락**: LLM에 넘기는 건 부모 페이지 원문. 300자가 아니라 페이지 전체라서 조건·예외까지 함께 전달됨.
3. **중복 제거**: 같은 페이지의 여러 청크가 동시에 상위에 올라도 결과 목록엔 그 페이지가 한 번만 등장.

### 코드 진입점

```python
retriever = ParentDocumentRetriever(
    child_collection=ChromaDB_collection,
    parent_docs=json.load("parent_docs.json"),
)

# 방법 1: 자체 검색부터 복원까지 (챕터 9 원본 API)
docs = retriever.search("병가 며칠까지 가능해?", top_k=3)

# 방법 2: 외부에서 정렬한 자식 결과를 받아 복원만 (RagPipeline 경로)
docs = retriever.resolve(reranked_children, top_k=3)
```

`RagPipeline`은 `resolve(...)`를 씁니다. 이미 `EnsembleRetriever`·`CrossEncoderReranker`가 자식 단위로 검색·재정렬을 끝낸 뒤라, 이 단계에서는 자식 → 부모 복원과 중복 제거만 맡는 거예요.

## 왜 이 순서인가 (아주 중요)

네 단계의 순서는 바꾸면 안 됩니다. 각 단계가 **앞 단계의 결과 모양**에 맞춰져 있어요.

| 순서 | 왜 여기인가 |
|-----|------------|
| 1. QueryExpander | 모든 검색 단계에 영향을 주므로 **맨 앞**. 뒤에서 확장하면 벡터도 BM25도 원래 질문만 본 상태가 됨 |
| 2. EnsembleRetriever | 수천 개 문서에서 **후보를 수십 개로** 압축. CE를 쓸 수 있게 만드는 전제 |
| 3. CrossEncoderReranker | 좁혀진 후보에서 **진짜 답에 가까운 순서**로 재정렬. 정확도의 마지막 거름망 |
| 4. ParentAssembler | LLM 전달 직전 **문맥 복원**. 순서가 이미 확정된 뒤에 해야 "상위 k"의 의미가 유지됨 |

직관으로는:

> **1: 묻는 방식을 다듬는다 → 2: 넓게 훑어 모은다 → 3: 꼼꼼히 고른다 → 4: 완전한 형태로 만든다**

## 한 번에 따라가 보는 예시

질문: `"WFH 며칠까지 가능해?"`

```
[1] QueryExpander
    → "WFH(재택근무) 며칠까지 가능해?"

[2] EnsembleRetriever (k*3 = 9개 후보)
    Vector 사서: 의미 비슷한 9개 (재택·출장·원격 관련)
    BM25 사서:   단어 겹치는 9개 (WFH·재택근무·며칠 정확 매칭)
    합친 후 점수 재정렬 → 상위 9개

[3] CrossEncoderReranker (top_k = k*2 = 6개)
    9개 각각 (질문, 문서) 쌍으로 모델 채점
    → "재택근무는 주 2회까지 가능..." 1위로 고정
    → 관련 없는 "출장 일수" 등은 아래로 밀림

[4] ParentAssembler
    상위 6개 중 같은 페이지인 것 묶기
    → 취업규칙 p.3 전체가 한 덩어리로
    → 최종 상위 k=3개 = 3개 페이지 단위 컨텍스트

LLM이 이 3개 페이지를 보고 최종 답변 생성.
```
