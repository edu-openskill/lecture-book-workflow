# 저자 특강 FAQ — 독자 자주 묻는 질문

이 책을 읽는 독자(또는 저자 특강 청중)가 자주 묻거나 앞으로 물을 가능성이 높은 질문을 **수준별**로 정리했다. 본문에 녹이지 못한 **설계 결정의 배경**과 **확장 방향**을 담았다.

- 🟢 **초급** — RAG를 처음 듣는 사람, "왜 이렇게 하지?"가 궁금한 독자
- 🟡 **중급** — 실습을 마치고 "내 프로젝트에 어떻게 적용?" 고민하는 독자
- 🔴 **고급** — 실제 운영·확장·비즈니스 결정을 해야 하는 독자

## 🟢 초급 — 개념과 선택의 이유

### Q1. 왜 RAG인가요? Fine-tuning 하면 안 되나요?

RAG는 **외부 지식을 그때그때 참조**하는 방식이고, Fine-tuning은 **모델 가중치에 지식을 새겨 넣는** 방식입니다. 사내 규정처럼 **자주 바뀌는 문서**는 RAG가 압도적으로 유리합니다. 규정이 바뀌면 벡터 DB만 업데이트하면 되지만 Fine-tuning은 모델을 통째로 재훈련해야 하죠. 비용과 갱신 주기를 비교했을 때, 문서 기반 Q&A에는 RAG가 기본입니다. (Fine-tuning은 **말투·포맷**을 바꿀 때 더 유용합니다)

### Q2. 왜 Ollama를 쓰나요? ChatGPT API 쓰면 더 쉽지 않나요?

두 가지 이유 때문입니다.

1. **데이터 주권**. 사내 규정·개인정보를 외부 API로 보내면 컴플라이언스 문제가 생깁니다. Ollama는 로컬에서 돕니다.
2. **비용**. 수만 건 문서를 임베딩하거나 매일 수백 건 질문을 받는 시스템에서 OpenAI API 과금은 빠르게 불어납니다.

물론 OpenAI가 더 빠르고 정확합니다. 이 책은 **두 옵션을 전환할 수 있도록** 설계했으니, 실제 서비스에서는 "민감도 낮은 질문 → OpenAI, 사내 규정 → Ollama" 같은 하이브리드도 선택지입니다.

### Q3. 벡터 DB가 왜 필요한가요? 그냥 검색엔진(Elasticsearch) 쓰면 안 되나요?

검색엔진은 **키워드**를 맞춥니다. "휴가"라고 물으면 "휴가"가 들어간 문서를 찾죠. 그런데 문서에는 "**연차유급휴가**"라고 적혀 있다면? 키워드가 안 맞으니 놓칩니다. 벡터 DB는 **의미의 거리**를 잽니다. "휴가"와 "연차유급휴가"의 의미 벡터가 가까우면 찾아냅니다. 챕터 9에서 Query Rewrite로 이 문제를 더 깊이 다루지만, 기본 전제는 "단어가 달라도 뜻이 같으면 찾는다"는 **의미 검색**입니다. (실무에서는 BM25(키워드) + 벡터 검색을 **하이브리드**로 섞는 게 표준입니다 — 챕터 8 참조)

### Q4. 왜 임베딩 모델이 `ko-sroberta-multitask`인가요? OpenAI의 `text-embedding-3-small`이 더 좋지 않나요?

품질만 놓고 보면 OpenAI 임베딩이 대체로 좋습니다. 다만 이 책에서 `ko-sroberta-multitask`를 쓴 이유는 세 가지입니다.

1. **한국어 특화**. OpenAI 임베딩은 다국어 모델이라 한국어에서 일반적이고, 한국어 전용 학습 모델이 특정 도메인(법률·규정)에서 더 안정적인 경우가 있습니다
2. **무료·로컬**. 수만 문서 임베딩 시 비용 0
3. **400MB 크기**. 노트북에서 즉시 돌릴 수 있음

실제 프로젝트에서는 본인 문서로 **둘 다 돌려보고 Recall@k를 비교**해 선택하시기 바랍니다.

### Q5. 왜 청킹이 500자인가요?

정답은 없습니다. 500자는 **한국어 규정 문서에서 조항 한두 개가 들어가는 경험적 길이**입니다. 챕터 8에서 확인하듯 너무 짧으면(100자) 맥락이 끊기고, 너무 길면(2000자) 한 조각에 여러 주제가 섞여 검색 정확도가 떨어집니다. 시맨틱 청킹(의미 경계로 자르기)도 대안이지만 임베딩 호출 비용이 추가됩니다. **먼저 500자 고정으로 돌려보고, Precision이 낮으면 시맨틱 청킹으로 전환**하는 순서를 권장합니다.

### Q6. 왜 SQLite를 안 쓰고 PostgreSQL을 쓰나요? (CH06부터)

두 가지 이유입니다.

1. **동시성**. SQLite는 쓰기 락이 DB 단위라, FastAPI에서 동시 요청이 여럿 들어오면 대기가 걸립니다. PostgreSQL은 row 단위 락이라 확장성이 좋습니다
2. **실무 환경 시뮬레이션**. 실제 사내 시스템은 대부분 PostgreSQL/MySQL을 쓰니, 처음부터 그 환경을 경험해보는 편이 나중에 배포할 때 덜 놀랍니다

다만 **SQLite도 완전히 배제하지 않았습니다**. 챕터 7의 `chroma.sqlite3`는 ChromaDB의 **내부 저장소**로 SQLite를 씁니다. 벡터 수백만 개까지는 SQLite로 충분하고, 별도 서버를 띄울 필요가 없어 **로컬 파일 기반 벡터 DB**로 좋은 선택입니다. 즉 이 책에서는 **정형 데이터 = PostgreSQL**, **벡터 = ChromaDB(내부는 SQLite)** 로 역할을 나눴습니다.

### Q7. 왜 챕터마다 도커 컨테이너를 새로 띄우나요? 하나 공유하면 안 되나요?

저자의 의도는 "**챕터마다 독립된 실습 환경**"입니다. 공유 컨테이너를 쓰면 학습자가 "DB가 어디서 나온 거지?" 혼란스러울 수 있고, 한 챕터에서 망가진 데이터가 다음 챕터까지 영향을 줍니다. 챕터마다 `docker compose up -d`를 반복하는 건 **번거롭지만 경계를 선명하게** 합니다. 대안으로는 각 챕터의 schema.sql을 공통으로 두고 컨테이너 이름만 바꾸는 방법이 있는데, 이 책은 **후자를 선택**했습니다(ex06~ex08 docker-compose.yml 비교).

### Q8. 왜 가상환경을 챕터마다 새로 만드나요? 하나로 통합하면 안 되나요?

세 가지 이유입니다.

1. **버전 충돌 격리**. 챕터별로 langchain·chromadb 버전이 미세하게 달라도 서로 영향 없게
2. **독립 실행**. 독자가 "챕터 5만 돌려보고 싶다"를 하기 쉽게
3. **실제 회사 환경 모방**. 프로젝트별 venv는 실무 기본 패턴

디스크 낭비가 마음에 걸리면 `uv`(Rust로 짠 pip 대체제)를 써보세요. `uv venv && uv pip install -r requirements.txt` 조합이 pip보다 10~100배 빠르고, 같은 패키지의 **여러 venv를 링크로 공유**하기도 합니다.

## 🟡 중급 — 설계 판단과 대안

### Q9. 왜 챕터 5는 `deepseek-r1:8b`, 챕터 6부터는 `llama3.1:8b`로 바뀌나요?

모델 특기가 다르기 때문입니다.

- **deepseek-r1**: 긴 추론(chain-of-thought)에 강함. RAG Q&A 답변 품질이 좋음. 하지만 **Tool Calling 미지원**
- **llama3.1**: 추론은 평균이지만 **Tool Calling을 안정적으로 지원**. 에이전트 구축에 필수

챕터 6부터 에이전트가 도구를 호출해야 하니 `llama3.1:8b`로 교체합니다. 실제 프로덕션에서는 "대화는 llama, 깊은 추론은 deepseek"처럼 **모델 라우팅**도 가능합니다. (챕터 6의 `:::tip`에 이 내용이 적혀 있습니다)

### Q10. Chroma 말고 Pinecone, Weaviate, Milvus는 안 쓰나요?

선택 기준은 **규모와 운영 부담**입니다.

| 옵션 | 장점 | 단점 | 적합 규모 |
|------|------|------|----------|
| **ChromaDB** | 로컬 파일, 무료, Python 한 줄 | 분산·레플리카 약함 | 수만~수백만 벡터 |
| **Pinecone** | 완전 관리형, 빠름 | 유료, 외부 서비스 | 중~대규모 SaaS |
| **Weaviate** | 하이브리드 검색·GraphQL 내장 | 설치·운영 부담 | 중~대규모 자체 호스팅 |
| **Milvus** | 수십억 벡터 확장, 고성능 | 운영 복잡도 매우 높음 | 대기업 대규모 |
| **pgvector** | PostgreSQL 플러그인. 하나로 관리 | 기능 한정 | 작은 서비스 |

**책은 ChromaDB를 선택**했습니다. 왜냐하면 설치가 `pip install chromadb` 한 줄, 데이터가 파일 하나(`chroma.sqlite3`)로 저장되어 **학습과 프로토타입에 최적**이기 때문입니다. 실제 서비스에서 수십만 이상 벡터를 다룬다면 pgvector나 Weaviate로 이동하세요.

### Q11. 고정 청킹 vs 시맨틱 청킹, 뭐가 나은가요?

**문서 유형에 따라 다릅니다**.

- **고정 청킹 (500자 + 100자 오버랩)**: 규정집처럼 **구조가 일정한 문서**에 좋음. 구현도 빠름
- **시맨틱 청킹**: 의미 경계에서 자름. **보고서·매뉴얼처럼 섹션 길이가 들쭉날쭉한 문서**에 유리. 다만 임베딩 호출이 추가돼 속도가 느리고 비용이 있음

챕터 8에서 둘을 비교 실험합니다. 일반적으로 **고정 청킹으로 먼저 돌려보고 Precision@k가 낮을 때 시맨틱으로 업그레이드**하는 순서가 합리적입니다. 또 하나의 팁 — **원본 문서를 Markdown으로 먼저 변환해두면** (CH04의 파싱 단계) 헤더 기준 청킹(`MarkdownHeaderTextSplitter`)이라는 세 번째 옵션도 열립니다.

### Q12. 왜 하이브리드 파서가 `if text: text_layer else: vision`으로 단순한가요? 이미지 크기를 보면 더 정교하지 않나요?

정교한 로직이 있었는데 **실측 중 오작동**해서 단순화했습니다.

- 초기안: `page.get_images()`로 이미지 유무를 판단 → 이미지 있으면 Vision
- 실제: 한국 규정 PDF는 대부분 **로고·워터마크가 임베드**돼 `get_images()`가 늘 True → 텍스트 PDF도 전부 Vision으로 가버림
- 해법: **pypdf가 글자를 반환하면 쓰고, 빈 문자열이면 Vision 호출** (한 줄 판단)

교훈: **의도보다 실측**. 책을 쓰는 과정에서도 설계 변경이 일어났습니다. 더 정교한 로직(이미지 크기·밀도 기반)을 원한다면 `page.get_images(full=True)`로 bbox를 얻어 페이지 면적 대비 비율을 재는 방법이 있습니다. 다만 대부분 PDF에서는 단순 로직이면 충분합니다.

### Q13. Vision LLM은 왜 이렇게 느리고 환각이 심한가요? 쓸 만한가요?

로컬 Vision LLM(minicpm-v, llama3.2-vision 등)은 **양자화된 소형 모델**이라 GPU 가속 없이는 페이지당 수십 초~수 분이 걸립니다. 한국어 품질도 완벽하지 않아 **환각(예: "스보보안 서의서"를 "AI 비즈니스 인사 관리"로 재해석)** 이 섞입니다. 그래서 이 책은 Vision LLM을 **"절대적 해결책"이 아니라 "마지막 수단"** 으로 배치했습니다.

실무 권장:

1. **원본 PDF를 얻을 수 있다면 최우선** (스캔을 다시 안 하는 게 최고)
2. **OpenAI Vision** (`gpt-4o-mini`)이 품질·속도 모두 로컬보다 훨씬 좋습니다. 민감 문서가 아니면 유료가 경제적
3. **Vision LLM 도메인 특화 모델** (예: Claude Sonnet 4.5, Gemini 2.5 Pro)도 실험해볼 가치가 있음
4. **특수 목적 OCR** (네이버 CLOVA, UPSTAGE Document Parse 등)이 한국어 스캔본에 더 정확할 수 있음

### Q14. RAG 평가 지표, Precision@k 외에 뭐가 있나요?

챕터 10에서 4가지를 다룹니다.

| 지표 | 측정 대상 | 계산 |
|------|----------|------|
| **Precision@k** | 상위 k개 중 정답 비율 | 정확히 맞는 청크 / k |
| **Recall** | 전체 정답 중 발견율 | 찾은 정답 / 전체 정답 |
| **MRR** | 정답이 몇 위에 처음 등장했나 | 1/rank 평균 |
| **Hallucination Rate** | 답변 중 근거 없는 문장 비율 | LLM 판정 |

추가로 실무에서 봐야 할 것:

- **Faithfulness** — 답변이 제공된 context를 얼마나 충실히 따르는지 (RAGAS 지표)
- **Answer Relevance** — 답변이 질문과 관련 있는지
- **Context Relevance** — 가져온 문서가 질문과 맞는지
- **Latency p50/p95/p99** — 응답 지연 분포
- **Token cost per query** — 비용 추적

툴: **RAGAS**, **TruLens**, **LangSmith**, **Langfuse**가 자동 계산을 지원합니다.

### Q15. ReAct 패턴 외에 다른 에이전트 패턴은?

챕터 6에서 ReAct(Reason-Act-Observe)를 구현하지만, 이것 외에도 여러 패턴이 있습니다.

| 패턴 | 언제 쓰나 |
|------|----------|
| **ReAct** | 질문이 도구 1~3번으로 풀릴 때. 가볍고 빠름 |
| **Plan-and-Execute** | 복잡한 다단계 질문. 먼저 계획을 세우고 순차 실행 |
| **Reflexion** | 실패를 반성하고 재시도. 장시간 자율 에이전트 |
| **Tree of Thoughts** | 여러 추론 경로를 병렬 탐색. 난이도 높은 문제 |
| **Multi-agent** | 여러 에이전트(Planner, Researcher, Writer)가 협업 |

LangChain + LangGraph로 위 패턴들을 모두 구현할 수 있습니다. 복잡한 패턴일수록 **LLM 호출 비용과 응답 시간이 증가**하므로, ReAct로 시작해서 정확도가 안 나올 때만 한 단계 위로 올리는 걸 권장합니다.

### Q16. 테스트 PDF가 왜 `generate_real_pdfs.py`로 스캔본화됐나요? 원본 두고 써도 되는데요.

챕터 10의 교육 목적이 "**스캔 PDF**를 다루는 법"이기 때문입니다. 원본이 텍스트 레이어가 있는 PDF라면 pypdf로 그냥 읽히니 OCR/Vision 실습이 성립하지 않습니다. 그래서 **의도적으로 원본을 이미지 PDF로 변환**해 두었습니다. ex04~ex09 데이터는 원본 텍스트 PDF를 유지하고, **ex10만 스캔본**으로 바꾸는 구조입니다.

## 🔴 고급 — 운영·확장·비즈니스

### Q17. 프로덕션 배포 시 가장 먼저 고려할 것은?

체크리스트 순서는 이렇습니다.

1. **권한 관리** — 문서별 접근 제어(ABAC/RBAC). HR 규정을 인턴이 볼 수 있으면 안 되죠
2. **로그·관측성** — Langfuse·Langsmith로 모든 쿼리 기록. 어떤 질문이 실패하는지
3. **평가셋 자동화** — 매 배포 전 평가셋 돌려 회귀 감지
4. **레이트 리미팅** — 사용자·조직별 QPS 제한
5. **장애 격리** — Ollama/OpenAI 호출 타임아웃·Circuit Breaker
6. **비용 모니터링** — OpenAI 토큰 비용 대시보드
7. **PII 마스킹** — 개인정보를 LLM에 보내기 전 치환
8. **감사 로그** — 누가 언제 무엇을 물었고 어떤 답을 받았는지 보존

이 책의 챕터 7(캐시·옵저버빌리티)에서 일부를 다루지만, 실제 배포에는 **최소 2~3배의 시간**이 더 듭니다.

### Q18. 수백~수천 개 문서를 넘어 **수만~수십만**으로 확장하려면?

병목 순서대로:

1. **임베딩 인덱스** — ChromaDB는 수십만까지 괜찮지만 그 이상은 **HNSW 파라미터 튜닝**이 필요. 또는 Weaviate/Milvus로 이전
2. **검색 응답 시간** — k=5 Top-K 검색이 100ms 넘어가면 **사전 필터링**(부서·문서 유형)이나 **하이브리드 검색** 도입
3. **문서 업데이트 주기** — 매일 갱신이면 증분 임베딩(Changed Detection). Parent Document Retriever가 효과적
4. **멀티 테넌시** — 회사별로 격리. Collection 단위 분리 or DB 단위 분리
5. **관련성 튜닝** — 부서·문서 유형·날짜로 가중치. BM25 + 벡터 + 리랭커 3단 파이프라인

### Q19. Fine-tuning과 RAG를 함께 쓰는 사례는?

흔히 **하이브리드 전략**을 씁니다.

- **RAG**: 최신 지식(법규, 매뉴얼, 가격 정보)
- **Fine-tuning (LoRA)**: 말투·포맷·도메인 어휘

예: 고객지원 챗봇에서 "답변 구조(인사 → 요약 → 근거 → 안내)"는 Fine-tuning으로 고정, 실제 답변 내용은 RAG로 가져오는 방식입니다. Fine-tuning은 **수백~수천 건의 좋은 샘플**이 있어야 효과가 있으므로, **먼저 RAG로 6개월 운영하며 실사용 데이터를 모은 뒤** Fine-tuning을 고려하는 순서를 권장합니다.

### Q20. 환각을 0%로 만들 수 있나요?

**현실적으로 불가능**하지만 5% 이하로 낮추는 패턴은 있습니다.

1. **그라운딩 프롬프트** — "제공된 context에 없으면 '모른다'고 답하세요" (간단하지만 효과 큼)
2. **출처 강제** — 답변에 문서 id·페이지를 포함하게 강제
3. **근거 검증 단계** — 답변 생성 후 "이 문장이 context에 근거가 있는가"를 **별도 LLM이 판정**
4. **사용자 확인** — 답변에 "참고 문서를 열어보기" 링크를 늘 제공해 사용자가 검증 가능하게
5. **선택적 거부** — LLM이 신뢰도가 낮다고 판단하면 **답변 대신 관련 문서 목록 반환**

`Hallucination Rate < 5%` + `Faithfulness > 0.9`가 실무 배포 기준선입니다.

### Q21. 비용·속도·품질, 어떻게 타협하나요?

프리셋 3가지로 제안합니다.

| 프리셋 | 임베딩 | LLM | 벡터 DB | 예상 비용 (1만 QPM) | 지연 |
|--------|--------|------|---------|--------------------|------|
| **로컬** | ko-sroberta | Ollama llama3.1 | ChromaDB | $0 | 3~5초 |
| **하이브리드** | ko-sroberta | Ollama llama3.1 / OpenAI gpt-4o-mini 폴백 | ChromaDB | $30~50/월 | 1~3초 |
| **클라우드** | OpenAI text-embedding-3-small | OpenAI gpt-4o-mini | Pinecone | $200~500/월 | 0.5~1.5초 |

내부용 PoC는 **로컬**부터, 사용자 체감 품질이 중요하면 **하이브리드**, 정확도·속도 둘 다 양보 불가면 **클라우드**. 대부분 프로젝트는 **로컬 → 하이브리드 → 클라우드** 순으로 진화합니다.

### Q22. 이 책 이후에 공부하면 좋은 주제는?

단계별 추천입니다.

1. **LangGraph** — 이 책의 에이전트를 그래프로 재설계. 복잡한 워크플로우 가능
2. **RAGAS / TruLens** — 평가 자동화
3. **Langfuse** — 운영 환경 관측성
4. **LightRAG / GraphRAG** — 지식 그래프 기반 RAG (관계 많은 도메인에 강함)
5. **MCP (Model Context Protocol)** — 에이전트가 여러 도구를 표준 프로토콜로 연결 (챕터 6의 @tool을 대체 가능)
6. **DSPy** — 프롬프트 최적화 자동화
7. **Small-but-specialized 모델** — Phi-3, Llama 3.2 1B/3B 같은 초경량 모델로 엣지 배포

저자의 다음 책/강의 주제는 **Agent Orchestration + Production RAG**가 될 예정입니다.
