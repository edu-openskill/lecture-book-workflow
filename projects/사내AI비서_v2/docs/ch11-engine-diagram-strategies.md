# CH11 §11.4·§11.5 엔진 다이어그램 디자인 전략

> 챕터 11의 두 큰 다이어그램(파이프라인 엔진 + 시스템 아키텍처)을 "엔진 느낌"으로 재설계하기 위한 옵션 제안서. 유저가 하나를 고르면 구현으로 들어갑니다.

## 공통 제약

- **인쇄 폭**: A4 세로, 18mm 좌우 여백 → 본문 폭 ~620px
- **디자인 토큰만 사용** (`var(--color-*)`, `var(--space-*)`, `--font-mono`)
- **챕터 뱃지 제거** (챕터 X 라벨)
- **계단 피해야 할 패턴**: 이모지, em dash, 밝은 색 띠 border-left (structure.md 금지)
- **단색 + 액센트** 우선. 다색 스펙트럼 금지

## 가용 디자인 토큰 (palette)

| 토큰 | 값 | 쓰임 |
|---|---|---|
| `--color-accent` | `#4f46e5` 인디고 | 브랜드 메인. 핵심 강조 |
| `--color-accent-warm` | `#ff7849` 오렌지 | 보조 액센트. 입력·태그 |
| `--color-info` | `#3b82f6` 블루 | 차분한 정보 |
| `--color-success` | `#059669` 초록 | 출력·성공 |
| `--color-text-heading` | `#2d3748` 근흑 | 굵은 텍스트 |
| `--color-surface` | `#fafbfc` 연회색 | 카드 배경 |
| `--color-border-strong` | `#cbd5e0` | 강조 테두리 |
| `--font-mono` | JetBrains Mono | 기술 라벨 |

# §11.4 — 파이프라인 엔진 (6단계 흐름)

6단계: 파싱 → 청킹 → 쿼리 확장 → 검색 → 리랭킹 → 부모복원·LLM

## Option A. CPU 파이프라인 (Clocked Stages)

```
┌──────────────────────────────────────────┐
│  PIPELINE ENGINE          clock → → →     │
├─────┬─────┬─────┬─────┬─────┬─────────┤
│ IN  │ S1  │ S2  │ S3  │ S4  │   OUT    │
│ ─── │     │     │     │     │          │
│ 질문 │파싱│청킹│검색│리랭│LLM+근거│
│     │     │     │     │     │          │
└─────┴─────┴─────┴─────┴─────┴─────────┘
         ↓ 각 stage는 동일 width register 블록
```

- **느낌**: CPU 파이프라인. 슈퍼스칼라 프로세서 데이터시트 스타일
- **레이아웃**: 가로 5~6칸 세로 1줄 (A4에 다 못 들어가니 줄바꿈 허용 or 세로 접힘)
- **색**: `--color-accent` 테두리 + `--color-bg` 배경. 라벨은 `--font-mono`
- **장점**: 매우 엔진다움. 선명한 "단계" 개념
- **단점**: 세로 폭이 얇아 설명 텍스트 줄이기 힘듦. 모바일 PDF에서 줄바꿈 깨질 수 있음
- **적합**: 각 단계를 동등한 처리 슬롯으로 보여주고 싶을 때

## Option B. 엔진 블록 단면도 (Cross-Section)

```
╔══════════════════════════════════════════╗
║        [ CONNECTHR ENGINE ]               ║
║                                            ║
║  ◎ INTAKE ─┬─ 파싱 ─ 청킹 ─┬─            ║
║            │                │              ║
║   fuel     │   COMBUSTION   │  EXHAUST    ║
║ (문서)      ├─ 쿼리 확장 ──┤             ║
║            │  검색 · 리랭킹   │ → OUT     ║
║            │                │   (답변)    ║
║            └─ LLM 합성 ────┘             ║
║                                            ║
╚══════════════════════════════════════════╝
```

- **느낌**: 실제 내연기관 단면도. 흡입·압축·폭발·배기 4행정 은유
- **레이아웃**: 큰 outer 박스(엔진 블록) + 좌(intake) · 중(chambers) · 우(exhaust) 3 zone
- **색**: `--color-surface` 배경 + `--color-border-strong` 두꺼운 테두리. `--color-accent-warm` intake, `--color-accent` chamber, `--color-success` exhaust
- **장점**: 가장 "엔진스러움". 은유 강력
- **단점**: 정보 밀도 높아서 작은 크기에서 읽기 어려움. 구현 복잡
- **적합**: 챕터 peak 시각 오브제로 기억에 남기고 싶을 때

## Option C. 컨베이어 조립 라인 (Assembly Line)

```
  IN                                              OUT
  ▼    ●────●────●────●────●────●            ▼
 질문──┤파싱├→┤청킹├→┤확장├→┤검색├→┤리랭├→┤LLM ├──답변
       └─┬─┘  └─┬─┘  └─┬─┘  └─┬─┘  └─┬─┘  └─┬─┘
         │      │      │      │      │      │
      parse  chunk  expand retr  rank   gen
       툴    툴     툴   툴     툴    툴
```

- **느낌**: 공장 조립 라인. 벨트 위에 스테이션들
- **레이아웃**: 가로 단일 라인. 각 station 원형 또는 둥근 사각형. 벨트는 실선 화살표
- **색**: `--color-accent` 핵심. station 배경 `--color-bg`, 벨트 `--color-text-subtle`
- **장점**: 데이터가 "흐른다"는 감각이 강함. 단순 명료
- **단점**: 가로폭에 6칸 넣으면 좁음. 세로 접힘 필요할 수 있음
- **적합**: 선형성(순차 처리)을 강조하고 싶을 때

## Option D. 계측 패널 (Instrumentation Dashboard)

```
┌────────────────── ENGINE STATUS ───────────────────┐
│ STAGE 01 ────────────────────────────── [ACTIVE]   │
│   parse_pdf_hybrid       │ input: 5 docs           │
│ STAGE 02 ────────────────────────────── [ACTIVE]   │
│   semantic_chunking      │ chunks: 39               │
│ STAGE 03 ────────────────────────────── [ACTIVE]   │
│   QueryExpander          │ hit: WFH→재택근무        │
│ STAGE 04 ────────────────────────────── [ACTIVE]   │
│   EnsembleRetriever      │ top_k: 6                 │
│ STAGE 05 ────────────────────────────── [ACTIVE]   │
│   CrossEncoderReranker   │ top_k: 3                 │
│ STAGE 06 ────────────────────────────── [ACTIVE]   │
│   ParentDoc + LLM        │ tokens: 238              │
└────────────────────────────────────────────────────┘
```

- **느낌**: 서버 대시보드. SRE 관제 센터 뷰
- **레이아웃**: 세로 6행. 각 행에 stage 번호 + 함수명 + 실측 수치
- **색**: `--color-text-heading` 텍스트 + `--color-success` ACTIVE 배지. 거의 단색
- **장점**: 정보 밀도 높음. 실측 데이터를 자연스럽게 끼워 넣을 수 있음
- **단점**: "엔진" 보다 "모니터링 도구" 느낌
- **적합**: 기술적 정확성을 최우선할 때

## Option E. 파이프 다이어그램 (Pipe Schematic)

```
  ┌──── 질문 ────┐
  │              │
  ▼              │
 ━╋━━━ PARSE ━━━━╋━━
  │              │
  ▼              │
 ━╋━━━ CHUNK ━━━╋━━
  │              │
  ▼              │
 ━╋━━ EXPAND ━━━╋━━
  │              │
  ▼              │
 ━╋━ RETRIEVE ━━╋━━
  │              │
  ▼              │
 ━╋━━ RERANK ━━━╋━━
  │              │
  ▼              │
 ━╋━━ LLM ━━━━━━╋━━
  │              │
  ▼              │
  └──── 답변 ────┘
```

- **느낌**: 공정도·P&ID(배관 계장 다이어그램). 화학 플랜트 스타일
- **레이아웃**: 세로 파이프 1개 + 6개 밸브·탱크. 파이프 두께로 흐름 표현
- **색**: `--color-accent` 파이프. 각 밸브 `--color-bg` + `--color-accent-border`
- **장점**: 엔지니어링 정본 느낌. 인쇄물에 잘 어울림
- **단점**: 구현 시 CSS 선 그리기 복잡. 분기 없는 단일 파이프라 다소 단조
- **적합**: 산업용·플랜트 은유를 좋아하는 독자

# §11.5 — 시스템 아키텍처 (5층 스택)

구성요소: LLM(Ollama 3모델) / User+FastAPI / Agent Runtime(ReAct, Cache·Monitor·Retry) / Tools(4개) / Stores(Chroma·Postgres)

## Option 1. 데이터센터 랙 (Server Rack)

```
┌───────────────────────────────────┐
│  ┌─────────────────────────────┐ │
│  │  ⬤⬤⬤  LLM Layer            │ │
│  │        ollama · 3 models    │ │
│  └─────────────────────────────┘ │
│  ┌─────────────────────────────┐ │
│  │  ⬤⬤⬤  Gateway              │ │
│  │        FastAPI              │ │
│  └─────────────────────────────┘ │
│  ┌─────────────────────────────┐ │
│  │  ⬤⬤⬤  Agent Runtime        │ │
│  │        ReAct + 운영 트리오   │ │
│  └─────────────────────────────┘ │
│  ┌─────────────────────────────┐ │
│  │  ⬤⬤⬤  Tools (4 slots)      │ │
│  │        SQL · RAG            │ │
│  └─────────────────────────────┘ │
│  ┌─────────────────────────────┐ │
│  │  ⬤⬤⬤  Storage              │ │
│  │        Chroma · Postgres    │ │
│  └─────────────────────────────┘ │
└───────────────────────────────────┘
```

- **느낌**: 서버 랙 1U 단위가 쌓인 모습. 각 1U가 1 레이어
- **레이아웃**: 세로 스택. 각 레이어에 좌측 LED 3개(상태등) + 레이어 이름 + 내용
- **색**: `--color-surface` 랙 배경 + `--color-bg` 각 유닛. LED는 `--color-success`
- **장점**: "엔진"보단 인프라지만, 물리적 시스템 느낌. 세로라 폭 문제 없음
- **단점**: 가로 흐름(user→agent→tools) 표현이 약함

## Option 2. 레이어드 케이크 (Stacked Layers)

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  L5 · External ──  Ollama (LLM)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  L4 · Interface ── User + FastAPI
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  L3 · Runtime ──   Agent · ReAct
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  L2 · Tools ──     MCP Tools (4)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  L1 · Storage ──   Chroma + Postgres
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

- **느낌**: OSI 7-layer 같은 전통적 아키텍처 레이어 다이어그램
- **레이아웃**: 세로 5층. 각 층 가로 전체 폭 사용. L1이 바닥, L5가 상단
- **색**: 층별 `--color-surface-alt` 교차 (짝수/홀수). 레이어 번호는 `--color-accent`
- **장점**: 레이어 개념 명확. 책 인쇄에 최적
- **단점**: "엔진"이 아닌 "층". 흐름(양방향) 표현이 약함

## Option 3. 엔진룸 단면 (Engine Room Layout)

```
┌────────────── CONNECTHR SYSTEM ───────────────┐
│                                                │
│  ╔═══ LLM ═══╗                                │
│  ║ Ollama    ║  ← 외부 엔진 블록                │
│  ╚═══════════╝                                │
│        ↕                                       │
│  ┌────────────────────────────────────┐       │
│  │  User ─→ FastAPI ─→ Agent Runtime  │       │
│  │                      ReAct Loop    │       │
│  │                      Cache·Mon·Retry│      │
│  └──────┬─────────────────────────────┘       │
│         │                                      │
│         ▼                                      │
│  ┌─────────────────────┐                      │
│  │  Tools (4 slots)     │                      │
│  └──────┬──────────────┘                      │
│         │                                      │
│         ▼                                      │
│  ╔═══════════╗  ╔═══════════╗                │
│  ║ ChromaDB  ║  ║ PostgreSQL║                │
│  ╚═══════════╝  ╚═══════════╝                │
└────────────────────────────────────────────────┘
```

- **느낌**: 배·비행기 엔진룸 개략도. 외부 엔진(LLM) + 내부 섀시(서버) + 연료(저장소)
- **레이아웃**: outer 케이스 안에 4개 모듈. 상단 LLM 분리, 중간 runtime, 하단 stores
- **색**: `--color-border-strong` 두꺼운 바깥 테두리. 내부 블록 `--color-accent` 액센트
- **장점**: "엔진" 느낌 살림. §11.4 엔진 파이프라인과 통일감
- **단점**: 구현 복잡. 겹치는 박스 CSS

## Option 4. 허니컴 (Honeycomb / Tile Grid)

```
       ┌──── Ollama (LLM) ────┐
       │  3 models            │
       └──────────┬───────────┘
                  ↕
     ┌──── User ──┴── FastAPI ────┐
     │  Interface                  │
     └──────────┬──────────────────┘
                │
       ┌────────┴────────┐
       │                 │
  ┌────▼────┐     ┌──────▼──────┐
  │ Runtime │     │   Tools     │
  │  Agent  │ ↔ │  4 slots     │
  │  ReAct  │     │  SQL·RAG    │
  └────┬────┘     └──────┬──────┘
       │                 │
       └────────┬────────┘
                ▼
      ┌─────────┴──────────┐
      │  Stores            │
      │  Chroma · Postgres │
      └────────────────────┘
```

- **느낌**: 마이크로서비스 상호 연결. 블록 간 양방향 화살표
- **레이아웃**: Runtime과 Tools를 좌우 나란히. 전체는 다이아몬드 흐름
- **색**: `--color-accent` 블록, `--color-text-subtle` 화살표
- **장점**: 런타임과 도구의 상호작용이 시각적으로 명확
- **단점**: 폭 관리 어려움(좌우 병치가 620px에 빡빡)
