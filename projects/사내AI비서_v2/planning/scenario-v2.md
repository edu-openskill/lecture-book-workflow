# 시나리오 v2 — 통합 흐름

> book_v6 기준. 이야기/기술 파트 분리 제거. 하나의 멘토 스타일 흐름으로 재구성.

## 설정

**주인공**: 오픈이 (입사 3년 차 개발자, 신설 팀 첫 발령)
**배경**: 커넥트HR — 사원 800명 규모 중견기업. 사내 AI 비서를 직접 만든다.
**파격 규정 (이야기 장치)**: "신입사원 3년간 연차 없음, 매월 리프레시 데이 1일". 외부 LLM이 절대 알 수 없는 사내 규정 → 환각 증명의 핵심 장치

**캐릭터 삼각 구도**:
| 캐릭터 | 역할 | 대사 패턴 |
|--------|------|----------|
| 팀장 | 힌트 제공자 | 짧은 한마디로 비유 촉발 |
| 동료 | 문제 제기자 / 테스터 | 실사용 피드백으로 문제 공개 |
| 오픈이 | 실행자 / 독자 대리인 | 시행착오를 직접 겪음 |

## 감정 곡선 (챕터별 한 줄)

| # | 챕터 | 감정 곡선 |
|---|------|----------|
| 1 | 환각과 RAG의 첫 만남 | 좌절(환각) → 발견(문서 주면 됨) → 성취(RAG 돌아감) |
| 2 | 일단 사내 시스템부터 | 의욕(API로 감싸자) → 막힘(DB 연결) → 성공(REST 뚫림) |
| 3 | 어떤 문서를 넣을까 | 혼돈(형식 제각각) → 정리(규칙 확정) → 준비 완료 |
| 4 | 문서를 지식으로 바꾸다 | 자신감 → 파싱 지옥 → 청킹 후 빛 |
| 5 | 드디어 답해준다 | 기대 → 첫 답변 성공 → 흐름이 보인다 |
| 6 | 연차도, 규정도, 한번에 | 한계 체감(RAG만으로 안됨) → 에이전트 도입 → 통합 성공 |
| 7 | 실제로 써보니 | 배포 기쁨 → 속도/비용 문제 → 캐시·토큰 관리로 잡음 |
| 8 | 엉뚱한 문서를 가져온다 | 당혹(틀린 검색) → 청킹/리랭커/하이브리드 튜닝 → 개선 |
| 9 | 질문을 제대로 이해 못한다 | 답답(쿼리 뉘앙스) → 쿼리 변형/확장 → 질문 이해력 향상 |
| 10 | PDF 이미지까지 잡아라 | 마지막 허들(멀티모달) → VLM 투입 → 완성 |

## 전환점 (turning points)

| # | 챕터 | 전환의 정체 |
|---|------|------------|
| 1 | CH01 | "LLM 혼자로는 안 된다" — 환각을 직접 본 순간 |

| 3 | CH08~10 | "기본 RAG는 출발점이다. 튜닝으로 완성한다" — 튜닝 서사 |

## 톤·스타일 규칙 (5개)

1. **멘토 스타일** — 독자에게 말 걸듯. "~해보세요", "~해볼까요?"
2. **통합 흐름** — 이야기/기술 파트 분리 금지. 한 섹션 안에서 문제 제기 → 코드 → 개념 정의 → 실행 결과가 자연스럽게 흐른다
3. **미디엄 스토리 농도 (5:5)** — 캐릭터 감정과 기술 설명의 비중을 비슷하게. 한쪽에 치우치지 않는다
4. **챕터 미니맵** — 챕터 상단에 "오늘 만들 것" 3~5개 컴포넌트 박스. 전체 아키텍처는 서문에 한 번만
5. **목표 선언 박스** — 챕터 시작에 "이번 챕터가 끝나면" 3줄. 괄호로 용어 병기

## 서사의 뼈대

**큰 흐름 한 문장**: 오픈이는 LLM의 환각을 보고 좌절했다가, 문서를 읽어주는 법을 배우고, 사내 시스템과 붙이고, 실제 사용자의 불만을 받으며 튜닝의 길로 들어선다.

**세 막으로 나누면**:
- **1막 (CH01~05)**: 환각 발견 → RAG 기본 파이프라인 완성. "답해준다"
- **2막 (CH06~07)**: 에이전트 통합 → 실사용 배포. "써보니 문제가 보인다"
- **3막 (CH08~10)**: 튜닝으로 완성. "이제 진짜 비서 같다"

## 적용 규칙

- 캐릭터 2개 챕터 연속 부재 금지 (최소 1명 대사 등장)
- 이야기 섹션당 감각 묘사 1개 이상 (시각 + 청각/촉각)
- 챕터 오프닝은 내면독백 또는 대사로 시작 (설명문 시작 금지)
- 원형 서사: 챕터 초반 문제 제기자가 끝에서 검증자로 재등장
