5.3 배달 완료 라이프사이클 설계

구현 전, 배달 상태 전이가 기존 흐름과 어떻게 달라지는지 먼저 비교합니다. 지금까지는 배달이 생성되는 즉시 완료로 처리됐습니다. 현실에서는 배달 기사가 물건을 전달해야 완료인데, 이 단계가 빠져 있습니다.

기존 흐름 (BEFORE — 소형 상태도 유지)

그림 5-3. 기존 배달 흐름 - 생성 즉시 완료 처리

이제부터는 실제 택배처럼 접수하면 "배달 중(PENDING)"이 되고, 배달 기사가 수령인에게 전달한 뒤 "배달 완료(COMPLETED)" 버튼을 눌러야 비로소 완료가 됩니다. 그 완료가 어떤 단계를 거쳐 사용자에게 닿는지, CH4의 요청 흐름과 같은 방식으로 한 단계씩 봅니다.

새 흐름 — 1단계: 배달 생성 결과 수신 후 대기

그림 5-4. 1단계 — 배달 생성 이벤트 발행 → 오케스트레이터 수신 후 대기

새 흐름 — 2단계: 배달 기사 완료 → 배달 완료 이벤트

그림 5-5. 2단계 — 배달 완료 API → 배달 완료 이벤트 발행 → 오케스트레이터 수신

새 흐름 — 3단계: 주문 완료 명령 → 주문 완료 → 사용자 실시간 알림

그림 5-6. 3단계 — 주문 완료 명령 발행 → 주문 수신 → WebSocket 알림

세 단계를 코드로 옮기기 위해 이제 세 곳을 수정합니다.

  1. delivery-service에 배달 완료 API를 추가합니다.
  2. orchestrator가 delivery-completed 토픽을 처리하도록 합니다.
  3. order-service에 WebSocket Push를 구현합니다.

샘플 메모 — 제안 A입니다. CH4 카드 템플릿(서비스 박스·Kafka 우편함 누적·발행/수신 화살표·우측 패널)을 그대로 따르되, REST(배달 기사 PUT, 청록)와 WebSocket Push(보라)는 Kafka와 다른 채널이므로 별도 색 화살표로 카드 안에 표시했습니다. 그림 번호(5-3~5-6)·문구는 초안이며 승인 시 5렌즈로 다듬어 챕터에 반영합니다. 제안 B(레인 확장 단일 시리즈)를 보고 싶으면 말씀해 주세요.
