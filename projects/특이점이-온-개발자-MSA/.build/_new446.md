## 4.4 Kafka로 주고받기 - 발행과 구독

그림 4-6부터 4-11까지 본 화살표(발행·구독)는 코드로 옮기면 딱 두 가지 패턴입니다. 보내는 쪽은 토픽에 메시지를 넣고(발행), 받는 쪽은 그 토픽을 듣고 있다가 받습니다(구독). 이 챕터의 모든 Kafka 코드는 이 두 패턴의 반복입니다.

### 4.4.1 발행 - 토픽에 메시지를 넣는다

`KafkaTemplate`에 토픽 이름과 메시지를 넘기면 끝입니다. 주문 서비스가 주문을 저장한 뒤 '주문 생성 이벤트'를 발행하는 코드입니다.

`adapter/producer/OrderEventProducer.java`를 열고 아래 메서드를 작성합니다.

```java [실습 1] adapter/producer/OrderEventProducer.java. 주문 생성 이벤트 발행
@Component
@RequiredArgsConstructor
public class OrderEventProducer {
    private final KafkaTemplate<String, Object> kafkaTemplate;

    public void publishOrderCreated(OrderCreatedEvent event) {
        // "order-created" 토픽에 이벤트를 넣는다
        kafkaTemplate.send("order-created", event);
    }
}
```

`kafkaTemplate.send(토픽, 메시지)` 한 줄이 발행의 전부입니다. 어떤 이벤트를 보내든 토픽 이름만 바꿔 같은 방식으로 보냅니다.

### 4.4.2 구독 - 토픽을 듣고 있다가 받는다

받는 쪽은 메서드에 `@KafkaListener`를 붙이고 토픽 이름과 그룹 이름을 적습니다. 그 토픽에 메시지가 들어오면 이 메서드가 자동으로 실행됩니다.

`adapter/consumer/OrderCommandConsumer.java`를 열고 아래 메서드를 작성합니다.

```java [실습 2] adapter/consumer/OrderCommandConsumer.java. 주문 완료 명령 구독
@KafkaListener(topics = "complete-order-command", groupId = "order-service")
public void completeOrderCommand(CompleteOrderCommand command) {
    // 메시지를 받으면 주문을 완료 처리
    orderService.completeOrder(command.orderId());
}
```

`@KafkaListener(토픽, 그룹)` 메서드 하나가 구독의 전부입니다.

:::note
**`groupId`는 컨슈머 그룹 이름입니다.** 같은 그룹끼리는 메시지를 나눠 받고, 다른 그룹이면 같은 메시지를 각각 받습니다.
:::

이렇게 바뀌면서 order-service의 `createOrder`는 더 이상 product·delivery를 REST로 부르지 않습니다. 주문만 저장하고 '주문 생성 이벤트'를 발행한 뒤 즉시 끝납니다. 나머지 흐름은 orchestrator가 이어받습니다.

## 4.5 누가 무엇을 주고받는가

각 서비스의 코드는 4.4의 발행·구독 패턴에 토픽 이름만 바꿔 끼운 것입니다. 그래서 코드를 일일이 보지 않아도, 어떤 서비스가 무슨 토픽을 구독해 무엇을 하고 무엇을 발행하는지만 알면 전체가 보입니다.

| 서비스 | 구독(받음) | 처리 | 발행(보냄) |
|---|---|---|---|
| **order** | 주문 요청(REST) | 주문 저장(PENDING) | 주문 생성 이벤트 |
| **order** | 주문 완료·취소 명령 | 주문 상태 변경 | — |
| **product** | 재고 차감·복구 명령 | 재고 증감 | 재고 차감 이벤트 |
| **delivery** | 배달 생성·취소 명령 | 배달 생성·취소 | 배달 생성 이벤트 |
| **orchestrator** | 주문 생성·재고 차감·배달 생성 이벤트 | 다음 단계 결정 | 재고 차감·배달 생성·주문 완료 명령 (실패 시 복구·취소 명령) |

order·product·delivery는 명령을 받아 자기 일만 하고 결과를 발행할 뿐, 다음이 무엇인지 모릅니다. 전체 흐름을 아는 것은 orchestrator뿐입니다. 각 서비스의 전체 코드는 GitHub에서 확인하세요.

## 4.6 orchestrator - 흐름을 조율하는 코드

orchestrator만 "이벤트를 받아 다음 명령을 결정"하는 로직을 가집니다. 핵심은 한 패턴입니다 — **이벤트를 구독해 받고, 진행 상태를 기록하고, 다음 명령을 발행한다.** 1단계(주문 생성 이벤트 수신)를 코드로 보면 다음과 같습니다.

`handler/OrderOrchestrator.java`를 열고 아래 메서드를 작성합니다.

```java [실습 3] handler/OrderOrchestrator.java. 주문 생성 이벤트를 받아 재고 차감 명령 발행
@KafkaListener(topics = "order-created", groupId = "orchestrator")
public void orderCreated(OrderCreatedEvent event) {
    // 1. 주문 진행 상태를 메모리에 기록
    states.put(event.orderId(), new WorkflowState(
            event.orderId(), event.address(),
            event.productId(), event.quantity(), event.price()));

    // 2. 다음 단계: 재고 차감 명령 발행
    kafkaTemplate.send("decrease-product-command",
            new DecreaseProductCommand(event.orderId(),
                    event.productId(), event.quantity(), event.price()));
}
```

구독(`@KafkaListener`) → 상태 기록(`states.put`) → 발행(`kafkaTemplate.send`). 그림 4-6·4-7의 흐름이 이 한 메서드에 그대로 담겨 있습니다.

재고 차감 결과를 받는 `productDecreased`, 배달 생성 결과를 받는 `deliveryCreated`도 **똑같은 구조**입니다. 결과 이벤트를 구독해, 성공이면 다음 명령을, 실패면 보상(재고 복구·주문 취소) 명령을 발행합니다(그림 4-8~4-11, 실패 흐름은 그림 4-12). 전체 코드는 GitHub에서 확인하세요.

진행 상태를 담는 `WorkflowState`는 주문 한 건의 정보(주문 ID·상품·주소)를 메모리(`ConcurrentHashMap`)에 보관하는 단순한 객체입니다. orchestrator Pod가 재시작되면 진행 중이던 워크플로우는 사라지므로, 실무에서는 DB나 Redis에 상태를 보관합니다.

이제 Kubernetes에 Kafka와 orchestrator를 추가하고 전체 시스템을 배포합니다.
