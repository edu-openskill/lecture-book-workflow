# 스토리텔링 삽입 계획

> 작성일: 2026-04-11
> 목적: 프롤로그의 선배+오픈이 대화를 챕터로 확장
> 패턴: 문제 인식 → 학습 → 해결

---

## 설계 원칙

- 프롤로그 0.1~0.4의 대사/장면을 챕터에서 자연스럽게 재현한다
- CH05 클로징이 프롤로그 마지막 대사를 그대로 반복하여 원형 구조로 책을 마감한다
- 코드 구현 구간(2.6, 4.6, 5.6)에서는 대화 대신 내면 독백 1줄을 사용한다
- 대화는 짧게 유지한다 (2~4턴). 길어지면 기술서가 아니라 소설이 된다
- 대화 표기: **역할**: "대사" / 내면 독백: *이탤릭*

## 프롤로그-챕터 매핑

| 프롤로그 | 사건 | 챕터 |
|---------|------|------|
| 0.1 금요일 밤, 서버가 멈추다 | 모놀리식 장애 → MSA 검색 | CH01 |
| 0.2 서비스를 나누다 | 동기 REST + 보상 트랜잭션 → "같이 죽는" 문제 | CH02, CH03 |
| 0.3 비동기로 전환하다 | Kafka + 오케스트레이터 | CH04 |
| 0.4 실시간 알림을 추가하다 | WebSocket → 최종 완성 | CH05 |

**선배**: "한 건물에 가게를 다 넣어두면 그렇지. 한 층에 불나면 전체가 대피해야 하잖아."

**오픈이**: "그럼 어떻게 해야 하나요?"

**선배**: "백화점 대신 개별 상점을 열면 되지. 각자 건물을 가지면 한 곳에 불이 나도 나머지는 멀쩡하잖아."

**선배**: "품절 상품을 주문해봐. 그러면 네가 짠 보상 코드가 돌아가는 걸 눈으로 볼 수 있어."

*...아. 전화를 걸고 기다리니까, 상대가 느리면 나도 느려지는 거잖아.*

### 4-F. 클로징 (챕터 끝, "이것만은 기억하자" 앞)

> 프롤로그 연결: 0.3절 "주문 서비스는 멀쩡히 돌아갔고 메시지는 Kafka에 안전하게 쌓였습니다" + 0.4절 "주문이 정말 끝났는지 확인하려면 사용자가 직접 새로고침"

**선배**: "이제 사용자 입장에서 한번 생각해봐. 주문이 끝났는지 알려면 새로고침을 계속 눌러야 하잖아."

### 5-B. 브릿지 (5.2~5.3 사이, 현재 L44~L47)
