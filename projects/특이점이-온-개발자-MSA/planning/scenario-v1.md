# 시나리오 + 버전 설계

## 시나리오

### 한 줄 요약
모놀리식 쇼핑몰의 한계를 느끼고, 동기 MSA → 클린 아키텍처 + K8s → Kafka 비동기 → WebSocket 실시간으로 단계별 진화하는 여정

### 전체 흐름
1. **시작**: 모놀리식 쇼핑몰이 성장하면서 장애 전파, 독립 배포 불가, 확장 한계에 부딪힌다
2. **전개 1**: 서비스를 4개로 나누고 REST로 연결한다. 보상 트랜잭션을 직접 구현하면서 동기 통신의 복잡함을 체감한다
3. **전개 2**: 코드를 클린 아키텍처로 리팩토링하고, K8s에 배포한다. 운영 환경이 갖춰진다
4. **전환점**: 한 서비스의 장애가 전체를 멈추는 문제를 맞닥뜨린다. Kafka로 비동기 전환하고 사가 오케스트레이터를 도입한다
5. **완성**: 폴링의 한계를 느끼고 WebSocket으로 실시간 알림을 구현한다. 브라우저에서 주문하면 즉시 완료 알림이 뜬다

### 최종 결과물
브라우저에서 주문하면 Kafka를 통해 비동기로 재고 차감 → 배달 생성 → 주문 완료가 처리되고, WebSocket으로 실시간 알림이 뜨는 쇼핑몰 MSA 시스템

## 전환점 상세

**4장: 동기 → 비동기 전환**

2장에서 동기 REST로 서비스를 연결했지만, 한 서비스가 죽으면 호출한 서비스도 같이 멈춘다. 배달 서비스가 응답하지 않으면 주문 서비스도 대기하다가 타임아웃이 나고, 사용자는 에러를 만난다. "서비스를 나눴는데 왜 같이 죽지?" — 이 의문이 비동기 전환의 동기가 된다.

Kafka를 도입하면 주문 서비스는 이벤트만 발행하고 바로 응답한다. 배달 서비스가 죽어 있어도 주문은 접수되고, 배달 서비스가 살아나면 밀린 이벤트를 처리한다. 독자는 "동기의 한계 → 비동기의 해결"을 직접 체감한다.
