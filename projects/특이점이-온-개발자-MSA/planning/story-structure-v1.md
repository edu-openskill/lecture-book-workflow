# MSA 책 — 스토리 구조 v1

## 전체 인과 사슬

한 시스템이 챕터마다 한 번씩 한계를 만나고 진화한다. 사건은 발명되지 않고 **직전 챕터에서 만든 코드(ex01~ex04)의 한계가 현실로 닥치는 순간**에서 도출된다. 각 챕터는 같은 모양 — 사건 → 통증 인식 → 깨달음 → 학습 → 결과 → 다음 사건의 씨앗.

```
CH01 모놀리식 장애 → MSA 학습
        ↓ "이제 만들어 봐"
CH02 동기식 ex01 → 보상 코드가 손에 잡힘
        ↓ "새 규칙이 들어왔는데 어디 손대야?"
CH03 DDD + 클린 아키텍처 + Gateway + K8s → ex02
        ↓ "Pod 살아나는 30초 동안은?"
CH04 비동기 Kafka + Orchestration Saga → ex03
        ↓ "사용자는 끝났는지 모름"
CH05 WebSocket 알림 → ex04 (원형 서사 닫힘)
```

---

## CH01 — 모놀리식 한계 인식 + MSA 학습

### 사건 — 금요일 밤 장애
- **트리거**: 주문 폭주 → 한 덩어리 서버가 다 죽음. 로그인까지 막힘. 새벽 3시 알람·전화·터미널 콘솔
- **월요일 아침**: 팀장 호출. "왜 이렇게 됐는지, 어떻게 막을지 보고해라."

### 통증 인식
> *주문이 몰린 것뿐인데 왜 로그인까지 멈췄지? 주문과 로그인이 무슨 상관이지?*

한 서버 안에 회원·상품·주문·배달이 다 들어 있다. 한 곳이 무너지니 다 같이 무너졌다.

### 깨달음 — 선배의 백화점 비유
> **선배**: "한 건물에 가게를 다 넣어두면 그래요. 한 층에서 불나면 전체가 대피해야 하잖아요."

한 덩어리(모놀리식)의 본질을 비유로 인지 → 분리(MSA)가 필요한 이유.

### 학습
- 모놀리식 vs MSA
- 분산 트랜잭션 — DB가 분리되면 단일 트랜잭션 불가
- Saga 패턴 (Choreography·Orchestration)
- 이 책에서 만들 쇼핑몰 4 서비스 청사진

### 결과
"이번 분기는 MSA로 전환" 결정. 전체 5챕터 학습 흐름이 머리에 들어옴.

### CH02로 다리
> **선배**: "개념은 됐어요. 이제 직접 만들어 봐요. 제일 단순한 방법부터."

## CH02 — 학습한 MSA를 동기식으로 구성

### 사건 — 만들다 부딪힌 "되돌릴 수 없는" 호출
- **트리거**: 4 서비스 분리하고 RestClient로 연결. 처음 몇 단계는 잘 굴러간다 — 주문 생성, 재고 차감. 그러다 배달 단계에서 발견

### 통증 인식
재고 차감은 상품 서비스에 HTTP 요청을 보내 성공. 그다음 배달 생성 단계. 만약 배달 생성이 실패하면? 차감한 재고는?

> *모놀리식이라면 `@Transactional`이 자동 롤백했을 텐데, 다른 서버로 보낸 HTTP 요청은 못 되돌린다.*

오픈이의 손가락이 멈춘다. **CH01 학습에서 머리로 알던 분산 트랜잭션 문제를 손으로 처음 마주침.**

### 깨달음 — Saga 패턴의 실체
> **선배**: "기억나죠? Saga. 자동으로 안 되니 직접 짜는 거예요. 재고를 줄였으면 다시 늘리고, 배달을 만들었으면 취소하고. 어디까지 진행됐는지 플래그로 기록하고."

**Choreography Saga 보상 트랜잭션을 직접 손으로**. CH01에서 개념만 알던 것이 코드로 박힌다.

### 학습
- 4 서비스(user·product·order·delivery) 독립 Gradle 프로젝트
- REST + JWT 인증 + Authorization 헤더 전달 (RestClient 인터셉터)
- 플래그(`productDecreased`·`deliveryCreated`) 기반 try/catch 보상 트랜잭션
- Docker Compose로 네 서비스 한 번에 실행

### 결과 — ex01 완성
- 품절 상품(iPhone 15) 주문 시도 → 재고 차감 실패 → 즉시 에러
- 빈 주소로 주문 → 재고 차감 성공 → 배달 실패 → **재고 자동 복구**
- "된다!"

### CH03로 다리
ex01 코드를 다시 보니, `OrderService.createOrder`가 한 함수에 모든 걸 다 들고 있다. 새 비즈니스 규칙이 들어오면 어디 손대야 할지 막막.

## CH03 — DDD + 클린 아키텍처 + Gateway + K8s

### 사건 — 새 요구가 들어왔는데 어디 손대야 할지 모르겠다

**트리거**: ex01 완성 후 며칠. 기획자가 새 요구를 들고 온다.

> **기획자**: "주문 후 30분 이내만 취소 가능으로 해주세요."

오픈이가 `OrderService.cancelOrder`를 연다.

```java
public OrderResponse cancelOrder(int orderId) {
    Order findOrder = orderRepository.findById(orderId)...;
    if (findOrder.getStatus() == OrderStatus.CANCELLED) {
        throw new Exception400("이미 취소");
    }
    // 재고 복구, 배달 취소, 주문 취소
}
```

30분 검증을 어디에 끼울까. if를 하나 더 추가하면 되긴 한다. 그런데 코드를 다시 훑어보니 **같은 종류의 검증이 여기저기 흩어져 있다**.

- `ProductService` — 재고 부족 검증 if
- `ProductService` — 가격 일치 검증 if
- `DeliveryService` — 주소 빈 값 검증 if
- `DeliveryService` — 이미 취소 검증 if

새 비즈니스 규칙이 들어올 때마다 if를 어디에 추가할지 매번 고민이다.

### 통증 인식 — 두 갈래

**갈래 1 — 비즈니스 규칙이 흩어짐**
> **동료**: "근데 이상하지 않아? **`Order`가 자기가 취소 가능한지를 답해야 하는 거 아니야?** `findOrder.canBeCancelled()` 같이. 지금은 `Order`가 그냥 데이터 덩어리고 Service가 다 답하고 있잖아."

**갈래 2 — 컨트롤러가 구현체에 묶임**
컨트롤러가 `OrderService` 구현체를 직접 들고 있다. 환경별·테스트별로 다른 구현체를 끼울 수 없음.

### 깨달음 — 한 뿌리, 두 패턴

> **선배**: "사실 두 가지가 한 뿌리예요. **도메인이 코드의 중심에 와야 한다**는 거. 그런데 그러려면 두 단을 풀어야 해요."

| 단 | 통증 | 풀이 | 패턴 이름 |
|---|---|---|---|
| 1 | 비즈니스 규칙이 Service에 흩어짐. `Order`는 데이터 덩어리 | 도메인 객체에 행위를 캡슐화. `Order.validateCancelable()`·`Order.cancel()` | **DDD** (Rich Domain Model) |
| 2 | 컨트롤러가 `OrderService` 구현체에 직접 의존. 테스트·환경별 교체 안 됨 | 약속(UseCase 인터페이스)을 추출. 컨트롤러는 약속만 안다 | **클린 아키텍처** (의존성 역전) |

두 패턴이 합쳐져 **"도메인을 가운데에 두고 외부 기술이 도메인에 맞춰진다"** 는 한 그림이 된다.

> **선배**: "DDD는 도메인 객체가 자기 규칙을 가진다는 쪽, 클린 아키텍처는 컨트롤러·DB 같은 외부가 도메인 인터페이스에 맞추는 쪽. 둘이 같이 가야 도메인이 진짜 중심이 돼요."

### 학습 1 — DDD + 클린 아키텍처

**DDD (Rich Domain Model)**
- `Order.validateCancelable()`·`Order.cancel()`·`Order.complete()`·`Order.create()`
- 새 규칙이 와도 도메인 메서드만 추가
- Service는 흐름 조율만

**클린 아키텍처 (UseCase 인터페이스)**
- `CreateOrderUseCase`·`GetOrderUseCase`·`CancelOrderUseCase`
- 컨트롤러 → 인터페이스, OrderService → 인터페이스 구현
- 환경별·테스트별 구현체 교체 가능

**패키지 구조**
```
domain/        ← 엔티티 (규칙 캡슐화)
usecase/       ← 인터페이스 + Service 구현
repository/
web/           ← 컨트롤러 (인터페이스만 의존)
adapter/
core/
```

→ 도메인이 가운데, 외부가 그 주위에 맞춰진 모양.

### 학습 2 — 외부 진입 통합: Nginx API Gateway

코드는 정리됐다. 팀장이 본다.

> **팀장**: "좋네요. 이제 운영에 올립시다."

올리려고 보니 4 서비스가 `:8081`·`:8082`·`:8083`·`:8084` 따로 노출. 클라이언트가 네 포트를 다 알 수 없다.

> **선배**: "안은 정리했으니 밖에서 들어오는 문도 하나로 모아요. Nginx 한 대가 URL 경로로 나눠주면 돼요."

`/login` → user, `/api/orders` → order, `/api/products` → product, `/api/deliveries` → delivery.

### 학습 3 — 운영 인프라: K8s

Docker Compose로 운영 서버에 띄움. 컨테이너가 죽으면 사람이 직접 살림. 환경 변수도 코드에 박혀 있음.

> **운영팀**: "K8s 환경 만들어줬어요. 매니페스트만 작성해서 올리면 돼요."

- MySQL + dev/prod 프로파일 분리
- ConfigMap·Secret·Deployment·Service·Ingress
- minikube로 검증

### 결과 — ex02 완성
`minikube tunnel`로 외부에서 들어옴. 도메인이 자기 일을 알고, 컨트롤러는 약속만 알고, 밖은 한 입구, 위에서는 K8s가 Pod이 죽어도 살린다.

> *DDD가 왜 필요한지 알겠다. 비즈니스 규칙이 도메인 안에 있어야 새 요구가 와도 안 무섭다.*

### CH04로 다리
> **선배**: "K8s가 Pod 살려주죠. 그런데 product-service Pod이 살아나는 30초 동안은 어떻게 될까요?"

오픈이가 시도. **주문 요청이 그대로 멈춘다.** K8s가 살려주는 시간 동안에도 order는 굳어 있다. CH04 사건의 시작점.

## CH04 — 비동기 전환 (클라이맥스)

### 사건 — 한쪽이 잠시 멈췄는데 전체가 멈췄다

**트리거**: ex02 운영 중. 어느 날 product-service Pod이 메모리 누수로 재시작. K8s가 자동으로 살려주는 데 30초. 그 30초 동안 **주문 요청이 모두 504**. 사용자 클레임 폭주.

### 통증 인식 — 두 갈래

**갈래 1 — 동기 호출의 연쇄 장애**
> *K8s가 Pod 살려준다고 좋아했는데, 그 살아나는 30초 동안 주문도 같이 굳잖아. 분리한 의미가 없잖아.*

`productClient.decreaseQuantity()`가 product가 살아날 때까지 그냥 기다린다. order가 product 응답을 동기로 기다리니 product의 장애가 그대로 order에 전염된다.

**갈래 2 — 보상 try/catch가 무서워진다**

```java
try {
    // 1. 주문 생성
    // 2. 재고 차감
    productDecreased = true;
    // 3. 배달 생성
    deliveryCreated = true;
    // 4. 주문 완료
} catch (Exception e) {
    if (deliveryCreated) { ... 배달 취소 ... }
    if (productDecreased) { ... 재고 복구 ... }
}
```

새 단계가 들어가면 if 플래그가 늘고 catch 분기도 늘어난다. 점점 깊어져 손대기 무섭다.

### 깨달음 — 메시지를 사이에 두자
> **선배**: "전화를 끊고 메시지를 남기는 방식으로 바꿔요. order는 'order-created' 한 줄만 던지고 즉시 PENDING으로 반환. 누가 받아서 처리할지, 실패하면 누가 되돌릴지는 가운데에 **지휘자**를 따로 두는 거예요. orchestrator."

두 통증이 한 뿌리 — **서비스끼리 직접 전화 걸지 말자**. Kafka가 우체통, orchestrator가 지휘자.

### 학습
- Kafka(Topic·Producer·Consumer·Consumer Group)
- 8개 토픽 + Orchestration Saga
- orchestrator 새 서비스 (REST API 없고 Kafka 이벤트만 처리)
- order-service의 try/catch 보상이 통째로 사라짐 (80 → 56줄)

### 결과 — ex03 완성
product-service Pod 일부러 내려본다. 주문은 PENDING으로 받음. product 복구되면 Kafka에 쌓인 메시지가 자동으로 처리. 자동 롤백도 OK.

> *진정한 독립을 이룬 기분.*

### CH05로 다리
잘 굴러가는데 사용자 입장에서 한 가지가 답답하다 — 주문이 PENDING으로 반환되고 끝났는지 알 수 없다.

## CH05 — WebSocket 실시간 알림

### 사건 — 사용자는 깜깜하다

**트리거 1**: 사용자 클레임 — "주문했는데 끝났는지 모르겠어요. 새로고침 계속 해야 해요."
**트리거 2**: 동료 의문 — "배달은 만들자마자 COMPLETED네? 기사가 직접 전달한 시점이 반영 안 되네."

### 통증 인식
ex03 안쪽은 깔끔하지만 두 가지가 사용자 경험을 망친다.

1. 비동기로 바꿔서 PENDING으로 즉시 반환 → 클라이언트는 "끝났는지" 알 길이 폴링뿐
2. 배달이 생성 즉시 자동 완료 — 현실의 배달 기사가 없는 가짜 모델

### 깨달음 — 서버가 먼저 말을 걸자 + 배달도 진짜로
> **선배**: "택배 기다리며 현관문 5분마다 여는 거 같잖아요. 초인종을 달아요. 서버가 클라이언트에게 먼저 말을 거는 거예요. 그리고 배달도 진짜 라이프사이클로 — 생성 시 PENDING, 기사가 완료 API 누르면 COMPLETED."

두 통증이 한 뿌리 — **사용자 경험까지 시스템이 책임지자**.

### 학습
- WebSocket + STOMP — 서버가 클라이언트에게 push
- 배달 라이프사이클: PENDING → COMPLETED (기사 API 호출 시점)
- `delivery-completed` 토픽 추가 — orchestrator가 배달 완료를 받아 주문 완료 명령 발행
- 프론트엔드 — SockJS로 `/topic/orders/{userId}` 구독
- Nginx WebSocket 프록시 (Upgrade 헤더)

### 결과 — ex04 완성
브라우저가 토큰 받고 주문 → PENDING 표시. 배달 기사가 `PUT /api/deliveries/{id}/complete`를 호출 → 사용자 화면이 1초도 안 되어 "주문 완료!"로 바뀜.

> *시스템이 살아 움직이며 사용자에게 말을 걸어온다.*

### 마무리 — 원형 서사
책 첫 장면(금요일 밤 한 덩어리가 같이 죽음) 회수. 한 덩어리에서 시작해서 사용자에게 직접 말을 거는 분산 시스템까지 왔다.

- 학습이 깨달음에서 **자연 도출** ✓
