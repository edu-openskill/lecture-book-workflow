## CH01 — MSA란 무엇인가 (5개)

### [삽입] ## 1.1 앞

월요일 아침, 오픈이는 선배를 찾아갔습니다. 금요일 밤 장애의 악몽이 아직 가시지 않았습니다.

**오픈이**: "선배님, 금요일에 주문이 몰렸는데 로그인까지 같이 죽었어요. 주문이랑 로그인이 무슨 상관인데 같이 멈추는 건지 모르겠어요."

**선배**: "한 건물에 가게를 다 넣어두면 그렇지. 한 층에서 불나면 전체가 대피해야 하잖아."

### [삽입] ## 1.2 앞

*이래서 서버가 전부 같이 죽었구나. 한 건물에 다 넣어뒀으니까.*

**오픈이**: "그럼 어떻게 해야 하나요?"

**선배**: "백화점 대신 개별 상점을 열면 되지. 각자 건물을 가지면 한 곳에서 불이 나도 나머지는 멀쩡하잖아."

### [삽입] ## 1.3 앞

**오픈이**: "우리 쇼핑몰로 치면 어떻게 나누면 되나요?"

**선배**: "회원, 상품, 주문, 배달. 이 네 개부터 떼어내 봐."

### [삽입] ### 1.4.3 앞

*재고는 줄였는데 배달이 실패하면... 자동으로 안 돌아간다고?*

**오픈이**: "선배님, 이거 되돌리는 방법이 없나요?"

**선배**: "은행 송금 실패하면 환불 절차 밟잖아. 그거랑 같은 원리야. Saga 패턴이라고 해."

### [삽입] ## 이것만은 기억하자 앞

**선배**: "개념은 이 정도면 됐어. 이제 직접 만들어봐. 제일 단순한 방법부터."

*손가락이 움직이지 않았던 게 엊그제인데, 이제 뭘 만들어야 할지는 보인다.*

## CH02 — 동기식 MSA 구현 (5개)

**선배**: "일단 각자 따로 띄워서 REST로 연결해봐. 서로 전화를 거는 것처럼 직접 호출하는 거야. 제일 단순한 방법이지."

### [삽입] ## 2.6 앞

*회원, 상품, 배달까지는 각자 제 일만 하면 된다. 진짜 문제는 이것들을 엮는 주문 서비스다.*

**오픈이**: "재고를 줄인 다음에 배달 만들다가 실패하면요?"

**선배**: "그때를 대비해서 되돌리는 코드를 직접 짜는 거지. 어디까지 진행됐는지 기록해 뒀다가, 실패하면 역순으로 취소해."

### [삽입] ### 2.6.10 OrderService 앞 (createOrder 코드 직전)

**선배**: "이제 진짜 핵심이야. 각 단계마다 플래그를 꽂아두고, 실패하면 플래그 보고 되돌려."

*테이프를 감듯 역순으로... 그 말이 이거였구나.*

### [삽입] ### 2.7.3 시나리오 테스트 앞

**오픈이**: "전부 띄웠는데, 진짜 되돌아가는지 어떻게 확인하죠?"

**선배**: "품절 상품을 주문해봐. 그러면 네가 짠 보상 코드가 돌아가는 걸 눈으로 볼 수 있어."

### [삽입] ## 이것만은 기억하자 앞

*품절 상품을 주문하니까 에러가 나고, 재고가 자동으로 복구됐다. 된다!*

**선배**: "잘 됐네. 근데 한 가지 더 생각해봐. 상품 서비스가 느려지면 주문 서비스는 어떻게 될까?"

*...아. 전화를 걸고 기다리니까, 상대가 느리면 나도 느려지는 거잖아.*

## CH03 — 클린 아키텍처와 Kubernetes (4개)

### [삽입] ## 3.1 앞

**오픈이**: "선배님, 동작은 하는데 코드가 좀 지저분해요. 서비스 교체하려면 컨트롤러까지 뜯어야 하고요."

**선배**: "구조를 정리하고, 제대로 된 운영 환경에도 올려봐야지. 두 가지 숙제야."

### [삽입] ## 3.4 앞

**오픈이**: "구조가 깔끔해졌는데, H2는 서버 끄면 데이터가 날아가잖아요."

**선배**: "운영에서는 MySQL 쓰고, DB 접속 정보는 코드 밖에서 넣어줘야 해. 프로파일로 분리하면 돼."

### [삽입] ## 3.6 앞

*Docker Compose로 한 번에 띄우는 건 편한데, 서비스가 죽으면 내가 직접 다시 띄워야 한다.*

**선배**: "Kubernetes는 네가 원하는 상태를 적어두면, 알아서 그 상태를 유지해줘. 서비스가 죽으면 자동으로 다시 띄우고."

### [삽입] ## 이것만은 기억하자 앞

**오픈이**: "Kubernetes에서도 주문이 잘 되네요!"

**선배**: "그런데 product-service가 잠깐 다운되면 어떻게 될 것 같아?"

*하나가 죽으면 나머지도 같이 멈추는 건... 아직 안 풀렸다.*

## CH04 — 비동기 MSA, Kafka (6개)

### [삽입] ## 4.1 앞

**오픈이**: "선배님, 상품 서비스 껐다 켰더니 주문까지 멈춰요. 서비스 분리한 의미가 없는 것 같아요."

**선배**: "서로 직접 전화를 걸어서 기다려주니까 그렇지. 이제 전화를 끊고 메시지를 남기는 방식으로 바꿔봐. Kafka라는 녀석이 있어."

### [삽입] ## 4.3 앞

**오픈이**: "메시지 방식으로 바꾸니까, 전체 흐름을 누가 관리해야 할지 모르겠어요."

**선배**: "오케스트레이터를 만드는 거야. 지휘자가 전체 악보를 보고 순서를 정하는 것처럼, 각 서비스는 자기 일만 하고 결과만 보고하게 해."

### [삽입] ## 4.4 앞

**선배**: "설계는 끝났어. 이제 직접 짜봐. order-service부터 REST 호출을 걷어내고 Kafka 이벤트로 바꿔."

### [삽입] 4.6 중간 — 2단계(productDecreased) 뒤, 3단계(deliveryCreated) 소제목 앞

*실패하면 이미 줄인 재고만 골라서 되돌리고, 전부 성공하면 배달을 만들라고 명령한다. 지휘자가 정말로 악보를 보고 있구나.*

### [삽입] ### 4.8.5 롤백 확인 앞

**오픈이**: "PENDING이었다가 COMPLETED로 바뀌었어요! 그런데 롤백도 자동으로 되나요?"

**선배**: "품절 상품으로 주문해봐. orchestrator 로그를 보면 보일 거야."

### [삽입] ## 이것만은 기억하자 앞

*상품 서비스를 꺼봤다. 주문 서비스는 멀쩡히 돌아갔고, 메시지는 Kafka에 쌓였다. 다시 켜자 밀려 있던 메시지가 순서대로 처리됐다. 단순히 서비스를 나눈 것을 넘어, 진정한 독립을 이룬 기분이다.*

**선배**: "이제 사용자 입장에서 한번 생각해봐. 주문이 끝났는지 알려면 새로고침을 계속 눌러야 하잖아."

## CH05 — 실시간 알림, WebSocket (5개)

### [삽입] ## 5.1 앞

**오픈이**: "사용자가 새로고침을 누르지 않아도, 주문 완료를 바로 알게 하고 싶어요."

**선배**: "그럼 WebSocket을 붙여봐. 서버가 먼저 사용자에게 말을 걸 수 있게 통로를 열어주는 거야."

### [삽입] ## 5.3 앞

**오픈이**: "근데 지금은 배달이 생성되자마자 완료되잖아요. 배달 기사가 진짜 전달한 다음에 완료 처리해야 하지 않나요?"

**선배**: "맞아. 배달 상태를 PENDING으로 바꾸고, 기사가 완료 API를 호출해야 COMPLETED가 되게 고쳐."

### [삽입] ## 5.6 앞

*delivery-service가 완료 이벤트를 보내고, orchestrator가 주문 완료를 지시한다. 이제 남은 건 사용자에게 알려주는 것뿐이다.*

### [삽입] 5.7 테스트 중간 — 배달 완료 API 호출 스크린샷 뒤, 주문 조회 단계 앞

*배달 완료 버튼을 눌렀다. 이제 주문도 바뀌었을까?*

### [삽입] ## 이것만은 기억하자 앞

*직접 주문을 넣어보았습니다. 화면에 "처리 중"이 표시되었고, 배달 완료 신호를 보내자 1초도 안 되어 화면이 저절로 바뀌었습니다. "주문 완료!" 새로고침을 누르지 않았는데도 시스템이 살아 움직이며 저에게 말을 걸어온 것입니다.*

완성된 시스템을 선배에게 보여주었습니다.

**오픈이**: "금요일 밤에 서버가 터졌을 때는 정말 막막했는데, 한 단계씩 부딪히며 오다 보니 결국 해내게 되네요."

**선배**: "처음부터 모든 정답을 알고 시작하는 사람은 없어. 부딪히고, 고치고, 다시 만드는 것. 그게 개발자에게는 가장 빠른 길이란다."
