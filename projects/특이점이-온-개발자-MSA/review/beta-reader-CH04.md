## 공통 피드백

### 긍정

1. **토픽 맵 표(4.2.3)** — 9개 토픽을 한눈에 보여주는 표가 큰 도움이 됨. 구현 전에 전체 그림을 보여주는 접근이 정확함.
2. **Orchestration Saga 흐름 표** — 성공/실패 흐름을 단계별 표로 표현한 것이 직관적임.
3. **WorkflowState 설명** — `decreasedProductIds`(어떤 상품)와 `processed`(몇 개)를 분리한 이유를 명확히 설명.
4. **카페 비유** — 카운터 대기 vs 진동벨은 비동기 개념을 직관적으로 전달함.
5. **Kafka yaml 주석** — `kafka-deploy.yml` 내 각 환경변수 설명이 상세하고 실용적임.
6. **롤백 확인 실습** — 품절 상품 주문으로 실패 흐름을 직접 확인하는 구성이 좋음.

### 부정

1. **이야기 파트 분량 부족** — 4.1 전체가 단 2개 절(카운터/진동벨)로 끝남. 캐릭터(오픈이)가 등장하지 않아 스토리텔링 부재.
2. **"이것만은 기억하자" 없음** — 챕터 닫기 섹션이 "정리"로 대체되어 있음. 규칙 위반.
3. **product/delivery 컨슈머 실습 공백** — 4.5절에서 "order-service와 동일한 패턴"이라고 넘기고 GitHub 참조. 핵심 실습 서비스를 독자가 직접 작성하지 않음.
4. **이벤트/커맨드 실제 JSON 메시지 미표시** — 메시지가 토픽에 올라갈 때 어떻게 생겼는지 JSON 예시가 없음. 스타일 규칙 "데이터는 보여준다" 위반.
5. **3→4장 난이도 점프 안내 없음** — 이 챕터가 전체 책에서 가장 어렵다는 사전 안내가 없음. 독자 심리 준비 부재.
6. **blockquote 헤더 형식 불일치** — `> 한 줄 요약:` / `> 핵심 개념:` 형식 없이 버전/설명이 혼재.

## 페르소나별 상세

### P1 — Java 백엔드 2년차, Kafka 처음 (주요 독자)

**통과 여부**: 조건부 통과

**이야기 파트**: 카운터/진동벨 비유는 바로 이해됨. 그런데 비유가 두 단락으로 끝나고 기술 파트로 넘어가는 게 너무 급함. "아, 이게 왜 필요한지 알겠다"는 감정까지는 전달되지만 "내가 겪어본 그 상황"으로 연결되지 않음.

**기술 파트**: 토픽맵 표가 구현 전에 나오는 게 큰 도움. `@KafkaListener` 어노테이션이 어떻게 동작하는지 처음에 간략히 설명해줬으면 좋겠음. `groupId` 옵션이 4.4.4에서 properties로만 설명되다가 4.4.8에서 어노테이션 안에 직접 쓰이는데 왜 두 군데에 다 있는지 혼란.

**실습**: OrderEventProducer, OrderCommandConsumer, OrderService는 잘 따라갈 수 있음. 문제는 4.5절 — product/delivery는 "동일한 패턴"이라고 GitHub로 넘기는데, 이걸 직접 작성해보지 않으면 Orchestrator 코드를 이해할 때 흐름이 끊김. OrderOrchestrator 코드가 길고 분기가 많아 처음 보면 당황함. 각 단계를 흐름도 이미지가 받쳐줘서 그나마 버팀.

**막히는 지점**: `ConcurrentHashMap`과 `newKeySet()`이 왜 쓰이는지 주석만으로 이해되지 않음. `states`가 메모리에 있다는 의미(재시작하면 날아감)를 모름.

### P2 — Java 백엔드 5년차, MSA 경험 약간

**통과 여부**: 통과

**이야기 파트**: 도입부가 짧아서 오히려 빠르게 기술로 넘어가기 좋음. 5년차에게는 비유보다 흐름 표가 더 유용.

**기술 파트**: 토픽맵, 성공/실패 흐름 표가 패턴을 파악하는 데 충분함. WorkflowState 상태 추적 방식이 메모리 기반임을 명시한 게 좋음.

**아쉬운 점**: Choreography Saga와의 비교가 없음. 왜 Choreography를 안 쓰고 Orchestration을 선택했는지 이유를 한 줄이라도 제시해줬으면 좋겠음.

**실습**: KRaft 설정 yaml 설명이 상세해서 좋음. product/delivery를 직접 구현하지 않는 것은 이 독자에겐 괜찮음.

### P3 — Python 백엔드, Java/Kafka 낯설음

**통과 여부**: 탈락

**이야기 파트**: 비유는 이해됨. 문제는 기술 파트에서 Java 어노테이션(`@Component`, `@KafkaListener`, `@Transactional`, `@Data`)이 설명 없이 쏟아짐. 처음에 간략한 어노테이션 역할 요약이라도 있었으면 좋겠음.

**막히는 지점**: `@KafkaListener`가 어떻게 연결되는지 Spring이 무엇을 자동으로 해주는지 감이 안 잡힘. `KafkaTemplate<String, Object>`에서 제네릭 타입이 왜 필요한지.

**결론**: 이 책의 대상 독자(Spring Boot 경험자)에 속하지 않아 탈락이 예상되지만, Python 개발자 중 Java 전환을 고려하는 독자가 있다면 기본 어노테이션 한 줄 설명이 큰 도움이 됨.

### P5 — PM/기획자, 비개발자

**통과 여부**: 조건부 통과

**이야기 파트**: 카운터/진동벨 비유로 "왜 바꿔야 하는지"는 완전히 납득됨. "하나가 멈추면 다른 하나도 멈춥니다"가 핵심을 한 문장으로 전달함.

**막히는 지점**: 4.2부터 기술 코드가 나오면서 완전히 이탈. PM에게는 4.1(이야기)과 4.3(흐름 표) 정도가 읽을 수 있는 한계.

**결론**: 이 책의 주 독자가 아니므로 탈락 예상. 그러나 4.1과 4.3.1~4.3.2의 흐름 표는 비개발자에게도 전달력이 있음.

### P6 — 비전공 전직자, 이벤트 개념 처음

**통과 여부**: 탈락

**이야기 파트**: 비유는 이해됨. 그러나 "이벤트"와 "커맨드"의 차이가 토픽맵 표에서만 암묵적으로 드러날 뿐 명확한 정의가 없음. "command가 붙은 토픽은 명령, 없으면 결과" 설명이 있지만 개념 자체를 처음 접하는 독자에게 부족함.

**막히는 지점**: Orchestration Saga라는 단어 자체가 낯섦. "챕터 1에서 소개한"이라고 나오는데 챕터 1을 기억하지 못하는 독자에게는 단절.

### P7 — K8s/인프라 시니어

**통과 여부**: 통과

**K8s 배포 품질**: kafka-deploy.yml의 각 환경변수 설명이 상세하고 정확함. KRaft 모드 선택 이유(ZooKeeper 불필요)가 명확히 설명됨. 클러스터링 개념(CLUSTER_ID, NODE_ID, QUORUM_VOTERS)도 잘 정리됨.

**아쉬운 점**: Kafka PersistentVolume 설정이 없음. 실습용이므로 괜찮지만, "실습용으로 볼륨은 생략합니다 — Pod 재시작 시 메시지가 소멸합니다" 한 줄 경고가 있었으면 좋겠음. `kubectl wait` 실패 시 대처 방법 없음.

**긍정**: 배포 순서(Kafka 먼저 → ready 대기 → 나머지)가 명확히 안내됨.

### P8 — MSA+Kafka 시니어

**통과 여부**: 조건부 통과

**기술 파트**: Orchestration Saga 구현 자체는 정확함. WorkflowState 설계 의도 설명이 명확함.

**핵심 아쉬움**: Choreography와 Orchestration의 선택 이유가 없음. 이 책의 독자가 아니라 동료에게 추천하거나 설득할 때 이 부분이 중요함. "중앙 집중 추적이 가능하다"는 Orchestration의 장점 한 줄만 있어도 충분함.

**기술적 우려**: WorkflowState가 메모리(`ConcurrentHashMap`)에 저장되는데, orchestrator Pod가 재시작되면 진행 중인 워크플로우가 모두 소멸함. 이를 "의도 밖 심화"로 넘기더라도 "주의: 이 구현은 메모리 기반입니다" 한 줄은 있어야 함.

### P9 — CS 학생, 분산 시스템 이론

**통과 여부**: 통과

**이론 연계**: Kafka의 오프셋 기반 소비 방식 설명이 간결하게 잘 됨. RabbitMQ와의 비교(읽어도 삭제 안 됨)가 이론과 실무를 연결함.

**기술 파트**: OrderOrchestrator의 분산 상태 관리(`ConcurrentHashMap`) 코드가 이론 지식과 연결되어 흥미로움. 다만 "왜 ConcurrentHashMap인가"에 대한 설명("Kafka 리스너가 멀티스레드로 동작하므로")이 주석으로만 처리된 게 아쉬움 — 본문에 한 문장으로 뽑아내면 더 좋을 것.

**막히는 지점**: `kafkaTemplate.send(topic, key, value)` 에서 두 번째 인자(key)가 왜 `String.valueOf(orderId)`인지 설명 없음. 파티셔닝과 관련된 내용이지만 "의도 밖"이므로 한 줄 설명만으로 충분.

---

### P10 — 스타트업 CTO

**통과 여부**: 조건부 통과

**복잡도 대비 효과 판단**: 챕터를 보면 서비스 3개(order/product/delivery) → 서비스 4개(+ orchestrator) + Kafka 추가. 코드량도 크게 늘어남. 이게 어떤 문제를 해결하는지 4.1이 명확히 설명하지만, "이 복잡도를 감수할 가치가 있는가"에 대한 답이 부족함.

**아쉬운 점**: "언제 Kafka가 필요한가 / 언제 REST로 충분한가"에 대한 한 줄 기준이 없음. CTO 입장에서 가장 궁금한 질문.

**긍정**: 챕터 마지막 "정리"에서 "이제 product-service가 잠깐 다운되어도 order-service는 이벤트를 Kafka에 올려두고 즉시 반환합니다"가 도입 효과를 명확히 설명함.

"정리" 섹션 앞에 다음 형식으로 추가.

```markdown
## 이것만은 기억하자

- 동기 REST 호출은 하나가 느려지면 전체가 느려진다. Kafka는 이 결합을 끊는다.
```
