## 심각도별 이슈

### 높음

**[이슈 H-1] "이것만은 기억하자" 섹션 없음**

스타일 규칙(글쓰기 원칙 8번)에 따르면 챕터 닫기는 "이것만은 기억하자" + 다음 챕터 예고 형식이어야 합니다. 현재 파일은 "정리"라는 일반 요약으로 대체되어 있습니다. "이제 직접 코드를 작성할 시간입니다..."라는 예고 문장은 있으나, 핵심 요약 섹션의 제목이 규칙과 불일치합니다.

- 영향 페르소나: 1, 2, 4, 6, 9 (5명)
- 수정 방향: `## 정리` → `## 이것만은 기억하자` 로 제목 변경, 내용은 현행 유지 가능

### 중간

**[이슈 M-1] 챕터 헤더 blockquote 없음**

구조 규칙에 따르면 챕터 제목 바로 아래에 `>` blockquote로 버전/요약/핵심개념을 작성해야 합니다. 대신 `### 학습 목표` 섹션이 있는데, 이는 교과서적 형식으로 이야기 스타일과 맞지 않습니다.

- 영향 페르소나: 1, 3, 6, 9 (4명)
- 수정 방향: `### 학습 목표` 삭제 후 blockquote 형식으로 교체

**[이슈 M-2] 이미지 캡션 접두어 규칙 위반**

구조 규칙에 따르면 캡션에 "그림 N-N:" 접두어를 붙이면 안 됩니다(Typst가 자동 부여). 현재 캡션 상당수에 `*그림 1-1: ...*`, `*그림 1-7: ...*` 형태가 포함되어 있습니다.

위반 위치:
- `*그림 1-1: 백화점 — 모든 매장이 한 건물에 모여 있다*`
- `*그림 1-2: 모놀리식 쇼핑몰 — 모든 기능이 하나의 서버에*`
- `*그림 1-3: 개별 상점 — 각 매장이 독립된 건물로 운영*`
- `*그림 1-4: 마이크로서비스 쇼핑몰 — 기능별로 서비스를 분리*`
- `*그림 1-7: 서비스별 독립 데이터베이스*`

(그림 1-5, 1-6, 1-8, 1-9, 1-10은 접두어 없이 올바르게 작성됨)

- 영향 페르소나: 2, 7, 8 (3명) — 실제로는 PDF 빌드 시 전체 독자에게 영향
- 수정 방향: 접두어 제거, 설명만 남김

**[이슈 M-3] 이야기 파트(등장인물 서사) 부재**

책의 핵심 컨셉은 "교과서가 아니라 이야기"이며, 스타일 규칙 원칙 1번에서 "모든 챕터는 이야기 파트로 시작한다"고 명시합니다. CH01은 백화점 비유를 쓰지만 등장인물(오픈이, 팀장, 선배)이 없고 서사 흐름이 없습니다. 전체적으로 설명문 구조입니다.

- 영향 페르소나: 1, 4, 6, 9, 10 (5명)
- 심각도 판단: 1장은 개요 챕터 특성상 어느 정도 허용될 수 있으나, 등장인물이 전혀 없으면 이후 챕터와의 연속성이 단절됨
- 수정 방향: 오픈이가 처음 팀에서 "모놀리식 서버가 터진 날" 경험하는 짧은 도입 장면 추가 권장

### 낮음

**[이슈 L-1] 1.1, 1.2 섹션 제목에 콜론 포함**

스타일 규칙에서 "문장 끝 콜론 금지"를 명시합니다. 섹션 제목인 `1.1 모놀리식 : 쇼핑몰을...`, `1.2 마이크로서비스 : 역할을 나눈다`에 콜론이 사용되었습니다. 섹션 제목의 콜론이 "문장 끝"에 해당하는지는 해석 여지가 있으나, 일관성을 위해 제거 또는 `—` 대시로 교체하는 것이 좋습니다.

- 영향 페르소나: 7, 8 (2명)
- 수정 방향: `:` → `—` 또는 부제목 제거

**[이슈 L-2] Java 코드 블록 위치의 맥락 불명확**

1.4.1에 `@Transactional` 코드가 `[참고]` 태그와 함께 삽입되어 있습니다. 코드 자체는 이야기 파트와 기술 파트가 혼재하는 구간에 배치되어 있고, `[참고]` 레이블은 올바른 분류이나 앞뒤 설명이 "이야기 비유 → 코드 → 이야기 비유"로 이어지는 구조라 흐름이 살짝 끊깁니다.

- 영향 페르소나: 2, 3, 8 (3명)
- 수정 방향: 코드 블록을 1.4.1 끝으로 이동하거나, 코드 앞 전환 문장 보강

## 페르소나별 상세

### 1. 김도현 (Java 백엔드 2년차, MSA 처음)

**[이야기]** 백화점 비유가 직관적으로 와닿았습니다. 모놀리식의 한계를 "배포가 두렵다", "장애가 전파된다" 같은 구체적 표현으로 정리해줘서 고개가 끄덕여졌습니다. 다만 처음부터 설명 위주라 약간 교과서 느낌이 납니다. 오픈이 같은 인물이 등장했다면 더 몰입했을 것 같습니다. "학습 목표" 섹션이 목차처럼 읽혀서 긴장감이 풀렸습니다.

**[기술]** Saga 패턴을 Choreography/Orchestration 두 가지로 구분하고 비유(동네 가게 vs 웨딩 플래너)로 설명한 것이 명확했습니다. 각 정의 blockquote 처리가 좋았습니다. 분산 트랜잭션 개념을 은행 비유로 풀어낸 것도 이해에 도움이 됐습니다. 챕터 2~5 아키텍처 로드맵이 있어서 전체 흐름이 보였습니다.

**[총평]** 첫 챕터로서 개념은 잘 잡아줍니다. 다만 "이것만은 기억하자" 형식이 아닌 "정리"로 끝난 게 아쉽습니다. 다음 챕터 예고 문장("처음에는 단순하게 시작합니다. 그 단순함이 나중에 어떤 문제를 만드는지 직접 느껴보는 것이 챕터 2의 핵심입니다.")은 호기심을 자극해서 좋았습니다.

**[점수]** 이야기 3/5 | 기술 4/5

### 2. 이수진 (Java 백엔드 5년차, 모놀리식, MSA 도입 고려)

**[이야기]** 백화점-개별 상점 비유는 이미 알고 있는 비유지만, 표로 정리된 비교(전기 배선 교체 시 임시 폐쇄 vs 전 층 영업 중단)가 실무 감각에 딱 맞았습니다. "팀이 커지면 충돌이 잦아진다"는 표현이 실제로 경험한 상황이라 공감이 됐습니다.

**[기술]** `@Transactional` 코드가 `[참고]` 태그로 나온 건 적절합니다. 이미 아는 내용이라 빠르게 넘길 수 있었습니다. Choreography vs Orchestration 비교가 명확한데, 두 방식의 장단점 한 줄 요약이 있었으면 더 좋았을 것 같습니다. 이미지 캡션 일부에 "그림 1-1:" 접두어가 있는데, 이건 출판 시 중복 번호가 생길 수 있는 문제입니다.

**[총평]** MSA 도입을 고려하는 시니어 개발자에게 필요한 포인트(독립 DB, Saga 패턴, 전체 로드맵)를 잘 짚었습니다. 5년차 입장에서는 분량이 적당하고 빠르게 읽힙니다.

**[점수]** 이야기 3/5 | 기술 4/5

### 3. 박민준 (Python 백엔드, Django/Flask, Spring 낯섦)

**[이야기]** 백화점 비유는 언어와 관계없이 이해 가능합니다. MSA 개념 자체는 언어 독립적이라 따라갈 수 있었습니다. 다만 1.4.1에 Java 코드가 갑자기 나와서 흐름이 끊겼습니다. `@Transactional`이 뭔지 모르는데 그냥 넘어가도 되는 건지 불안했습니다.

**[기술]** 코드 주석 "// 모놀리식: 하나의 트랜잭션으로 처리 가능"과 "// 실패 시 세 가지 모두 자동 롤백"이 코드 내용을 충분히 설명하고 있어서 Java 몰라도 흐름은 이해했습니다. 하지만 "동작 이해용입니다"라는 레이블 외에 "Java를 몰라도 주석만 읽으면 됩니다"라는 한 문장이 있었으면 안심이 됐을 것 같습니다.

**[총평]** Python 개발자도 MSA 개념 이해에는 문제없습니다. 코드 블록 1개가 부담이 됐지만 무시하고 읽어도 흐름이 이어졌습니다.

**[점수]** 이야기 3/5 | 기술 3/5

### 4. 최지은 (프론트엔드 React 3년차, 백엔드 구조 궁금)

**[이야기]** 백화점 비유가 부드럽게 읽혔습니다. 프론트엔드에서 본 경험과 연결이 됐습니다(API 하나가 느려지면 화면 전체가 굳는 상황). 이야기 파트에 등장인물이 없어서 소설 같은 몰입감은 없었지만, 설명은 친절했습니다.

**[기술]** 챕터 2~5 로드맵 그림이 있어서 이 책을 다 읽으면 어디까지 이해하게 되는지 그림이 그려졌습니다. 분산 트랜잭션을 은행 비유로 설명한 부분이 백엔드 구조를 처음 접하는 입장에서 이해하기 쉬웠습니다.

**[총평]** 프론트 입장에서 백엔드 아키텍처를 이해하는 데 충분한 첫 챕터입니다. "이것만은 기억하자" 같은 핵심 요약 마커가 있었으면 중요한 개념을 더 잘 캐치할 수 있었을 것 같습니다.

**[점수]** 이야기 3/5 | 기술 4/5

### 5. 정윤아 (PM/기획자, 비개발자)

**[이야기]** 백화점-개별 상점 비유는 비개발자에게도 완벽하게 와닿습니다. 배포/장애/확장 문제를 비유로 풀어준 방식이 회의 자료로도 쓸 수 있을 것 같습니다. 은행 송금 비유로 분산 트랜잭션을 설명한 부분도 명확했습니다.

**[기술]** Java 코드 블록이 갑자기 등장해서 잠깐 당황했습니다. `[참고]` 표시가 있어서 넘어가긴 했지만, "개발자가 아니면 이 코드는 건너뛰어도 됩니다"라는 안내가 있었으면 좋았을 것 같습니다. Saga 패턴 이름 자체는 어색하지만 비유 설명 덕분에 개념은 이해했습니다.

**[총평]** MSA 도입 의사결정에 필요한 핵심 포인트(왜 분리하는가, 어떤 문제가 생기는가, 어떻게 해결하는가)가 모두 담겨 있습니다. 비개발자도 1장은 읽을 수 있습니다.

**[점수]** 이야기 4/5 | 기술 2/5

### 6. 한승호 (비전공 전직자, 부트캠프 수료, Spring Boot 입문)

**[이야기]** 백화점 비유는 이해했지만, "모놀리식", "마이크로서비스", "분산 트랜잭션", "Saga", "Choreography", "Orchestration"이 연속으로 쏟아져서 용어 압박이 컸습니다. `### 학습 목표` 섹션부터 이미 긴장됐습니다. 등장인물이 있었다면 "나도 저 사람처럼 해보면 되겠구나" 하는 심리적 안전감이 있었을 텐데요.

**[기술]** Java 코드 블록이 등장하는 순간 호흡이 막혔습니다. `@Transactional`이 뭔지 모르는 상태에서 "동작 이해용"이라는 설명만으로는 부족했습니다. Choreography/Orchestration 비유(동네 가게, 웨딩 플래너)는 좋았지만 이름 자체가 너무 낯설어서 기억하기 어려웠습니다.

**[총평]** 부트캠프 수료 수준에서는 1장에서 이미 탈락 위기입니다. 용어 도입 속도가 너무 빠르고, 각 개념 사이 숨돌릴 여유가 부족합니다. 스타일 규칙대로 이야기 파트에 등장인물을 넣고 "나도 저 상황이었다"는 공감대를 먼저 형성한 뒤 개념을 풀었다면 훨씬 나았을 것 같습니다.

**[점수]** 이야기 2/5 | 기술 2/5

### 7. 오재원 (K8s/인프라 시니어, MSA 배포 경험)

**[이야기]** 비유 설명이 군더더기 없이 깔끔합니다. 인프라 관점에서 "확장이 어렵다"는 항목이 가장 공감됩니다. 개발자 관점에서 왜 서비스를 분리하는지 정리해주는 1장의 목적에 잘 맞습니다.

**[기술]** Saga 패턴 두 가지를 이미 알고 있지만, 개발 측면에서 어떻게 구현하는지 로드맵이 명확하게 제시되어 있어 좋습니다. 이미지 캡션 일부에 "그림 1-1:", "그림 1-2:" 접두어가 있는데, Typst 조판 시 중복 번호 문제가 생깁니다. 인프라 경험상 이런 작은 오류가 쌓이면 나중에 골치 아픕니다. 챕터 3의 Kubernetes 언급이 있어서 기대됩니다.

**[총평]** 인프라 시니어에게도 개발 측면 MSA 정리에 유용한 챕터입니다. 빠르게 읽을 수 있는 분량과 구성입니다.

**[점수]** 이야기 4/5 | 기술 4/5

### 8. 강하은 (MSA 시니어, Kafka/마이크로서비스 실무)

**[이야기]** 비유와 설명이 무난합니다. 이미 알고 있는 내용이지만 입문자를 위한 첫 챕터로는 적절한 수준입니다. "팀이 커지면 충돌이 잦아진다"는 표현이 실무 경험과 딱 맞습니다.

**[기술]** Choreography와 Orchestration 구분이 명확합니다. 다만 실무에서는 두 방식의 선택 기준이 중요한데, "두 방식을 모두 직접 구현해보면 각각의 장단점이 자연스럽게 체감됩니다"라고만 넘어갑니다. 이미 알고 있는 독자에게는 약간 아쉽지만, 입문서 특성상 맞는 접근입니다. 이미지 캡션 접두어 문제는 의도 밖으로 보이지만 출판 시 수정 필요합니다.

**[총평]** MSA를 처음 배우는 사람에게 맞게 설계된 1장입니다. 시니어에게는 빠르게 훑고 챕터 2로 넘어가는 용도로 충분합니다.

**[점수]** 이야기 3/5 | 기술 4/5

### 9. 윤성민 (CS 학생, 분산 시스템 이론, 실무 없음)

**[이야기]** 분산 시스템 이론을 알고 있어서 개념은 빠르게 이해했습니다. 비유가 이론과 잘 연결됩니다. 다만 이야기 파트가 없고 전체가 설명문 형식이라 이 책이 "이야기를 통해 배운다"는 컨셉을 아직 체감하지 못했습니다.

**[기술]** Saga 패턴의 정의 blockquote 처리가 논문에서 정의를 읽는 것과 유사해서 익숙했습니다. 그런데 "분산 트랜잭션" 정의 blockquote에 "입니다"로 끝나는 문장이 있는데 이건 약간 어색합니다. 챕터 헤더 blockquote(버전/요약/핵심개념)가 없어서 이 챕터의 버전 정보와 핵심 개념을 한눈에 파악하기 어려웠습니다.

**[총평]** 이론 배경이 있는 학생에게는 빠르게 읽히는 챕터입니다. "이것만은 기억하자" 형식이 없어서 중요 개념을 스스로 추려야 했습니다.

**[점수]** 이야기 3/5 | 기술 4/5

### 10. 서태준 (스타트업 CTO, 풀스택, MSA 도입 여부 판단)

**[이야기]** MSA 도입 여부를 판단하는 입장에서 모놀리식 한계 4가지(배포/장애/확장/팀 충돌)가 명확하게 정리되어 있어 유용합니다. 백화점 비유보다는 실제 트레이드오프 비교 표(1.2.2)가 더 설득력 있었습니다. 이야기 파트(등장인물 서사)가 없는 건 경영자 입장에서 오히려 편합니다.

**[기술]** Choreography vs Orchestration 선택 기준이 로드맵 형태로 제시된 게 좋습니다. 챕터 2는 단순하게 시작해서 그 한계를 느끼는 방식으로 진행된다는 예고가 명확합니다. 아키텍처 이미지(1.3.1, 1.3.2)가 있어서 전체 구조가 한눈에 보였습니다.

**[총평]** CTO 입장에서 MSA 도입 판단에 필요한 "왜 분리하고, 어떤 문제가 생기고, 어떻게 해결하는가"가 깔끔하게 정리된 챕터입니다. 팀에게 MSA를 소개할 때 이 챕터를 먼저 읽히고 싶습니다.

**[점수]** 이야기 3/5 | 기술 4/5
