## 심각도별 이슈

### 높음

1. **이야기 파트 부재**: 2.1 "이야기의 시작"이라는 제목을 달았지만 실제 이야기(오픈이, 팀장 등 등장인물 서사)가 전혀 없음. 챕터 1이 이야기로 시작했다면 챕터 2도 동일한 기대를 가지고 들어오는 독자가 이탈함. 글쓰기 원칙 1번("이야기 먼저") 직접 위반.

2. **RestClient/인터셉터 개념 사전 설명 없음**: [작성] 코드인 `RestClientConfig.java`에서 `ClientHttpRequestInterceptor`, `RequestContextHolder`, `ServletRequestAttributes` 등이 등장하지만 이 개념에 대한 비유나 설명이 없음. 2년차 독자도 처음 보면 막힘. 실습 경계 원칙 위반(실습에 나오는 것은 설명).

3. **[작성]/[참고] 파일 분류 불일치**: 파일 트리(2.1.1)에서 `RestClientConfig.java`가 `[작성]`으로 표시되었으나, 2.6.8절 본문에도 `[작성]`으로 표시됨(일치). 그러나 파일 트리 상단 주석(`# [작성] JWT 헤더 전달 인터셉터`)과 실제 섹션 태그가 일치하는지 재확인 필요. `OrderController.java`가 본문 정보 기준에서는 [실습] 5개 중 하나인데 파일 트리에는 `[참고]`로 표시됨 — 심각도 높음.

### 중간

4. **챕터 헤더 blockquote 누락**: `structure.md` 규칙에 따라 챕터 제목 바로 아래에 `> 한 줄 요약`, `> 핵심 개념` blockquote가 있어야 하는데 버전 태그와 GitHub URL만 있음.

5. **이야기 파트 내 용어 공식 정의**: 2.1에서 보상 트랜잭션을 blockquote로 공식 정의함. "이야기 파트에서 비유로 먼저 풀고, 기술 파트에서 정식 정의"라는 원칙과 충돌. 이야기 파트에서는 비유로만 언급하고 정의는 2.6.4 또는 2.6.5로 이동해야 함.

6. **"이것만은 기억하자" 형식 미준수**: "## 정리" 섹션이 있으나 스타일 가이드의 "이것만은 기억하자" 형식이 아님. 다음 챕터 예고도 마지막 단락에 묻혀 있고 독립 섹션이 아님.

7. **보상 트랜잭션 흐름도 이미지 플레이스홀더 처리**: `images/fig-2-1.png`, `images/fig-2-2.png`가 직접 참조되어 있는데 실제 파일 존재 여부 불명. `[GEMINI PROMPT]` 플레이스홀더로 처리되어야 하는 개념도가 PNG 경로로 직접 삽입됨.

8. **2.6.4 흐름 설명의 text-block 시각화 아쉬움**: 보상 트랜잭션 설계를 `text` 코드블록으로 표현했는데, D2 다이어그램이나 `[GEMINI PROMPT]`로 처리하는 것이 "데이터는 보여준다" 원칙에 더 부합.

9. **`docker-compose.yml` [작성] vs [참고] 표기**: 챕터 정보에서는 docker-compose.yml이 [실습] 5개 중 하나로 명시되었으나 본문 2.7에서는 `[참고]`로만 표시됨. 독자가 직접 작성해야 하는지 불명확.

### 낮음

10. **2.2 공통 설정 헤더에 콜론**: `## 2.2 공통 설정 : 모든 서비스가 공유하는 뼈대` — 헤더에 콜론이 포함됨(문장 끝 콜론 금지 규칙. 헤더 내부의 콜론이므로 엄밀히는 문장 끝은 아니나, 일관성을 위해 제거 권장).

11. **2.7.3 JSON 요청/응답 블록 언어 태그**: `POST http://localhost:8083/login` 형태로 시작하는 블록이 ` ```json` 태그를 달고 있으나 내용이 URL + body 혼합. `http`나 별도 `text` 태그로 분리하는 것이 더 명확.

12. **더미 데이터 세 섹션(2.3.2, 2.4.3, 2.5.3, 2.6.3) 반복**: `data.sql` 설명이 각 서비스마다 반복되어 지면을 차지. 하나의 섹션으로 묶거나 "GitHub에서 확인" 처리 권장.

## 페르소나별 상세

### 1. 김준혁 (Java 2년차, MSA 처음 — 주요 독자)

**[이야기]** "이야기의 시작"이라는 섹션 제목을 보고 기대했는데, 서비스 구조 설명이 바로 시작됩니다. 챕터 1에서 오픈이가 MSA를 처음 접했던 흐름이 기억나는데, 챕터 2는 그 감각이 없습니다. 비유도 없고, 왜 지금 이 네 서비스를 이 순서로 만드는지 맥락이 설명 없이 시작됩니다.

**[기술]** 공통 설정(2.2)의 표가 역할 이해에 도움이 됩니다. JwtAuthenticationFilter가 무엇을 하는지 한눈에 파악됩니다. RestClient 인터셉터(2.6.8)에서 `RequestContextHolder`가 처음 등장하는데 설명이 없어 막혔습니다. "왜 이게 필요하지?"라는 의문을 해소할 단 한 줄이라도 필요합니다.

**[실습]** 보상 트랜잭션 코드(2.6.10)는 플래그 추적 힌트(`productDecreased`, `deliveryCreated` 두 변수를 추적하면서 읽어보세요)가 있어 이해하기 쉬웠습니다. Docker Compose 실행 시나리오(2.7.3~2.7.5)는 실제로 따라할 수 있어 좋았습니다. 단, `docker-compose.yml`을 직접 작성해야 하는지 `[참고]`인지 불명확합니다.

**[총평]** 기술 내용은 충실하나 이야기 파트 부재로 주요 독자의 감정적 연결이 끊어집니다. 실습 시나리오는 이 챕터의 강점. RestClient 개념 설명 보완 후 재평가 필요.

**[점수]** 이야기 2/5 | 기술 3.5/5 | 실습 4/5

**[기술]** 코드 분류([참고]/[작성])가 명확합니다. ProductClient, DeliveryClient 분리 구조가 깔끔합니다. 그러나 `cancelOrder`에서 재고 복구 실패 시 롤백 처리가 없다는 점이 눈에 띕니다 — 보상 트랜잭션의 보상 트랜잭션 문제. 책의 의도(입문 실습서)상 언급 생략을 의도한 것으로 보이나, "이런 한계가 있다"는 footnote라도 있으면 좋겠습니다.

**[총평]** 실무 경력자 관점에서 내용 자체는 충분합니다. 챕터 구조 형식(헤더 blockquote, 이것만은 기억하자) 미준수가 책 전체 일관성을 해칩니다.

### 3. 박소연 (Python/Django 3년차, Spring 낯설음)

**[이야기]** 챕터 시작부터 Gradle, Spring Boot 패키지 구조가 나옵니다. Django 개발자 입장에서 `com.metacoding.order.core.config.RestClientConfig` 같은 경로는 문화적으로 낯섭니다. 이야기가 먼저 있었다면 낯선 코드를 만나기 전에 개념적으로 준비할 수 있었을 것입니다.

**[기술]** `@Transactional`이 단일 서비스 DB에만 작동하고 서비스 간 HTTP 호출에는 작동하지 않는다는 설명이 2.6.4에 간략히 있습니다. 이 차이는 Spring을 모르는 독자에게 핵심인데 더 강조해야 합니다.

**[실습]** Hoppscotch 안내는 친절합니다. 그러나 Docker Desktop 설치부터 서비스 4개를 띄우는 과정이 처음 Spring을 접하는 독자에게는 부담입니다. 빌드 실패 시 대처 방법이 없습니다.

**[총평]** Spring 생태계 배경 지식이 없는 독자에게는 진입 장벽이 높습니다. 이야기 파트로 개념을 먼저 설명하고 기술 파트로 넘어가는 구조가 반드시 필요합니다.

**[점수]** 이야기 1/5 | 기술 2.5/5 | 실습 3/5

### 4. 최지호 (프론트엔드 React 3년차)

**[실습]** Hoppscotch를 사용한다는 점이 반갑습니다. 그러나 실제 이미지가 없어(플레이스홀더) 따라하기가 어렵습니다.

**[총평]** 이야기 파트가 없어 진입이 어렵습니다. API 표는 이 독자에게 가장 유용한 요소.

**[점수]** 이야기 1/5 | 기술 3/5 | 실습 3/5

### 6. 한도영 (비전공 전직자, Spring Boot 입문)

**[이야기]** 부트캠프에서 Spring Boot를 배웠지만 MSA는 처음입니다. 챕터 1에서 개념을 배웠는데 챕터 2가 바로 코드로 시작해서 당황했습니다. 이야기로 다시 맥락을 잡아줬으면 좋겠습니다.

**[기술]** `@Builder`, `@Getter`, `@NoArgsConstructor` 어노테이션이 설명 없이 등장합니다. 이 독자층은 이런 어노테이션을 배웠지만 완전히 소화하지 못한 상태입니다. 엔티티 코드가 [참고]이므로 넘어가도 된다는 안내가 명확하지 않습니다.

**[실습]** Docker Compose 실행이 첫 경험일 수 있습니다. "빌드에 5~10분 걸릴 수 있습니다" 같은 한 줄 안내가 불안감을 줄여줍니다.

**[총평]** 이야기 파트와 개념 사전 설명이 없어 이 독자층에게는 가장 어려운 챕터가 됩니다.

**[점수]** 이야기 1/5 | 기술 2/5 | 실습 2.5/5

### 7. 강민준 (K8s/인프라 시니어, Spring 코드 약함)

**[이야기]** 이야기 파트 유무는 크게 중요하지 않습니다.

**[기술]** Docker Compose 설정(2.7.1)이 간결하고 핵심만 보여줍니다. `msa-network` 설명도 명확합니다. 인프라 관점에서 각 서비스의 포트 매핑이 표로 정리되어 있어 좋습니다. Spring 코드 부분은 [참고]가 많아 훑고 넘어갈 수 있습니다.

**[실습]** `docker compose up` 이후 서비스 정상 기동 확인 방법(health check, log 확인 명령어)이 없습니다. 이미지 없이 플레이스홀더만 있는 상태는 실습서로서 아쉽습니다.

**[총평]** 인프라 관점에서는 충분히 유용합니다. K8s로 가기 전 단계의 Docker Compose 구성 이해가 명확합니다.

**[점수]** 이야기 2/5 | 기술 3.5/5 | 실습 3.5/5

### 8. 오세진 (MSA 경험 시니어, 빠르게 훑음)

**[이야기]** 필요하지 않습니다. 빠르게 코드와 설계를 파악합니다.

**[기술]** 플래그 기반 보상 트랜잭션 패턴이 입문서 수준에 적합합니다. `cancelOrder`에서 보상 자체가 실패하는 경우(멱등성, 재시도) 언급이 없는데, 이는 의도된 범위 조정으로 이해됩니다. Orchestration Saga와 Choreography Saga 구분이 2.6.4에서 한 줄로 언급되는데, 챕터 1 내용을 전제로 하는 것으로 보입니다. 연결 고리가 더 있으면 좋겠습니다.

**[실습]** 시나리오 3개(성공/재고부족/주소누락)로 보상 트랜잭션 동작을 검증하는 구성이 교육적으로 훌륭합니다.

**[총평]** 입문서로서 적절한 범위와 깊이입니다. 형식적 규칙 위반(헤더, 이것만은 기억하자)만 수정하면 완성도가 높아집니다.

**[점수]** 이야기 N/A | 기술 4/5 | 실습 4.5/5

### 9. 윤채원 (CS 학생, 이론 있음, 실무 없음)

**[이야기]** 분산 트랜잭션 이론(2PC, Saga)은 알고 있습니다. 그 이론이 실제 HTTP 호출로 어떻게 구현되는지 보고 싶었는데, 코드 바로 앞에 간략한 설계도(2.6.5)가 있어 좋았습니다. 이야기 파트로 현실 맥락을 먼저 설정했으면 더 좋았겠습니다.

**[기술]** `@Transactional`이 서비스 경계를 넘지 못한다는 설명(2.6.4)이 이론 공부와 연결됩니다. `productDecreased` 플래그 패턴이 2PC의 Prepare 단계와 유사한 역할을 한다는 연결고리가 있으면 이 독자층에게 깊이가 생깁니다(의도 밖 심화는 제외하더라도 한 줄 언급은 가능).

**[실습]** Docker 환경이 익숙하지 않아 처음 실행에 시간이 걸릴 수 있습니다. 빌드 시간과 로그 확인 방법 안내가 필요합니다.

**[총평]** 이론-실습 연결이 이 챕터의 핵심 가치인데, 이야기 파트 부재로 연결의 첫 단계가 빠졌습니다. 코드 내용 자체는 충실합니다.

**[점수]** 이야기 2/5 | 기술 4/5 | 실습 3/5

### 10. 류태환 (스타트업 CTO, MSA 도입 판단 중)

**[이야기]** 이야기 파트가 없어도 핵심 파악은 됩니다. "이 구조의 한계" 섹션(정리 부분)이 실질적인 도입 판단에 도움이 됩니다. 동기 호출의 지연 전파, 보상 코드 복잡도 증가, 운영 환경 미지원이라는 세 한계를 명확히 제시한 점이 좋습니다.

**[기술]** RestClient 인터셉터로 JWT를 자동 전달하는 구조가 실용적입니다. ProductClient/DeliveryClient를 adapter 패키지로 분리한 설계도 도입 시 참고할 수 있습니다.

**[실습]** Docker Compose로 4개 서비스를 로컬에서 바로 실행해볼 수 있다는 점이 의사결정 데모로 활용 가능합니다.

**[총평]** CTO 관점에서 "이것을 쓰면 어떤 장단점이 있는가"가 핵심인데, 한계 섹션이 잘 정리되어 있습니다. 형식 규칙 준수 후 완성도가 높아집니다.

**[점수]** 이야기 2/5 | 기술 4/5 | 실습 4/5
