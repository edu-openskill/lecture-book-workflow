## 공통 피드백

### 긍정

- UseCase 인터페이스 개념 설명(3.2절)이 코드 비교와 함께 제시되어 이해가 빠르다. 2장 vs 3장 ASCII 다이어그램이 특히 효과적이다.
- ConfigMap/Secret/Deployment/Service/Ingress를 order 예시 하나로 순서대로 펼치는 구성이 좋다. 반복 패턴이 명확하다.
- "코드 작성 없이 결과를 살펴본다"는 접근 방식은 대부분의 독자에게 납득된다. K8s YAML 파일 자체가 실습이 되고, 이전까지 손으로 쓴 코드와 분리되어 부담이 줄어든다.
- 정리 섹션에서 4장(Kafka) 예고가 자연스럽게 연결된다. "동기 호출의 한계"로 다음 문제를 제시하는 방식이 깔끔하다.

### 부정

- **이야기 파트가 없다.** 챕터 전체가 기술 파트로만 구성되어 있다. 3.1절이 문제 제시 역할을 하지만 이야기/비유로 풀지 않고 곧바로 기술 설명으로 들어간다. 글쓰기 원칙 1번("이야기 먼저") 위반.
- **챕터 닫기 형식 불일치.** "이것만은 기억하자" 섹션이 없고 "## 정리" 불릿 목록으로 대체되어 있다. 글쓰기 원칙 8번 위반.
- **학습 목표 섹션(### 학습 목표)**이 이야기 흐름을 끊는다. 다른 챕터와 일치하는지 확인 필요. 교과서 형식 느낌을 준다.
- K8s 환경 구축 전제 조건(Minikube 설치, Docker Desktop 실행)이 3.5절 중간에 묻혀 있다. 챕터 시작 전 명시가 필요하다.
- Secret의 보안 한계(`stringData` 평문 문제)에 대한 설명이 없다. 실무에서 그대로 쓰면 문제가 생기는 지점인데 언급이 없다.

## 심각도별 이슈

### 심각 (수정 필수)

**[이슈 1] 이야기 파트 부재 — 글쓰기 원칙 1번 위반**

챕터 전체에 이야기 파트가 없다. 3.1절이 2장 숙제를 제시하지만, 비유나 등장인물 없이 곧바로 기술 설명으로 진입한다. 다른 챕터와 톤이 달라 독자 이탈을 유발한다.

수정 방향: 오픈이가 2장의 코드를 운영에 올리려다 막히는 장면을 짧게(1~2페이지) 삽입. UseCase와 K8s를 처음 마주하는 감정 흐름을 만들고, 기술 파트로 전환.

**[이슈 2] "이것만은 기억하자" 섹션 누락 — 글쓰기 원칙 8번 위반**

"## 정리"의 불릿 목록은 요약이지, 독자에게 핵심 하나를 심어주는 형식이 아니다. 다른 챕터와 일관성을 위해 "이것만은 기억하자" 형식으로 변환 필요.

---

### 중간 (수정 권장)

**[이슈 3] K8s 선행 조건이 본문 중간에 매몰**

3.5절 첫 문장에 "Minikube가 설치되어 있지 않다면 공식 사이트에서 설치 후..."가 등장한다. 이 정보는 챕터 시작 또는 blockquote 박스에 선행 안내로 올려야 한다. 3.7절까지 읽고 나서야 환경 구축이 안 돼 있다는 걸 아는 독자가 생긴다.

**[이슈 4] Secret의 보안 한계 미언급**

`stringData` 필드는 평문을 Base64로 인코딩할 뿐, 암호화가 아니다. etcd에서 Secret을 읽을 수 있는 권한만 있으면 디코딩이 가능하다. 실무에서 Secret을 Git에 올리면 안 되는 이유, Sealed Secrets / Vault 같은 키워드 한 줄이 필요하다. 이 책의 "의도 밖" 범위(보안 심화)이므로 한 줄 언급 + 심화 키워드 제시로 충분하다.

**[이슈 5] `minikube tunnel` 실행 후 흐름 단절**

3.7.5절에서 `minikube tunnel`을 실행하면 터미널이 점유된다. 새 터미널에서 테스트해야 한다는 안내가 없다. 초보자(P1, P6)가 터널 실행 후 같은 터미널에서 curl/Hoppscotch를 실행하려다 막히는 지점이다.

**[이슈 6] "학습 목표" 섹션의 교과서 느낌**

챕터 직후 `### 학습 목표` 불릿 목록이 등장한다. 이야기 파트가 없는 상태에서 학습 목표 → 기술 설명 순서는 강의자료처럼 보인다. 이야기 파트 삽입 후 학습 목표를 blockquote 안으로 합치거나 제거 검토 필요.

---

### 경미 (검토 권장)

**[이슈 7] `### 3.2 UseCase : 왜 인터페이스인가` 섹션 — 설명이 이야기 파트 없이 시작**

UseCase를 처음 접하는 독자(P1, P6)에게 "컨트롤러는 구현체가 아닌 UseCase 인터페이스에 의존합니다"라는 첫 문장은 추상적이다. 전원 어댑터 비유 이미지가 아래에 있지만, 비유를 텍스트로 먼저 풀고 이미지로 연결하는 흐름이 자연스럽다.

**[이슈 8] 패키지 구조 트리가 너무 길다**

3.3.1절의 패키지 구조 트리가 order 서비스 하나에 대한 것인데, 26줄에 달한다. `[참고]` 태그가 모든 줄에 붙어 있어 정보 밀도가 높아 보인다. 핵심 구조만 남기고 "전체는 GitHub 참조"로 줄이는 것이 좋다.

**[이슈 9] `imagePullPolicy: Never` 설명 없음**

Deployment YAML에 `imagePullPolicy: Never`가 있는데, 주석이나 설명이 없다. Minikube 환경에서 로컬 이미지를 사용하기 위한 설정인데, 이것이 왜 필요한지 모르면 실제 클러스터에 그대로 쓰려는 독자(P8)가 생긴다.

**[이슈 10] `kubectl delete all --all -n metacoding` 부연 설명 없음**

정리 명령어가 본문 말미에 갑자기 등장한다. "테스트가 끝났으면 정리합니다"로 짧게 소개되지만, 이 명령이 Pod/Service/Deployment 등 모든 리소스를 삭제한다는 점, ConfigMap/Secret은 삭제되지 않는다는 점을 짚어주는 한 줄이 필요하다.

## 페르소나별 상세 피드백 (10명)

### P1 — Java 백엔드 2년차, MSA 처음 (주요 독자)

**판정: 통과 (단, 이야기 파트 삽입 후 재평가 필요)**

UseCase 코드 비교(2장 vs 3장)가 명확해서 "왜 인터페이스를 쓰는가"를 이해했다. K8s YAML을 처음 보는데 ConfigMap → Secret → Deployment → Service → Ingress 순서로 하나씩 쌓아가는 구성 덕분에 무섭지 않았다.

막힌 지점: 3.7.5절에서 `minikube tunnel` 후 새 터미널에서 테스트해야 한다는 안내가 없어서 잠깐 헷갈렸다. `imagePullPolicy: Never`가 왜 필요한지 궁금했는데 설명이 없었다.

감정: 이야기 파트가 없어서 챕터 도입이 딱딱하다고 느꼈다. 오픈이가 등장하지 않으니 앞 챕터와 다른 책 같은 느낌이었다.

---

### P2 — Java 백엔드 5년차, MSA 도입 고려 중

**판정: 통과**

UseCase 패턴이 왜 실용적인지 3.3.4절 OrderService 코드에서 바로 납득됐다. 엔티티에 비즈니스 로직을 위임하는 `validateCancelable()` 패턴도 실무에서 바로 쓸 수 있을 것 같다.

막힌 지점: 없음. 오히려 "완전한 클린 아키텍처가 아니라 핵심만 적용한다"는 안내 blockquote가 적절한 기대 조정이었다.

요청: Secret의 Git 커밋 금지, Vault 키워드를 한 줄이라도 언급해줬으면 좋겠다. 실무에서 첫 번째로 실수하는 지점이다.

---

### P3 — Python 백엔드, Django/Flask 경험, Spring 낯설음

**판정: 부분 통과 (이탈 위험)**

K8s YAML 자체는 선언적이라 이해했다. Nginx 설정도 익숙하다. 그러나 Spring Boot 특유의 `@Transactional(readOnly = true)` 어노테이션, `@Service`, JPA 관련 설명이 전제 지식처럼 쓰인다. OrderService 코드 블록에서 "2장과 동일"이라는 주석이 많은데, Python 독자는 2장도 Java라서 누적 이해가 어렵다.

요청: Spring 어노테이션을 처음 보는 독자를 위한 짧은 각주 또는 사이드 설명이 있으면 좋겠다.

---

### P4 — 프론트엔드, React 3년차, 인프라 궁금

**판정: 부분 통과 (이탈 위험)**

Nginx API Gateway 개념(3.5.1절)은 프론트엔드 관점에서 바로 와닿았다. "클라이언트는 80번 포트 하나만 알면 된다"는 설명이 명확했다.

이탈 지점: UseCase 인터페이스와 DIP 개념은 백엔드 설계 지식이 없어서 "왜 굳이?"라는 의문이 해소되지 않았다. 이야기 파트(비유)가 있었다면 훨씬 쉬웠을 것이다.

---

### P5 — PM/기획자, 비개발자

**판정: 탈락**

3.1절까지는 문제 설명이 명확했다("운영에는 MySQL이 필요하고, 설정값을 외부에서 주입받아야 한다"). 그러나 3.2절부터 코드가 바로 등장하고, 이야기 흐름이 없어서 완독 동기가 사라졌다. 이야기 파트 삽입 없이는 PM/기획자 독자는 3.3절 이후 이탈한다. 단, 이 챕터의 주요 독자가 아니므로 치명적이지는 않다.

---

### P6 — 비전공 전직자, 부트캠프 수료, 인프라 처음

**판정: 부분 통과 (이탈 위험)**

K8s 개념(ConfigMap, Secret, Deployment)이 처음인데 설명이 친절하다. YAML 주석이 있어서 따라가기 쉬웠다.

이탈 지점: `minikube start` 전에 Docker Desktop이 실행 중이어야 한다는 전제가 본문 중간에 있다. 챕터 초반에 "이 챕터를 시작하기 전 필요한 것: Minikube 설치, Docker Desktop 실행"이 명시됐으면 좋겠다. 이걸 놓치면 `minikube start`가 실패하고 에러 메시지를 해석하기 어렵다.

---

### P7 — K8s/인프라 시니어, Spring 코드 약함

**판정: 통과**

K8s YAML이 이 책 수준에서 다룰 수 있는 것들만 간결하게 다루고 있다. `imagePullPolicy: Never`가 설명 없이 등장하는 것과 Secret의 보안 한계 미언급이 아쉽다. 실무 K8s 환경과 차이가 있다는 주석이 없으면 독자가 그대로 가져다 쓴다.

요청: "이 YAML은 Minikube 학습용입니다. 실제 클러스터에서는 ImagePullPolicy, Secret 관리 방식이 달라집니다"라는 한 줄 안내 필요.

---

### P8 — MSA 경험 시니어, 실무 적용 관점

**판정: 통과**

UseCase 패턴의 가치를 이미 알고 있어서 3.2~3.3절은 빠르게 넘겼다. K8s YAML 구조도 친숙하다. 이 책이 학습용으로 "단일 MySQL 공유" 구성임을 명시한 점이 좋다(실무와 다른 이유가 설명됨).

요청: Saga 패턴이 4장에서 다뤄진다는 예고가 있는데, 3장에서 분산 트랜잭션 이슈가 이미 잠재되어 있다는 언급이 있었으면 연결이 더 자연스러웠을 것이다.

---

### P9 — CS 학생, K8s 실습 처음

**판정: 통과**

이론(DIP, 인터페이스 추상화)은 이해가 빠르다. K8s의 "선언적 배포" 개념이 처음인데, "원하는 상태를 파일에 적어두면 K8s가 유지한다"는 설명이 명확했다. `kubectl apply -f`가 어떤 원리인지 궁금했는데, 설명은 없지만 실습에서 직접 확인할 수 있어서 납득됐다.

요청: `kubectl get pods` 출력에서 STATUS가 `ContainerCreating`이면 기다려야 한다는 안내가 있었으면 좋겠다. 이미지 캡처(그림 3-7)가 모두 Running으로 나오는데, 실제로는 잠시 다른 상태가 나온다.

---

### P10 — 스타트업 CTO, MSA+K8s 도입 여부 판단

**판정: 통과**

"단일 MySQL 공유 → 실제 MSA에서는 서비스별 독립 DB"라는 주석이 실무 판단에 도움이 된다. Nginx API Gateway를 K8s Ingress로 대체하는 방향도 명확하다. 이 챕터만으로도 "소규모 팀이 MSA를 도입할 때 어디서 시작할지"를 파악할 수 있다.

요청: Secret 보안 한계와 운영 K8s 환경 차이(imagePullPolicy 등)에 대한 한 줄 언급이 CTO 관점에서는 필수다. 이 설정 그대로 운영에 올리면 안 된다는 신호가 필요하다.

## 수정 제안

### 우선순위 1 (필수)

**이야기 파트 삽입 (3.1절 앞)**

오픈이가 챕터 2 코드를 팀장에게 "이제 배포할 수 있습니다"라고 보여주는 장면. 팀장이 "그래, 근데 MySQL은? 서버 재시작하면 데이터 날아가잖아"라고 묻는 장면. 오픈이가 OrderController에서 OrderService를 교체하려다가 컨트롤러까지 바꿔야 하는 상황에 당황하는 장면. 이 두 가지 좌절이 3.1절 "두 가지 숙제"로 자연스럽게 연결된다.

**Secret 보안 한계 + imagePullPolicy 설명**

Secret YAML 설명 뒤에 한 줄: "stringData는 Base64 인코딩이지 암호화가 아닙니다. Secret을 Git에 커밋하면 안 됩니다. 운영 환경에서는 Sealed Secrets, Vault 같은 도구를 검토하세요."

Deployment YAML 주석에: `imagePullPolicy: Never # Minikube 로컬 이미지 사용. 실제 클러스터에서는 Always 또는 IfNotPresent 사용`
