## 챕터 1 — 3건 (高 0건 / 中 1건 / 低 2건)

이야기 시작 챕터답게 비유(백화점·개별 상점, 오케스트라)와 캐릭터 대사가 절마다 잘 분포되어 있다. 다만 후반부 1.4 분산 트랜잭션 절에서 비유 없이 "기술적 사실"을 빠르게 풀어내는 단락이 몇 군데 있다.

### [1.4.2 / L266] 독립 DB 정의가 두 사실을 한 문장에 압축
- 직접 인용: "DB를 공유하면 한 서비스의 테이블 변경이 다른 서비스에 영향을 주고, 배포와 확장도 함께 묶이기 때문입니다. 상품 서비스의 DB와 배달 서비스의 DB가 물리적으로 분리되어 있으므로, 하나의 트랜잭션으로 묶을 방법이 없습니다."
- 진단: "각 서비스가 독립 DB를 갖는 이유" + "그래서 트랜잭션을 묶지 못한다" 두 가지가 한 단락에 압축됨. 도커&쿠버네티스 책을 본 독자라도 "왜 DB를 공유하면 안 되는가"를 한 번 더 짚어주면 좋은 자리.
- 해당 기준: A-3 (복잡한 인과 압축), A-6 (인지 부담)
- 우선순위: 中
- 수정 방향: "왜 분리?" → "그래서 트랜잭션이 안 묶임" 두 단락으로 쪼개고 첫 단락에 비유(별도 가게의 별도 금고) 추가 검토

### [1.4.2 / L268~269 note] 시점 모순 가능성
- 직접 인용: ":::note 이 책은 학습 편의를 위해 하나의 MySQL 인스턴스를 공유합니다. 실무에서는 서비스별로 DB를 분리하는 것이 원칙입니다. 다만 DB 인스턴스가 같다고 해서 트랜잭션이 묶이지는 않습니다. ..."
- 진단: 이야기상 챕터 1에서는 아직 MySQL이 등장하지 않은 시점인데 "이 책은 하나의 MySQL을 공유"라는 챕터 3 결정이 미리 새어 나옴. 의도된 미리보기일 수 있으나, 독자가 "어? 그게 뭔데?" 하고 멈출 수 있음.
- 해당 기준: A-4 (선행 지식 가정 — 챕터 3에서 결정될 사실의 선행 노출)
- 우선순위: 低
- 수정 방향: 이 note를 챕터 3 도입부로 이동 검토 또는 "챕터 3에서 자세히" 한 줄 가이드 추가

### [1.5 / L383 직후 단락] "이야기" 메타 표현

## 챕터 2 — 6건 (高 2건 / 中 3건 / 低 1건)

이야기는 2.1, 2.6, 2.7에서 비유·대화가 살아 있다. 다만 2.2~2.5의 "공통 설정 + 회원 + 상품 + 배달 [참고] 코드" 블록 4절이 연속되며 비유·캐릭터가 사라지고 메서드 표·엔티티 코드만 줄지어 등장한다. 또한 2.6.10의 "보상 트랜잭션 핵심" 직후 한 단락이 4가지 사실을 한 호흡에 우겨 넣어 부담이 크다.

### [2.2~2.5 / L112~368 절 블록] 4개 절 연속으로 비유·캐릭터 없음
- 직접 인용 (2.2 도입): "서비스 구조를 잡았으니, 먼저 공통 기반을 만들겠습니다. MSA에서는 서비스마다 서버가 다르므로 세션을 공유할 수 없습니다. 대신 JWT 토큰을 발급하고, 각 서비스가 토큰만 검증하는 방식을 사용합니다."
- 진단: 2.2(공통 설정) → 2.3(회원) → 2.4(상품) → 2.5(배달) 네 절 약 260줄 동안 캐릭터 대사 0건, 비유 0건, 표·엔티티 코드만 반복. 챕터 2 중반이 갑자기 "참고 문서"처럼 변하면서 독자가 이야기 흐름을 잃을 수 있음. 2.5 끝(L369) "*회원, 상품, 배달까지는 각자 제 일만 하면 된다. 진짜 문제는 이것들을 엮는 주문 서비스다.*" 한 줄이 가까스로 톤을 회복하지만, 그 사이 4개 절이 메마름.
- 해당 기준: C-2 (절마다 톤 들쭉날쭉), C-3 (캐릭터 분포 불균등)
- 우선순위: 高
- 수정 방향: 2.2 도입부에 비유 한 문장(예: "각 가게가 공통으로 쓰는 출입증 발급 창구")과 2.3·2.4 사이 캐릭터 짧은 대사 한 줄 추가 검토. [참고] 블록은 유지하되 절 사이 호흡을 넣는다

### [2.6.4 / L487~497] 보상 트랜잭션 도입 단락이 추상적
- 직접 인용: "단일 서비스였다면 DB 트랜잭션이 자동으로 롤백해줍니다. 하지만 이미 상품 서비스에 보낸 HTTP 요청은 되돌릴 수 없습니다. 우리가 직접 '재고를 다시 늘려줘'라는 HTTP 요청을 보내야 합니다."
- 진단: 챕터 2의 핵심 개념인 "보상 트랜잭션"을 캐릭터 대사나 비유 없이 곧장 정의로 들어감. 챕터 1에서 "Saga 패턴"으로 한 번 풀었던 개념이지만, 챕터 2에서 다시 만나는 이 단락은 이론 한 단락 + 곧바로 설계로 이어지면서 비유 회수 없음.
- 해당 기준: B-2 (비유 없는 이론), C-1 (챕터 1에서 비유로 풀었던 개념이 비유 없이 등장)
- 우선순위: 中
- 수정 방향: "이미 보낸 HTTP 요청은 되돌릴 수 없다" 자리에 챕터 1의 "오케스트라" 또는 새 비유(택배 발송 취소 — 보낸 택배는 받는 사람에게 다시 회수 요청해야 함) 한 문장 추가 검고

### [2.6.5 / L495] 한 단락에 사실 4개 압축
- 직접 인용: "핵심 아이디어는 진행 상태를 플래그로 추적하는 것입니다. 주문 PENDING 저장 → 재고 감소 → 배달 생성 → 주문 완료 순서로 진행되며, `productDecreased`와 `deliveryCreated` 두 boolean 변수가 각각 재고 감소·배달 생성의 성공 여부를 기록합니다. 예외가 발생하면 이 플래그를 보고 켜진 단계만 골라 역순으로 외부 서비스를 보상하고, 마지막 `throw`로 `@Transactional`이 주문 row 자체를 롤백합니다."
- 진단: (1) 플래그 추적이라는 아이디어, (2) 4단계 순서, (3) 2개 변수의 역할, (4) catch 동작 + Transactional 롤백 동작 — 4가지 사실이 4문장에 들어 있음. 이게 그림 2-1, 2-2 직전이라 의도된 압축일 수 있지만 코드(2.6.10)를 보기 전 독자는 흐름을 머리에 담아두기가 부담.
- 해당 기준: A-3 (복잡한 인과 압축), A-5 (단일 문장 다중 정보)
- 우선순위: 中
- 수정 방향: 4가지를 (a) 아이디어 한 문단, (b) 플래그가 켜지는 순간을 그림 2-1과 연결한 한 문단으로 분할 검토. 또는 짧은 다이어그램(플래그 ON/OFF 흐름)을 추가

### [2.6.10 / L721] 가장 어려운 단락 — 트랜잭션 의미론 압축
- 직접 인용: "한 가지 짚어둘 점이 있습니다. catch 블록 마지막의 `throw new Exception500(...)` 은 `@Transactional` 메서드 안에서 던지는 RuntimeException이므로, Spring이 주문 트랜잭션을 자동으로 롤백합니다. 결과적으로 외부 서비스(재고와 배달)는 직접 보상 호출로 원상복구되고, 주문 row는 트랜잭션 롤백으로 처음부터 없던 일처럼 사라집니다. CANCELLED 상태로 남는 것이 아니라, 아예 DB에 흔적이 남지 않는다는 뜻입니다."
- 진단: 챕터 2 최고 난도 단락. (1) `@Transactional`이 RuntimeException으로 롤백된다는 Spring 동작, (2) 그래서 외부 서비스는 보상으로 / 내부는 자동 롤백으로 처리된다는 이중 의미론, (3) CANCELLED 상태 아님 = 흔적 없음. 도커·Spring 기본은 알지만 `@Transactional` 롤백 규칙(unchecked/checked 차이)까지 모르는 독자에게 부담. 챕터 4에서 "이력이 남지 않는 게 문제"라고 풀게 되는 복선이 여기에 압축되어 있음.
- 해당 기준: A-1 (갑자기 깊이), A-3 (인과 압축), A-4 (선행 지식 가정)
- 우선순위: 高
- 수정 방향: (a) Spring `@Transactional`이 RuntimeException에 자동 롤백을 건다는 한 줄 보조 설명 추가 (b) "외부는 직접 보상, 내부는 자동 롤백"을 두 문장으로 분리. (c) "이력이 남지 않는다"는 메모 또는 :::note 박스로 분리하여 챕터 4 복선 표시

### [2.6.8 / L568] 인터셉터 비유는 좋지만 "택배 스티커"가 다소 모호

### [2.7.5 / L928] "처음부터 없던 일처럼 DB에서 사라진다" 다시 등장

## 챕터 3 — 5건 (高 1건 / 中 3건 / 低 1건)

이 챕터는 책 전체에서 가장 톤이 메마른 챕터. K8s YAML이 매니페스트 5종(ConfigMap·Secret·Deployment·Service·Ingress)을 차례로 나열하며, 각 매니페스트 직후 1~2문장 "역할 설명"만 붙고 비유·캐릭터 대사가 거의 등장하지 않는다. K8s 디테일 자체는 도커&쿠버네티스 책의 선수 지식이므로 일부 메마름은 "의도된 깊이"로 분류 가능하지만, 챕터 1·2의 톤과 비교했을 때 격차가 가장 큰 챕터다.

### [3.1 / L51~61] 두 숙제 도입이 비유 없이 사실 나열
- 직접 인용: "첫 번째 문제: 코드 구조. OrderController는 OrderService에 직접 의존합니다. 나중에 OrderService를 교체하거나 가짜 데이터(Mock)로 바꿔 테스트하려면 Controller 코드도 건드려야 합니다. ... 두 번째 문제: 운영 환경. 챕터 2에서는 H2 인메모리 DB로 로컬에서만 실행했습니다."
- 진단: 챕터의 두 축을 도입하는 자리인데 사실 두 가지를 짧게 던지고 곧장 해결책으로 진입. 챕터 2 도입의 "전화 거는 방식" 비유 같은 톤이 부재.
- 해당 기준: C-1 (다른 챕터는 비유로 풀었음), B
- 우선순위: 中
- 수정 방향: 첫 문제는 "어댑터 없이 콘센트와 가전을 직접 납땜한 상태" 같은 비유, 두 번째는 "내 노트북에서만 동작하는 카페" 비유 추가 검토. 그림 3-1(전원 어댑터)이 첫 비유에는 이미 깔려 있어 도입부에 한 문장만 끌어와도 됨

### [3.2~3.3.2 / L69~157] 클린 아키텍처 + UseCase 인터페이스 절 비유 의존도 낮음
- 직접 인용 (3.3.2): "주문 생성·조회·취소를 각각 별도 인터페이스로 표현합니다. 인터페이스 하나가 하나의 행위(Use Case)를 표현하도록 합니다."
- 진단: 그림 3-1(전원 어댑터)·3-2가 충실한 비유를 제공하지만, 3.3에서 실제 패키지 구조와 코드로 옮겨가면서 비유가 멈추고 "구현체에서 인터페이스로 의존을 바꿉니다" 같은 정의문이 반복. 도커&쿠버네티스 책을 본 독자라도 "왜 굳이 인터페이스 3개로 분리하는가" 같은 의도가 비유로 회수되지 않음.
- 해당 기준: B (긴 정의문), C-1 (그림 비유와 본문 비유 회수 불일치)
- 우선순위: 中

- 우선순위: 中
- 수정 방향: 5.6.3 도입부에 "초인종 비유로 돌아오면, `/api/ws/orders`는 초인종 단자, `/topic/orders/{userId}`는 사용자별 채널 번호" 같은 회수 추가 검토. 또는 그림 한 장으로 (a) WebSocket 연결 단자 (b) STOMP 채널 분기 두 단계를 시각화

### [5.6.4 / L355~370 + 5.6.8 / L432~448] WebSocket Upgrade 헤더·Nginx 프록시 — 깊이 적절성 보고
- 직접 인용 (5.6.8): "Nginx가 중간에 있으면 WebSocket 연결이 일반 HTTP로 처리되어 끊어집니다. `/api/ws/` 경로에 Upgrade 헤더를 설정하여 '이 연결은 WebSocket이니 끊지 말라'고 알려줘야 합니다."
- 진단: HTTP/1.1 Upgrade 헤더, `proxy_http_version 1.1`, `Connection: upgrade` — 도커&쿠버네티스 책에 다루지 않았을 가능성이 높은 디테일. 비유는 한 문장("끊지 말라고 알려줘야") 들어 있지만 충분치 않음. 한편 WebSocket 운영 환경 구성의 필수 디테일이라 의도된 깊이일 수 있음.
- 해당 기준: A-4 (선행 지식), A-1 (갑자기 깊이)
- 우선순위: 中 (의도된 깊이 여부 사용자 확인 필요)
- 수정 방향: term-box `Nginx WebSocket 프록시(Upgrade 헤더)`(L434~436)가 이미 정의하고 있으므로 통과 가능. 다만 본문에 "왜 일반 HTTP면 끊어지는가" 한 줄(HTTP는 요청-응답 1회로 끝나는 프로토콜이므로) 보강 검토

### [5.7.5 / L561~668 통합 테스트 시나리오] 캐릭터 부재 100여 줄
- 직접 인용: "Step 1: WebSocket 연결 및 주문 생성 (클라이언트 역할) ... Step 2: 배달 완료 (배달 기사 역할) ... Step 3: 주문 완료 및 웹소켓 응답 확인"
- 진단: 책의 마지막 통합 시나리오 100여 줄이 거의 캡처+JSON 요청 나열로 진행. L639에 내면독백 한 줄("*배달 완료 버튼을 눌렀다. 이제 주문도 바뀌었을까?*"), L708~714 끝맺음에 캐릭터 대사 등장으로 가까스로 회복하지만, 시나리오 중반에 캐릭터·비유 부재가 길게 이어짐.
- 해당 기준: C-3 (캐릭터 분포 불균등)
- 우선순위: 低
- 수정 방향: Step 1·2·3 각 시작에 짧은 1~2줄 캐릭터 행동/독백 추가 검토 (예: Step 2 직전에 "*배달 기사 시점으로 바꿔보겠다.*")

## 종합 요약

### 가장 큰 패턴

1. **챕터 3 K8s 5종 매니페스트 절 + 챕터 4 Kafka KRaft 옵션 절이 책 전체 톤 격차의 양대 봉우리**. 두 챕터 모두 "도커&쿠버네티스 책 선수 지식을 가정한 의도된 깊이"로 분류할 여지가 있지만, 챕터 1·2·5의 비유·캐릭터 풍부함과 비교했을 때 격차가 너무 큼. 챕터 3은 의도적으로 메마르게 가져갈지(빠르게 훑는 챕터), 톤을 끌어올릴지 결정 필요.

2. **챕터 2 중반(2.2~2.5)의 [참고] 코드 행렬 4개 절이 비유·캐릭터 공백**. 챕터 1과 챕터 2.6의 풍부한 톤 사이에 끼인 메마른 4개 절이 톤 일관성을 깬다. 절 사이에 짧은 캐릭터 대사 1~2개만 끼워도 호흡이 살아남.

3. **챕터 4의 4.6.3 WorkflowState 정의 시점 어긋남**과 **챕터 2.6.10 `@Transactional` 롤백 의미론 압축**이 단일 단락 단위에서 가장 어려운 두 지점. 둘 다 "Spring 내부 동작 + 새 자료구조"를 한 호흡에 던지는 패턴.

4. **챕터 1에서 풀어둔 비유(백화점/개별 상점, 오케스트라/지휘자)가 후속 챕터에서 회수되지 않는 지점이 다수**. 챕터 3 API Gateway 도입, 챕터 4 orchestrator 코드 절, 챕터 5 STOMP 절 모두 챕터 1의 비유로 한 줄 회수만 추가해도 톤 일관성이 크게 회복됨.
