## A. 결정 필요 자리 (2개, 같은 룰 한 번 결정)

### I-4 / I-7 — 비유 위치 옵션

CH01 I-5에서 결정한 동일 패턴 (사실/이유 사이에 비유가 끼는 자리). 두 자리 모두 옵션이 같음:

#### I-4. 2.6.3 보상 트랜잭션 도입 (v4 L267)

**v4 현재**:
> 단일 서비스였다면 DB 트랜잭션이 자동으로 롤백해줍니다. 하지만 이미 상품 서비스에 보낸 HTTP 요청은 되돌릴 수 없습니다. 우리가 직접 "재고를 다시 늘려줘"라는 HTTP 요청을 보내야 합니다. 이것이 **보상 트랜잭션**입니다. 챕터 1에서 소개한 **Choreography Saga**를 직접 HTTP 호출로 구현하는 방식입니다. 주문 서비스가 직접 각 서비스를 호출하고, 실패 시 역순으로 보상합니다.

**옵션 A — 비유를 단락 끝으로 매듭** (CH01과 일관, mid-recheck FIX-5와 같은 룰):
> 단일 서비스였다면 DB 트랜잭션이 자동으로 롤백해줍니다. 하지만 이미 상품 서비스에 보낸 HTTP 요청은 되돌릴 수 없습니다. 우리가 직접 "재고를 다시 늘려줘"라는 HTTP 요청을 보내야 합니다. 이것이 **보상 트랜잭션**입니다. 챕터 1에서 소개한 대로, 주문 서비스가 직접 각 서비스를 호출하고 실패 시 역순으로 보상하는 방식입니다. 이미 발송한 택배를 취소하려면 받는 사람에게 직접 "되돌려달라"고 요청해야 합니다. MSA의 보상도 마찬가지입니다.

**옵션 B — 비유를 별도 단락 첫 줄로 도약**:
> 단일 서비스였다면 DB 트랜잭션이 자동으로 롤백해줍니다. 하지만 이미 상품 서비스에 보낸 HTTP 요청은 되돌릴 수 없습니다.
>
> 이미 발송한 택배를 취소하려면 받는 사람에게 직접 "되돌려달라"고 요청해야 합니다. MSA의 보상도 마찬가지입니다. 우리가 직접 "재고를 다시 늘려줘"라는 HTTP 요청을 보내야 합니다. 이것이 **보상 트랜잭션**입니다. 챕터 1에서 소개한 대로, 주문 서비스가 직접 각 서비스를 호출하고 실패 시 역순으로 보상하는 방식입니다.

#### I-7. 2.6.7 RestClient 택배 스티커 비유 (v4 L353)

**v4 현재**:
> 주문 서비스가 다른 서비스를 호출할 때 필요한 RestClient를 먼저 설정합니다. 인터셉터는 HTTP 요청이 나갈 때 자동으로 JWT 토큰을 챙겨 붙여주는 장치입니다. **택배를 보낼 때 보내는 사람 정보를 매번 적는 대신, 스티커를 자동으로 붙여주는 것과 같습니다.** RestClient에 인터셉터(`ClientHttpRequestInterceptor`)를 등록하면 외부 API를 호출할 때 Authorization 헤더가 자동으로 전달되어 하위 서비스의 JWT 인증을 통과합니다.

**옵션 A — 비유를 단락 끝으로 매듭**:
> 주문 서비스가 다른 서비스를 호출할 때 필요한 RestClient를 먼저 설정합니다. 인터셉터는 HTTP 요청이 나갈 때 자동으로 JWT 토큰을 챙겨 붙여주는 장치입니다. RestClient에 인터셉터(`ClientHttpRequestInterceptor`)를 등록하면 외부 API를 호출할 때 Authorization 헤더가 자동으로 전달되어 하위 서비스의 JWT 인증을 통과합니다. 택배를 보낼 때 보내는 사람 정보를 매번 적는 대신, 스티커를 자동으로 붙여주는 것과 비슷합니다.

**옵션 B — 비유를 별도 단락 첫 줄로 도약**:
> 주문 서비스가 다른 서비스를 호출할 때 필요한 RestClient를 먼저 설정합니다. 인터셉터는 HTTP 요청이 나갈 때 자동으로 JWT 토큰을 챙겨 붙여주는 장치입니다.
>
> 택배를 보낼 때 보내는 사람 정보를 매번 적는 대신, 스티커를 자동으로 붙여주는 것과 비슷합니다. RestClient에 인터셉터(`ClientHttpRequestInterceptor`)를 등록하면 외부 API를 호출할 때 Authorization 헤더가 자동으로 전달되어 하위 서비스의 JWT 인증을 통과합니다.

## B. 자동 적용 자리 11개 — Before/After 핵심만

### I-1. 2.1 두 동작 압축
**v4 L132**: "주문을 생성하려면 상품 서비스에서 재고를 줄이고, 배달 서비스에서 배달을 만들어야 합니다."
**적용 후**: "주문을 생성하려면 두 작업이 필요합니다. 상품 서비스에서 재고를 줄이고, 배달 서비스에서 배달을 만들어야 합니다."

### I-2. 2.2 한 단락 5사실 → 두 단락
**v4 L145** 현재 5사실 한 단락 → JWT 채택까지 한 단락 / "4개 서비스 모두 JWT 인증, 표준 응답 형식..."부터 새 단락

### I-3. 2.6.3 "Choreography Saga" 영문 제거 (CH01 I-9 일관성)
"챕터 1에서 소개한 **Choreography Saga**를 직접 HTTP 호출로 구현하는 방식입니다" → "챕터 1에서 소개한 대로, 주문 서비스가 직접 각 서비스를 호출하고 실패 시 역순으로 보상하는 방식입니다"

### I-5. 2.6.4 단계 명세 + @Transactional 롤백 명시
polish가 2.6.5에 추가한 "주문 PENDING 저장 → 재고 감소 → 배달 생성 → 주문 완료 순서로 진행됩니다. 각 단계가 성공할 때마다 `productDecreased`·`deliveryCreated` 두 boolean을 켜둡니다. 예외가 발생하면 이 플래그를 보고 켜진 단계만 골라 역순으로 외부 서비스를 보상하고, 마지막 `throw`로 `@Transactional`이 주문 row 자체를 롤백합니다." — v4 본문 점검 후 같은 의미 자리에 적용

### I-6a. 그림 2-2 aria-label 정확성
"배달 실패 후 재고 복구**·주문 취소**를 역순으로 실행" → "배달 실패 후 재고 복구**와 주문 트랜잭션 롤백**을 역순으로 실행"

### I-6b. 그림 2-2 ⑤ 텍스트
"⑤ 주문 취소 (CANCELLED)" → "⑤ 주문 트랜잭션 롤백" (코드는 `throw`만 하고 `cancel()` 호출 없음. v4 ⑤가 코드 동작과 불일치)

### I-8. 2.6.9 OrderService 코드 직후
`throw`로 `@Transactional`이 주문 트랜잭션까지 롤백하는 동작 명시 한 줄 추가 (2.7.5 시나리오 3 + 마무리 단락 + :::remember 진화 서사와 연결)

### I-9. 2.7.3 시나리오 1 명사형 풀기
"이는 서버에게 ... 알려주는 **것입니다**" → "이 헤더는 서버에게 ... 알리는 **역할을 합니다**"

### I-10. 2.7.5 시나리오 3 결과 보강
"... 차감된 재고가 복구됩니다." → "... 차감된 재고가 복구되고, 주문 row는 트랜잭션 롤백으로 처음부터 없던 일처럼 DB에서 사라집니다."

### I-11. 2.8 마무리 단락 분할 + 진화 서사
한 단락 4문제 압축을 두 단락으로 + "실패한 주문은 트랜잭션 롤백으로 row 자체가 사라지므로 '어떤 주문이 왜 실패했는지' 이력도 남지 않습니다" 추가

### I-12. 2.8 :::remember 재구성 + bullet 추가
4 bullet → 5 bullet로 재구성:
- MSA에서 분산 트랜잭션은 자동으로 풀리지 않습니다...
- 4개 서비스를 REST로 연결하고...
- 플래그 기반 보상 트랜잭션으로 외부 서비스(재고와 배달)를 원상복구했습니다.
- **단 주문 row 자체는 트랜잭션 롤백으로 사라지므로 실패 이력은 남지 않습니다.** ← 신규 (CH04 4.3 진화 서사 복선)
