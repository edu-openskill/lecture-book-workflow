## A. 자동 적용 자리 (2개)

### I-1. 4.1 번역투 풀기 (v4 L76)

**v4 현재**:
> 생각해 보면 챕터 3에서 마주친 30초 다운은 **MSA의 목표**, 즉 *"각 서비스를 독립적으로 배포하고 장애를 격리한다"* 에 정면으로 위배되는 일이었습니다. 직접 호출하는 한 격리는 환상입니다.

**적용 후**:
> 챕터 3에서 마주친 30초 다운은 **MSA의 목표**인 *"각 서비스를 독립적으로 배포하고 장애를 격리한다"* 와 정면으로 충돌합니다. 직접 호출하는 한 격리는 환상입니다.

**의도**: "X의 Y, 즉 Z에 W" → "X의 Y인 Z와 W"로 풀어 번역투 제거. "생각해 보면" 군더더기 제거 + 과거형 "위배되는 일이었습니다" → 현재형 "충돌합니다".

## B. 결정 필요 자리 (1개)

### I-3. 그림 4-2 — 단순 1대1 흐름 유지 vs polish 다중 토픽 구조 교체

**v4 현재** (L125~L156): viewBox 1000x240, 단순 1대1 흐름
- Producer(order-service) → Topic(order-created) → Consumer(orchestrator) 1줄 흐름
- 캡션 "그림 4-2. 메시지 큐 개념"

**polish 의도**: 다중 토픽 Kafka 구조도 (viewBox 1080x400)
- Producer A/B/C 세 개 → Kafka 브로커 안 Topic A/B/C 세 개 → Consumer α/β/γ 세 개
- 캡션 "Kafka — 이름 붙은 우체통(토픽)이 여러 개 있는 메시지 저장소"

**이전 사용자 결정**: B (polish 다중 토픽 채택) — 캡션 문구와 그림이 더 일치한다는 이유

**옵션 A — v4 단순 그림 유지**:
