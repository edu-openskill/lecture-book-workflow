## 공통 피드백

### 잘 된 것

1. **Kafka 토픽 맵(5.5.3) 탁월함**: 10개 토픽 전체를 한 표로 정리해 "챕터 5 신규" 레이블까지 붙인 것은 10명 중 8명이 긍정적으로 언급. 전체 흐름을 한눈에 파악하는 데 결정적으로 기여.
2. **nginx 프록시 설정 설명 신선함**: WebSocket Upgrade 헤더 필요성을 독자가 "왜?"를 묻기 전에 먼저 답해준 구조. P7 등 인프라 독자 만족.
3. **5.7.6 전체 흐름 요약 ASCII 다이어그램**: 10단계 흐름을 순서대로 따라가며 볼 수 있어 통합 테스트의 맥락을 잡는 데 효과적.
4. **폴링 vs WebSocket 비유(현관문/초인종)**: 직관적이고 과장 없음. 기술 이해 배경 무관하게 납득 가능.

### 공통으로 부족한 것

1. **이야기 파트 사실상 부재**: 5.1~5.2의 문장들이 이야기가 아니라 설명 목록. 오픈이/팀장 등장이 전혀 없음.
2. **"이것만은 기억하자" + 책 전체 마무리 없음**: 마지막 챕터임에도 "정리" 섹션이 bullet 나열로 끝남. 독자를 배웅하는 문장이 없음.

4. **STOMP 메시지 실제 구조 미제시**: "데이터는 보여준다" 원칙 적용 부족. STOMP 프레임이 어떻게 생겼는지 독자가 눈으로 확인할 수 없음.

## 심각도별 이슈

### CRITICAL

#### C1. "이것만은 기억하자" + 책 전체 마무리 문장 없음

파일 끝(625행)이 bullet 나열인 `## 정리` 섹션으로 종료됨. 마지막 챕터이므로 책 전체 여정을 마무리하는 서사가 필수. 챕터 닫기 공식(`이것만은 기억하자` + 다음 예고)도 누락. "다음 버전 예고"가 마지막 챕터에서는 "이 여정을 마치며" 형식으로 전환되어야 하는데 없음.

- **영향 페르소나**: P1, P5, P6, P10 (직접적 불만족)
- **위치**: 619~625행 (`## 정리`)

#### C2. 이야기 파트 부재 — 전체가 기술 설명 구조

5.1이 이야기 파트를 자처하지만, 오픈이/팀장 등 인물이 전혀 등장하지 않고, 현관문/초인종 비유도 그래프 설명처럼 서술됨. "이야기 먼저" 원칙(글쓰기 원칙 1번) 및 "이야기 파트에 코드 없음" 규칙의 전제 자체가 무너짐. 챕터 전체가 기술 파트만으로 구성된 것과 동일한 상태.

- **영향 페르소나**: P1, P5, P6, P10
- **위치**: 16~46행 (5.1~5.2.1)

### HIGH

#### H1. 학습 목표 bullet이 챕터 최상단에 등장

7~13행의 `### 학습 목표` 블록은 강의 슬라이드 형식. 이야기 책에서 챕터를 시작하는 방식이 아님. 독자의 기대값을 "읽는 책"이 아닌 "공부하는 책"으로 전환시킴. 스타일 규칙 위반.

- **영향 페르소나**: P5, P6, P9
- **위치**: 7~13행

#### H2. STOMP 메시지 실제 구조 미제시 ("데이터는 보여준다" 원칙 위반)

WebSocket/STOMP 설명에서 실제 STOMP 프레임 구조나 서버가 보내는 메시지의 실제 JSON 형태가 없음. `Map.of("orderId", orderId)`가 전송된다는 것은 코드에서 추론 가능하지만, 클라이언트가 실제로 받는 메시지가 어떻게 생겼는지(`{"orderId": 4}`) 명시적으로 보여주지 않음. 글쓰기 원칙 12번 위반.

- **영향 페르소나**: P1, P4, P9
- **위치**: 309~328행 (5.6.5)

#### H3. blockquote 내 폴링 정의가 이야기 시작 전에 등장

22~23행: `> **폴링(Polling)**: 클라이언트가...` — 5.1에서 비유(음식 배달 앱 비유)가 먼저 나오고 그 뒤에 정의 blockquote가 배치된 것은 "비유 → 정의 2단계"의 의도에 맞으나, 이 정의가 실제로는 비유 문장 중간에 삽입되어 흐름을 끊음. 비유가 완전히 끝난 후 기술 파트에서 정의가 나와야 함(글쓰기 원칙 4번).

- **영향 페르소나**: P6, P9
- **위치**: 20~23행

### MEDIUM

#### M1. 헤더 blockquote 버전 정보가 chap04로 기재 (오기)

3행: `> chap04 · 실행 환경: 컨테이너 · WebSocket(STOMP/SockJS)` — 챕터 5인데 `chap04`로 표기. GitHub URL도 `chap04`를 가리킴. 독자 혼란 유발.

- **위치**: 3~4행

#### M2. index.html 실습 분류 불일치

목차에는 index.html이 `[실습]` 파일로 명시되어 있으나 본문 352~365행에서 `[참고]`로 분류됨. 코드 규칙(`[실습]`/`[설명]`/`[참고]` 분류 일관성) 위반.

- **위치**: 352행

#### M3. 5.7.5 통합 테스트에서 토큰 발급 방법 미안내

497행: "여기서 `Bearer` 접두사를 제외한 토큰 값만 입력합니다" — 토큰을 어디서 발급받는지 안내가 없음. 이전 챕터에서 다뤘더라도 마지막 챕터 통합 테스트에서 "로그인 API로 발급" 한 줄이 필요.

- **위치**: 496~500행

#### M4. 5.7.6 흐름 요약에 이모지 포함

615행: `10. 클라이언트: 실시간 알림 수신 ✓` — 체크마크(✓)가 이모지로 처리될 수 있음. 스타일 규칙 "이모지 전면 금지"와 충돌 가능성.

- **위치**: 615행

## 페르소나별 상세 평가 (10명)

### P1. Java 백엔드 2년차 — 조건부 통과

4장을 따라온 주요 독자로, 기술 내용 자체는 충분히 이해 가능. delivery-service → orchestrator → order-service 수정 흐름이 명확히 안내되어 실습 따라가기에 무리 없음. Kafka 토픽 맵도 4장과 5장의 차이를 한눈에 보여줘 만족.

그러나 이야기 파트가 없어 오픈이가 어디서 뭘 고민하는지 모른 채 바로 코드로 들어감. 챕터를 마쳤을 때 "끝났다"는 감각이 없음 — "이것만은 기억하자"가 없고 정리 bullet으로 끝나기 때문. STOMP 메시지가 실제로 어떻게 생겼는지 JSON을 한 번이라도 보고 싶었음.

**통과 조건**: 이야기 파트 추가, 챕터 닫기 추가.

### P2. Java 백엔드 5년차 — 통과

WebSocket 경험이 없지만 5년차라 코드 구조를 빠르게 파악. `WebSocketConfig`, `SimpMessagingTemplate` 사용법이 간결하게 제시되어 "이 정도면 혼자 할 수 있겠다" 수준. nginx Upgrade 헤더 설명이 특히 실용적. 빠르게 훑고 구현하기 좋음.

이야기 파트 부재는 이 페르소나에게 오히려 장점으로 작용. 단, 마지막 챕터 마무리 문장은 있었으면 함.

### P3. Python 백엔드 — 통과

Django Channels에서 channel layer와 group_send 개념을 쓴 경험이 있어, STOMP의 `/topic/` 채널 개념이 자연스럽게 연결됨. Spring 구현 방식이 Django보다 설정 중심적이라는 차이를 본문에서 명시해주지 않지만, 코드만 보고 파악 가능.

SockJS 클라이언트가 왜 필요한지(미지원 브라우저 폴백) 설명이 있어 좋음. `withSockJS()` 한 줄이 하는 일을 본문에서 명시해준 것이 특히 가치 있음.

### P4. 프론트엔드 개발자 — 조건부 통과

index.html WebSocket 연결 코드(354~365행)가 핵심 관심사. SockJS 객체 생성, STOMP `connect()`, `subscribe()` 패턴이 명확히 보임. 토큰을 쿼리 파라미터로 전달하는 방식도 납득.

그러나 index.html 전체 코드를 `[참고]` 처리하며 GitHub으로 넘긴 점이 아쉬움. 핵심 JavaScript 구독 코드 외에 로그인 처리, 에러 핸들링 부분이 궁금함. 또한 STOMP 서버에서 받는 메시지의 실제 JSON 구조(`{"orderId": 4}`)를 명시했으면 클라이언트 파싱 코드를 스스로 작성하기 쉬웠을 것.

**통과 조건**: 수신 메시지 JSON 구조 명시.

### P5. PM/기획자 — 탈락

폴링 vs WebSocket 비유(현관문/초인종)는 이해함. "폴링이 왜 나쁜지"는 납득.

그러나 챕터 전체가 기술 명세서처럼 전개되어 중간에 흥미를 잃음. "오픈이가 이 기능을 구현하면서 어떤 걸 깨달았는지"가 없으니 이야기로 읽히지 않음. 5장이 마지막인데 완성된 시스템에 대한 감동 서사가 없음. `## 정리`의 bullet 4개로 끝나는 챕터에서 "이 책 읽길 잘했다" 감각을 얻기 어려움.

**탈락 이유**: 이야기 파트 부재 + 마무리 서사 없음.

### P6. 비전공 전직자 — 탈락

4장에서 Kafka 개념에 지쳐 있는 상태. 5장 시작부터 `### 학습 목표` bullet 5개가 등장하니 "또 공부해야 하나" 부담감. 이야기가 없으니 동기 부여를 받을 수 없음.

폴링 비유(음식 배달 앱에서 5초마다 확인)는 납득했으나, 곧바로 WebSocket, STOMP, SimpMessagingTemplate이 쏟아지며 압도됨. 이 페르소나에게 가장 필요한 것은 "지금 우리가 뭘 만들고 있는지"를 직관적으로 보여주는 이야기 파트.

**탈락 이유**: 이야기 파트 부재 + 학습 목표 bullet 첫머리 배치.

### P7. K8s/인프라 시니어 — 통과

이 챕터에서 가장 관심 있는 내용인 nginx WebSocket 프록시 설정(5.6.6)이 `frontend/nginx.conf`, `gateway/nginx.conf` 두 파일로 명확히 분리되어 제시됨. Upgrade 헤더 필요 이유 설명도 간결.

K8s 매니페스트 구조(5.7.1)에서 Ingress가 gateway 대신 frontend를 가리키도록 바뀐 이유 설명도 충분. minikube tunnel 사용법도 실습 맥락에 맞게 배치됨.

### P8. MSA 시니어 — 통과

전체 흐름 완성도 검토 관점에서 5.5.3의 Kafka 토픽 맵이 결정적으로 가치 있음. 10개 토픽의 발행 주체/구독 주체/목적이 한 표에 정리된 것은 MSA 설계 관점에서 참조 가치가 높음.

orchestrator의 상태 관리(`states` 맵) 처리—배달 완료는 별도 리스너에서 처리하므로 `states.remove()` 호출—가 코드 주석으로 설명된 점도 납득. 단, delivery-completed 리스너에서 `states` 맵을 참조하지 않는 이유(직접 orderId로 연결)를 명시적으로 설명했으면 더 좋았음.

### P9. CS 학생 — 통과

WebSocket 이론을 알고 있어 개념 설명은 빠르게 소화. STOMP 프로토콜이 WebSocket 위에 얹히는 레이어라는 설명(44행 blockquote)이 명확. `/topic/orders/{userId}` 채널이 개별 사용자에게만 전달되는 pub/sub 구조도 이해.

다만 STOMP 프레임 자체의 실제 생김새(SUBSCRIBE, MESSAGE 명령 구조)가 없어 "원리는 알겠는데 실제 프로토콜이 어떻게 생겼지?" 궁금증이 남음. 이 챕터 난이도는 적절 — WebSocket 배경 없이 읽는 독자에게는 조금 빠를 수 있으나 CS 학생에게는 쾌적.

### P10. 스타트업 CTO — 조건부 통과

마지막 챕터로 책 전체 투자 가치를 판단하는 관점. 기술 흐름과 아키텍처 완성도는 납득 — 동기→비동기→실시간 알림으로 단계적으로 성장한 구조가 명확.

그러나 마지막 챕터임에도 "이 책을 읽으면 뭘 얻었는가"를 독자에게 상기시켜 주는 마무리 문장이 없음. bullet 4개짜리 정리로 끝나는 책은 완결감이 없음. 책 전체의 핵심 메시지("동기에서 비동기로, 단계별로 바꿔보면 MSA의 구조가 보인다")를 마지막에 한 번 더 울려줘야 투자 가치가 확인됨.

**통과 조건**: 책 전체 마무리 서사 추가.

## 수정 제안

### 우선순위 1 (CRITICAL — 즉시 수정)

**마지막 챕터 닫기 추가**

현재 `## 정리` 섹션을 아래 구조로 교체:

```markdown
## 이것만은 기억하자

- 폴링은 클라이언트가 반복 요청한다. WebSocket은 서버가 먼저 Push한다.
```
