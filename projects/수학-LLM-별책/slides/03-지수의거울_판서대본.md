# 3강 · 지수의 거울 — 판서 대본

## 슬라이드 1 — 확률을 곱했더니 0으로 사라졌다
대본:
오늘은 오픈이가 강의를 듣다 걸린 작은 사건에서 시작할게요. 언어 모델이 문장의 확률을 계산한다는 대목이었어요. 모델은 단어를 하나씩 예측하는데, 각 단어가 맞을 확률을 쭉 곱한다는 거예요. "나는"이 나올 확률 0.2, 다음에 "밥을"이 나올 확률 0.1, 그다음 "먹었다"가 0.05. 오픈이가 종이에 곱해봤더니 0.2 곱하기 0.1 곱하기 0.05는 0.001이에요. 겨우 세 단어인데 벌써 0.001입니다. 그럼 단어가 백 개, 천 개인 문장은요? 0.1 같은 수를 수백 번 곱하면 결과는 0.000…0001처럼 0에 한없이 달라붙어요. 나중엔 컴퓨터조차 "이건 그냥 0이야" 하고 포기해버릴 만큼 작아집니다.

## 슬라이드 2 — 그래서 문제는: 다 사라지면 뭘 보고 판단하지?
대본:
여기서 오픈이가 멈칫합니다. 확률을 곱하는 건 알겠어요. 그런데 이렇게 다 0으로 사라져버리면, 모델은 대체 뭘 보고 "이 문장이 저 문장보다 그럴듯하다"를 판단하죠? 곱셈을 그대로 두면 안 되겠다는 직감이 옵니다. 작아지기만 하는 이 곱셈을, 뭔가 다루기 쉬운 형태로 갈아탈 도구가 필요하다는 거예요. 오늘 우리가 챙길 게 바로 그 도구입니다.

## 슬라이드 3 — 중심 비유: "1024겹이 됐어. 몇 번 접은 거지?"
대본:
도구를 찾으러 지난 시간의 종이접기로 돌아가 볼게요. 종이를 한 번 접을 때마다 두께가 두 배가 됐죠. 열 번 접으면 1024겹. 그때 우리가 던진 질문은 "몇 번 접으면 얼마나 두꺼워지나"였어요. 곱해 나가는 방향이었습니다. 그런데 오늘은 질문을 거꾸로 뒤집어봅니다. "두께가 1024겹이 됐어. 그럼 몇 번 접은 거지?" 답은 열 번이죠. 결과를 손에 쥐여주고, 거기서 접은 횟수를 되묻는 겁니다. 이 거꾸로 묻기가 바로 로그예요.
개념 — 로그(logarithm):
밑 a를 몇 번 곱해야 x가 되는가를 묻는 값. log_a x = n 이라는 건 a^n = x 라는 뜻과 완전히 같은 말이다. 비유로는 "결과를 주고 접은 횟수를 되묻기".

## 슬라이드 4 — 핵심 개념: 로그는 지수의 거울
대본:
그러니까 로그는 지수 앞에 세운 거울이에요. 지수가 "밑을 몇 번 곱했나"를 곱셈으로 키워가는 도구라면, 로그는 그 결과를 보고 "몇 번 곱한 거야?"를 되읽는 도구입니다. 둘은 같은 사건을 양쪽에서 바라본 한 쌍이에요. 예를 들어 log₂ 8은 3입니다. 왜냐고요? 2를 세 번 곱하면 8이니까요. 2의 3승이 8이라는 식과, log₂ 8이 3이라는 식은 방향만 다르지 똑같은 말이에요. 종이를 몇 번 접어야 8겹이 되느냐, 세 번. 그게 log₂ 8 = 3입니다.
개념 — 역함수 관계:
y = a^x 와 y = log_a x 는 서로 역연산이다. 지수가 키운 것을 로그가 되돌린다. 그래서 "지수의 거울".

## 슬라이드 5 — 거울의 재주: 곱셈이 덧셈이 된다
대본:
이 거울에는 묘한 재주가 하나 더 있어요. 거울 앞에 서면 곱셈이 덧셈으로 바뀝니다. 이미 세 번 접어 8겹이 된 뭉치를 생각해봐요. 여기서 멈추지 않고 네 번 더 접으면, 두께가 16배로 불어나 128겹이 됩니다. 곱셈이 일어난 거죠, 8 곱하기 16은 128. 그런데 이걸 "접은 횟수"로 세면? 처음 3번에 4번을 더해서 그냥 7번이에요. 두께를 16배로 키우는 곱셈이, 접은 횟수에서는 4를 더하는 덧셈으로 가라앉은 겁니다. 식으로 쓰면 log(xy) = log x + log y. 오픈이 눈이 번쩍였어요. 확률을 곱하면 0으로 사라지지만, 로그라는 거울을 통과시키면 곱셈이 덧셈이 되고, 덧셈은 아무리 더해도 갑자기 0으로 사라지지 않잖아요.
개념 — 로그의 곱셈 법칙:
log_a(xy) = log_a x + log_a y. x = a^m, y = a^n 으로 두면 xy = a^(m+n) 이니, xy를 만드는 지수는 m+n. 곧 곱이 합이 된다. 2강의 지수법칙 "곱하면 지수끼리 더한다"가 거울에 비친 모습.

## 슬라이드 6 — 직관 한 컷: log₂(8×16) = 3 + 4 = 7
대본:
말로만 보면 미끄러지니까 숫자로 끝까지 풀어볼게요. log₂(8 곱하기 16)을 두 가지 방법으로 구해봅니다. 첫째, 먼저 곱하는 방법. 8 곱하기 16은 128이고, 2를 일곱 번 곱하면 128이니까 답은 7이에요. 둘째, 로그 법칙으로 쪼개는 방법. log₂(8 곱하기 16)을 log₂ 8 더하기 log₂ 16으로 풀어요. 각각은 "몇 번 접었나"로 바로 읽히죠. 2의 3승이 8이니 log₂ 8은 3, 2의 4승이 16이니 log₂ 16은 4. 그래서 3 더하기 4, 7입니다. 두 방법 다 7이에요. 그런데 첫 번째는 128이라는 큰 수를 거쳐 갔고, 두 번째는 곱셈을 아예 덧셈 3 더하기 4로 갈아타서 큰 수를 만들지 않고 곧장 도착했어요. 확률 수백 개를 다룰 때 왜 두 번째 방식이 안전한지가 여기서 보입니다.

## 슬라이드 7 — 이것만은 기억하자
대본:
오늘 챙긴 걸 한 줄로 묶어봅시다. 로그는 지수의 거울이에요. "몇 번 곱했나"를 거꾸로 되묻는 역연산입니다. 그리고 로그의 가장 큰 재주는 곱셈을 덧셈으로 바꾸는 것, log(xy)는 log x 더하기 log y. 그래서 0으로 사라질 만큼 작은 확률의 곱도, 로그를 씌우면 안전한 더하기로 다룰 수 있어요. 별것 아닌 거울 같지만, 곱셈을 덧셈으로 바꿀 줄 안다는 건 AI가 다루는 수많은 확률을 안전하게 계산하는 첫 번째 열쇠입니다. 이게 나중에 만날 로그우도와 손실 함수의 출발점이에요.
개념 — 로그우도(log-likelihood):
확률의 곱에 로그를 씌워 합으로 바꾼 값. 언어 모델이 문장 확률을 단어별 확률의 곱으로 계산할 때, 그대로 두면 0으로 무너지므로 로그를 씌워 합으로 다룬다. 7강·15강에서 손실 함수로 본격적으로 돌아온다.
