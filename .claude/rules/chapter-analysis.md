# 챕터 본문 분석 절차

사용자가 챕터 본문 일부를 인용해 "분석"을 요청하면 반드시 따른다. `.claude/hooks/analyze-trigger.sh`(UserPromptSubmit hook)가 매 메시지마다 절차를 주입하지만, 이 파일은 **영구 규칙**으로 매 세션 자동 적용된다.

## 인용된 문장 주변만 보지 않는다

표면적 어휘 교정으로 끝내면 같은 지적이 반복된다. 인용 문장은 단락의 한 부분이고, 단락은 절(H3)의 흐름 한 칸일 뿐이다. 한 칸만 손보면 앞 단락 결론·뒤 단락 전제와 어긋난다.

## 응답 형식 강제 (위반 시 분석 무효)

### 1. 첫 출력 — H3 전체 흐름 표

인용 문장이 속한 H3(절) 전체를 Read한 뒤, 다음 표를 분석 응답 첫 출력으로 둔다.

```markdown
## H3 전체 흐름

| # | 단락 | 역할 |
|---|------|------|
| 1 | (도입 문장) | 5.x.y 호기심 받음 |
| 2 | (송장 본문) | 비유 설명 |
| 3 | **그림 N-N** | 시각화 |
| ... | ... | ... |
```

절 안의 모든 단락·그림·캡션·매핑을 순서대로 나열한다.

### 2. 두 번째 출력 — 수정 영향

수정 대상 단락의 앞·뒤 단락이 어떻게 받는지·전제하는지 명시한다.

```markdown
## 수정 영향

- **앞 단락(#N) 끝**: "..." 으로 마침
- **본 단락(#N+1) 역할**: 앞 끝을 받고, 뒤 시작을 깔아 준다
- **뒤 단락(#N+2) 시작 전제**: "..." 을 받는다고 가정
- **수정·삭제 시 끊기는 지점**: ...
```

### 3. 그 다음에야 옵션 제시

위 두 섹션 위에서만 옵션 제시. 옵션은 흐름 매핑과 수정 영향에 근거해야 한다. 단어 단위 어휘 교체만 나열하면 안 된다.

## 사용자 인터럽트 신호

응답에 흐름 매핑·수정 영향 섹션이 없으면 사용자는 **"흐름부터 다시"** 라고만 응답하면 된다. 그 시점에 다시 1번부터 시작한다.

## 관련 규칙

- `style.md` — 톤·편집·금지패턴
- `storytelling.md` — 캐릭터·비유·챕터 패턴 (스토리텔링 흐름 점검 시)
- `writing-chapters.md` — 챕터 작성 패턴
- `.claude/hooks/analyze-trigger.sh` — 매 메시지 절차 주입 (분석 단어 감지 시)
