# 글로벌 코드 규칙

**코드블록·실습 규칙·Git 레포·실험 스크립트** 단일 근거. 톤은 `style.md`, 스토리텔링은 `storytelling.md`, 구조는 `structure.md` 참조.

---

## 코드 블록 기본 규칙

- **언어 태그 필수** (`java`, `python`, `bash` 등)
- 구조: 목적 한 줄 → 코드 → 설명 → 실행 결과
- 핵심만 **10~15줄** (전체는 완성 레포 참조)
- bash·shell. 상세 설명 불필요
- **번호 주석(①②③) 금지**
- 코드 내 단계 주석은 `# N. 설명` 형태로 **코드 윗줄**에 작성. 독자가 각 줄의 역할을 알 수 있게 한다
- **핵심 로직만 남긴다**. 방어 코드(None 체크, fallback, try·except, 로깅)는 챕터 코드블록과 완성 코드 모두에서 제외
- **챕터 코드블록 ↔ 완성 코드 동기화**. 챕터 수정 시 해당 `rag-end` 코드의 TODO 주석·단계 번호(`# 1.`, `# 2.`)·함수 시그니처가 일치하는지 반드시 확인

## 파트별 코드 규칙

- 이야기 파트에 코드 블록 절대 없음
- 기술 파트에서만 코드 등장

---

## 코드 파일 유형

- **[실습] 파일**. 예제 레포에 TODO+pass로 제공. 코드블록 내부에 `# TODO:` 주석 + `# N.` 단계 번호
- **[설명] 파일**. 핵심 흐름만 발췌. 전체 코드는 완성 레포 참조
- **[참고] 파일**. 코드 블록 없음. 메서드 표로 요약

---

## Git 레포 구조

- **code/**: 원천 소스코드. 수정하지 않는다
- **rag-end/**: 완성 코드. `code/`에서 챕터별로 분해한 동작 코드. 정답 확인용
- **rag-start/**: 예제 스켈레톤. `rag-end`를 복사한 뒤 핵심 로직을 TODO+pass로 비운 것. 독자가 클론하여 챕터를 따라 작성
- **실습 흐름**: rag-start 클론 → `cd exNN` → 챕터 따라 TODO 채우기 → 막히면 rag-end 참고
- **수정 범위**: 챕터 코드블록이나 핵심 로직을 수정할 때는 **rag-end만 수정**. rag-start는 TODO+pass이므로 핵심 로직 변경의 영향을 받지 않는다. 보일러플레이트(함수 시그니처·import·`__init__`)가 바뀔 때만 rag-start도 함께 수정
- **챕터 cd 경로**: 독자가 작업하는 곳은 rag-start의 `exNN`

## 실습 코드블록 작성 규칙

### [실습] 코드블록 앞 필수 안내

`[실습]` 코드블록 **바로 위**에 반드시 다음 두 가지를 포함한다.

1. **파일 경로**: `` `tuning/step1_query_rewrite/hyde.py` `` 형태로 독자가 열어야 할 파일 안내
2. **파일 열기 + TODO 안내**: "를 열고 TODO의 `pass`를 지우고 아래 코드를 작성합니다"

예시:
```
`tuning/step1_query_rewrite/hyde.py`를 열고 TODO의 `pass`를 지우고 아래 코드를 작성합니다.
```

이 안내 없이 `[실습]` 코드블록만 던지지 않는다.

### 코드블록 헤더 형식

```python [실습 N] tuning/.../파일명.py. 제목
def 함수명(...):
    # TODO: 설명
    # 1. 첫 번째 단계
    ...
    # 2. 두 번째 단계
    ...
```

- `[실습 N]` 태그 + 파일 경로 + 제목
- 내부에 `# TODO:` (핵심 로직 위치) + `# N.` 단계 번호
- 보일러플레이트(import·`__init__`·prompt)는 미리 채워져 있음

### 상수·매개변수 첫 등장 시 설명

코드에 상수나 매개변수가 처음 등장하면 **코드블록 앞에서** 역할·기본값·어디서 넘어오는지를 설명한다.

예시:
- `chunk_size`(기본값 500) — 한 조각의 최대 글자 수. `main.py` CLI 옵션으로 전달
- `embedding_model_name` — `"jhgan/ko-sroberta-multitask"`. 이야기 파트에서 선택한 모델

상수가 3개 이상이면 테이블로 정리한다.

### 보조 함수 등장 시 설명

`[참고]` 파일에서 import하는 보조 함수가 코드에 등장하면, 직접 작성하지 않는 함수임을 밝히고 역할을 한 줄씩 설명한다.

### 코드블록 친절 설명

코드블록 하나를 설명할 때 아래 요소를 갖춘다.

1. **코드블록 위**: 이 코드가 왜 필요한지(동기), 이야기 파트의 비유와 연결
2. **코드블록 내부**: 단계 주석(`# N. 설명`)으로 흐름 안내
3. **코드블록 아래**: 핵심 동작 1~2문장 설명 + 주요 변수·함수 역할 풀이
4. **보조 함수 테이블**: import하는 외부 함수가 2개 이상이면 테이블 (파일·함수명·역할)
5. **데이터 흐름**: 입력→출력이 명확하지 않으면 예시 데이터(JSON·터미널 출력)를 함께 보여준다

불친절한 코드 설명 = 독자가 멈추는 지점. 코드만 던지고 넘어가지 않는다.

### 코드블록 위·아래 역할 분담

코드블록 **위**에는 이 코드가 무슨 역할인지(왜 만드는지), **아래**에는 코드 흐름의 핵심 동작(어떻게 작동하는지). 위·아래에 같은 내용을 반복하지 않는다.

### 실습 간 연결

이전 실습에서 만든 함수를 다음 실습에서 사용할 때 반드시 언급한다. "실습 1에서 만든 `split_text_into_chunks`가 여기서 호출됩니다"처럼 부품이 조립되는 과정을 독자가 상상할 수 있게 한다.

### 앞 내용과 연결 (비유 회수)

기술 파트의 모든 설명은 앞에서 사용한 비유·캐릭터·장면을 **구체적으로 회수**해야 한다. "이번 실습에서 이 흐름을 하나씩 만들어 봅니다" 같은 **템플릿 문장 금지**. 대신 해당 챕터의 비유를 직접 언급.

적용 범위:

1. 실습 순서 전체 흐름 설명 — 다이어그램 아래에서 비유로 흐름을 풀어줌
2. 각 코드블록 위 — 이 코드가 비유의 어떤 부분인지 연결
3. 실습 도입부 — 이전 실습과의 관계를 비유로 연결

- BAD: "질문이 들어오면 라우터가 분류하고 도구를 호출하여 답변을 생성합니다."
- GOOD: "질문이 안내데스크(QueryRouter)에 도착하면, 안내 직원이 DB 담당자에게 보낼지 문서실로 보낼지 판단합니다."

### 이야기에서 다루지 않은 용어

기술 파트에서 이야기 파트에 없던 새 용어·함수·개념이 등장하면 즉시 한두 문장으로 설명한다. 설명 없이 넘어가지 않는다.

## 실험·실습 스크립트 출력

- 터미널 출력에 **장식선 금지** (`console.rule()`, `───` 등의 장식 구분선 사용 금지)
- 제목은 `console.print("[bold]제목[/bold]")` 으로 간결하게 표시
- Rich Table의 `title` 파라미터로 섹션 구분 (장식선 대신)
- 모든 스크립트 출력은 Rich Console을 통해 출력 (print 대신 `console.print`)
- 캡처 워크플로우: 스크립트가 Rich Console로 출력 → `book_capture.py --cmd`로 캡처
- 캡처 실행 전에 대상 스크립트에 `console.rule()` 잔존 여부를 grep으로 점검

## 코드 일치 점검

완성 코드가 변경되면 챕터 기술 파트가 불일치할 수 있다.

### writer 점검

- `chapters/*.md`의 코드 블록이 해당 `rag-end/` 코드와 일치하는지 확인
- 불일치 시 챕터를 `rag-end/`에 맞춤

### editor 점검

- `chapters/*.md`의 코드 블록과 `rag-end/` 사이 불일치 감지
- 함수명·파라미터·임포트 경로 변경 여부 확인
- 불일치 발견 시 경고 + 수정 대상 챕터 목록 제시
