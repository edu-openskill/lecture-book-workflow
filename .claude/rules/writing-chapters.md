# 챕터 집필 규칙

chapters/ 하위 마크다운 파일을 수정·작성할 때 자동 적용된다.

---

## 필수 스킬 호출

| 작업 | 스킬 |
|------|------|
| 챕터/프롤로그/에필로그 작성 | writing |
| 초안 완성 후 AI 패턴 교정 | humanizer |
| 검토 시 | review |
| 다이어그램 생성 시 | visual |
| 코드 분석/설명 시 | code |

## 비유 일관성 전략

책 유형에 따라 자동 결정한다.

- **버전업형** (하나의 기술을 챕터마다 발전): 하나의 비유로 끝까지. 챕터가 넘어가도 같은 비유를 확장
- **독립형** (챕터마다 다른 기술): 챕터별로 각각의 비유. 기술이 다르면 비유도 달라야 함
- STEP 1(씨앗)에서 책 유형 판단 → seed.md에 기록 → 이후 비유 전략 자동 적용

## 이미지 플레이스홀더 (Phase 5a)

챕터 작성 시 아래 플레이스홀더를 삽입한다.

| 유형 | 플레이스홀더 | 담당 |
|------|-------------|------|
| Mermaid/D2 다이어그램 | MD 안에 mermaid 코드블록 | writer |
| 플로우 카드 (시퀀스/단계 흐름) | `[FLOW CARD: id]` + desc 서술 | writer 삽입 → 유저가 visual 스킬 호출 |
| 개념도 | `[GEMINI PROMPT: 설명]` | writer 삽입 → 유저가 Gemini에 입력 |
| 실행 결과 캡처 | `[CAPTURE NEEDED: 설명]` | writer 삽입 → 유저가 screenshot 스킬 호출 |

경로 규칙: `assets/CH{N}/{gemini|terminal|diagram}/{NN}_{id}.png`

### 플로우 카드 플레이스홀더 형식

시퀀스/단계 흐름 다이어그램(ReAct, 파이프라인, 요청-응답 등)에 사용한다. Writer는 **어떤 그림이 필요한지 자연어로 서술**만 한다. 유저 요청 시 메인 세션에서 visual 스킬이 서술을 읽고 맥락에 맞는 JSON을 설계하여 4가지 레이아웃 변형을 생성한다.

**Writer는 JSON 구조나 화살표 흐름을 직접 기술하지 않는다.** 서술(desc)에 이 그림이 보여줘야 할 것, 강조 포인트, 챕터 내 위치의 맥락을 적는다.

```markdown
<!-- [FLOW CARD: {NN}_{identifier}]
path: assets/CH{N}/diagram/{NN}_{identifier}.png
desc: {이 그림이 보여줘야 할 것을 자연어로 서술.
  어떤 흐름인지, 어디가 강조 포인트인지, 앞뒤 맥락은 무엇인지.}
-->
![](../assets/CH{N}/diagram/{NN}_{identifier}.png)
*{캡션}*
```

예시:
```markdown
<!-- [FLOW CARD: 06_sequence-crud]
path: assets/CH06/diagram/06_sequence-crud.png
desc: CRUD 질문이 통합 에이전트에서 처리되는 전체 흐름.
  사용자 질문이 QueryRouter를 거쳐 분류되고,
  ReAct 패턴(Think→Act→Result→Observe)으로 DB를 조회한 뒤 답변.
  각 단계가 한눈에 보이는 플로우 카드.
-->
![](../assets/CH06/diagram/06_sequence-crud.png)
*CRUD 질문의 처리 흐름입니다*
```

```markdown
<!-- [FLOW CARD: 07_cache-highlight]
path: assets/CH07/diagram/07_cache-highlight.png
desc: 전체 에이전트 흐름 중 ResponseCache(메모장) 부분만 강조.
  질문이 들어오면 메모장을 먼저 확인하고, 없으면 LLM을 거쳐 답을 구한 뒤
  메모장에 저장하는 구조. 캐시 조회/저장 두 곳이 이번 실습 범위.
  나머지 단계는 흐리게 처리하여 실습 범위를 명확히 보여준다.
-->
![](../assets/CH07/diagram/07_cache-highlight.png)
*파란색이 실습 1에서 만드는 부분입니다*
```

## 기술 파트 실습 규칙

기술 파트도 이야기다. 코드를 설명할 때도 `rules/storytelling.md`의 대화체·스토리텔링 기법을 적용한다.

### 상수·매개변수 첫 등장 시 설명

코드에 상수나 매개변수가 처음 등장하면 **코드블록 앞에서** 역할·기본값·어디서 넘어오는지를 설명한다. 독자가 "이 값은 뭐지?"라고 멈추지 않게 한다.

예시:
- `chunk_size`(기본값 500) — 한 조각의 최대 글자 수. main.py CLI 옵션으로 전달
- `embedding_model_name` — `"jhgan/ko-sroberta-multitask"`. 이야기 파트에서 선택한 모델

상수가 3개 이상이면 테이블로 정리한다.

### 보조 함수(인프라) 등장 시 설명

`[참고]` 파일에서 import하는 보조 함수가 코드에 등장하면, 직접 작성하지 않는 함수임을 밝히고 역할을 한 줄씩 설명한다. 독자가 "이 함수는 어디서 온 거지?"라고 혼란스럽지 않게 한다.

### 앞 내용과 연결 (비유 회수)

기술 파트의 모든 설명은 앞에서 사용한 비유·캐릭터·장면을 **구체적으로 회수**해야 한다. "이번 실습에서 이 흐름을 하나씩 만들어 봅니다" 같은 **템플릿 문장 금지**. 대신 해당 챕터의 비유를 직접 언급한다.

적용 범위:
1. **실습 순서 전체 흐름 설명** — 다이어그램 아래에서 비유로 흐름을 풀어줌
2. **각 코드블록 위** — 이 코드가 비유의 어떤 부분인지 연결
3. **실습 도입부** — 이전 실습과의 관계를 비유로 연결

- BAD: "질문이 들어오면 라우터가 분류하고 도구를 호출하여 답변을 생성합니다."
- GOOD: "질문이 안내데스크(QueryRouter)에 도착하면, 안내 직원이 DB 담당자에게 보낼지 문서실로 보낼지 판단합니다. 담당자가 결과를 가져오면 에이전트가 하나로 묶어 답변합니다."

### 실습 간 연결

이전 실습에서 만든 함수를 다음 실습에서 사용할 때 반드시 언급한다. "실습 1에서 만든 `split_text_into_chunks`가 여기서 호출됩니다"처럼 부품이 조립되는 과정을 독자가 상상할 수 있게 한다.

### 코드블록 설명 중복 금지

코드블록 **위**에는 이 코드가 무슨 역할인지(왜 만드는지), **아래**에는 코드 흐름의 핵심 동작(어떻게 작동하는지)을 쓴다. 위/아래에 같은 내용을 반복하지 않는다.

### 이야기에서 다루지 않은 용어

기술 파트에서 이야기 파트에 없던 새 용어/함수/개념이 등장하면 즉시 한두 문장으로 설명한다. 설명 없이 넘어가지 않는다.

### 코드블록 친절 설명 (CH04 기준)

코드블록 하나를 설명할 때 아래 요소를 갖춘다.

1. **코드블록 위**: 이 코드가 왜 필요한지(동기), 이야기 파트의 비유와 연결
2. **코드블록 내부**: 단계 주석(`# N. 설명`)으로 흐름 안내
3. **코드블록 아래**: 핵심 동작 1~2문장 설명 + 주요 변수/함수 역할 풀이
4. **보조 함수 테이블**: import하는 외부 함수가 2개 이상이면 테이블로 정리 (파일, 함수명, 역할)
5. **데이터 흐름**: 입력→출력이 명확하지 않으면 예시 데이터(JSON, 터미널 출력 등)를 함께 보여준다

불친절한 코드 설명 = 독자가 멈추는 지점. 코드만 던지고 넘어가지 않는다.

### 캡션 톤

이미지 캡션(`*설명*`)과 다이어그램 캡션은 **존댓말(해요체/합니다체)** 로 작성한다. 반말 종결(-다, -이다)을 사용하지 않는다.

- BAD: "파란색이 이번 실습에서 만드는 부분이다."
- GOOD: "파란색이 이번 실습에서 만드는 부분입니다."

### 다이어그램 실습 번호 표시

전체 흐름 다이어그램에서 특정 실습 범위를 하이라이트할 때, `rect` 블록 안에 `Note`로 실습 번호를 표시한다.

```
rect rgb(214, 234, 248)
    Note over AI 비서: 실습 1
    ...
end
```

### 소스코드 준비 섹션

- **CH01(최초)**: git clone, 레포 테이블, 파일 트리 풀 버전
- **CH02 이후**: "클론한 레포에서 이번 챕터의 폴더로 이동합니다" + `cd rag-start/exNN` + 파일 트리만 유지. git clone URL, 레포 테이블, "아직 클론하지 않으셨다면" 분기 제거

## 금지 패턴 (why-log 기반)

- "이번 버전: exNN → exNN" 메타데이터를 챕터에 넣지 않는다
- 실습 순서를 다이어그램으로 만들지 않는다. 번호 목록으로 작성한다
- **"이야기 파트", "기술 파트" 메타 라벨 금지** — 챕터 본문에서 파트 구조를 직접 언급하지 않는다. "이야기 파트에서 다룬"이 아니라 "앞에서 다룬", "1.3에서 다룬"으로 자연스럽게 참조한다
