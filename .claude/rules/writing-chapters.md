# 챕터 집필 규칙

`chapters/` 하위 마크다운 파일을 수정·작성할 때 자동 적용. **워크플로우·스킬 호출·이미지 플레이스홀더·메모 박스** 관리.

## 함께 로드되는 규칙

챕터 파일을 편집할 때는 **이 규칙과 함께 반드시 다음 파일을 로드·준수**한다. 모두 `projects/*/chapters/*.md` paths 스코프가 걸려 있어 자동 적용된다.

| 규칙 | 파일 | 핵심 |
|------|------|------|
| **스토리텔링** | `.claude/rules/storytelling.md` | 캐릭터 삼각 구도(팀장·동료·오픈이), 대화체 톤, Show-Don't-Tell, Try/Fail, 비유 규칙 |
| **챕터 포맷** | `.claude/rules/chapter-format.md` | 실습 챕터 12단계 포맷(목표 → 준비 → 섹션 → 용어 → 기억하자) |

위 두 파일을 먼저 훑지 않고 챕터 본문을 작성·수정하지 않는다. 특히 **대화 장면**을 쓰거나 고칠 때는 `storytelling.md`의 "캐릭터 삼각 구도", "대화체 톤 규칙", "기대 전복", "Show Don't Tell" 섹션을 반드시 확인한 뒤 손을 댄다.

## 그 외 전역 규칙

- 톤·편집·포맷 → `style.md`
- 산출물 형식·캡션·이미지 규칙 → `structure.md`
- 코드블록·실습 규칙 → `code.md`

## 코드 분석 참조 (최우선)

챕터 집필·수정 전에 반드시 **`projects/[책이름]/planning/code-analysis-vN.md`** (가장 최근 버전)를 읽는다. 완성 코드의 구조·재사용 관계·기술 스택·의도 밖 제외 목록을 정리한 단일 근거.

| 시점 | 동작 |
|------|------|
| 챕터 신규 작성 | `code-analysis-vN.md` 전체 읽기 → 해당 챕터의 핵심 기능·재사용 매트릭스 확인 |
| 챕터 수정 | 수정 대상이 코드블록·파일 경로를 포함하면 해당 ex 구조를 code-analysis에서 재확인 |
| 코드블록 추가 | 참조 함수·파일이 재사용 원칙에 맞는지 점검 (직접 import 금지) |

**내용 불일치 시**: 본문이 code-analysis와 다르면 **code-analysis가 우선**. 본문을 고치거나 코드·분석을 먼저 동기화한 뒤 집필한다.

**버전 규칙**: 코드 구조가 바뀌면 `code-analysis-vN.md` 새 버전 생성(덮어쓰기 금지). 챕터 작성 시 최신 `vN` 선택.

## 비유 일관성 전략

책 유형에 따라 자동 결정한다.

- **버전업형** (하나의 기술을 챕터마다 발전): 하나의 비유로 끝까지. 챕터가 넘어가도 같은 비유를 확장
- **독립형** (챕터마다 다른 기술): 챕터별로 각각의 비유. 기술이 다르면 비유도 달라야 함
- STEP 1(씨앗)에서 책 유형 판단 → `seed.md`에 기록 → 이후 비유 전략 자동 적용

## 메모 박스 자율 호출

이야기 파트에서 캐릭터가 **수첩·포스트잇·쪽지**에 내용을 정리하는 장면이 나오면 `:::memo` directive를 삽입한다.

### 호출 조건 (하나라도 해당 시)

| 장면 | 예시 |
|------|------|
| 실험·관찰 결과 정리 | OCR 결과의 한계를 수첩에 적는 장면 |
| 시행착오 기록 | "두 번째 시도에서도 실패. 원인은…" |
| 문제를 좁히는 생각 | 다음 단계로 넘어가기 전 요점 |
| 꼭 기억할 규칙·발견 | "이 구조는 나중에 또 쓰일 것 같다" |

**제외**: 챕터 닫기 총정리(`:::remember`), 용어 확정(`:::term-box`), 보조 정보(`:::tip`).

### 삽입 방식

1. 반드시 **캐릭터의 물리적 행동**이 선행. 예: "수첩을 펼쳤습니다", "서랍에서 수첩을 꺼냈습니다"
2. `:::memo` directive로 작성. 상세 템플릿 → `components-catalog/boxes/memo.md`
3. 항목은 1~4개. 5개를 넘기면 수첩 느낌이 사라진다
4. 같은 챕터에서 3개 초과 금지

```markdown
:::memo
**— 제목 —**

1. **항목 이름**
   - 상세 1
   - 상세 2
2. **항목 이름**
   - 상세 1
:::
```

### 제목 톤

- 실험 결과: `— 실험 메모 —`
- 문제 식별: `— 문제 정리 —`
- 생각 정리: `— 생각 정리 —`
- 체크리스트: `— 기억할 것 —`
- 한 줄 쪽지: `— 쪽지 —`

## 이미지 플레이스홀더

챕터 작성 시 아래 플레이스홀더를 삽입한다. **다이어그램 형식 선택 원칙**(수치 있는 구조도 → HTML inline, 개념·비유 → 이미지)과 캡션·경로 규칙은 `structure.md` 참조.

| 유형 | 플레이스홀더 | 담당 |
|------|-------------|------|
| Mermaid/D2 다이어그램 | MD 안에 mermaid 코드블록 | writer |
| 플로우 카드 (시퀀스·단계 흐름) | `[FLOW CARD: id]` + desc 서술 | writer 삽입 → 유저가 visual 스킬 호출 |
| 개념도 | `[GEMINI PROMPT: 설명]` | writer 삽입 → 유저가 Gemini에 입력 |
| 실행 결과 캡처 | `[CAPTURE NEEDED: 설명]` | writer 삽입 → 유저가 screenshot 스킬 호출 |

**경로 규칙**: `assets/CH{N}/{gemini|terminal|diagram}/{NN}_{id}.png`

### 플로우 카드 플레이스홀더 형식

시퀀스·단계 흐름 다이어그램(ReAct, 파이프라인, 요청-응답 등)에 사용. Writer는 **어떤 그림이 필요한지 자연어로 서술**만 한다. 유저 요청 시 메인 세션에서 visual 스킬이 서술을 읽고 맥락에 맞는 JSON을 설계하여 4가지 레이아웃 변형을 생성.

**Writer는 JSON 구조나 화살표 흐름을 직접 기술하지 않는다.** 서술(desc)에 이 그림이 보여줘야 할 것, 강조 포인트, 챕터 내 위치의 맥락을 적는다.

```markdown
<!-- [FLOW CARD: {NN}_{identifier}]
path: assets/CH{N}/diagram/{NN}_{identifier}.png
desc: {이 그림이 보여줘야 할 것을 자연어로 서술.
  어떤 흐름인지, 어디가 강조 포인트인지, 앞뒤 맥락은 무엇인지.}
-->
![](../assets/CH{N}/diagram/{NN}_{identifier}.png)
*{캡션}*
```

예시:
```markdown
<!-- [FLOW CARD: 06_sequence-crud]
path: assets/CH06/diagram/06_sequence-crud.png
desc: CRUD 질문이 통합 에이전트에서 처리되는 전체 흐름.
  사용자 질문이 QueryRouter를 거쳐 분류되고,
  ReAct 패턴(Think→Act→Result→Observe)으로 DB를 조회한 뒤 답변.
-->
![](../assets/CH06/diagram/06_sequence-crud.png)
*CRUD 질문의 처리 흐름입니다*
```
