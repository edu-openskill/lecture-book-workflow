# 글로벌 구조 규칙

**산출물 구조·버전 관리·캡션·이미지·워크플로우 진행** 규칙. 톤·편집은 `style.md`, 소설 기법은 `storytelling.md`, 코드블록은 `code.md` 참조.

---

## 버전 관리

- 파일명에 `-vN` 접미사. 절대 덮어쓰지 않음 (`seed-v1.md → seed-v2.md`)

---

## 질문/선택 규칙

- 선택형 UI. 일반 4~10개, 추천은 첫 번째 + (Recommended)
- **STEP 1(씨앗) 의도 파악 시에만**. 선택지 3개 + "다시 추천해줘" 1개 (총 4개)
- 답변 즉시 `answers.md`에 기록
- 상수는 질문하지 않음 (톤, 스타일, 코드 분류 등)
- **저자 경험 질문 금지**. AI가 "처음 접하는 사람의 전형적 경험"으로 범용 작성

---

## 워크플로우 진행

1. **한 STEP씩 진행**: 각 STEP의 산출물이 완성된 후 다음 STEP으로 넘어간다
2. **자연스러운 대화**: 질문을 딱딱하게 나열하지 않고, 대화하듯 자연스럽게 진행한다
3. **답변 즉시 저장**: 모든 STEP의 질문 답변을 `answers.md`에 기록한다. 유저의 수정 요청은 `review/revision-log.md`에 기록
4. **STEP 완료 안내**: 각 STEP이 끝나면 다음 STEP 명령어를 안내한다
5. **되돌아가기 허용**: 이전 STEP 산출물을 수정하고 싶으면 새 버전으로 생성한다
6. **기본값 제안**: 답변이 어려운 항목은 합리적인 기본값을 제안한다
7. **인사이트 질문**: 단순히 "뭐야?"가 아니라 "이건 생각해봤어?" 형태로 묻는다
8. **의도가 필터다**: `seed.md`의 의도가 이후 모든 결정의 기준이다
9. **상수는 질문하지 않는다**: 톤·스타일·챕터 구조·코드 분류 등은 자동 적용한다

## progress.json 운영 규칙

세션이 끊기거나 다시 시작할 때 현재 상태를 복구하는 파일.
템플릿: `.claude/progress-template.json` / 예시: `.claude/progress-example.json`

1. **STEP 시작 시**: 해당 step의 status를 `in_progress`로, `current_step` 업데이트
2. **산출물 생성 시**: artifact 경로와 artifact_version 업데이트
3. **검토 완료 시**: review 결과(pass/conditional_pass/fail) 업데이트
4. **STEP 완료 시**: status를 `done`으로
5. **챕터 추가 시**: step_5_chapters.chapters 배열에 push
6. **세션 재개 시**: progress.json 읽고 `current_step` + 미완료 항목 확인 후 이어가기
7. **`현재 상태` 명령어**: progress.json 기반으로 요약 출력

## 산출물 형식

- Markdown 형식, 파일명은 번호 접두사 (예: `01-시작하기.md`)
- 이미지 삽입. `![](경로)` — alt 텍스트 비움

### 다이어그램 형식 선택

다이어그램·그림을 만들 때 **내용 성격**으로 형식을 결정한다. 이 선택이 캡션 형식을 자동으로 결정한다.

| 내용 성격 | 형식 | 캡션 |
|----------|------|-----|
| **수치·실측값이 포함된 구조도·결과표** (예: 실험 결과 표, 성능 비교, 파이프라인 단면에 수치 표시) | HTML inline `<div>` + CSS 변수 | HTML 내부 `<div class="xx-caption">` 또는 `<figcaption>` |
| **개념·비유 일러스트** (예: 사서·확대경·눈 은유, 추상 흐름도) | 이미지 `![](경로)` (Gemini/visual 스킬 생성) | 외부 마크다운 이탤릭 `*그림 N-N. 설명*` |
| **단계 흐름·시퀀스** (수치 없이 단계만 표시) | 이미지 또는 HTML 중 맥락에 맞게 (수치 추가 예상 시 HTML) | 대상에 맞춰 |

**판단 기준**:
1. 나중에 **수치가 바뀔 가능성**이 있으면 HTML (재생성 비용 없음)
2. **정적 개념·비유** (사람·물건 은유, 한 번 그리면 끝)은 이미지 (CSS로 표현 어려움)
3. **단계 흐름**은 수치 여부로 갈림. 수치 없고 흐름만이면 이미지도 OK, 수치 덧붙을 여지 있으면 HTML

판단이 애매할 땐 HTML을 기본값으로 택한다. 나중에 수치를 추가하기 쉽고, CSS 변수(`var(--color-*)`, `var(--fs-*)`)로 테마 적응이 자동이기 때문이다.

### 이미지 캡션

- **마크다운 이미지 캡션**. 이미지 바로 아래 `*그림 N-N. 설명*` 이탤릭으로 별도 작성 (빌드 시 `<div class="caption">`으로 자동 변환)
- **HTML 다이어그램 내부 캡션**. `<div class="xx-caption">그림 N-N. 설명</div>` 또는 `<figcaption>` 컴포넌트 일부로 유지
- **중복 금지**. 다이어그램 안에 캡션이 있으면 밖에 마크다운 캡션을 중복으로 넣지 않는다
- **캡션 톤**. 존댓말(합니다·입니다체) 필수. 반말(~다·~이다) 금지
- **캡션 정렬**. 가운데 정렬. 마크다운 이탤릭 캡션은 빌드 시 자동 중앙 정렬. HTML 내부 캡션도 `text-align:center` 필수
- **`그림 N-N:` 접두어 금지**. HTML 파이프라인은 자동 번호 미지원이므로 수동 번호 (`그림 N-N.`) 필수. Typst 파이프라인만 자동 부여

### 그림 번호 규칙

- 챕터 내 순번: `그림 N-1`, `그림 N-2`, ... (N = 챕터 번호)
- 번호 삽입 순서대로 증가. 중간에 새 그림을 끼우면 뒤 번호를 전부 시프트

### 이미지 플레이스홀더

- `[GEMINI PROMPT]` — 개념도 (유저가 Gemini에 입력)
- `[CAPTURE NEEDED]` — 실행 결과 캡처 (유저가 screenshot 스킬 호출)
- `[FLOW CARD]` — 시퀀스·단계 흐름 (유저가 visual 스킬 호출)
- 상세 형식 → `writing-chapters.md`

### `<img>` 경로

- 경로 변경 시 실제 파일도 이동 필수

## 챕터 작성 규칙

### 파트 라벨 H2 금지

"## 이야기 파트" / "## 기술 파트" 같은 라벨형 H2 금지. H2는 해당 내용을 담은 자연스러운 제목으로 작성. `---` 수평선도 사용 금지. 대신 전환 문장만으로 자연스럽게 넘김 (예: "이제 직접 만들어 보겠습니다.")

### 챕터 본문에서도 파트 지칭 금지

본문에서 "이야기 파트에서 다룬"이 아니라 "앞에서 다룬", "1.3에서 다룬"으로 자연스럽게 참조한다.

### 레포 소개 (기술 파트 시작 직후)

- 레포 소개 블록 필수. URL 테이블만으로 끝내지 마라
- 형식: git clone → 파일 트리([실습]·[설명]·[참고]) → start·end 안내
- 파일 트리는 `outline-vN.md` 구조 그대로
- 닫기: "챕터를 따라 하며 코드를 작성하고, 막히면 완성 코드를 참고하세요"

### 챕터 공통

- 구조: 문제 → 기술 소개 → 해결 → 결과 (상세 → `storytelling.md`)
- 닫기: "이것만은 기억하자" + 다음 예고
- 이미지 플레이스홀더 필수

### 챕터 점검 규칙

- 챕터 점검 시 기존 챕터 패턴을 먼저 확인. 구조 문제는 리라이트
- STEP 7 머릿말 작성 시 저자 정보(수, 이름, 순서)를 반드시 질문
- 상세 체크리스트 → `review` 스킬

## 에이전트 디스패치 규칙

1. 각 STEP의 workflow 파일 헤더에 명시된 에이전트를 순서대로 디스패치한다
2. STEP 완료 전 디스패치 체크리스트 대조 — 헤더에 명시된 에이전트 목록과 실제 호출된 에이전트 목록을 비교하여, 누락이 있으면 STEP을 완료하지 않고 누락분을 호출
3. `illustrator`는 유저 요청 시에만 디스패치. writer·editor는 자동, 이미지 생성은 유저가 코드 완성을 확인한 후 요청
