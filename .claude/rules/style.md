# 글로벌 스타일 규칙

모든 에이전트가 준수하는 **톤·편집·포맷** 규칙. 소설 기법·캐릭터는 `storytelling.md`, 산출물 구조·캡션은 `structure.md`, 코드블록 규칙은 `code.md` 참조.

---

## 톤

### 서술어 규칙 — `~합니다·~입니다` 원칙

기술 서적 본문은 **`~합니다`·`~입니다`·`~습니다`** 로 통일한다. 독자에게 직접 행동을 지시하는 **명령·권유 문장**에서만 `~세요`/`~해 주세요`를 허용한다.

| 문장 유형 | 종결 | 예시 |
|-----------|------|------|
| 기술 설명·사실 서술 | `~합니다`/`~입니다` | "LLM이 생성한 답변이 돌아옵니다." |
| 개념 정의 | `~입니다`/`~이다` | "벡터는 의미를 담은 숫자 배열입니다." |
| 독자에게 행동 지시 | `~세요`, `~해 주세요` | "터미널에서 실행해 주세요." |
| 저자 선언 | `~보겠습니다` | "다음 절에서 도구를 만들어 보겠습니다." |
| 대화 / 내면독백 | 캐릭터성 그대로 | `storytelling.md` 대화체 톤 규칙 참조 |

**원칙**: 한 문장 단위로 위 표의 다섯 종결 중 하나로 맺는다. 한 문단 안에서 톤을 섞지 않는다. 표에 없는 종결은 본문에서 쓰지 않는다(대화·독백 예외).

### 종결어미 반복 처리

같은 어미가 3문장 연속되면 리듬이 죽는다. `~합니다`·`~입니다`·`~습니다` 안에서 변화를 준다.

| 대체 어미 | 용도 |
|-----------|------|
| `~합니다` / `~입니다` | 기본 |
| `~습니다` | 과거·상태 서술 |
| `~됩니다` / `~십니다` | 수동·높임 |
| 명사구·동사 생략 | 짧은 호흡 ("한 페이지.", "여기서 끝.") |

### "봅시다"·"봅니다" 금지

독자 권유형 "~봅시다"와 합니다체 시도형 "~봅니다" 모두, 저자 선언형 "~보겠습니다"로 쓴다.

- BAD: "다음 절부터 도구를 만들어 봅시다."
- BAD: "다음 절부터 도구를 만들어 봅니다."
- GOOD: "다음 절부터 도구를 만들어 보겠습니다."

## 편집

- **볼드 띄어쓰기**. 볼드 내부에 괄호·따옴표 등 특수문자가 있으면 닫힌 뒤 띄어쓰기 (`**통행증(Access Token)** 을`). 순수 텍스트 볼드는 붙여쓰기 (`**Docker**라는`, `**팀장**이`)
- **문장 끝 콜론 금지** (`.` 사용)
- **능동태 우선** ("~되어진다" 금지)
- **영어 용어 처음 등장**. 한국어(English) 형태. 이후 한국어만
- **이모지 전면 금지** (박스 스타일 포함 모든 곳)
- **대시 `--`·`—`(em dash)·`–`(en dash) 금지**. 본문에서 부연 설명이 필요하면 **괄호 `()`** 로 묶는다. 섹션 제목(H2/H3)에서 부연이 필요하면 **단일 하이픈 `-` 하나**만 사용 (예: `## 10.2 확대경 달기 - OCR`)
- **문맥 전환 시 빈 줄 삽입**
- **문단 구성**. 같은 맥락의 문장은 빈 줄 없이 이어 쓴다. 한 줄짜리 문장이 연속으로 나열되지 않게 한다
- **숨돌리기 빈 줄**. (1) 핵심 키워드/개념 도입 문장 전후 (2) 감정·반전 전환 지점 (3) 비유 마무리 직후에만 빈 줄 삽입. 단순 설명 구간은 문단으로 묶는다

## 디자인 제약

- 박스 테두리에 색상 띠(border-left 굵은 색상) 금지. 일반 테두리 + 뱃지·배경색으로 구분
- 전자책 표지 스펙. JPG, 2MB 이하, 750×1110px, 72ppi
- 표지 생성 후 Read로 겹침·침범 시각 검증 필수

---

## 금지 패턴

- 설교 ("이것이야말로...")
- 반복 강조 ("이것이 바로...이유")
- 정의 전 용어 사용
- 진부한 표현 ("두말할 필요 없이")
- 주관적 감탄 ("정말 놀라운 기능!")
- 시간 의존 표현 ("최근에 나온")
- 불필요한 감정 추임새 ("잠깐 기뻤습니다", "순간 설렜습니다" 등 서사에 기여하지 않는 한 줄짜리 감정 표현)

## 글쓰기 원칙

1. **이야기 먼저**: 모든 챕터는 이야기 파트로 시작 (상세 구조 → `storytelling.md`)
2. **비유는 자유롭게**: 길이 제한 없음. 용어의 정식 설명이 길어지면 기술 파트로 (상세 → `storytelling.md` 비유 규칙)
3. **기술은 뒤에서**: 이야기 파트가 끝난 후 기술 파트에서 상세 설명
4. **비유 → 정의 2단계**: 이야기 파트에서 비유로 먼저 풀고, 기술 파트에서 정식 정의. 비유로 넘어간 용어는 기술 파트에서 반드시 정식 정의
5. **한 챕터 한 버전**: 챕터가 끝나면 결과물이 한 단계 성장
6. **점진적 깊이**: 쉬운 것 → 어려운 것
7. **코드 최소화**: 이야기 파트에 코드 없음. 기술 파트에서만 (상세 → `code.md`)
8. **챕터 닫기**: "이것만은 기억하자" + 다음 버전 예고
9. **기술은 저자가 결정**: AI가 제안, 저자가 최종 결정
10. **실패 먼저**: 정답부터 알려주지 않는다. 실수 → 에러 → 해결 순서
11. **실습 경계 원칙**: 실습 코드에 등장하는 것만 설명. 실습에 없는 개념은 "이런 게 있다" 한 줄 언급만. 심화를 넣기 시작하면 끝이 없다
12. **데이터는 보여준다**: "데이터가 흘러간다" 같은 추상적 설명에는 실제 데이터 구조를 함께 보여준다. 시각화 도구는 상황에 맞게 선택
    - **코드블록**: JSON·Python dict·터미널 출력 (```` ```json ````)
    - **HTML 다이어그램**: 흐름도·시퀀스·아키텍처 (inline HTML + CSS 변수)
    - **개념도 이미지**: 추상 개념의 시각화 (`[GEMINI PROMPT]` 플레이스홀더)
    - **터미널 캡처**: 실행 결과·에러 (`[CAPTURE NEEDED]` 플레이스홀더)

## 출력 형식

- 모든 파일은 Markdown 형식으로 작성한다
- 파일명은 한글 사용, 번호 접두사를 붙인다 (예: `01-시작하기.md`)
- 코드 블록에는 반드시 언어 태그를 명시한다
- 이미지 삽입 형식은 `structure.md`의 "산출물 형식" 참조
