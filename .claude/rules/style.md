# 글로벌 스타일 규칙

모든 에이전트가 준수하는 톤, 편집, 포맷 규칙.

---

## 톤

### 서술어 혼합 규칙 (책 전체 공통)

기술 서적이므로 본문은 기본 **하십시오체(~합니다)** 를 중심으로 쓰되, 독자를 직접 끌어당기는 문장에서만 **해요체(~요/~세요)** 를 섞는다. 혼합 비율은 "단단한 설명(합니다) + 따뜻한 권유(요)".

| 문장 유형 | 종결 | 예시 |
|-----------|------|------|
| 기술 설명·사실 서술·챕터 흐름 안내 | `~합니다` (하십시오체) | "LLM이 생성한 답변이 돌아옵니다." "이번 챕터에서는 RAG 파이프라인을 조립합니다." |
| 독려·권유·독자 직접 지시 | `~세요`, `~해주세요`, `~해요` | "터미널에서 실행해 주세요.", "가볍게 따라오세요." |
| 감탄·질문·가벼운 전환 | `~요?`, `~죠`, `~군요` | "실행해 볼까요?", "이게 바로 환각이죠." |
| 대화 / 내면독백 | 캐릭터성 그대로 | "**팀장**: '언제까지야?'", "*잠깐, 뭐라고?*" |

**금지 패턴**:
- 본문 설명 전체가 "~요/~에요/~이에요"로만 끝남 (가벼워 보임)
- 기술 용어 정의·사실 서술을 "~같아요", "~되는 거예요" 같이 구어체로 마무리
- 한 문단에 `~합니다`와 `~해요`를 무작위 섞어 리듬이 깨지는 경우

**원칙**: 한 문단 내에서 톤을 바꾸지 않는다. 설명 문단은 `~합니다`로 통일, 독려 문장은 문단 끝이나 별도 문단에 배치.

### 멘토링 스타일

독자에게 친절하게 권장하거나 안내하는 어조("~해 볼까요?", "~해 보세요", "~죠")를 적절히 섞어 멘토처럼 설명한다. 단, 위의 혼합 규칙 범위 안에서 사용한다.

### 캐릭터 표기

- 대화 표기. **역할**: "대사"
- 내면 독백. *이탤릭* 사용 (`*독백 내용*`). 앞뒤 빈 줄로 독립 행 배치. 예: `*잠깐, 이거 아닌데?*`
- 등장인물. 주인공 **오픈이**, 주변인물 **팀장**/**선배**/**동료** (역할명만, 실명 금지)

### 구조 금지

"## 이야기 파트" / "## 기술 파트" 같은 라벨형 H2 금지. H2는 해당 내용을 담은 자연스러운 제목으로 작성 (`---` 수평선 사용 금지).

## 편집

- **볼드** 띄어쓰기. 볼드 내부에 괄호/따옴표 등 특수문자가 있으면 닫힌 뒤 띄어쓰기 (`**통행증(Access Token)** 을`). 순수 텍스트 볼드는 붙여쓰기 (`**Docker**라는`, `**팀장**이`)
- 문장 끝 콜론 금지 (`.` 사용)
- 능동태 우선 ("~되어진다" 금지)
- 영어 용어 처음 등장. 한국어(English) 형태. 이후 한국어만
- 이모지 전면 금지 (박스 스타일 포함 모든 곳에서 금지)
- 문맥 전환 시 빈 줄 삽입
- 문단 구성. 같은 맥락의 문장은 빈 줄 없이 이어 쓴다. 한 줄짜리 문장이 연속으로 나열되지 않게 한다
- 숨돌리기 빈 줄. (1) 핵심 키워드/개념 도입 문장 전후 (2) 감정/반전 전환 지점 (3) 비유 마무리 직후에만 빈 줄 삽입. 단순 설명이 이어지는 구간은 문단으로 묶는다

## 금지 패턴

- 설교 ("이것이야말로...")
- 반복 강조 ("이것이 바로...이유")
- 정의 전 용어 사용
- 진부한 표현 ("두말할 필요 없이")
- 주관적 감탄 ("정말 놀라운 기능!")
- 시간 의존 표현 ("최근에 나온")
- 불필요한 감정 추임새 ("잠깐 기뻤습니다", "순간 설렜습니다" 등 서사에 기여하지 않는 한 줄짜리 감정 표현)

## 글쓰기 원칙

1. **이야기 먼저**: 모든 챕터는 이야기 파트로 시작한다.
2. **비유는 자유롭게**: 길이 제한 없음. 용어의 정식 설명이 길어지면 기술 파트로 넘긴다.
3. **기술은 뒤에서**: 이야기 파트가 끝난 후 기술 파트에서 상세 설명.
4. **비유 → 정의 2단계**: 어려운 개념은 반드시 2단계로 설명한다. 이야기 파트에서 비유로 먼저 풀고, 기술 파트에서 정식 정의를 내린다. 비유로 넘어간 용어는 기술 파트에서 반드시 정식 정의.
5. **한 챕터 한 버전**: 챕터가 끝나면 결과물이 한 단계 성장.
6. **점진적 깊이**: 쉬운 것 → 어려운 것.
7. **코드 최소화**: 이야기 파트에 코드 없음. 기술 파트에서만.
8. **챕터 닫기**: "이것만은 기억하자" + 다음 버전 예고.
9. **기술은 저자가 결정**: AI가 제안, 저자가 최종 결정.
10. **실패 먼저**: 정답부터 알려주지 않는다. 실수 → 에러 → 해결 순서.
11. **실습 경계 원칙**: 실습 코드에 등장하는 것만 설명한다. 실습에 없는 개념은 "이런 게 있다" 한 줄 언급만 하고 넘어간다. 독자가 직접 찾아볼 수 있도록 키워드만 남긴다. 심화를 넣기 시작하면 끝이 없다.
12. **데이터는 보여준다**: "데이터가 흘러간다", "결과가 돌아온다" 같은 추상적 설명에는 실제 데이터 구조를 함께 보여준다. 독자가 "실제로 어떻게 생겼는지"를 눈으로 확인할 수 있어야 한다. 사용 가능한 시각화 도구는 상황에 맞게 선택한다.
    - **코드블록**: JSON, Python dict, 터미널 출력 등 실제 데이터 구조 (```` ```json ````)
    - **D2 다이어그램**: 흐름도, 시퀀스, 아키텍처 등 구조/관계 (`[GEMINI PROMPT]` 또는 D2 직접 작성)
    - **개념도 이미지**: 추상 개념의 시각화, 비유의 도식화 (`[GEMINI PROMPT]` 플레이스홀더)
    - **터미널 캡처**: 실행 결과, 에러 메시지 (`[CAPTURE NEEDED]` 플레이스홀더)

## 출력 형식

- 모든 파일은 Markdown 형식으로 작성한다.
- 파일명은 한글 사용, 번호 접두사를 붙인다 (예: `01-시작하기.md`).
- 코드 블록에는 반드시 언어 태그를 명시한다.
- 이미지 위치는 `![](이미지경로)` 형태로 삽입. alt 텍스트는 비워둠. 캡션은 바로 아래 `*설명*` 이탤릭으로 작성.
