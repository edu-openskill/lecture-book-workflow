# Boxes

> `:::`/`::::` directive 기반 박스 블록. 챕터의 구조적 박스(목표·준비·팁·주의·용어·기억)를 담당한다.

## 속함

6개 directive: `:::goal`, `::::prep`, `:::tip`, `:::note`, `:::term-box`, `:::remember`

## 속하지 않음

- 책 전체 구성도 박스 → [`../fullmap/`](../fullmap/)
- 청크/임베딩 단위 카드 → [`../cards/`](../cards/)
- 비교형(A vs B) 블록 → [`../comparisons/`](../comparisons/)

## 컴포넌트 목록

### :::goal

**언제 쓰는가**: 챕터 첫 머리에 "이번 챕터가 끝나면 무엇을 할 수 있는가"를 3~5개 불릿으로 제시. 독자가 챕터의 도달점을 먼저 파악하고 본문으로 진입하게 한다.

**사용 챕터**: CH01~CH10 (전 챕터 공통 도입부)

**Markdown 사용 예** (CH04 L3-10):
```markdown
:::goal
**이번 챕터가 끝나면**

- PDF, DOCX, XLSX를 **Markdown**으로 변환하는 통합 파서를 이해합니다 *(Parsing)*
- **고정 크기 500자 + 100자 오버랩** 청킹으로 문맥을 보존한 조각을 만듭니다 *(Chunking)*
- **ko-sroberta-multitask** 임베딩으로 텍스트를 768차원 벡터로 바꾸고 **ChromaDB**에 저장합니다
- CLI 검색으로 "휴가 규정" 같은 쿼리가 실제로 원하는 문서를 찾아오는 걸 눈으로 확인합니다
:::
```

**렌더 CSS**: `components.css:22-26` `.goal-box` (연두 그라디언트 배경 + 좌측 4px 초록 보더), `.goal-box .gl` (L24, 제목 라벨), `.goal-box ul/li` (L25-26)

**변형(Variants)**: 없음 (단일 스타일)

**피해야 할 것**
- 목표 1개만 두지 말 것 (3~5개 권장)
- 코드블록/이미지/표 삽입 금지 (순수 텍스트 불릿만)
- "기술 용어 나열" 대신 "~을 이해합니다", "~을 만듭니다", "~을 눈으로 확인합니다" 같은 행동 문장
- 이탤릭으로 괄호 안 영문 키워드(*(Parsing)*)를 덧붙이는 건 허용하지만, 불릿 본문을 영문 전문용어로 채우지 말 것

### ::::prep

**언제 쓰는가**: 챕터 실습을 시작하기 직전, "한 번만 설정하면 되는 준비 작업"을 모아 담는다. 폴더 이동, 가상환경 생성, 라이브러리 설치, 실습 순서 안내처럼 본문 리듬을 끊는 설정성 내용을 격리시키는 컨테이너. 네 개(`::::`) 콜론으로 여는 이유는 내부에 `:::tip` 같은 3콜론 directive를 중첩하기 위함.

**사용 챕터**: CH02~CH10 (실습이 있는 챕터)

**Markdown 사용 예** (CH04 L12-82):
```markdown
::::prep
**준비하기**. 실습 시작 전 한 번만 설정

### 1. 실습 폴더 이동

```bash [터미널] 폴더 이동
cd rag-start/ex04
```

### 2. 실습 환경 구축

```bash [터미널] 환경 구성. macOS / Linux
cd ex04
cp .env.example .env
python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

:::tip
**임베딩 모델은 첫 실행 시 약 400MB 다운로드**

`ko-sroberta-multitask`는 최초 실행 시 HuggingFace에서 자동 다운로드됩니다.
:::

### 3. 사용할 라이브러리

| 패키지 | 역할 |
|-------|------|
| `pypdf` | PDF 텍스트 추출 |
...

### 4. 실습 순서
1. `python src/main.py --step 1`. 파싱 결과만 먼저 확인
2. ...
::::
```

**렌더 CSS**: `components.css:263-292` `.prep-section-md` (흰 배경 + 14px radius 카드, L264-270), `> p:first-child` (헤더 문단 굵게, L271-277), `.prep-section-md h3` (좌측 3px 파란 보더 + padding-left 10px, L278-285), `.prep-section-md ol/ul/table` (L288-292). 내부 `:::note`는 `.prep-note`로 렌더(L294-305).

**변형(Variants)**: 내부에 `:::tip`, `:::note` 중첩 가능. 중첩 시 각자 고유 스타일로 렌더.

**피해야 할 것**
- 본문 이야기/기술 파트 내용을 prep 안에 넣지 말 것 (한 번만 실행하는 설정 성격의 내용만)
- `:::` 3콜론으로 열지 말 것. 내부 directive 중첩을 위해 반드시 `::::` 4콜론
- 첫 줄은 `**준비하기**. ...` 패턴을 유지 (렌더 시 `.prep-section-md > p:first-child` 스타일을 받아 섹션 제목처럼 표시됨)
- 실습 순서 외의 일반 해설을 prep 안에 섞어 장문화하지 말 것

### :::tip

**언제 쓰는가**: 본문 흐름과 직결되지는 않지만 독자가 알아두면 유용한 **보조 정보**를 노란 박스로 분리. 모델 다운로드 용량, 이후 챕터에서 다룰 비교, 선택 가능한 대안 등.

**사용 챕터**: CH01, CH04 등 (챕터마다 필요에 따라 0~수 개)

**Markdown 사용 예** (CH01 L349-353):
```markdown
:::tip
**이 챕터의 임베딩 모델**

`nomic-embed-text`를 사용합니다. 챕터 4에서 한국어에 최적화된 `ko-sroberta-multitask`로 교체합니다.
:::
```

또 다른 예 (CH04 L60-64, `::::prep` 내부 중첩):
```markdown
:::tip
**임베딩 모델은 첫 실행 시 약 400MB 다운로드**

`ko-sroberta-multitask`는 최초 실행 시 HuggingFace에서 자동 다운로드됩니다. 이후부터는 로컬 캐시를 사용하므로 한 번만 다운받으면 됩니다.
:::
```

**렌더 CSS**: `components.css:54-59` `.tip` (아이보리 배경 `#fffaf0` + 1px 주황 보더 `#feebc8`, 10px radius), `.tip .tip-title` (L56, 황토색 제목 + `::before`로 전구 이모지 자동 삽입), `.tip p` (L58-59, 짙은 브라운 톤 본문)

**변형(Variants)**: 없음 (단일 스타일). 단, `::::prep` 내부와 본문 최상위 어디든 배치 가능.

**피해야 할 것**
- 본문 필수 설명을 tip 안에 넣지 말 것 (tip을 건너뛰어도 이해에 지장이 없어야 함)
- 경고·주의·오류 안내 용도로 쓰지 말 것 (그런 내용은 별도 컴포넌트로)
- 제목(`**...**`) 없이 본문만 넣지 말 것. 첫 줄에 볼드 제목 → 빈 줄 → 본문 순서를 유지
- 이모지 문자를 직접 타이핑하지 말 것. `.tip-title::before`가 자동으로 삽입

### :::note

**언제 쓰는가**: 챕터 진행 중 독자에게 **맥락 안내**를 박스로 전달. "이 챕터는 코드 실습이 없습니다" 같은 구조 공지, `::::prep` 내부 설정 주의사항 등. tip이 "알아두면 좋은 정보"라면 note는 "알아야 하는 맥락".

**사용 챕터**: CH03 등 (필요 챕터에 한정)

**Markdown 사용 예** (CH03 L11-13):
```markdown
:::note
**이번 챕터는 코드 실습이 없습니다.** 챕터 4에서 문서를 실제로 파싱하고 벡터 DB에 저장하기 전에, "어떤 문서를 어떤 규칙으로 넣을 것인가"를 먼저 정리합니다. 규칙 없이 바로 구현하면 검색 품질이 엉망이 됩니다.
:::
```

**렌더 CSS**: `components.css:294-305` `.prep-note` (연한 파랑 배경 `#f0f7ff` + 좌측 3px 파란 보더 `var(--color-info)`, 오른쪽만 8px radius), `.prep-note strong` (L304, info-text 컬러 강조), `.prep-note p` (L305, 상속 컬러·폰트사이즈)

**변형(Variants)**: 없음. 본문 최상위/`::::prep` 내부 어디든 배치 가능.

**피해야 할 것**
- 긴 문단을 note에 몰아넣지 말 것 (1~3문장이 이상적)
- 팁과 혼용하지 말 것. 경험 공유·선택지는 `:::tip`, 구조·맥락 공지는 `:::note`
- 이야기 파트의 서사를 note 안에 넣지 말 것. 박스는 서사를 끊는다
- 경고(주의/에러)용으로 남용하지 말 것. 맥락 안내에 한정

### :::term-box

**언제 쓰는가**: 본문 중 핵심 용어가 처음 등장하고 **짧게 정의를 확정해야 할 때** 인라인에 가깝게 덧붙인다. 테이블처럼 여러 용어를 나열하는 게 아니라 "이 용어 하나만 지금 확실히 짚고 넘어가자"는 맥락.

**사용 챕터**: CH01, CH04 등 (용어가 문단 흐름 속에서 처음 등장할 때)

**Markdown 사용 예** (CH04 L341-343):
```markdown
:::term-box
**ChromaDB란?** 텍스트 의미를 담은 **벡터**(768차원 숫자 배열)를 저장하고, 주어진 질문 벡터와 가장 가까운 벡터를 빠르게 찾아 주는 **벡터 데이터베이스**입니다. 일반 관계형 DB(PostgreSQL, MySQL)가 "이름이 홍길동인 행"처럼 정확한 값을 조회한다면, 벡터 DB는 "이 질문과 의미가 비슷한 문서 Top-K"를 조회합니다. 로컬에 파일로 영속 저장이 가능해 서버 재시작 후에도 데이터가 남아 있고, Python 한 줄로 띄울 수 있을 정도로 가볍습니다. 실무에서는 Pinecone, Weaviate, Milvus 같은 유료/대규모용 옵션도 쓰지만, 이 책처럼 로컬 실습, 소규모 운영에는 ChromaDB가 표준에 가깝습니다.
:::
```

**렌더 CSS**: `components.css:307-318` `.term-box` (연한 슬레이트 배경 `#f8fafc` + 좌측 3px 파란 보더 `#2563eb`, 13px 작은 글씨, 오른쪽 6px radius), `.term-box .tb-name` (L313-315, info-text 컬러로 용어명 강조), `.term-box .tb-desc` (L316-318, muted 톤 설명)

**변형(Variants)**: 없음. 단 첫 줄을 `**용어란?** 설명...` 패턴으로 쓰면 볼드 부분이 용어명 역할을 한다.

**피해야 할 것**
- 여러 용어를 한 `:::term-box`에 나열하지 말 것 (복수 용어는 "용어 정리" 테이블로)
- 본격 정의가 필요한 경우에는 본문 문단 또는 표로 풀고, term-box는 한 용어의 빠른 확정용으로만
- 이야기 파트에 사용하지 말 것 (비유가 용어보다 먼저라는 원칙과 충돌). 기술 파트에서 처음 등장할 때만
- 박스 안에 코드블록·이미지를 넣지 말 것. 텍스트 정의만

### :::remember

**언제 쓰는가**: 챕터 맨 끝, "이것만은 기억하자" 마무리 섹션. 챕터에서 독자가 꼭 가져가야 할 핵심을 3개 내외 불릿으로 정리하고, 필요 시 다음 챕터 예고를 덧붙인다. 챕터 닫기 패턴(`rules/storytelling.md` "챕터 닫기")의 시각적 구현체.

**사용 챕터**: CH01~CH10 (전 챕터 공통 마무리)

**Markdown 사용 예** (CH04 L746-752):
```markdown
:::remember
**이것만은 기억하자**

- **문서 → 지식은 네 단계를 거칩니다.** 파싱(손질), 청킹(다지기), 임베딩(양념), ChromaDB 저장(냉장고). 어느 한 단계라도 소홀하면 검색이 엉망이 됩니다.
- **파싱 결과를 먼저 눈으로 확인하세요.** 챕터 3의 **Garbage In, Garbage Out** 원칙이 여기서 적용됩니다. 텍스트가 깨져 있으면 뒤에서 아무리 고쳐도 소용없습니다. `data/markdown/`의 결과를 반드시 검증합니다.
- **벡터 검색은 키워드 매칭이 아닙니다.** "휴가 규정"이라고 물어도 "연차 유급휴가"가 잡힙니다. 의미가 가까우면 찾아냅니다. 다음 챕터 5에서는 이 검색 결과를 **LCEL 체인**에 붙여 실제 답변을 만들어 냅니다.
:::
```

**렌더 CSS**: `components.css:62-115` `.remember` (아이보리 배경 `#fffdf7` + 겹친 그림자로 노트 북마크 느낌, 상단에 `::before` 주황 띠 `#dd6b20`가 포스트잇처럼 붙음, L62-82), `.remember h2` (L83-89, 주황 텍스트 + `::before` 원형 도트), `.remember ul/li` (L98-105, 리스트 마커 제거 후 `→` 커스텀 마커 L106-112), `.remember li strong/code` (L113-114, 주황 강조), `.remember .next` (L115, 점선 상단 보더 + 다음 챕터 안내)

**변형(Variants)**: `.next` 클래스로 "다음 챕터 예고" 별도 영역을 상단 점선으로 분리 가능 (`<p class="next">...</p>`).

**피해야 할 것**
- 불릿 1개로 끝내지 말 것 (챕터 총정리는 2~4개 요약이 적정)
- 새로운 개념을 remember에서 처음 꺼내지 말 것 (본문에서 다룬 것만 회수)
- 챕터 중간에 넣지 말 것. 반드시 챕터 최하단
- `h2` 제목을 마크다운 `## ...`으로 직접 쓰지 말 것. 첫 줄 `**이것만은 기억하자**`가 관례이며 렌더러가 `.remember h2`로 변환
- 다음 챕터 예고는 마지막 불릿에 문장으로 녹이거나 `.next` 영역을 사용. 중간 불릿에 섞지 말 것
