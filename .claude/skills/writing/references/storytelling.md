## 구조: 문제 → 기술 소개 → 해결 → 결과

모든 챕터는 이 흐름을 따른다:

1. **문제 등장**: 주인공(독자)이 구체적 상황에서 문제를 만난다
2. **기술 소개**: 비유로 먼저, 그다음 개념 설명
3. **해결 과정**: 시행착오 포함 (실수 → 에러 → 해결)
4. **결과 확인**: 눈에 보이는 성과

## 톤: 대화체

- "여기서 한 가지 궁금한 점이 생기지 않나요?"
- "혹시 이런 경험 있으신가요?"
- "자, 이제 이 코드를 한번 살펴볼까요?"

### 금지 톤 패턴

| # | 패턴 | 이유 |
|---|------|------|
| 1 | 설교/강제 깨달음 ("이것이야말로 진정한...") | 독자가 스스로 느껴야 함 |
| 2 | "이것이 바로...이유입니다" 반복 | 진부함 |
| 3 | 과도한 "~하는 이유는...때문입니다" | 설명문이 아니라 이야기 |
| 4 | 습관/태도 설교 | 책은 선생님이 아님 |
| 5 | 정의 전 용어 사용 | 비유가 먼저 |
| 6 | 영어 용어 설명 없이 사용 | 처음 등장 시 비유+한글 필수 |
| 7 | 과도한 챕터 간 참조 (섹션당 1개 이하) | 독립적으로 읽을 수 있어야 |

## 비유 규칙

### 원칙
- **독자가 100% 아는 일상적인 것**으로만 비유
- 기술 용어로 비유 금지: "세션은 JWT와 비슷하다" ❌
- 길이 제한 없음. 비유가 길어져도 괜찮다
- 비유로 넘어간 용어는 기술 파트에서 반드시 정식 정의

### 비유 주의사항
- 진부한 비유 금지 (너무 흔한 것은 피할 것)
- 오해를 유발하는 비유 금지 (기술과 비유가 충돌하면 안 됨)
- 페이지당 비유 3개 이하 (과하면 산만)

## 코드 규칙 (스토리텔링 모드)

이야기 파트에서는 **코드 블록이 등장하지 않는다.**
기술 파트에서 코드를 보여줄 때도:

- **개념이 코드보다 먼저** — 코드 블록 최소화
- 핵심 로직만 코드 블록으로 보여줌 (10~15줄 이내)
- 나머지는 실행 명령어 + 요약 + 결과 화면
- 번호 주석(①②③) 대신 자연어로 흐름 설명
- 이야기 속 인물이 기술 섹션 사이에 등장 가능

## 소설 작법

프롤로그와 이야기 파트에 적용하는 소설 기법. 기술 서적이지만 읽히는 글은 소설의 기법을 따른다.

### Show, Don't Tell

감정을 직접 서술하지 말고 행동/장면으로 보여준다.

**BAD**: "그는 매우 화가 났습니다."
**GOOD**: "규정집을 들고 있는 손에 힘이 들어갔습니다."

**BAD**: "뿌듯했습니다. 동료에게 보여줬습니다."
**GOOD**: "동료 자리로 걸어가서 화면을 보여줬습니다."

### 감각 묘사 (2+1)

시각 + 청각/촉각 중 1개를 조합한다. **이야기 파트 섹션당 1개 이상 필수.**

**BAD**: "사무실에서 코딩했습니다."
**GOOD**: "키보드 소리만 또록또록 울리는 사무실에서 모니터를 노려봤습니다."

### Try/Fail 루프

첫 시도에서 성공하면 긴장감이 없다. 성공 전에 최소 2회 실패를 보여준다. **각 챕터 이야기 파트에 최소 1회 실패 장면 필수.**

```
시도 1: 프롬프트에 문서를 통째로 넣었다 → 토큰 한도 초과
시도 2: 잘라서 넣었다 → 잘린 부분에서 다시 환각
시도 3: 질문마다 관련 문서만 골라 넣었다 → 됐다 (= RAG)
```

### 문장 리듬

짧은 문장은 충격/긴장, 긴 문장은 사유/여유. 교차해야 독자가 숨 쉰다.

**BAD (균일한 리듬)**:
```
시스템은 복잡했습니다. 많은 부분이 얽혀 있었습니다. 하나를 손대면 다른 것이 망가졌습니다.
```

**GOOD (변화 있는 리듬)**:
```
손댔습니다.
망가졌습니다.
하나를 고쳤는데 의존하던 세 곳이 연쇄로 깨져나갔습니다.
```

**금지**: 같은 어미(-습니다, -했다 등) 3연속 사용

### 장면 전환

`---` 전후가 완전히 단절되면 독자가 떨어진다. 이전 장면의 마지막 이미지를 다음 장면 첫 줄에 겹치거나 연결한다.

**BAD**:
```
한 달이 걸렸습니다.
---
도서관은 완성됐습니다.
```

**GOOD**:
```
여기까지 오는 데 한 달이 걸렸습니다.
---
도서관은 완성됐습니다. 서가에 문서가 꽂혀 있고 검색하면 관련 문서가 나옵니다. 그런데 사서가 없었습니다.
```

### 구체적 디테일

추상적 서술 대신 시간, 장소, 물리적 행동으로 장면을 만든다.

**BAD**: "며칠간 고민했습니다."
**GOOD**: "이틀 동안 모니터 앞에서 구조를 그렸다 지웠다를 반복했습니다."

**BAD**: "밤늦게까지 작업했습니다."
**GOOD**: "퇴근길 지하철에서 노트북을 꺼냈습니다."

### 한국어 호흡

- 쉼표 리듬: "시간이 흐르고, 또 흐르고, 아무것도 해결되지 않았습니다" — 쉼표로 호흡을 만든다
- 문장 끝맺음 변화: -습니다, -입니다, -했습니다를 의도적으로 섞는다. 같은 끝맺음이 3번 이상 반복되면 지루하다
- 주어 생략: 한국어는 주어를 생략해도 자연스럽다. 매 문장 주어를 달면 번역투가 된다

### 캐릭터 삼각 구도

세 캐릭터가 역할을 분담하여 이야기를 끌고 간다.

| 캐릭터 | 역할 | 대사 패턴 |
|--------|------|----------|
| **팀장** | 힌트 제공자 (소크라테스식 질문) | 짧은 한마디로 비유를 촉발. 답을 주지 않고 방향만 던짐 |
| **동료** | 문제 제기자 / 테스터 | 실사용 피드백으로 새 문제를 열어줌. 날것의 불만 |
| **오픈이** | 실행자 / 독자 대리인 | 시행착오를 직접 겪으며 독자가 동일시할 수 있는 경험 제공 |

**규칙**: 2개 챕터 연속 캐릭터 부재 금지. 최소 1명은 대사로 등장해야 한다.

### 비유는 대화에서 발견

핵심 비유 최소 1개는 캐릭터 대사에서 촉발된다. 서술자가 직접 "~라고 생각해 보겠습니다"로 비유를 꺼내는 것은 차선.

**BAD**: "RAG를 오픈북 시험이라고 생각해 보겠습니다."
**GOOD**: **팀장**: "전부 외우게 하지 말고 필요한 것만 찾아서 읽게 해." → 오픈이가 오픈북 시험을 떠올림

팀장의 한마디가 비유의 단초가 되고, 오픈이가 스스로 비유를 발견하는 구조.

### 기대 전복

낙관 직후 실패를 배치하여 독자의 기대를 의도적으로 무너뜨린다. Try/Fail과 결합하여 "되는 줄 알았는데 안 됨" 순간을 만든다.

**패턴**: 낙관 내면독백 → 즉시 실패

**GOOD**:
```
*오, 이거면 되는 거 아니야?*

서랍에서 규정집을 꺼냈습니다. 첫 줄부터 달랐습니다.
```

### 의인화

AI/시스템을 사람처럼 묘사하여 독자의 감정적 투사를 유도한다. 사서 비유 체계와 자연스럽게 연결.

**GOOD**: "ConnectHR이 멈칫했습니다." / "사서가 다른 사람이 된 것 같았습니다."

### 문제 체인

기술을 나열하지 않고 "이걸 풀었더니 저 문제가 터졌다" 식의 인과 연쇄로 전개한다. 챕터 간 연결과 챕터 내부 모두 적용.

**BAD**: "다음으로 캐싱을 알아보겠습니다."
**GOOD**: "빨라지니까 쓰는 사람이 늘었고 늘어나니까 그동안 보이지 않던 문제가 하나씩 터지기 시작합니다."

### 원형 서사

챕터 초반 문제 제기자(동료)가 끝에서 검증자로 재등장하여 서사를 완결한다.

**패턴**: 동료 불만(오프닝) → 해결 과정 → 동료 검증(클로징)

**GOOD**:
- 오프닝: **동료**: "직원 이름도 모르는 AI 비서예요?"
- 클로징: **동료**: "이제 진짜 비서 같은데요."

## 챕터 패턴

### 도입

```
(주인공)은 (문제)로 고생하고 있었다.
(구체적 상황: 시간, 장소, 하고 있던 일).
이번 챕터에서는 (기술)로 이 문제를 해결한다.
```

예시:
> "밤 11시, 배포 직전에 서버가 뻗었다.
> 로그를 보니 세션이 메모리를 잡아먹고 있었다.
> 이번 챕터에서는 세션이 뭔지, 왜 터졌는지 알아보자."

### 전환
```
(주인공)이 선택한 해결책은 (기술)이었다.
```

### 닫기
```
### 이것만은 기억하자
- (비유로 된 한 줄 기억)
- 다음 챕터에서는 (다음 버전 예고)
```

## 머릿말 패턴 (STEP 7)

머릿말은 챕터와 다르다. 이야기 파트/기술 파트 구분 없이, **독자에게 직접 말하는 단일 톤**으로 쓴다.

### 머릿말의 역할 분담

| 파트 | 역할 | 톤 |
|------|------|-----|
| 프롤로그 | 저자(캐릭터)의 성장 이야기 | 소설 내러티브 |
| **머릿말** | **왜 이 기술들을 배워야 하는가** | **독자에게 말하듯 대화체** |
| 맺음말 | 독자에게 보내는 마지막 메시지 | 대화체 |

### 머릿말 문단 패턴

각 기술을 소개하는 문단은 이 흐름을 따른다.

```
(현실 상황) → (기술 이름 볼드) → (왜 필요한가 한 문장)
```

예시:
> 서버가 한 대일 때는 세션을 메모리에 두면 됩니다.
> 그런데 트래픽이 늘어서 서버를 두 대로 늘리면,
> 한쪽에서 로그인한 사용자가 다른 쪽으로 넘어갈 때 로그인이 풀립니다.
> **Redis**는 이런 상태 공유 문제를 해결하는 가장 흔한 선택지입니다.

### 머릿말에서 하지 않는 것

- 코드 블록
- 비유 (머릿말은 비유 대신 현실 상황으로 설명)
- 저자 경험담 (프롤로그가 담당)
- 기술별 번호 매기기 ("1장은..., 2장은...")
- 감사 인사

### 맺음말 패턴

맺음말은 프롤로그와 대칭이다. 프롤로그가 시작이면 맺음말은 회수.

```
(프롤로그 이미지 회수) → (지금 독자의 위치) → (다음 방향) → (짧은 닫기)
```

- 프롤로그에서 사용한 이미지(사수, 티켓, 터미널 등)를 다시 꺼내되, 성장한 시점에서 바라본다
- 기술 요약을 반복하지 않는다 (머릿말이 담당)
- 설교 없이 "다음은 당신 차례"를 전달한다
- 머릿말보다 짧다 (1~2페이지)

## 버전별 성장 (project-buildup 연계)

각 챕터는 프로젝트의 한 버전에 대응한다.
챕터가 끝나면 독자 눈에 **보이는 결과물**이 있어야 한다.

- v0.1 → "Hello World"가 뜬다
- v0.2 → 글 목록이 보인다
- v0.3 → 데이터가 저장된다

내부 리팩터링만 하는 챕터는 독자가 지친다.
반드시 눈에 보이는 변화를 포함할 것.

## 이야기 파트 체크리스트

챕터 이야기 파트 작성/검토 시 아래를 확인한다.

- [ ] 감각 묘사 1개 이상 (시각 + 촉각/청각)
- [ ] Try/Fail 최소 1회 (첫 시도 성공 금지)
- [ ] 캐릭터 대화 최소 1회 (팀장/동료/오픈이)
- [ ] 내면독백 최소 1회 (`*이탤릭*` 표기)
- [ ] 이야기 파트에 코드 블록 0건
- [ ] 비유 중 최소 1개가 캐릭터 대사에서 촉발
- [ ] 감정 곡선에 기복이 있음 (낙관→좌절 또는 문제→해결)
- [ ] 2개 챕터 연속 캐릭터 부재 금지
- [ ] 이야기 파트 내 분석적 비교표 금지 (기술 파트로 이동)
