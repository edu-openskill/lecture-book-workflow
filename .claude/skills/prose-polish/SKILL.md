# prose-polish: 원고 완성 후 톤 수술기

## 사용 시점

**원고가 완성된 후** 출판 직전 일괄 패스. 집필 중간에는 사용하지 않습니다.

- humanizer = 집필 중간 점검 (보편 AI 흔적)
- **prose-polish = 출판 직전 최종 톤 수술 (책 고유 톤)**

두 스킬은 처리 결이 다릅니다. humanizer는 쉼표·복수형·접속사 같은 보편 한국어 AI 패턴을, prose-polish는 *이 책의 톤 결정*과 *누적된 금지 표현*을 잡습니다.

## 호출

```
/prose-polish [챕터 N]                  # 예: /prose-polish 6
프로젝트의 chapters/NN-*.md 중 가장 최신 버전 자동 감지
```

또는 자연어: "챕터 6 톤 정리", "CH06 prose-polish 돌려"

## 처리 원칙

이 스킬은 메모리·CLAUDE.md의 다음 룰과 정합합니다.

- **분석·판단이 본업**: 사용자에게 매 자리 묻는 스킬이 아닙니다. 카탈로그·화이트리스트·맥락 분석으로 *명확하게 판정 가능한 자리*는 스킬이 직접 결정해 자동 Edit. 적용 결과는 *한 줄 보고*만. 사용자는 *진짜 판단이 필요한 자리*에만 시간을 씁니다.
- **내용 보존 절대 원칙**: 이 스킬은 *문장만 교정*합니다. 정보·사실·예시·코드·캐릭터 행동·서사 단계는 *지우거나 바꾸지 않습니다*. 같은 정보를 평이한 표현으로 풀어 쓰는 *치환*만 허용됩니다.
- **삭제·추가는 무조건 사용자 결정**: 한 단어 이상을 *지우거나 추가*해야 자연스러워지는 경우 자동 적용 금지. 반드시 사용자에게 묻습니다. 수식어 한 개 삭제(예: "변하지 않는 고정 진입점" → "고정 진입점")도 *삭제*이므로 카드로 제시.

## 자동 적용 vs 사용자 결정 분기

| 자동 적용 (Edit + 한 줄 보고) | 사용자 결정 카드 |
|---|---|
| 정보·예시·코드 *완전 보존* | 한 단어 이상 *삭제*가 들어가는 안 |
| 단순 단어 치환 (한두 단어 교체) | 문장 *추가* (의미·서사 보강) |
| 카테고리 룰·화이트리스트에 *명확히* 해당 | 동등하게 합리적인 다중 톤 안 (A/B/C 톤 선택은 저자 영역) |
| 회수 자리에서 정식 용어로 교체 | 화이트리스트 신규 비유 후보 발견 (등록 여부) |
| 의인화 동사 → 평이체 (정보 동일) | *도입 영향권* vs *회수 자리* 판단이 모호한 경계 |
- **grep 단독 판단 금지**: grep은 후보 좁히기에만 보조. 모든 판단은 챕터 전체 Read + 자리별 맥락 분석.
- **일괄 grep/replace 금지**: 자리마다 본문 맥락을 읽고 풀어쓰기. replace_all 사용 금지.
- **자리별 사용자 결정**: 판단이 모호하면 자체 결정 금지, 사용자 결정 우선.
- **챕터 수정 후 자동 빌드**: Edit 직후 같은 턴에 `pub-html-build` 실행. 브라우저는 띄우지 않음.
- **수정 요청 시 디스크 재확인**: Edit 전 항상 Read로 디스크 현재 상태 다시 확인.

## 허용/금지 처리 유형

| 처리 | 자동 적용 | 예시 |
|---|---|---|
| **표현 치환** (정보 동일) | OK (카드 제시 후 결정) | "변하지 않는 문" → "고정 진입점" / "흔들리지 않습니다" → "그대로 남습니다" |
| **수식어 삭제** (한 단어 이상 빠짐) | **사용자 결정 필수** | "변하지 않는 고정 진입점" → "고정 진입점" |
| **문장 삭제** | **사용자 결정 필수** | "한 단계 더 무거운 문제였습니다." 삭제 |
| **문장 추가** (의미·서사 보강) | **사용자 결정 필수** | 인과 사슬 한 칸 누락 보강 |
| **조사·접속어 보강** (10a, 한 단어, 의미 동일) | 묶음 승인 후 자동 | "X합니다. Y합니다." → "X했고, Y합니다." |
| **사실·예시·코드·캐릭터 행동 변경** | **절대 금지** | (스킬 처리 대상 아님) |

## 워크플로우

```
1. 챕터 전체 Read (가장 최신 버전 자동 감지)
2. PASS 1 — 제거형 후보 발견
   ├─ 1. 수사적 비유 (이미 도입된 개념을 비유로)
   ├─ 2. 시적·서정 형용사 (흔들리지 않는·살아 있는·잠든·머무는)
   ├─ 3. 분위기 보강 수사 (또렷이 그려졌습니다·와닿았습니다)
   ├─ 4. 강한 의인화 동사 (휘청거리다·자리를 잡다)
   ├─ 5. AI 흔적·금지 종결 (셈입니다·~봅시다·~고요·본문 ~지요)
   ├─ 6. em-dash 부연구절 (본문 한정)
   ├─ 7. 콜론 뒤 명사구 종결
   ├─ 8. 따옴표 안 명사구 마침표 종결
   └─ 9. 누적 금지 단어 (부품·안쪽·모양새 등)
3. PASS 2 — 연결 분석
   ├─ 10a 미세 연결 (조사·접속어·전환어 보강안 묶음 제시)
   └─ 10b 의미 누락 (인과·서사·비유 마무리 끊김 — 자리별 카드)
4. 통합 카드 묶음 → 사용자 결정
   ├─ AskUserQuestion (자리당 옵션 3~4개)
   └─ 또는 한 표로 묶음 토글 (자리 수가 많을 때)
5. Edit + 자동 빌드
```

## 카테고리별 판정 가이드

상세 룰: `references/banned-categories.md`

핵심 원칙:

- **개념 도입용 비유는 유지**: 책에서 처음 개념을 풀어낼 때 쓰는 핵심 비유는 살림. 화이트리스트는 `projects/<책>/planning/prose-polish-allowlist.md`.
- **수사적 비유는 제거**: 이미 도입돼 정식 용어가 손에 있는 상태에서 다시 비유로 풀어 쓴 것.
- **쿠버네티스 관용 의인화는 유지**: "죽은 Pod·살리는·올린다·뜬다" 같은 k8s 커뮤니티 일상 표현.
- **약한 일상 비유는 유지**: "박아 두면·문제가 터지다·잡아 둔·짚다" 같은 평이 비유.
- **캐릭터 말투는 보존**: 대사·내면독백의 호흡·존댓말 강도는 손대지 않음. 단, 그 안의 *수사적 비유·시적 형용사*는 잡음.

## 책별 화이트리스트

스킬은 자동으로 `projects/<책>/planning/prose-polish-allowlist.md`를 찾아 로드합니다. 없으면 `references/concept-allowlist-template.md`를 복사해 만들도록 안내합니다.

화이트리스트에 등록된 비유는 *처음 도입 시점에서는 무조건 유지*, *반복 회수에서는 위치에 따라 판단*합니다.

## 출력 형식

모든 카드(후보 카드·수정 후 점검 카드)는 **원본 + 수정 결과 + 분석**을 한 자리에 묶어 표시합니다. 한쪽만 보여주면 사용자가 무엇이 바뀌었고 왜 문제인지 판단할 수 없습니다.

### PASS 1 후보 카드

```
[챕터 N · L142] 카테고리 1 — 수사적 비유
원본: "두 매듭이 풀려야 챕터 3의 통합 사이트를 K8s 위로 옮길 수 있겠다는 그림이 머리에 그려졌습니다."

문제: "매듭"은 책 화이트리스트에 없는 분위기 보강 은유.
     "해결할 두 가지"라는 의미를 시각적 은유로 치환.

대안 시안:
  A. "두 가지가 해결되어야 통합 사이트를 K8s 위로 올릴 수 있겠다는 그림이 그려졌습니다."
  B. "이 두 가지를 정리해야 통합 사이트를 K8s 위로 올릴 수 있었습니다."
  C. 유지
```

### 수정 후 점검 카드 (이해·흐름·톤 결합 우려)

```
[챕터 N · L1550] 점검 — 두 용어 동단락 공존
수정 전: "Service는 Pod의 변하지 않는 고정 진입점입니다. ... 변하지 않는 주소를 제공하며"
수정 후: "Service는 Pod의 고정 진입점입니다. ... 고정 주소를 제공하며"

우려: "고정 진입점"과 "고정 주소" 두 용어가 동단락 공존.
     거의 같은 의미인데 미세하게 결이 달라 입문 독자가 한 박자 멈출 수 있음.

다듬기 옵션:
  A. 현행 유지 (5.1.1 도입에서도 둘 다 등장하므로 정합성 있음)
  B. 진입점 통일
  C. 주소 통일
```

## 메모리 룰 정합 점검

이 스킬은 다음 메모리 룰을 *반드시* 준수합니다 (어기면 작업 중단).

- "분석" 트리거 시 챕터 전체 Read 의무
- 분석 시 grep 사용 금지 (보조만)
- 일괄 grep/replace 수정 금지
- 사용자 선택 라인은 정확히 그 범위만
- 수정 시 앞뒤 단락·챕터 호 점검
- 응답 끝 메타 요약 금지
- 챕터 수정 후 자동 빌드
- "빌드만 요청은 브라우저 안 띄우기"
