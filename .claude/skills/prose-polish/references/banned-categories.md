# prose-polish 금지 카테고리

10개 카테고리. PASS 1은 1~9번(제거형), PASS 2는 10번(연결 분석).

## 절대 원칙 — 내용 보존

이 스킬은 *문장 표현*만 교정합니다. **정보·사실·예시·코드·캐릭터 행동·서사 단계는 절대 지우거나 바꾸지 않습니다.**

처리 단위는 다음 셋입니다.

1. **표현 치환** (정보 동일) — 정식 용어·평이체로 단어를 바꾸기. 한 단어 이상 *삭제·추가가 없을 때만* 자동 카드 시안으로 제시.
2. **수식어·문장 삭제** — 형용사·부사·문장 한 개 이상이 빠지는 경우. **항상 사용자 결정 카드로 제시**. 자동 적용 금지.
3. **문장 추가** (카테고리 10b 의미 누락) — **항상 사용자 결정 카드로 제시**. 자동 적용 금지.

조사·접속어 한 단어 보강(카테고리 10a)만 *묶음 승인 후 일괄 자동* 처리할 수 있습니다.

---

각 카테고리는 다음 구조로 정리합니다.

- **정의**: 카테고리가 무엇을 잡는지
- **판정 가이드**: 후보를 발견하면 무엇을 확인해 잡을지 결정하는지
- **허용 예외**: 잡지 않는 경우
- **교체 가이드**: 어떻게 풀어쓰는지 (치환 / 삭제 / 추가 표시)
- **예시**: BAD → GOOD

## 1. 수사적 비유

**정의**: 이미 책에 도입된 개념·정식 용어가 손에 있는 상태에서, 그 개념을 다시 비유로 풀어 쓴 표현.

**판정 가이드**:
- 이 비유가 가리키는 개념이 *해당 챕터 이전 또는 이번 챕터 앞 절*에서 이미 도입됐는가? → **잡는다**
- 이 비유가 *이번에 처음 도입되는 개념*을 풀어내는 것인가? → **유지**
- 화이트리스트(`projects/<책>/planning/prose-polish-allowlist.md`)에 등록된 비유인가? → 위치별 판단

**허용 예외**:
- 화이트리스트의 핵심 비유 (상자·공장·관제센터·가맹점·메뉴판/비밀레시피 등)
- 사용자 실제 경험 회수 ("도커 때 Nginx 앞에 세웠던 것처럼")

**교체 가이드**:
- 비유 단어 → 정식 용어
- "변하지 않는 문" → "고정 주소" / "고정 진입점"
- "두 매듭" → "두 가지" / "두 문제"

**예시**:
- BAD: *"도커 때 Nginx 한 대를 앞에 세웠던 것처럼, 변하지 않는 문 하나가 있어야 하는데."*
- GOOD: *"도커 때 Nginx 한 대를 앞에 세웠던 것처럼, 고정된 진입점이 있어야겠는데."*

## 2. 시적·서정 형용사·동사구

**정의**: 기술 대상에 *생명체* 또는 *서정적 속성*을 부여하는 형용사·관형어.

**판정 가이드**:
- 이 형용사가 기술적 사실을 *직접 서술*하는가, *생명체 표현*인가?
- 평이체 대안이 있는가?

**허용 예외**:
- 쿠버네티스 관용 의인화: "죽은 Pod·살아 있는 컨테이너·잠든 컨테이너" 중 k8s 커뮤니티 일상 표현 (책 화이트리스트로 관리)
- 캐릭터 대사·내면독백 내 *기분 표현* (예: "마음에 걸렸다")

**교체 가이드**:
| BAD | GOOD |
|---|---|
| 흔들리지 않는 진입점 | 고정 진입점 |
| 변하지 않는 문/주소 | 고정 주소 |
| 살아 있는 Pod | 떠 있는 Pod / 실행 중인 Pod |
| 잠든 컨테이너 | 정지된 컨테이너 |
| 머무는 데이터 | 남는 데이터 / 보존되는 데이터 |
| 자리를 잡았다 | 정착했다 (단어 자체를 빼는 게 안전) |

## 3. 분위기 보강 수사

**정의**: 정보를 더하지 않으면서 *감정·체감*을 강조하기 위해 넣은 수식어구.

**판정 가이드**:
- 이 표현이 *사실*을 더하는가, *기분*을 더하는가?
- 빼도 의미가 그대로 통하는가?

**허용 예외**:
- Show Don't Tell 묘사 ("손이 키보드 위에서 멈췄습니다")
- 감각 묘사 (시각·청각·촉각)

**교체 가이드** (삭제 케이스는 *항상 사용자 결정*):

| BAD | 처리 유형 | GOOD |
|---|---|---|
| 또렷이 그려졌습니다 | 치환 | 분명해졌습니다 / 보였습니다 |
| 와닿았습니다 | 삭제 → 사용자 결정 | (단락 흐름 검토 후 빼거나 직접 서술로 교체) |
| 한 단계 더 무거운 문제 | 삭제 → 사용자 결정 | "더 큰 문제" 치환 / 통째 빼기 둘 다 카드 |
| 머릿속에 잠시 머물렀습니다 | 치환 | 떠올랐습니다 |
| 한꺼번에 도드라졌습니다 | 치환 | 함께 보였습니다 |

## 4. 강한 의인화 동사

**정의**: 시스템·코드·도구가 *사람처럼* 움직이는 것으로 묘사하는 동사구.

**판정 가이드**:
- 주어가 *기술 대상*인가? + 동사가 *인간 행동*인가?
- k8s 관용 표현(올린다·뜬다·죽는다·살린다)인가?

**허용 예외**:
- k8s 관용 의인화 (위와 동일)
- 캐릭터의 *물리적* 동작 (오픈이가 펜을 들었습니다)

**교체 가이드**:
| BAD | GOOD |
|---|---|
| 사이트가 휘청거립니다 | 운영에 문제가 생깁니다 |
| 줄을 맞춰 도는 사이트 | 정상적으로 동작하는 사이트 |
| 시스템이 멈칫했다 | 시스템 응답이 늦어졌다 |
| 자리를 잡았다 | (위치 명시 또는 생략) |

## 5. AI 흔적·금지 종결

**정의**: 메모리 룰에 누적된 부자연스러운 종결어미·구문.

**판정 가이드**: 본문(서술자 시점). 캐릭터 대사·내면독백은 별도 룰.

**금지 종결 목록**:
- `~셈입니다` / `~셈이었습니다` / `~는 셈` *(절대 금지)*
- `~봅시다` / `~봅니다` → `~보겠습니다`
- `~고요` / `~구요` (연결어미 + 요) → `~네요·~이에요`로 끊거나 두 문장
- 본문 `~지요` / `~예요` / `~네요` → 합니다체·입니다체로 통일
- 본문 `~었습니다·~았습니다` 분위기 보강용 과거형 회수 (수정안에 한정 — 메모리 룰)

**예시**:
- BAD: "정리해 둔 셈입니다."
- GOOD: "정리해 두었습니다." / "정리됐습니다."

## 6. em-dash 부연구절 (본문 한정)

**정의**: 본문에 `X — 부연 — Y` 또는 `X — 부연.` 패턴 삽입.

**판정 가이드**:
- 위치가 *본문*인가? (제목·캡션은 허용)
- 부연구절을 *두 문장으로 쪼개기*가 가능한가?

**허용 예외**:
- 섹션 제목 (H2/H3): 단일 하이픈 `-` 하나 허용 (`## 10.2 확대경 달기 - OCR`)
- 캡션
- 코드블록 내 주석

**교체 가이드**: 두 문장으로 분리하거나 괄호 `()`로 묶기.

## 7. 콜론 뒤 명사구 종결

**정의**: `**X** : 명사구.` 또는 리스트·박스 항목에서 명사구로 끊는 패턴.

**판정 가이드**: 동사로 매듭 짓는가?

**교체 가이드**:
- BAD: "**ConfigMap** : 설정값을 저장하는 자리."
- GOOD: "**ConfigMap**은 설정값을 저장하는 자리입니다."

## 8. 따옴표 안 명사구 마침표 종결

**정의**: 내면독백·대사 안에서 `*'X1과 X2. 둘 다 ~'*`처럼 명사구로 끊고 다시 시작하는 패턴.

**판정 가이드**: 따옴표 안의 한 호흡을 *명사구.* 로 끊는가?

**교체 가이드**: 한 문장으로 묶거나 두 문장으로 자연스럽게 분리.

- BAD: *'두 매듭. 둘 다 풀어야 한다.'*
- GOOD: *'두 가지를 다 해결해야 한다.'*

## 9. 누적 금지 단어

**정의**: 메모리 피드백에 누적된, *현실 대화에서 안 쓰는 문어체 어휘* + *추상 위치·역할 명사*.

**현재 목록** (계속 누적):

문어체 어휘:
- 부품
- 안쪽 (추상적·은유적 의미일 때)
- 모양새

추상 위치·역할 명사 (의미 모호 + 문장 자체 이해도 낮춤):
- **자리** ("~하는 자리입니다", "그 자리를 맡습니다")
- **역할** ("~하는 역할입니다") — 단, 정식 정의에서 *"~의 역할이다"*는 OK
- **구조** ("~하는 구조라", "한 단계 위에 ~ 두는 구조")
- **층** ("한 층 위에", "한 단계 위에")
- **곳/지점** ("~한 곳입니다", "~한 지점에서")
- **것** (명사구 종결 "~하는 것입니다" 빈번 사용)

**판정 가이드**:
- 일반인이 일상에서 쓰는 단어인가? + 같은 의미의 더 자연스러운 동사 종결로 풀 수 있는가?
- 추상 명사가 *주어 부재* 상황에서 등장하면 카테고리 11(문장 자체 이해 어려움)도 함께 검토

**교체 가이드**:
| BAD | GOOD |
|---|---|
| Ingress는 ~ 묶어 주는 자리입니다 | Ingress는 ~ 묶어 줍니다 |
| 그 일을 맡는 역할입니다 | ~을 합니다 / ~을 담당합니다 |
| 한 단계 위에 두는 구조라 | (문장 통째 재구성 — 카테고리 11) |
| ~를 처리하는 곳입니다 | ~를 처리합니다 |

## 11. 문장 자체 이해 어려움 (재구성)

**정의**: 단편 단어 치환으로 풀리지 않는 자리. 문장을 처음부터 끝까지 읽었을 때 입문 독자가 *무슨 말이지* 멈추는 자리. *문장 통째 재구성*이 필요.

**판정 가이드**:
- *주어 부재*: 동사·서술어만 떠 있고 행위 주체가 맥락에서 추론해야 함
- *추상 위치 표현*: "위/아래/안/바깥/한 단계/한 층/구조라"가 *무엇과 무엇 사이*인지 명시 없이 등장
- *인과 직관성 부족*: "~기 때문에 ~ 할 수 있습니다"의 양쪽이 입문 독자에게 자연스럽게 이어지지 않음
- *정보 압축 과다*: 한 문장에 3~4겹 정보

**처리 방식**:
- **재구성 필수** — 단어 치환 금지. 문장 통째 다시 쓰기.
- **자동 적용 금지** — 항상 사용자 결정 카드 (정보 순서·분할 단위에 저자 톤 영역)
- PASS 1 단편 치환 결과가 이 카테고리에 걸리면 *치환 되돌리고 재구성*하는 옵션도 함께 제시

**예시** — CH05 L511

수정 전(v8): "한 단계 위에 또 한 명의 문지기를 두는 구조라" — *문지기*가 누구인지·*위에 둔다*가 어디인지 모호
수정 후(v9 단편 치환): "Service 위에 라우팅 한 단계를 더 두는 구조라" — *위에*·*한 단계*가 여전히 추상. 의인화만 뺐을 뿐 의미 모호 동일

→ 재구성 안: "**그렇게 만들어진 Service 여러 개를 외부 도메인 하나로 묶어 주는 자리가 Ingress입니다. 외부 요청은 도메인 하나로 들어와 URL 경로에 따라 알맞은 Service로 갈라집니다.**" — *위/아래* 표현 빼고 *Ingress의 기능*을 직설로 풀어냄

**정의**: 한 단락 끝과 다음 단락 시작 사이에 *내용 한 칸*이 빠진 자리.

**판정 가이드**:
- A 문제 발견 → C로 해결 사이에 *B 시도하다 실패*가 빠졌는가?
- 비유 마무리와 다음 단락 시작이 *맞물리는가* (`storytelling.md` 장면 전환 룰)?
- 캐릭터 행동·반응이 빠졌는가?

**처리 방식**: 자리별 카드로 제시.

```
[L97 → L99 사이] 내용 누락 가능성
앞 단락 끝: "Service의 selector 한 줄로 식구를 가립니다."
뒤 단락 시작: "라벨이 일치하는 Pod 두 대만 Service 뒤로 묶이고..."

분석: selector 동작 설명(라벨 매칭)이 뒤 단락에서 갑자기 결과로 점프.
     사이에 "어떻게 라벨로 매칭하는지" 한 문장이 빠져 있을 수 있음.

대안:
  A. 한 문장 추가 (selector가 라벨로 Pod를 찾는 메커니즘 설명)
  B. 패스 (앞 단락에 이미 충분히 설명됨)
  C. 사용자 직접 작성
```
