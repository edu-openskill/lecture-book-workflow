# Educational Simulator Builder (교육용 인터랙티브 시뮬레이터)

어려운 개념을 **"직접 만져보며 배우는"** 단일 HTML 시뮬레이터로 만든다. 이 분야의 이름은
*explorable explanations*(Bret Victor, Nicky Case)이고, 핵심은 "글로 설명"이 아니라
"플레이로 발견"하게 만드는 것이다.

결과물은 **항상 의존성 없는 단일 `.html` 파일** (외부는 폰트 CDN만 허용). 다크 네이비 테크 테마
고정. 한국어, 초보자 대상, 내용은 전문적.

## 전체 흐름 (이 순서를 지킬 것)

1. **요청 해석 + 부족하면 1~2개만 질문.** `references/requesting.md`의 체크리스트로 빠진 정보를
   채운다. 특히 *한 줄 질문*, *비유*, *상호작용 방식*이 비면 짧게 묻는다. 참고 스케치/스크린샷이
   있으면 그대로 따른다(이미지는 말 100줄보다 정확하다).
2. **교육 설계를 먼저 한다 (코드 X).** `references/pedagogy.md`를 읽고, 어떤 질문으로 시작해
   어떤 단계로 올라가 어떤 반전으로 끝낼지 *시나리오*를 한 문단으로 잡는다. 이야기 → 문제 →
   해결(여러 개면 하나씩)의 형태여야 한다. **하나의 구체적 예시(스레드)를 끝까지 끌고 간다**
   (`pedagogy.md`의 "예시 연속성").
2-1. **내용(개념)이 사실인지 먼저 검증한다 (코드 X).** 시나리오에서 *단정하는 문장*을 모두 뽑아
   한 줄씩 "이게 사실인가?"를 따진다. 흔한 오개념(folk model)을 그대로 그리지 않는다. 확신이
   안 서면 권위 있는 출처(1차 문서·논문)로 확인한다. `references/verification.md` E절을 따른다.
   — **개념이 틀리면 그리지 않는다.**
2-2. **표준 도식이 있는 주제면 웹에서 1~2개 조사한다 (해당될 때만).** 어텐션·트랜스포머·역전파·
   RAG처럼 *널리 쓰이는 표준 그림*이 있는 주제는, 코드 전에 그 도식을 찾아 **구조(입력→연산→
   출력, 흐름 방향)** 를 맞춘 뒤 비유로 옷을 입힌다. 그림을 베끼는 게 아니라 **불변 구조만 추출**
   한다. 표준과 어긋나면 학습자가 다른 자료와 충돌한다. (단, 표준 도식이 없는 일상 개념엔 불필요 —
   조사는 보조일 뿐, 정확성의 핵심 레버는 3·아래 '메커니즘 검증'이다.)
2-3. **다룰 범위·층위를 먼저 못 박는다.** 한 시뮬에 섞으면 안 되는 경계를 빌드 전에 정한다. (예:
   *추론(forward)* vs *학습(역전파)* — 둘을 한 화면에 섞으면 오개념. 무엇을 넣고 무엇을 뺄지 정하고,
   필요하면 사용자에게 한 줄로 확인한다.)
2-4. **과정 전체를 열거한다.** 파이프라인/알고리즘이면 그리기 전 체크: ① 입구 변환에 **역변환**이
   있는가(예: 임베딩↔언임베딩)? ② 전체가 **반복(루프)** 되는가(예: 자기회귀)? ③ **한계/엣지**에
   닿으면? — 빠뜨리기 쉬운 '되돌아오는·반복되는' 단계를 미리 잡는다.
3. **수학/좌표를 먼저 검증한다.** 시각화(특히 SVG 벡터·정사영·그래프류)는 **숫자가 틀리면
   무의미**하다. 화면을 그리기 전에 `references/verification.md`의 방식으로 좌표를 계산하고
   viewBox 안에 들어오는지, 의도한 승자/값이 나오는지 node로 확인한다.
4. **빌드한다.** `references/design-system.md`(테마 토큰)와 `references/build-recipe.md`(구조·
   애니메이션 패턴)를 따른다. `assets/template.html`을 출발점으로 쓸 수 있다.
5. **다시 검증한다.** node `--check` 문법 검사 + 좌표/수학 재확인 + (가능하면) jsdom으로 버튼
   클릭 런타임 오류 0건 확인 + **내용(개념) 재확인**(화면의 단정 문장이 여전히 사실인가, 비유의
   성립 조건을 어기지 않았는가 — verification.md E절). `references/verification.md` 참고.
   기술 검증을 통과해도 개념이 틀리면 미완성이다. 통과 전엔 전달하지 않는다.
6. **전달 + 1줄 반전 유도.** 사용자가 브라우저에서 새로고침해 확인하게 하고, 더 다듬을 곳을 묻는다.

## 절대 원칙 (실제 작업들에서 비싸게 배운 것)

- **고정 좌표계.** 사용자가 버튼을 누를 때 **장면 전체를 다시 그리거나 회전시키지 마라.** 기준이
  되는 요소(데이터 벡터, 축, 노드)는 그 자리에 **고정**하고, 상호작용은 *새 요소 하나만 추가*하는
  느낌이어야 한다. "검색하면 선 하나만 그어진다"가 직관이다.
- **한 화면에 한 개념.** 메커니즘을 분리해 하나씩 가르치고 나중에 합친다. 한 번에 다 보여주면 망친다.
- **깔끔함 > 화려함.** 라벨·숫자·보조선이 많아지면 핵심이 묻힌다. 강조하고 싶은 *하나*(예: 정사영
  길이)만 또렷하게. 중복 정보(같은 값을 그림+패널에 두 번)는 한 곳으로 몰아라.
- **구체적이고 친숙한 예시 > 추상적 예시.** "프로그래밍 언어 질문 → 파이썬과 가깝다"가 "Rust
  소유권 모델"보다 100배 잘 와닿는다. 학습자가 *아는* 소재로 비유하라.
- **두께·색·폰트 일관성.** 새로 그리는 요소는 기존 요소의 두께/스타일과 맞춘다. 한 요소만 과하게
  굵으면 뒤를 가린다.
- **검증 없이는 미완성.** 좌표가 화면 밖으로 나가거나, 정사영 값이 그림과 안 맞거나, 버튼이 콘솔
  에러를 내면 "잘 만든 것"이 아니다.
- **개념이 사실이어야 한다(내용 검증).** 코드가 돌아도 내용이 틀리면 오개념을 심는다. "그럴듯한
  직관"을 그리기 전에 *주체(모델 vs 앱)와 조건*을 확인하라. 비유가 핵심 메커니즘과 어긋나면, 잘
  만든 게 아니라 잘못 가르친 것이다.
- **하나의 스레드를 끝까지.** 단계마다 새 예시로 갈아타지 마라. 반전(그런데)에서 던진 문제는
  해결(그래서)에서 *같은 소재*로 푼다. 앞 단계의 객체를 뒤 단계가 재사용하는 콜백이 곧 학습이다.
- **정확성 > 비유 일관성.** 시리즈 세계관(예: RPG)이 있어도 그것에 *갇히지* 마라. 비유가 핵심
  사실을 왜곡·은폐하는 순간(예: "성/서버" 비유가 "프롬프트는 텍스트 그대로 전송"을 가림) **그
  지점만 비유를 갈아끼우거나 아예 직설적 그림으로** 간다. 개념마다 최적의 비유를, 때로는 비유
  없이. 더 좋은 비유가 보이면 기존 틀을 버리고 그쪽을 택한다.
- **메커니즘 먼저 — 시뮬이 '실제 연산'을 수행해야 한다.** 가장 비싼 실패는 *느낌만* 애니메이션하는
  것이다(예: "층을 지나니 화살표가 그냥 커진다" → 실제 동작 아님). 코드 전에 실제 연산을 한 줄로
  적고(예: "이웃과 **내적** → 결과 Δ를 **더해(residual)** 누적 → 다음 층"), 시각화가 그 연산을
  *문자 그대로* 수행하게 만든다. 예쁜 움직임은 메커니즘이 맞은 *다음* 일이다. (이 연산이 화면의
  어떤 요소로 보이는지 캔버스에 라벨로 드러낸다.)
- **다단계 과정은 step으로 분해.** RAG 4단계·어텐션의 내적→소프트맥스→가중합처럼, 한 번에 다
  펼치지 말고 step 버튼/하이라이트로 한 단계씩 진행하게 한다.

## 참고 파일 (필요할 때 읽기)

- `references/requesting.md` — 사용자가 요청을 어떻게 하면 좋은지 + 빠진 정보 묻는 법
- `references/pedagogy.md` — explorable explanation 5패턴, "그래서/그런데" 구조, 비유 설계
- `references/design-system.md` — 다크 네이비 테마 CSS 토큰(색·폰트·카드·버튼·뱃지·토큰) 전체
- `references/build-recipe.md` — 파일 구조, 탭 전환, 버튼→데이터흐름 애니메이션, SVG 벡터/정사영 패턴
- `references/verification.md` — node 문법검사 + 좌표 viewBox 검증 + 수학 검증 + jsdom 런타임 검증
- `assets/template.html` — 다크 테마 탭 시뮬레이터 최소 스캐폴드 (복사해서 시작)

## 결과물 규칙

- 단일 `.html`. CSS/JS 인라인. 외부 의존은 폰트(pretendard, JetBrains Mono) CDN만.
- 파일명은 내용을 담아 한국어로 OK (예: `시뮬레이터_어텐션.html`).
- 작업 폴더에 저장 후 `present_files`로 전달.
- 큰 파일(>50KB)을 다룰 때 보조 작업창(샌드박스)이 읽기 오류를 낼 수 있으니, 진위 판단은 파일
  도구(Read/Grep)를 기준으로 한다. (`references/verification.md`의 주의사항 참고)
