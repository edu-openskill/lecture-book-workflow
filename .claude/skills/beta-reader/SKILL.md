# Beta Reader 스킬

출간 전 시험 독자 패널. 다양한 배경의 가상 독자 10명이 챕터를 읽고 실습하며 피드백을 제공한다.

## 사용법

```
/beta-reader CH04          # 단일 챕터
/beta-reader CH01-CH03     # 범위 (각각 순차 실행)
```

## 페르소나 (10명)

| # | 이름 | 배경 | 코드 수준 | 관점 |
|---|------|------|----------|------|
| 1 | 신입 백엔드 | Java Spring 1년차 | Python 처음 | pip/venv 혼란, 언어 차이 |
| 2 | 프론트엔드 | React 3년차 | 서버/DB 약함 | API 호출 경험만, 인프라 개념 부족 |
| 3 | 데이터 분석가 | pandas/SQL 능숙 | ML 기초 있음 | 임베딩 개념은 아는데 RAG는 처음 |
| 4 | PM/기획자 | 비개발자 | 코드 읽기만 | 터미널 서툼, 큰 그림 위주로 이해 |
| 5 | CS 학생 | 전공 3학년 | 이론 있음 | 실무 경험 없음, 교과서적 사고 |
| 6 | SI 개발자 | Java/Oracle 10년 | Python 기초만 | Docker 안 써봄, 레거시 사고방식 |
| 7 | ML 엔지니어 | PyTorch 능숙 | RAG 처음 | 임베딩/벡터 익숙, 파이프라인 관점 |
| 8 | DevOps | 인프라 전문 | 코드보다 환경 | Docker/포트/의존성에 민감 |
| 9 | 비전공 전직자 | 부트캠프 수료 | 기초 겨우 | 에러 나면 당황, 용어 장벽 큼 |
| 10 | CTO/리드 | 풀스택 15년 | 빠르게 훑음 | 아키텍처/확장성/실무 적용 관점 |

## 실행 흐름

### 1단계: 컨텍스트 로드

```
읽어야 할 파일:
- planning/seed-v1.md (의도)
- planning/outline-v1.md (해당 챕터 섹션)
- chapters/{해당 챕터}.md (전문)
- code/{해당 ex}/ (실습 코드)
```

### 2단계: 10명 병렬 디스패치

Agent 도구로 10명을 병렬 실행한다. 각 에이전트에게 다음을 전달한다.

```
프롬프트 구조:
- 페르소나 설명 (배경, 코드 수준, 관점)
- 챕터 전문
- 실습 코드 경로
- 평가 체크리스트 (아래)
```

### 3단계: 평가 체크리스트

각 페르소나는 아래 항목을 평가한다.

#### 이야기 파트

| 항목 | 질문 |
|------|------|
| 몰입 | 이야기에 빠져드는가? 중간에 지루한 구간은? |
| 비유 | 내 배경에서 비유가 와닿는가? 억지스러운 비유는? |
| 톤 | AI스러운 문장("이것이야말로", "기억하십니까?")은 없는가? |
| 용어 도입 | 새 용어가 갑자기 나오지 않는가? 비유 → 정의 순서가 자연스러운가? |
| 등장인물 | 오픈이/팀장의 대화가 자연스러운가? |

#### 기술 파트

| 항목 | 질문 |
|------|------|
| 용어 연결 | 이야기 파트의 비유와 기술 파트의 정의가 연결되는가? |
| 코드 설명 | 코드가 내 수준에서 이해 가능한가? 설명이 부족한 부분은? |
| 데이터 시각화 | 추상적 설명에 실제 데이터(JSON/출력)가 함께 보이는가? |
| 파일 트리 | [실습]/[설명]/[참고] 분류가 명확한가? 어디서부터 시작할지 알겠는가? |

#### 실습

| 항목 | 질문 |
|------|------|
| 환경 구축 | pip install, venv, Docker 등 막히는 지점은? |
| 실행 결과 | 챕터에 나온 결과와 실제 결과가 일치하는가? |
| TODO | TODO를 채우는 데 힌트가 충분한가? |
| 에러 대응 | 에러가 나면 챕터에 해결법이 있는가? |

#### 전체

| 항목 | 질문 |
|------|------|
| 학습 곡선 | 이전 챕터 대비 난이도 점프가 적절한가? |
| 분량 체감 | 길다/적당하다/짧다? |
| 다음 챕터 | 예고가 흥미를 유발하는가? |

### 4단계: 종합 리포트 생성

```
review/beta-reader-CH{N}.md
```

#### 리포트 구조

```markdown
# Beta Reader 리포트: CH{N}

## 요약
- 통과: N/10명
- 주요 이슈: N건

## 공통 피드백 (3명 이상 동일 의견)
| 영역 | 피드백 | 언급 페르소나 |
|------|--------|-------------|
| ... | ... | 1, 4, 9 |

## 심각도별 이슈
### 높음 (학생이 막힘)
- ...
### 중간 (혼란스럽지만 진행 가능)
- ...
### 낮음 (개선하면 좋음)
- ...

## 페르소나별 상세

### 1. 신입 백엔드 (Java Spring 1년차)
**[이야기]** ...
**[기술]** ...
**[실습]** ...
**[점수]** 이야기 4/5 | 기술 3/5 | 실습 3/5

### 2. 프론트엔드 (React 3년차)
...

## 수정 제안 (우선순위순)
| # | 위치 | 제안 | 심각도 |
|---|------|------|--------|
| 1 | ... | ... | 높음 |
```

## 제약

- 실습 코드는 실제로 실행하지 않고 **코드 리딩으로 평가** (실행은 별도 검증에서)
- 페르소나의 "막힘"은 해당 배경에서 합리적으로 추론
- seed.md의 독자 상수("배경지식 있음, 이 주제만 모름")를 기본 전제로 함
- style.md 12번 규칙("데이터는 보여준다")을 평가 기준에 포함
