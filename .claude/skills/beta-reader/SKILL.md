# Beta Reader 스킬

출간 전 시험 독자 패널. 다양한 배경의 가상 독자 10명이 챕터를 읽고 실습하며 피드백을 제공한다.
어떤 책이든 seed.md에서 독자 정보를 읽고, 해당 주제에 맞는 페르소나를 자동 생성한다.

## 사용법

```
/beta-reader CH04          # 단일 챕터
/beta-reader CH01-CH03     # 범위 (각각 순차 실행)
```

## 실행 흐름

### 0단계: 프로젝트 감지

현재 작업 중인 프로젝트를 자동 감지한다.

```
1. projects/ 아래 가장 최근 수정된 폴더 → 프로젝트 경로
2. {프로젝트}/planning/seed-v*.md 읽기 → 책 주제, 독자 상수, 핵심 메시지
3. {프로젝트}/planning/outline-v*.md 읽기 → 해당 챕터 섹션
4. {프로젝트}/chapters/ → 챕터 .md 파일
5. {프로젝트}/code/ → 실습 코드 (없으면 코드 없는 개념 챕터로 판단)
```

### 1단계: 페르소나 자동 생성

seed.md에서 **독자 상수**와 **책 주제**를 읽고 10명의 페르소나를 생성한다.

#### 생성 규칙

```
입력:
- 책 주제 (예: "RAG 파이프라인", "React 컴포넌트 설계", "쿠버네티스 운영")
- 독자 상수 (예: "배경지식 있음, 이 주제만 모름")
- 기술 스택 (예: Python, LangChain, FastAPI)

생성할 10명:
- 2명: 주제와 가까운 배경 (관련 기술 경험 있음, 이 주제만 처음)
- 2명: 주제와 먼 배경 (다른 분야 전문가, 기술 스택 자체가 낯설음)
- 2명: 비개발 배경 (PM, 기획자, 비전공 전직자)
- 2명: 숙련자 (빠르게 훑는 CTO/리드, 아키텍처 관점의 시니어)
- 1명: 초보 (부트캠프 수료, 에러에 당황)
- 1명: 학생 (이론 있음, 실무 없음)
```

#### 페르소나 출력 형식

각 페르소나는 다음 필드를 가진다.

```
이름: (역할 한 줄)
배경: (경력/전문 분야)
코드 수준: (이 책의 기술 스택 기준)
관점: (이 사람이 특히 주목하거나 막힐 포인트)
```

#### 예시: RAG 파이프라인 책

| # | 이름 | 배경 | 코드 수준 | 관점 |
|---|------|------|----------|------|
| 1 | 신입 백엔드 | Java Spring 1년차 | Python 처음 | pip/venv 혼란, 언어 차이 |
| 2 | 데이터 분석가 | pandas/SQL 능숙 | ML 기초 있음 | 임베딩은 아는데 RAG는 처음 |
| 3 | 프론트엔드 | React 3년차 | 서버/DB 약함 | API는 호출만, 인프라 부족 |
| 4 | SI 개발자 | Java/Oracle 10년 | Python 기초만 | Docker 안 써봄, 레거시 사고 |
| 5 | PM/기획자 | 비개발자 | 코드 읽기만 | 터미널 서툼, 큰 그림 위주 |
| 6 | 비전공 전직자 | 부트캠프 수료 | 기초 겨우 | 에러 나면 당황, 용어 장벽 큼 |
| 7 | ML 엔지니어 | PyTorch 능숙 | RAG 처음 | 벡터/임베딩 익숙, 파이프라인 관점 |
| 8 | DevOps | 인프라 전문 | 코드보다 환경 | Docker/포트/의존성에 민감 |
| 9 | CS 학생 | 전공 3학년 | 이론 있음 | 실무 경험 없음, 교과서적 사고 |
| 10 | CTO/리드 | 풀스택 15년 | 빠르게 훑음 | 아키텍처/확장성/실무 적용 관점 |

#### 예시: React 컴포넌트 설계 책

| # | 이름 | 배경 | 코드 수준 | 관점 |
|---|------|------|----------|------|
| 1 | jQuery 개발자 | 프론트 5년 | React 처음 | 컴포넌트 사고 전환 |
| 2 | Vue 개발자 | Vue 3년차 | React 문법 낯설음 | 비교 관점 |
| 3 | 백엔드 | Spring 3년차 | JS 기초만 | 프론트 전체가 낯설음 |
| ... | ... | ... | ... | ... |

### 2단계: 컨텍스트 로드

```
읽어야 할 파일:
- {프로젝트}/planning/seed-v*.md (의도, 독자 상수)
- {프로젝트}/planning/outline-v*.md (해당 챕터 섹션)
- {프로젝트}/chapters/{해당 챕터}.md (전문)
- {프로젝트}/code/{해당 ex}/ (실습 코드, 있으면)
- {프로젝트}/review/beta-reader-*.md (이전 리포트, 있으면 — 개선 확인용)
```

### 3단계: 10명 병렬 디스패치

Agent 도구로 10명을 **병렬** 실행한다. 각 에이전트에게 다음을 전달한다.

```
프롬프트 구조:
- 페르소나 설명 (이름, 배경, 코드 수준, 관점)
- 책의 독자 상수 (seed.md에서)
- 챕터 전문
- 실습 코드 (있으면)
- 평가 체크리스트 (아래)
- "이 페르소나의 관점에서 솔직하게 평가하라. 좋은 점도, 나쁜 점도 숨기지 마라."
```

### 4단계: 평가 체크리스트

각 페르소나는 아래 항목을 평가한다.

#### 이야기 파트

| 항목 | 질문 |
|------|------|
| 몰입 | 이야기에 빠져드는가? 중간에 지루한 구간은? |
| 비유 | 내 배경에서 비유가 와닿는가? 억지스러운 비유는? |
| 톤 | AI스러운 문장("이것이야말로", "기억하십니까?")은 없는가? |
| 용어 도입 | 새 용어가 갑자기 나오지 않는가? 비유 → 정의 순서가 자연스러운가? |
| 등장인물 | 등장인물의 대화가 자연스러운가? (등장인물이 있는 책만) |

#### 기술 파트

| 항목 | 질문 |
|------|------|
| 용어 연결 | 이야기 파트의 비유와 기술 파트의 정의가 연결되는가? |
| 코드 설명 | 코드가 내 수준에서 이해 가능한가? 설명이 부족한 부분은? |
| 데이터 시각화 | 추상적 설명에 실제 데이터(JSON/출력/다이어그램)가 함께 보이는가? (style.md 12번 규칙) |
| 파일 트리 | [실습]/[설명]/[참고] 분류가 명확한가? 어디서부터 시작할지 알겠는가? |

#### 실습 (코드가 있는 챕터만)

| 항목 | 질문 |
|------|------|
| 환경 구축 | pip install, venv, Docker 등 막히는 지점은? |
| 실행 결과 | 챕터에 나온 결과와 실제 결과가 일치하는가? |
| TODO | TODO를 채우는 데 힌트가 충분한가? |
| 에러 대응 | 에러가 나면 챕터에 해결법이 있는가? |

#### 전체

| 항목 | 질문 |
|------|------|
| 학습 곡선 | 이전 챕터 대비 난이도 점프가 적절한가? |
| 분량 체감 | 길다/적당하다/짧다? |
| 다음 챕터 | 예고가 흥미를 유발하는가? |

### 5단계: 종합 리포트 생성

리포트를 `{프로젝트}/review/beta-reader-CH{N}.md`에 저장한다.

#### 리포트 구조

```markdown
# Beta Reader 리포트: CH{N} — {챕터 제목}

## 프로젝트
- 책: {책 제목}
- 독자 상수: {seed.md에서}

## 페르소나 목록
| # | 이름 | 배경 | 통과 |
|---|------|------|------|
| 1 | ... | ... | O/X |

## 요약
- 통과: N/10명
- 주요 이슈: N건

## 공통 피드백 (3명 이상 동일 의견)
| 영역 | 피드백 | 언급 페르소나 | 심각도 |
|------|--------|-------------|--------|
| ... | ... | 1, 4, 9 | 높음 |

## 심각도별 이슈

### 높음 (학생이 막힘)
- ...

### 중간 (혼란스럽지만 진행 가능)
- ...

### 낮음 (개선하면 좋음)
- ...

## 페르소나별 상세

### 1. {이름} ({배경})
**[이야기]** ...
**[기술]** ...
**[실습]** ...
**[총평]** ...
**[점수]** 이야기 N/5 | 기술 N/5 | 실습 N/5

### 2. {이름} ({배경})
...

## 수정 제안 (우선순위순)
| # | 위치 (라인/섹션) | 제안 | 심각도 | 관련 페르소나 |
|---|-----------------|------|--------|-------------|
| 1 | ... | ... | 높음 | 4, 6, 9 |
```

## 제약

- 실습 코드는 실제로 실행하지 않고 **코드 리딩으로 평가** (실행은 별도 검증에서)
- 페르소나의 "막힘"은 해당 배경에서 합리적으로 추론
- seed.md의 독자 상수를 기본 전제로 함
- style.md의 글쓰기 원칙을 평가 기준에 포함
- 코드가 없는 개념 챕터는 실습 평가를 건너뛰고 이야기+기술만 평가
- 이전 beta-reader 리포트가 있으면 "지적 사항이 개선되었는가"도 확인
